### Cross-Cutting Decisions / Tradeoffs

- **Validator chain insertion point.** Each new validator inserts into the `LoadWithOptions` chain (load.go:188-225) following the established "fail-fast, kind-vocabulary-first" ordering. Concrete insertion order: (D1) `validateAgentMapKeys` after `validateMapKeys` (line 191) — D1 extends the same closed-enum check to the runtime `[agents.<kind>]` map W0 ships; (D2) `validateAgentBindingNames` after `validateAgentBindingFiles` (line 221) — both are agent-resolution-related and the existing `validateAgentBindingFiles` is warn-only, so D2's hard-fail companion sits next to it; (D3) `validateChildRuleCyclesExtended` REPLACES the existing `validateChildRuleCycles` (line 197) — the existing implementation already detects parent→child cycles, D3 expands its scope to also walk `BlockedByParent`-induced edges in a unified DFS; (D4) `validateChildRuleRecursionDepth` after the cycle validator (after line 197 / D3's drop-in); (D5) `validateBlockedByAcyclicity` after D4's depth check; (D6) `validateClaimVsImplCoherence` last in the chain (post-`validateTillsyn` line 222). The builder adjusts the precise insertion line if Drop 4c.6 W0 (the runtime config schema) reshapes Template's field set first — but W0 is an independent sub-plan and W0.5 ships before W0's runtime-config consumers per the Drop 4c.7 ordering.

- **Sentinel error pattern.** Each new validator gets one new package-scoped sentinel (e.g. `ErrChildRuleRecursionTooDeep`) following the existing `var ErrXxx = errors.New("...")` block at load.go:232-395. Every wrapped message names the offending kind / TOML field / TOML line where `pelletier/go-toml/v2` provides one. The existing chain uses `fmt.Errorf("%w: %s", ErrSentinel, detailString)` — D1-D6 mirror the same shape so `errors.Is` continues to route deterministically.

- **Existing `validateChildRuleCycles` already exists.** load.go:517-561 ships a parent→child cycle DFS today. D3's directive (`[[child_rules]]` cycle detector) is REINFORCEMENT not REPLACEMENT — D3 extends the existing DFS to also walk `BlockedByParent: true` edges in a unified graph. This is a small surgical extension (~20-30 LOC) plus a fixture exercising the new cycle path. D3 keeps the existing `ErrTemplateCycle` sentinel; the wrapped message expands to name the edge type (parent→child vs blocked_by) so adopters see which kind of cycle fired. **If the builder finds the existing DFS already covers the new edge type via accidental closure, D3 collapses to "audit + fixture-and-test only, no production code change" — but the L1 directive lists it as a distinct droplet, so the slot stays.**

- **Today's `BlockedBy` graph in templates is degenerate.** `templates.ChildRule` has only `BlockedByParent bool` (schema.go:385). The induced kind-level blocked_by graph today has only child→parent edges, so it's trivially acyclic. **D5 ships the validator scaffolding + a synthetic fixture that injects a hypothetical kind-level cycle via the existing `BlockedByParent` walk.** Concretely: the fixture authors a child_rule chain A→B with `BlockedByParent: true` AND a sibling child_rule B→A also with `BlockedByParent: true` (auto-create rules that, when fired, would produce action items each blocked by the other) — the validator walks the induced kind-graph and rejects with `ErrTemplateBlockedByCycle`. **The validator's primary forward-looking value is preventing future schema additions (e.g. a `BlockedByKinds []domain.Kind` field on ChildRule) from introducing cycles silently.** D5's wrapped message names the participating kinds in path order, mirroring D3's `formatCyclePath` helper.

- **Recursion-depth bound is on `[[child_rules]]` parent→child chains.** D4 walks the same parent→child kind graph D3 / `validateChildRuleCycles` already builds, but tracks max depth from any root. Default bound = 5 (per L1 sub-plan container Acceptance bullet + `SKETCH.md` § 26.W0.5 "default 5; configurable post-MVP via template"). The bound is hardcoded for Drop 4c.6 — no schema-level `[tillsyn] recursion_depth_max = N` field; that lands post-MVP. D4 fails loud when any kind's reachable depth exceeds 5.

- **Agent .md resolver shape (D2).** D2 implements the 3-tier resolution priority per `SKETCH.md` §3.4: project `<projectRoot>/.tillsyn/agents/<name>.md` → user `~/.tillsyn/agents/<group>/<name>.md` → embedded `internal/templates/builtin/agents/till-<group>/<name>.md`. **`projectRoot` and `group` are NOT available at template load time** — Load takes only an `io.Reader`, not a project context. The project-tier and user-tier checks happen at SPAWN time via the existing `validateAgentBindingFiles` (which today checks `~/.claude/agents/<name>.md`) — D2 either retargets that warn-only check to the new 3-tier embedded check OR adds a sibling hard-fail validator that ONLY checks the embedded tier. **The builder picks the smaller diff after reading the existing `validateAgentBindingFiles` body.** Either way, D2's load-time check is "does at least the embedded fallback exist for every `agent_bindings.<kind>.agent_name`" — that's the floor of the 3-tier resolution; project + user tiers can legitimately be absent (project hasn't run `till init` yet, user has no machine library).
