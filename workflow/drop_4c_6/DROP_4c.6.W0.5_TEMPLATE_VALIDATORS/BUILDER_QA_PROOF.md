**Verdict on the two design refinements: both justified, neither is scope creep.**

- **`[K ~string]` vs `[K comparable]`** (PLAN bullet at line 128 / worklog Design decisions line 141): The PLAN's literal helper signature was `[K comparable]`. Sorting requires either projection-from-string + back at every call site, or a constraint that supports string conversion at the helper. The `~string` underlying-type constraint is the smallest constraint that lets the helper self-sort without forcing every caller to project. Every cascade kind-keyed graph in this package keys by `domain.Kind` (a `~string` enum, verified at `internal/domain/kind.go:15`), so the constraint is sufficient for D3/D4/D5. Drift to a different key type (e.g., a future custom struct key) would surface as a compile error — the helper would need re-parameterisation. Acceptable Go-idiomatic choice, narrower than PLAN's literal but covers every documented caller and preserves the deterministic-root-iteration intent.

- **Two-graph walk vs literal-merged unified graph** (worklog Design decisions line 138): A literal merged-edge approach would falsely flag every well-formed `BlockedByParent=true` rule as a 2-cycle (each such rule contributes A→B in parent→child AND B→A in blocked_by, which combined are a degree-2 cycle). The validator builds the two edge sets separately and runs `dfsDetectCycle` on each, reporting the first cycle found with its edge-type label. This honors the PLAN's "unified graph" wording (one DFS pass per edge set, unified caller, edge-type label injected) without producing the false-positive. The 5 test rows demonstrate the contract: the acyclic-blocked_by row passes despite three `BlockedByParent=true` rules whose induced child→parent edges in a literal-merged graph would cycle. Acceptable refinement that preserves spec intent while avoiding a known false-positive.
