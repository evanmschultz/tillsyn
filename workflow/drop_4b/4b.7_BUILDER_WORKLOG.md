## Design Choices

### Per-project goroutine model (Option B from WAVE_C_PLAN.md §4.5)

`Start(ctx)` calls `projectsLister.ListProjects(ctx, false)` exactly once at startup and spins one goroutine per project ID. Each goroutine subscribes to `LiveWaitEventActionItemChanged` for that project's broker channel via the existing 4a.15 `subscribeBroker` helper. The single MVP dogfood (Drop 5) needs multi-project support without per-process management; per-project subscription is the cheapest path.

New projects added during runtime are NOT picked up — Start enumerates projects once. Per WAVE_C_PLAN.md §4.5: project-creation-during-runtime is a Drop 4c / Drop 5 dogfood refinement requiring a `LiveWaitEventProjectCreated` event the broker does not yet publish. This limitation is documented in `subscriber.go`'s package doc-comment.

### Lifecycle synchronization via `sync.Mutex` + `sync.WaitGroup`

All four lifecycle fields (`started`, `stopped`, `subCancel`, `subWG`) are guarded by `subMu`. The mutex serializes Start vs Stop transitions:

- **Start**: locks `subMu`, checks `started || stopped` (rejects with `ErrAlreadyStarted` if either is true), enumerates projects, derives a child ctx via `context.WithCancel(context.Background())` (NOT from Start's input ctx — the input ctx is for the one-shot `ListProjects` call only; a short-lived input ctx must NOT tear down the long-lived subscribers), `wg.Add(1)` per project, spawns goroutines, sets `started = true`, unlocks.
- **Stop**: locks `subMu`, checks `!started || stopped` (returns nil for idempotent no-op), captures `cancel` reference, sets `stopped = true`, unlocks. Then calls `cancel()` outside the lock so the cancel propagation does not contend with concurrent Start/Stop callers. Waits for `wg` via a goroutine + ctx-deadline `select`.

The "lock on state transition, unlock before external work" pattern is critical — holding the mutex across `cancel()` or `wg.Wait()` would deadlock concurrent Start callers.

### `ctx context.Background()` for subscriber goroutines (NOT Start's ctx)

Subscriber goroutines run under a context derived from `context.Background()` plus an explicit `cancel` func stored on the dispatcher. Start's input ctx is only used for the one-shot `ListProjects` call. This decouples the long-lived subscriber lifetime from the short-lived call ctx — important for cmd/till where Start is called inside a request-scoped ctx but the subscribers must outlive the request.

Stop owns cancellation via the stored `cancel` reference; Start's ctx cancellation is irrelevant to the subscribers' lifetime. This matches the lifecycle pattern of HTTP server `Shutdown` flows.

### Re-start of stopped dispatcher rejected

A second Start after Stop returns `ErrAlreadyStarted`. The state machine is `unstarted → started → stopped`; stopped is terminal. Re-start would require resetting `subWG`, `subCancel`, and the `stopped` flag — race-prone. Callers construct a fresh `*dispatcher` instead. Test `TestDispatcherStartStopConcurrent` pins this contract.

### `hylla_reingest` gate stub DROPPED per WC-A10

WAVE_C_PLAN.md §4.4 originally called for a `hyllaReingestGate` stub registering against the gate registry with a "log + skip" body. WC-A10 dropped the stub: the "log + skip" semantic violates `feedback_orchestrator_runs_ingest.md` which enforces orchestrator-only Hylla ingest with `enrichment_mode=full_enrichment` from the GitHub remote. A silent-skip stub would mask the contract violation; a fail-loud `ErrGateNotRegistered` (already wired in 4b.2) is the better signal. The `hylla_reingest` gate kind stays in 4b.1's closed enum for forward-compat with Drop 4c F.7's real implementation; no `gate_hylla_reingest.go` file exists in this droplet.

### `dispatcherFactoryFunc` test-injection seam in cmd/till

`runServe` consumes the dispatcher via the local `dispatcherLifecycle` interface (`Start` + `Stop` only) and a package-level `dispatcherFactoryFunc` variable that production wires to `dispatcher.NewDispatcher`. Tests stub the factory (matching the existing `serveCommandRunner` pattern) to inject a `stubDispatcherLifecycle` that records `startCalls` / `stopCalls` atomic counters. The stub lets `TestRunServeIntegrationStartsAndStopsDispatcher` assert both lifecycle calls fired during the serve flow without standing up the real subscriber graph.

### Fail-loud on dispatcher Start failure in runServe

`runServe` returns a wrapped error if `dispatcher.NewDispatcher` or `disp.Start(ctx)` fails — the HTTP / MCP servers do NOT boot in that case. Per WAVE_C_PLAN.md §4.4 the sub-spec called for warning + continuing; I chose fail-loud per the MVP-discipline rule "fail-loud beats silent-skip." Test `TestRunServeIntegrationFailsOnDispatcherStartError` pins this contract (asserts `serveCommandRunner` is NOT invoked when Start fails). If dogfood reveals that dispatcher misconfig should not block the HTTP serve flow, future Drop 5 refinement can flip to logged-warning behavior.

### Errors swallowed inside the subscriber loop

`handleSubscriberEvent` calls `walker.EligibleForPromotion` + `RunOnce` per eligible item; both return values are intentionally swallowed (`_, _ = d.RunOnce(...)`). Rationale: a single eligibility-walk error or spawn failure must NOT kill the subscriber loop because the next state-change event gives the dispatcher another chance to promote the item. This matches the broker_sub.go contract that any non-context error from `Wait` exits the loop, but RunOnce is invoked separately and its failures are recoverable.

Pre-cascade-dogfood there is no logger wired into the dispatcher package; surface logging arrives with Drop 4c F.7's spawn pipeline rewrite. Until then, errors are silent inside the subscriber.
