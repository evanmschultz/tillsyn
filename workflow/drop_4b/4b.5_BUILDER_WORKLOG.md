## Design Choices

### Multi-session iteration (per WC-A2)

`RevokeSessionForActionItem` lists ALL sessions through the configured backend
(`AuthSessionFilter{}` — no scope filter at the backend boundary because
`ApprovedPath` parsing happens in the app layer, not in the backend), then
iterates and revokes EVERY session matching `path.ScopeID == actionItemID`.
Returning early on the first match would leak the rest — retries / fix-builder
cycles can leave 2–3 sessions tied to the same action item across the
dispatcher's lifetime, and any unrevoked session continues to validate against
the ApprovedPath until its TTL expires.

The test `TestRevokeSessionForActionItemRevokesAllSessions` seeds 3 matching
sessions plus 1 sibling (different action item) and asserts exactly 3 revokes
fired with the 3 expected session IDs in the recorder.

### Lease cascade (per WC-A1)

`s.authBackend.RevokeAuthSession` revokes only the autent session row — it
does NOT cascade to tillsyn-side capability leases, which live in a separate
SQLite table reachable through `s.repo.RevokeCapabilityLeasesByScope`. The
plan's WC-A1 mitigation is to invoke BOTH per matching session:

```go
s.authBackend.RevokeAuthSession(ctx, session.SessionID, reason)
// then
s.repo.RevokeCapabilityLeasesByScope(ctx, projectID, capScope, scopeID, now, reason)
```

Scope-tuple resolution: `session.ApprovedPath` parses to a `domain.AuthRequestPath`
whose `ScopeType` is a `ScopeLevel` (project / branch / phase) and whose `ScopeID`
is the leaf scope ID. `ScopeLevel.ToCapabilityScopeType()` (existing domain
helper) maps to the lease layer's `CapabilityScopeType`. The cascade scope is
`(path.ProjectID, path.ScopeType.ToCapabilityScopeType(), path.ScopeID)` —
identical to the tuple the lease was issued under.

Test `TestRevokeSessionForActionItemRevokesLeaseAndSession` seeds a single
session + lease pair and asserts both surfaces revoke.

### `errors.Join` aggregation across all attempts

Each per-session attempt (session-revoke + lease-revoke) appends any non-nil
error to `errs []error`; the final return is `errors.Join(errs...)`. This
preserves the load-bearing safety property that one backend hiccup cannot
strand the rest of the cascade — and the cleanup hook's existing
`errors.Join` aggregation at `cleanup.go:218-237` survives unchanged because
my method already aggregates internally.

`TestRevokeSessionForActionItemIteratesPastSessionRevokeError` configures the
stub backend to return an error on the first session and pin the second
session's call — assertion is that BOTH calls happen and the final error
satisfies `errors.Is(err, revokeErr)` so callers can detect the failure
class.

### Constructor binding seam — function-typed via `actionItemAuthRevoker` interface

The cleanup hook's existing pattern is interface-typed external dependencies
collapsed to function-typed internal seams (e.g. `monitorUnsubscriber` ⇒
`unsubscribeMonitor func(string)`). I followed the same shape:

- New interface `actionItemAuthRevoker` with `RevokeSessionForActionItem(ctx, id) error`.
- `newCleanupHook(fileLocks, pkgLocks, monitor, authRevoker)` validates
  non-nil, then closes over `authRevoker` to bind `revokeAuthBundle` to a
  `func(ctx, id) error` closure.
- Tests can inject a struct-literal `cleanupHook{revokeAuthBundle: ...}`
  directly (existing in-package pattern preserved) OR pass a `stubAuthRevoker`
  through the constructor (new pattern for wiring assertions).

`*app.Service` satisfies `actionItemAuthRevoker` automatically via the new
`RevokeSessionForActionItem` method — no adapter shim needed.

### Seam signature change — added `ctx context.Context` to `revokeAuthBundle`

Pre-4b.5 the seam was `func(actionItemID string) error` because the Wave-2 stub
didn't need a ctx. The real revoke needs to pass ctx through to the autent
backend (cancellable session-revoke RPC) and the SQLite repo (lease-revoke
transaction) so the cleanup deadline is honored. I changed the field shape to
`func(ctx context.Context, actionItemID string) error` and forwarded the ctx
through `OnTerminalState`'s ctx parameter (which was already present but
underscored as `_ context.Context`).

This forced an update to 4 direct-struct-literal cleanup hook constructions
in `cleanup_test.go` and 1 in `dispatcher_test.go` — all mechanical, all
signature-only. The lock-release seams remain unchanged (no ctx) because
their implementations are in-process and synchronous.

### `ListAuthSessions` filter shape

`AuthSessionFilter{}` (zero value, no filtering) returns ALL sessions across
all projects. The matching logic is in the app layer (`ParseAuthRequestPath` +
`ScopeID` comparison), not at the backend boundary, because:

- The backend's filter shape (`SessionID`, `ProjectID`, `PrincipalID`,
  `ClientID`, `State`, `Limit`) doesn't include "scope_id" or
  "approved_path" — it can't filter on what we need.
- An action item's project_id isn't always available at cleanup time
  (`item.ProjectID` is in scope but pre-4a.12 fixtures may have it empty),
  whereas the `ApprovedPath` parse provides the project_id deterministically.
- Tail volume is bounded — per-project session counts in dogfood are
  expected in single digits, so iterating the full list per cleanup is
  acceptable. If profiling shows hot path post-MVP, the backend can grow
  a `ScopeID` filter alongside the existing fields.

### Skip-paths (idempotent no-ops)

Three documented degenerate cases return nil without backend calls:

1. `actionItemID == ""` after TrimSpace — cleanup hook's empty-ID
   short-circuit composes cleanly.
2. `s.authBackend == nil` — fixture compatibility for tests constructed
   without an auth backend; mirrors `RevokeAuthSession`'s nil-guard at
   `auth_requests.go:861-863`.
3. Sessions whose `ApprovedPath` fails `ParseAuthRequestPath` are skipped
   silently. Malformed paths are an upstream-bug surface that is not this
   method's concern; aborting the whole cleanup over one malformed session
   would block all the well-formed siblings.
4. Sessions with non-nil `RevokedAt` are skipped (already revoked
   externally — manual `till auth_request revoke` or autent TTL expiry).

Tests `TestRevokeSessionForActionItemEmptyIDIsNoop`,
`TestRevokeSessionForActionItemNilAuthBackendIsNoop`,
`TestRevokeSessionForActionItemSkipsRevokedSessions`, and
`TestRevokeSessionForActionItemSkipsMalformedApprovedPath` pin each path.

### Reason string — stable constant

The reason string passed to both `RevokeAuthSession` (audit-trail field on
the session row) and `RevokeCapabilityLeasesByScope` (audit field on the
lease row) is the package-level constant `terminalStateCleanupRevokeReason =
"terminal_state_cleanup"`. Stable + machine-grep-able for post-mortem
queries: filter the audit log on `revocation_reason = "terminal_state_cleanup"`
and you've found every cleanup-driven revoke.

The plan's earlier suggestion (`"dispatcher cleanup: terminal state " +
string(item.LifecycleState)`) embedded the lifecycle state in the reason
string, but the cleanup hook's `OnTerminalState` doesn't pass lifecycle
state to the seam — only the action_item ID. Adding the state to the reason
would require widening the seam signature again. The constant pattern is
simpler, machine-parseable, and keeps the audit field stable across the
3 terminal states (success / failed / archived all share one reason —
disambiguate via the action-item state itself if needed).

  - **Suggestion:** None — `hylla_node_full` was the right tool.

- **Query 3:** `hylla_search_keyword query="authBackend stubAuthBackend ListAuthSessions" test_mode=include_tests`.
  - **Missed because:** The multi-token query didn't return any of `app/auth_requests_test.go`'s `*authRequestServiceFixture` rows. I expected the test fixture's name + the backend symbol to score together. Fell back to grep.
  - **Worked via:** `Bash /usr/bin/grep -n "authBackend\|AuthBackend\|fakeAuthBackend" auth_requests_test.go` once the bash sandbox allowed grep.
  - **Suggestion:** Same gripe as 4b.4's worklog reported — multi-token searches across mixed visibility are weak. Add a "`tail_symbol` exact-match" mode that takes a single-token bare symbol name and returns every node where the tail matches, regardless of file or package.

- **Query 4:** `hylla_node_full node_id="github.com/evanmschultz/tillsyn/internal/domain/CapabilityLease"`.
  - **Result:** Returned the struct's content with all fields. Critical for catching the `InstanceID` (not `AgentInstanceID`) field name + the absence of `PrincipalID`/`PrincipalType`/`ClientID` on the input shape. Saved a compile-error round-trip.
  - **Suggestion:** None — pristine.

Ergonomic gripe that bit me twice during this droplet: the project-naming
inconsistency in node IDs. Some live under `github.com/evanmschultz/tillsyn/...`
(e.g. `Service.RevokeAuthSession`) while others live under
`github.com/hylla/tillsyn/...` (e.g. `domain.AuthRequestPath`,
`domain.ParseAuthRequestPath`). Both refer to the same source tree — but
`hylla_node_full` requires the exact prefix, so I had to guess-and-retry on
the first lookup. A canonicalization pass that stores nodes under one
project prefix (or a `hylla_search_keyword` mode that returns alternates
when a node is resolvable under multiple prefixes) would tighten the loop.
