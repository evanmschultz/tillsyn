## Findings

### F1 — `KindCatalog` does not carry `Gates`; plan promised it would

`WAVE_A_PLAN.md:81` says the `KindCatalogJSON` envelope "does not currently embed Gates — it lands here." 4b.1 did not extend `templates.KindCatalog` (the baked catalog struct) nor `Bake()` to copy `Template.Gates`. The runtime read path (`gateRunner.Run` per WAVE_A_PLAN 4b.2) takes the live `Template`, not the baked catalog, so this is **not a runtime defect** — but the planning doc and the worklog are inconsistent. Either:

- The plan was wrong and Gates does not need to live on the baked catalog (recommended — gates are a runtime concern that re-reads the template), in which case the fresh-DB invariant on `WAVE_A_PLAN.md:81` is vacuous and should be edited.
- Or 4b.2 / 4b.3 / 4b.4 will discover they need a baked-Gates path and 4b.1 has work backlogged.

**Action:** surface in 4b.1 worklog §2 + raise as a Drop 4b refinement. No 4b.1 code change required.

### F2 — Missing test `TestTemplateGatesEmptyMapDecodes`

Plan named explicitly (`WAVE_A_PLAN.md:115`), builder skipped. The "no `[gates.*]` table → nil/empty map" path is implicitly walked by `TestLoadValidTemplate` (which has no `[gates]`) but no assertion pins `tpl.Gates == nil`. A 5-line test would close the gap. Not blocking — none of the runtime paths the gate runner will use depend on this being asserted, and the existing positive `TestDefaultTemplateLoadsWithGates` plus negative `TestValidateGateKindsRejects*` tests cover the populated half.

**Action:** ship as a 4b.1 follow-up commit OR add to 4b.2 builder's task list (writing the runner naturally bumps into this case). Recommend the follow-up commit for cleanliness.

### F3 — Doc-comment on `IsValidGateKind` misrepresents `domain.IsValidKind`

`schema.go:96-99` reads:

> The check is exact-match against validGateKinds — no whitespace trimming or case folding. Template authors are responsible for canonical spelling (matches domain.IsValidKind's pattern, which also rejects whitespace-padded or mixed-case values at the load boundary).

The parenthetical is wrong: `domain.IsValidKind` (`internal/domain/kind.go:50-52`) DOES `TrimSpace + ToLower` before the membership check, so it **accepts** whitespace-padded and mixed-case values. The asymmetry is intentional (worklog §2.2 calls it out: `GateKind` is "stricter than `domain.Kind`"). The doc-comment should say so plainly.

**Suggested edit (drop-builder follow-up, ~2 lines):**

```go
// (matches domain.IsValidKind's pattern, which also rejects whitespace-padded
// or mixed-case values at the load boundary).
```

becomes

```go
// (deliberately stricter than domain.IsValidKind, which TrimSpace + ToLower-
// normalizes its input before membership check).
```

**Action:** trivial doc-comment patch in a follow-up commit OR fold into 4b.2's worklog.

### F4 — Project-wide: `validateMapKeys` accepts case-folded TOML keys due to `domain.IsValidKind` normalization

This is the load-time discoverability bug from A24. `domain.IsValidKind` is case-insensitive, so `[gates.BUILD]`, `[kinds.BUILD]`, `[agent_bindings.BUILD]` all survive validation, get stored under the literal mixed-case key, and then are **never matched** by canonical-case lookups (`tpl.Gates[domain.KindBuild]`, `tpl.AgentBindings[domain.KindBuild]`, …). Resulting behavior is silent absence — the template author thinks they configured `build`, the runtime sees no entry.

This is **not a 4b.1 regression** — the same hazard has existed on `[kinds.*]` and `[agent_bindings.*]` since Drop 3's `validateMapKeys` shipped. 4b.1 inherited the pattern unchanged. Two reasonable fixes, both project-wide:

1. **Strict on map keys**: replace `validateMapKeys`'s `domain.IsValidKind(k)` calls with a strict-equality membership check (`slices.Contains(validKinds, k)` against the literal key, no normalization). Most invasive but correct.
2. **Normalize-then-rewrite**: in `Load`, after strict-decode, walk each map and re-key using `domain.NormalizeKindAppliesTo`-style canonical lookup. Less invasive but adds a normalize pass.

**Action:** raise as a Drop 4c+ refinement. Out of 4b.1 scope. Document on `project_drop_4b_refinements_raised` memory.
