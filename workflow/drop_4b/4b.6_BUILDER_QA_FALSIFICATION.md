# 4b.6 BUILDER QA FALSIFICATION — GIT-STATUS PRE-CHECK ON `Service.CreateActionItem`

## Verdict

**FAIL — one CONFIRMED counterexample (C1) requires a fix-builder round before 4b.6 closes.** Two CONFIRMED test-coverage gaps (NIT-1, NIT-2 — `ErrGitNotFound` and nonexistent-path scenarios from WAVE_C_PLAN §3.5) plus three ergonomic NITs (NIT-3 through NIT-5) route to refinements regardless. - **Hypothesis:** N calls fire N×M `git status` subprocesses; race-safe?

## Counterexamples

### C1 — CONFIRMED — Ordering bug: pre-check fires BEFORE domain validity check on `Paths`

- **Surface:** `internal/app/service.go:923-934` (the `runGitStatusPreCheck` insertion point) vs `internal/domain/action_item.go:413` (where `normalizeActionItemPaths` runs inside `domain.NewActionItem`).
- **Reproduction:**
  1. Caller invokes `Service.CreateActionItem` with `in.Paths = []string{"", "internal/foo/bar.go"}` (an empty entry mixed with a real path).
  2. Line 930: `len(in.Paths) > 0` is true.
  3. Line 931: `runGitStatusPreCheck(ctx, in.ProjectID, in.Paths)` is called.
  4. Inside `runGitStatusPreCheck` → `defaultGitStatusChecker` → `runGitStatusPath`: `git status --porcelain -- ""` is invoked. With an empty pathspec, git either reports the entire worktree's status (a non-empty stdout if anything anywhere is dirty, even files unrelated to the declared paths) — flagging the action item dirty incorrectly — or returns a "fatal: empty string is not a valid pathspec" error class on newer git versions, which `runGitStatusPath` wraps as a non-nil error and propagates as `"git status pre-check ...: ..."`. Either branch is wrong: the input is structurally invalid and should reject with `domain.ErrInvalidPaths` per `normalizeActionItemPaths:713-714`, NOT with a wrapped git error or a misleading dirty-tree verdict.
  5. The caller never reaches `domain.NewActionItem` (line 936) where `normalizeActionItemPaths` would have rejected the empty entry with `ErrInvalidPaths`.
- **Same shape applies to:** `Paths = []string{`internal\foo.go`}` (backslash-bearing — `normalizeActionItemPaths:716-718` rejects with `ErrInvalidPaths`; pre-check on Unix runs `git status` against a literal `internal\foo.go` which doesn't exist → empty stdout → "clean" → bug-leakage as a clean verdict). And to `Paths = []string{"   "}` (whitespace-only — domain rejects after `TrimSpace` per line 712-714; pre-check runs `git status -- "   "` with the untrimmed entry).
- **Why this matters:** the spawn-prompt explicitly listed "ErrPathsDirty vs existing ErrInvalidPaths: which sentinel fires when?" as an attack surface. The current ordering means the answer depends on input shape: clean+invalid → wrapped git error or false-clean; dirty+invalid → wrapped git error or `ErrPathsDirty`; valid → either sentinel correctly. The contract should be: **structural-validity rejection always precedes semantic-validity rejection**. Today it's the reverse for the path-validity slice.
- **Builder worklog "Insertion-point ordering" rationale (lines 60-65) explicitly claims:** "After mutation-guard, template-nesting, and kind-resolution checks: a well-formed-but-dirty input still gets the dirty rejection (don't mask it behind another error class), but a malformed input still rejects on its own structural error first (don't run the expensive git pre-check on input we'd reject anyway)." The first half holds; the second half is contradicted by the actual code — `Paths` malformedness is checked INSIDE `domain.NewActionItem`, AFTER the pre-check.
- **Severity:** the malformed-path classes (empty, whitespace, backslash) are caller bugs that domain would reject anyway, so the production-impact blast radius is small in practice. But the test contract from the spawn prompt is broken: the answer to "which sentinel fires when?" is non-deterministic for malformed inputs, and no test pins the behavior. WAVE_C_PLAN §3.5 doesn't enumerate this scenario; the builder didn't catch it.
- **Recommended fix (for fix-builder round):** move the pre-check call to AFTER `domain.NewActionItem` succeeds (line 970), passing `actionItem.Paths` (the normalized slice) to `runGitStatusPreCheck` instead of `in.Paths`. Place it before `s.repo.CreateActionItem(ctx, actionItem)` at line 972 so a dirty-tree reject still doesn't mutate state. Add a regression test `TestCreateActionItemRejectsMalformedPathsBeforeGitCheck` that passes `Paths=[""]` and asserts `errors.Is(err, domain.ErrInvalidPaths)` AND `len(checker.calls) == 0`.
- **Alternative, lower-disruption fix:** call `domain.NormalizeActionItemPaths(in.Paths)` at the top of `runGitStatusPreCheck` and pass the normalized slice to the checker; let `ErrInvalidPaths` propagate before `git` is invoked. This preserves the pre-domain.NewActionItem ordering for the rest of the validation but normalizes paths first. Slightly redundant work (NewActionItem renormalizes), but cheap (no exec).
