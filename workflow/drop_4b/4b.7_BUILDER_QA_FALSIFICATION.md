# 4b.7 BUILDER QA FALSIFICATION — AUTO-PROMOTION SUBSCRIBER

## Verdict

**PASS.** No CONFIRMED counterexample. Two non-blocking REFUTED-with-residual observations recorded as Unknowns for orchestrator routing — both are documented limitations rather than counterexample-worthy bugs.

## Section 0 — Falsification Trace

Attacks below were attempted against the 4b.7 implementation. Each is labeled **REFUTED** (no counterexample produced) or **CONFIRMED** (counterexample produced).

### Attack 1 — Goroutine leak on Start: Add(1) ordering vs. goroutine spawn

**Premise:** `sync.WaitGroup` requires `Add` to happen-before the goroutine that calls `Done` is spawned, otherwise `Wait` may return prematurely.

**Evidence:** `internal/app/dispatcher/subscriber.go:85-86`:
```go
d.subWG.Add(1)
go d.runSubscriberLoop(subCtx, projectID)
```
`Add(1)` precedes `go` in the same goroutine that will later call `Wait` (via Stop). `runSubscriberLoop` defers `d.subWG.Done()` (line 150).

**Trace:** Single-writer (Start under `subMu`) ⇒ single-reader (Stop under `subMu`); `Add` is sequenced-before goroutine creation by program order. `mage ci` runs `-race` clean across 2265 tests including `TestDispatcherStopCancelsAllSubscribers`.

**Verdict:** REFUTED.

### Attack 2 — Concurrent Start + Stop deadlock via lock-then-external-call

**Premise:** Start calls `d.projectsLister.ListProjects(ctx, false)` while holding `subMu` (subscriber.go:64-70). A slow `ListProjects` call would block any concurrent `Stop` waiting for the same mutex.

**Evidence:** subscriber.go:64-89 — the entire Start critical section runs under `subMu.Lock()`, including the external DB-bound `ListProjects` call.

**Trace:** Stop (lines 108-115) takes `subMu.Lock()` first then unlocks BEFORE calling `cancel()` and waiting on the WaitGroup. Stop never holds `subMu` while doing external work. So:
- If Start wins the lock race: Stop blocks on `subMu` until Start returns (potentially seconds for a slow DB), but no deadlock — eventually Start finishes, Stop acquires.
- If Stop wins: Stop sees `!d.started || d.stopped` = `true || false` = `true`, returns nil **without setting `d.stopped = true`**. Start then proceeds and SUCCEEDS, leaving the dispatcher RUNNING despite the caller's Stop having returned.

**Verdict (deadlock attack):** REFUTED — no deadlock; Stop is bounded by the duration of `ListProjects` plus subscriber drain.

**Residual observation (NOT a counterexample, recorded under Unknowns):** the Stop-wins-race semantics are weaker than the worklog claims at line 31 ("Concurrent Start + Stop are safe"). A concurrent Stop that wins the race against a not-yet-Started dispatcher returns nil without preventing a subsequent Start from succeeding. The worklog's stronger framing in §"Lifecycle synchronization" oversells this: if a caller fires `go d.Start(ctx); go d.Stop(ctx)` and Stop wins, the dispatcher will end up running. **Test `TestDispatcherStartStopConcurrent` accepts both outcomes** so this is a documented contract gap, not a bug. Routed to orchestrator under Unknowns.

### Attack 3 — Stop's bounded-wait goroutine leak

**Premise:** Stop's `go func() { d.subWG.Wait(); close(done) }()` (subscriber.go:124-127) outlives the `select` if `ctx.Done()` fires first. The subscribers may still be running when Stop returns `ctx.Err()`.

**Evidence:** subscriber.go:123-138.

**Trace:** Yes, subscribers and the helper goroutine continue running after Stop returns ctx.Err(). They eventually drain when subscribers respond to the cancel. The helper goroutine writes to a channel no one reads, then exits and is GC'd. Not a leak in the unbounded sense — bounded by subscriber drain time. The subscriber.go docstring at line 96-97 says "Stop returns nil on clean drain, ctx.Err() if the supplied context expires before every goroutine exits" — which honestly describes the ctx-expiry path as "goroutines did not exit yet."

**Verdict:** REFUTED. The contract is documented. cmd/till/main.go:2627 caps the shutdown ctx at 5 seconds, which is generous for in-process subscribers responding to a cancel.

### Attack 4 — RunOnce panic kills subscriber permanently

**Premise:** `runSubscriberLoop` has no `defer recover()`. A panic from `RunOnce` (e.g. nil-pointer in a downstream gate) takes the goroutine down with `subWG.Done()` firing via the existing defer. The project's subscriber is gone for the rest of the process lifetime, with no restart path and no log.

**Evidence:** subscriber.go:149-164. The only defer is `d.subWG.Done()` at line 150. `handleSubscriberEvent` (lines 175-193) does not recover either.

**Trace:** A panic propagates up through `handleSubscriberEvent` → `runSubscriberLoop` → goroutine death. `subWG.Done()` fires via defer; `subWG.Wait()` returns when the OTHER subscribers also drain. The dispatcher does not log; per subscriber.go line 175 doc-comment, "Pre-cascade-dogfood there is no logger wired into the dispatcher package; surface logging arrives with Drop 4c F.7's spawn pipeline rewrite." Documented as known.

**Verdict:** REFUTED with residual. The doc-comment explicitly accepts silent error/panic handling for the MVP. No spec line in the plan or the WIKI requires panic-recovery in 4b.7. **Routed to Unknowns** as a Drop 4c / Drop 5 dogfood concern. If a builder agent panics-prone code lands downstream, the missing recover would silently kill the project's auto-promotion path. Worth a refinement note.

### Attack 7 — `subscribeBroker` returns mid-flight non-context error

**Premise:** Per Drop 4a R1+R2, the broker may emit non-context errors. `runBrokerSubscriber` (broker_sub.go:75-85) exits on any non-context error, closing `out`. The `runSubscriberLoop` then sees `!ok` from `<-events` and returns. The project's subscriber is permanently dead.

**Evidence:** subscriber.go:157-160 — the `_, ok := <-events` path exits the loop on channel close.

**Trace:** Permanent loss of subscription for that project. No restart logic. Pre-MVP this matches the broker_sub contract; the comment at broker_sub.go:81-85 documents the choice ("Exit so we do not spin").

**Verdict:** REFUTED. Documented contract.

### Attack 9 — `RunOnce` error: subscriber loop dies?

**Evidence:** subscriber.go:175-193.

**Verdict:** REFUTED. Intentional error-swallowing per doc-comment.

### Attack 10 — Idempotent Stop after first Stop returns ctx.Err()

**Premise:** First Stop times out (ctx.Err returned), subscribers still running. Second Stop sees `d.stopped == true`, returns nil immediately. Caller sees a successful Stop but goroutines continue.

**Evidence:** subscriber.go:108-115. `d.stopped = true` is set BEFORE the wait. Subsequent Stops short-circuit.

**Trace:** Caller assumption "Stop returned nil ⇒ all goroutines drained" is FALSE under this race. Documented contract: "Stop returns nil on clean drain, ctx.Err() if the supplied context expires before every goroutine exits." First Stop returned ctx.Err(). Caller HAS the signal. Second Stop is correctly idempotent — it does not re-arm the wait.

**Verdict:** REFUTED. Caller responsibility to honor the first Stop's verdict.

### Attack 11 — runServe shutdown ordering: dispatcher Stop vs. HTTP server Shutdown

**Premise:** `defer disp.Stop` runs after `serveCommandRunner` returns. If the HTTP server publishes events while shutting down, dispatcher subscribers may still be active. Concern: are subscribers torn down BEFORE or AFTER the HTTP/MCP server stops accepting requests?

**Evidence:** cmd/till/main.go:2616-2642. `serveCommandRunner(ctx, ...)` runs in the foreground; when it returns (via `ctx.Done()` ⇒ context.Canceled), the deferred Stop fires.

**Trace:** Order is: HTTP server returns → dispatcher Stop. So dispatcher subscribers continue running for the duration of HTTP shutdown, then are torn down. This is the safe ordering — events emitted DURING HTTP shutdown have a chance to be observed. Inverse order (dispatcher Stop first, then HTTP) would race.

**Verdict:** REFUTED. Order is correct.

## Unknowns (routed to orchestrator)

**U1 — Stop-wins-concurrent-race semantics gap.** When `go d.Stop(ctx)` runs ahead of `go d.Start(ctx)` against an unstarted dispatcher, Stop returns nil without poisoning subsequent Start. Caller sees a successful Stop but the dispatcher then starts and runs subscribers. The worklog claims "Concurrent Start + Stop are safe — Start holds the internal mutex across project enumeration + goroutine spawning so Stop never observes a half-spawned dispatcher" — accurate as far as it goes (no half-spawned state) but understates the gap. **Test `TestDispatcherStartStopConcurrent` already accepts both race outcomes**, so this is a documented contract gap rather than a bug. Recommended action: log a Drop 4c refinement to either (a) tighten Stop to mark `stopped` even when called pre-Start (poisoning the dispatcher), or (b) tighten the worklog wording to match the test's accept-both stance.

**U2 — No panic-recovery in subscriber goroutines.** A `RunOnce` panic kills the project's subscriber permanently with `subWG.Done` firing via defer but no log + no restart. Pre-MVP acceptable (subscriber.go line 175 doc-comment explicitly accepts silent failure pre-Drop-4c-F.7). Recommended action: log a Drop 4c / Drop 5 refinement to add `defer recover()` plus structured logging once the dispatcher gets a logger.
