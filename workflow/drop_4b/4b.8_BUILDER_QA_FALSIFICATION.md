## 1. Findings

### 1.1 Per-Method Publish Placement (REFUTED)

Read the five service.go publish call sites at lines 1163 / 1197 / 1414 / 1461 / 1739. Every publish sits AFTER the successful `s.repo.UpdateActionItem` (or `s.repo.DeleteActionItem`) AND any subsequent embedding-enqueue / search-index / lifecycle-cleanup steps that may return an error. If any earlier step short-circuits with a non-nil err, the publish never fires — correct semantics, the dispatcher would be told about a state change that never happened.

`service.go:1411-1415` (DeleteActionItem archive branch) deserved special inspection because the builder restructured the original tail-call `return s.repo.UpdateActionItem(...)` into the conventional `if err := ...; err != nil { return err }` shape. Spot-check via diff confirms the publish now lives between successful repo write and `return nil`. Identical pattern at `service.go:1428-1462` for the hard-delete branch. Both branches are equally protected.

### 1.2 service.go Write-Path Inventory (REFUTED missed-write attack)

### 1.3 ImportSnapshot Single-Bulk Publish Semantics (REFUTED)

Read `snapshot.go:349-372`. The publish loop runs only after every error-returning step (`UpdateColumn` / `CreateColumn` / `UpdateActionItem` / `CreateActionItem` / `importSnapshotComments` / `importSnapshotCapabilityLeases` / `importSnapshotHandoffs`) returns nil. Partial-fail short-circuits before the publish loop. The builder's worklog §4 explicitly accepts this ("Pre-MVP we don't try to roll back partial imports; the subscriber will re-discover on the next event"); the unified PLAN row 4b.8 accepts it ("single bulk publish per project_id touched at the end of the import"). Spec-acknowledged design choice, not a defect.

Edge-case verification:
- **Empty snapshot** — `len(projectIDs) == 0`, `for projectID := range projectIDs` is a no-op. Correct.
- **Snapshot with 5 distinct projects** — map dedup gives 5 publishes. Correct.
- **Project_id appears in BOTH `snap.Projects` AND `snap.ActionItems`** — `map[string]struct{}` collapses to one entry, one publish. Correct.
- **Whitespace-only or empty project_id** — `strings.TrimSpace` + `id != ""` guards on both source loops at 359-362 and 365-368. Correct.

### 1.4 Hard-Delete Observability (REFUTED)

The dispatcher subscriber re-walks the tree on every wakeup. After a hard delete, the deleted item is simply absent from the next walk — that IS the observable state change. The publish is meaningful in the "wake up and look again" sense, even though there is no specific item to inspect. Builder worklog §5.2 captures this rationale. No bug.

### 1.5 N-Amplification Regression Test Rigor (REFUTED, mutation-tested)

`TestImportSnapshotPublishesActionItemChangedPerProject` at `snapshot_test.go:1354` asserts `Latest.Sequence == 1` for each of two distinct projects with two action items each. The in-process broker stamps a monotonic per-(eventType, key) sequence (`live_wait.go:151-160`), so any per-item amplification regression would land Sequence at 2 (or 4 if both projects amplified) and the test would fail.

**Mutation test executed:**

```diff
+	// MUTATION: simulate per-item amplification — extra publish per action item
+	for _, actionItem := range snap.ActionItems {
+		s.publishActionItemChanged(strings.TrimSpace(actionItem.ProjectID))
+	}
```

Result: `TestImportSnapshotPublishesActionItemChangedPerProject [FAIL]` immediately. Test reverted; clean state restored. Test pins behavior precisely; not passing-for-wrong-reason.
