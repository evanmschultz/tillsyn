## 2. Design choices

### 2.1 `[gates]` vs reserved `[gate_rules]` distinction

These are two separate TOML keys decoding into two distinct fields on `Template`:

| TOML key      | Field            | Type                              | Purpose                                          |
| ------------- | ---------------- | --------------------------------- | ------------------------------------------------ |
| `gates`       | `Gates`          | `map[domain.Kind][]GateKind`      | Drop 4b Wave A — closed-enum gate sequence.       |
| `gate_rules`  | `GateRulesRaw`   | `map[string]any`                  | Drop 3 reserved seam — future per-gate config.    |

Acceptance bullet 5 holds: `GateRulesRaw` is untouched (its line moved from 85 to ~96 because the new `Gates` field was inserted above it, but the field, tag, and doc-comment intent are unchanged — the doc-comment was extended to call out the distinction with the new `Gates` field).

`TestTemplateGatesAndGateRulesCoexist` proves both decode independently when both tables appear in one TOML document.

### 2.2 Closed-enum rationale (locked decisions L1, L2, L6)

`GateKind` is an open-ended `string` underlying type but the only sanctioned values come from `validGateKinds`. `IsValidGateKind` does **exact match** — no whitespace trimming, no case folding. `domain.IsValidKind` normalizes its input via `strings.TrimSpace + ToLower`; my reading of the WAVE_A_PLAN A5 falsification mitigation ("no silent normalization") + the surrounding closed-enum doc-comment is that `GateKind` should be **stricter** than `domain.Kind` here — template authors are responsible for canonical spelling. The invalid-case tests pin this explicitly: `MAGE_CI`, ` mage_ci `, `mage-ci` all reject. Drop 4c will add `commit` + `push` to the slice; the existing closed-enum test will flip those two cases at that time.

### 2.3 Default template minimal landing per L6

`[gates.build] = ["mage_ci"]` only. No other kinds carry gate sequences. Deliberately minimal:

- Drop 4c will expand the `build` sequence to `["mage_ci", "commit", "push"]`.
- Drop 4b Wave C 4b.7 will register `hylla_reingest` for drop-end coordination, but that's not a `[gates.<kind>]` entry — it's gate-runner registry-side wiring with a different invocation surface.
- All other kinds (plan, plan-qa-proof, build-qa-proof, etc.) are absent from `[gates]` and the gate runner treats absence as "no gates" (returns `Success: true` immediately) per the 4b.2 doc-comment.

`TestDefaultTemplateLoadsWithGates` enforces both halves: `build` is present with exactly `[mage_ci]`, AND all 11 sibling kinds are absent.

### 2.4 Validator chain placement

`validateGateKinds` runs **last** in `Load`'s validator chain (after `validateChildRuleReachability`). Justification:

- Order 4a–4d (existing): kind/agent_bindings map keys → child-rule kind references → child-rule cycles → child-rule reachability (no-op).
- Order 4e (new): gate value-slice members.

Running last preserves existing pass order and lets all kind-vocabulary checks fire before the gate-vocabulary check. The map-key axis for `Gates` is folded into the existing `validateMapKeys` (4a) so a `[gates.bogus_kind]` typo reports as `ErrUnknownKindReference: gates map key "bogus_kind"` rather than `ErrUnknownGateKind`. This matches the existing pattern: map-key validation = closed-Kind enum; value validation = the relevant per-table closed enum.

### 7.1 F2 — Missing test for empty `[gates]` decode

Added `TestTemplateGatesEmptyMapDecodes` to `internal/templates/load_test.go` (chosen over `schema_test.go` because every other gate-related load-time decoding test already lives in `load_test.go` — `TestTemplateGatesAndGateRulesCoexist`, `TestValidateGateKindsRejectsUnknownKind`, `TestValidateGateKindsRejectsUnknownParentKind`).

Asserts the contract: a template TOML without a `[gates]` table loads cleanly and `tpl.Gates == nil`. The accompanying doc-comment notes that the gate runner treats nil-or-absent-entry as "no gates," so the nil-vs-empty distinction is not load-bearing for runtime semantics — but the test pins zero-value-on-absence so a future schema edit can't silently start initialising `Gates` to a non-nil empty map at load time, which would defeat the explicit-presence assertion in `TestDefaultTemplateLoadsWithGates`.

Test placement is alphabetically natural (between `TestValidateGateKindsRejectsUnknownParentKind` and `TestLoadSelfCycleSingleRule`).

### 7.2 F3 — Doc-comment factual error on `IsValidGateKind`

Rewrote the doc-comment on `IsValidGateKind` (`internal/templates/schema.go:95-99`). The original prose claimed `domain.IsValidKind` "rejects whitespace-padded or mixed-case values at the load boundary." That was wrong: `domain.IsValidKind` (`internal/domain/kind.go:50-52`) normalizes via `strings.TrimSpace + strings.ToLower` and accepts whitespace-padded / mixed-case input.

Replacement prose (verbatim from schema.go):

> Unlike `domain.IsValidKind` which normalizes via `strings.TrimSpace + strings.ToLower` (case-insensitive, whitespace-tolerant), `IsValidGateKind` does exact-match. Gate kinds are stricter than action-item kinds because templates author them explicitly and silent fold-matching would mask typos (e.g. " Mage_CI " quietly resolving to mage_ci) at load time, well before the dispatcher runs.

The exact-match-no-trim-no-fold behavior of `IsValidGateKind` itself is unchanged — only the doc-comment's claim about how it differs from `domain.IsValidKind` was corrected. The closed-enum invalid-case sub-tests in `TestGateKindClosedEnum` (`MAGE_CI`, ` mage_ci `, `mage-ci`) already pin the exact-match contract; this fix just stops the doc-comment from misrepresenting the comparison kind.
