## Design Choices

### `commandRunner` Seam Shape (4b.4 Reuse)

Interface signature locked at:

```go
type commandRunner interface {
    Run(ctx context.Context, dir string, name string, args ...string) (stdout []byte, stderr []byte, exitCode int, err error)
}
```

Four-tuple return is the smallest carrier that disambiguates the four observable outcomes a gate must distinguish:

1. **Pass** — `err == nil`, `exitCode == 0`. stdout/stderr discarded by gate.
2. **Non-zero exit** — `err == nil`, `exitCode != 0`. Streams routed to `tailOutput`.
3. **Process-start failure** — `err != nil`, `exitCode == 0`, streams empty. Gate wraps with `"mage ci start failed: %w"`.
4. **Ctx cancellation** — `err != nil` (wraps `ctx.Err()`), `exitCode == 0`. Gate disambiguates via `ctx.Err()` check BEFORE reading `err`.

The seam is `package-private` (lowercase). 4b.4's `mage_test_pkg` gate reuses `commandRunner` + `defaultCommandRunner` verbatim — adding a new mage gate does NOT require a new seam, just a new `gateFunc` calling `defaultCommandRunner.Run(ctx, dir, "mage", "testPkg", pkg)`.

`defaultCommandRunner` is a package-level `var` (not a struct field) so the gateFunc signature stays exactly `(ctx, item, project) GateResult` — runner injection happens through the package var, swapped via `withFakeCommandRunner` test helper using `t.Cleanup`.

### Output Capture Aggregation Order

`combineGateOutput(stdout, stderr)` produces stdout-then-stderr concatenation with a synthetic `\n` boundary inserted when stdout doesn't end with one. Rationale per WAVE_A_PLAN.md:

- **Stderr last** keeps the failure-summary tail (mage's structured logger emits failure frames on stderr) inside the bounded window when `tailOutput` byte-bounds at 8KB.
- **Boundary newline** keeps the line-tail bounding line-aligned — without it a stdout that ends mid-line could fuse with the first stderr line and inflate that line beyond its true length.

`TestGateMageCICombinesStdoutAndStderrInOrder` asserts the relative position invariant directly (stdout marker MUST appear before stderr marker in `Output`).

### Ctx Cancellation Propagation

Two-step disambiguation in `gateMageCI`:

```go
stdout, stderr, exitCode, runErr := defaultCommandRunner.Run(ctx, dir, "mage", "ci")
if ctxErr := ctx.Err(); ctxErr != nil {
    // ctx-cancel branch: wrap ctx.Err() with "mage ci context cancelled:"
}
if runErr != nil {
    // start-failure branch: wrap runErr with "mage ci start failed:"
}
```

`ctx.Err()` is checked FIRST because `exec.CommandContext` kills the child when ctx fires and the resulting `cmd.Wait()` error is opaque about the cause. Gate distinguishes ctx-cancel from start-failure by Err-prefix:

- `"mage ci context cancelled: context canceled"` → ctx-cancel.
- `"mage ci start failed: exec: ..."` → start-failure.
- `"mage ci failed: exit code N"` → non-zero exit.

Three distinct prefixes let dashboards / log greps route the failure mode.

### Process-Start vs Exit-Code Distinction (in `execCommandRunner.Run`)

`cmd.Start()` and `cmd.Wait()` are split rather than calling `cmd.Run()`:

- `cmd.Start()` error → returned raw with `exitCode=0`. Real-world cause: binary not on PATH, working dir missing, fork errno.
- `cmd.Wait()` returning `*exec.ExitError` → exitCode populated, err `nil`. Non-zero exit IS the failure signal.
- `cmd.Wait()` returning some other error → returned raw with `exitCode=0`. Real-world cause: ctx-cancel kills child (the gate routes this via the `ctx.Err()` check above).

The `errors.As(err, &exitErr)` check is the load-bearing line: it flips a `*exec.ExitError` into a clean `(streams, exitCode, nil)` return so the gate's exit-code branch sees `runErr == nil && exitCode != 0`.

### Empty-Worktree Guard

Mirrors `dispatcher.go:392`: `strings.TrimSpace(project.RepoPrimaryWorktree) == ""` rejects with a `"RepoPrimaryWorktree is empty"` error. Whitespace-only paths are functionally empty — `TestGateMageCIRejectsWhitespaceWorktree` proves the guard trims.

The guard short-circuits BEFORE the runner is invoked — `TestGateMageCIRejectsEmptyWorktree` asserts `fake.calls == 0` so the contract is not "runner returns gracefully on empty dir" but "gate never reaches the runner."
