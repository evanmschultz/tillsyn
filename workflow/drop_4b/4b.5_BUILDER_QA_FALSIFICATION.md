## 2. Findings (Detailed)

### 2.1 A1 — Multi-session matching (REFUTED)

**Premise.** WC-A2 requires every active session whose `ApprovedPath.ScopeID == actionItemID` is revoked, not just the first.

**Evidence.**
- `auth_requests.go:928-953` is a `for _, session := range sessions` loop with no early-exit on first match.
- `TestRevokeSessionForActionItemRevokesAllSessions` (`auth_revoke_for_action_item_test.go:145-183`) seeds 3 matching sessions + 1 noise sibling, asserts exactly 3 revoke calls hit only `sess-1`/`sess-2`/`sess-3` and NEVER `sess-other`.
- The test asserts `terminalStateCleanupRevokeReason` ("terminal_state_cleanup") on every call — pins both the iteration AND the audit-trail string in one shot.

**No counterexample.**

### 2.2 A2 — Project-scope vs action-item collision (NIT, theoretical)

**Premise.** Brief flags an edge case: "a project-scope session whose ID collides with an action-item ID gets revoked. Is the collision realistic? Documented?"

**Evidence.**
- `AuthRequestPath.Normalize` (`auth_request.go:295-296`) sets `ScopeID = AuthRequestGlobalProjectID` for the global path.
- Same `Normalize` (`auth_request.go:311-313`) sets `ScopeID = p.ProjectIDs[0]` for `ScopeLevelProjects`.
- Same `Normalize` (`auth_request.go:333-335`) sets `ScopeID = p.ProjectID` for project-scope paths.
- `RevokeSessionForActionItem`'s match-filter at `auth_requests.go:938` is `if path.ScopeID != actionItemID`. The filter does NOT examine `path.ScopeType` — so a project-scope session whose `ScopeID == actionItemID` (UUID collision) would falsely match.

**Counterexample probability.** Both `actionItemID` and `projectID` are UUIDs (per `domain.NewID` patterns); collision probability is `~10^-37` in any realistic deployment. Not a CONFIRMED counterexample for production behavior, but the absence of a `path.ScopeType` guard is a real defense-in-depth gap.

**Documentation.** The builder's worklog does NOT call out this collision angle. The `RevokeSessionForActionItem` doc-comment (`auth_requests.go:873-911`) describes ID-based matching without acknowledging that ID-only matching couples to ID-uniqueness across scope types.

**Recommendation (NIT, not blocking).** Add a `path.ScopeType == ScopeLevelBranch || path.ScopeType == ScopeLevelPhase` guard to the filter, OR add a doc-comment line documenting the ID-uniqueness assumption. Defer to refinement queue — the impact is "0 in practice; theoretically a project-scope revoke if UUIDs collide." Not blocking 4b.5 close.

### 2.3 A3 — Lease cascade ordering (REFUTED)

**Premise.** "Revoke session FIRST then lease, or vice versa? If session-first errors mid-way, lease still revoked? Builder claims yes (errors.Join continues)."

**Evidence.**
- `auth_requests.go:941-952`: session-revoke at line 941 runs first; on error it appends to `errs` (line 942) and falls through to the lease-revoke at line 950. There is NO `continue` or short-circuit between them.
- Both errors go into the same `errs` slice; the final `errors.Join(errs...)` at line 954 aggregates without losing either.

**Sanity check.** `TestRevokeSessionForActionItemIteratesPastSessionRevokeError` doesn't exercise the session-then-lease intra-session continuation, but the multi-session iteration test combined with a code-read of lines 941-952 is sufficient: the `if revokeErr != nil` clause at 941 only appends to errs, no flow control. **No counterexample.**

### 2.7 A11 — Multi-project session normalization (REFUTED)

**Premise.** "Builder docs an edge case re: `Normalize` setting `ScopeID = ProjectIDs[0]` for `ScopeLevelProjects`. Verify the scope-type guard properly excludes those from action-item-id matching."

**Evidence.**
- `auth_request.go:311-312` sets `ScopeID = p.ProjectIDs[0]` for the multi-project path. The result is `ScopeID == projectID`, NOT an action-item ID.
- The match-filter at `auth_requests.go:938` (`path.ScopeID != actionItemID`) correctly excludes these — a `ProjectIDs[0]` value will not equal an `actionItemID` value (UUID-disjoint, modulo the A2 collision concern which is identical mathematics).
- Edge case: `ScopeLevelProjects` with `ProjectIDs == [actionItemID]` would also be rejected by the same `Normalize` collapse to `ScopeLevelProject` (lines 305-309) — single-element projects collapse to project-scope before the ScopeID-set logic fires.

**No counterexample for normal UUID semantics.** The A2 collision concern is the only failure mode and is theoretical only.

### 2.8 A12 — Empty `CapabilityScopeType` (NIT)

**Premise.** Defense in depth: what if `path.ScopeType.ToCapabilityScopeType()` returns `""`?

**Evidence.**
- `level.go:120-136`: the default branch returns `CapabilityScopeType("")`.
- `auth_requests.go:948` calls `capScope := path.ScopeType.ToCapabilityScopeType()` without inspecting capScope.
- `auth_requests.go:950` passes `capScope` (possibly empty) into `RevokeCapabilityLeasesByScope`. The SQLite impl `repo.go::RevokeCapabilityLeasesByScope` calls `domain.NormalizeCapabilityScopeType` (`capability.go:203-214`) which preserves empty input as empty.
- The SQL `WHERE scope_type = ?` with `scope_type=""` would match zero rows (no leases issued with empty scope_type because `IsValidCapabilityScopeType` rejects empty at issue-time).

**Effective behavior on empty scope_type.** Zero leases match → zero revokes → no error. The cascade silently no-ops. Acceptable — this is reachable only if `path.ScopeType` is somehow not one of the 5 valid `ScopeLevel`s, which `Normalize` should prevent. NIT not CONFIRMED.

### 2.9 A14 — Reason string drift (REFUTED-WITH-NOTE)

**Premise.** Plan §2.4.5 says reason MUST be `"dispatcher cleanup: terminal state " + string(item.LifecycleState)`. Builder used the constant `"terminal_state_cleanup"`.

**Evidence.**
- Plan §2.4.5 (`WAVE_C_PLAN.md:72`): "machine-parseable: `\"dispatcher cleanup: terminal state \" + string(item.LifecycleState)`".
- Implementation (`auth_requests.go:871`): `terminalStateCleanupRevokeReason = "terminal_state_cleanup"` — single constant, no lifecycle-state component.
- Builder worklog §"Reason string": divergence is intentional + documented. Rationale: cleanup hook does NOT pass lifecycle state to the seam (only action-item ID), so embedding it would require widening the seam signature again. Constant pattern is simpler, machine-parseable, audit-trail-stable.

**Verdict.** The plan and the implementation diverge but the divergence is documented and the new contract is explicitly more grep-friendly for post-mortem audit (single constant string vs three lifecycle-variant strings). The integration test (`TestRevokeSessionForActionItemRevokesAllSessions`) pins the constant. Non-blocking — but the divergence does tell us the plan-QA-falsification round didn't catch this drift. Logged for awareness only.
