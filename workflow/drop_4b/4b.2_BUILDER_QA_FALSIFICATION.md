## 2. Findings

### 2.1 NIT — Data race on concurrent `Register` + `Run` (matches spec divergence)

`TestProbeConcurrentRunRegisterRace` triggered the Go race detector reading `gates.go:214` and writing `gates.go:184` on the same map. **CONFIRMED counterexample to a "concurrency-safe gate runner" claim.**

The implementation's doc-comment (lines 146–151) is honest: "gateRunner is NOT safe for concurrent Register/Run calls. The expected lifecycle is: construct → register every gate during dispatcher startup → call Run from the subscriber's serial event loop."

WAVE_A_PLAN.md line 143 EXPLICITLY required `mu sync.RWMutex` "defense-in-depth," and line 172 (A7 mitigation) called out a `TestGateRunnerConcurrentRuns` test. Both were dropped from the implementation. The choice is defensible — the subscriber in 4b.7 is single-goroutine — but it diverges from the locked spec without an in-line worklog rationale acknowledging the divergence.

**Why NIT, not FAIL:** the doc-comment names the constraint and the spec divergence does not change end-state behavior under the documented single-goroutine usage. The 4b.7 subscriber will exercise the runner serially. **Mitigation request for the worklog:** add §2.X explicitly noting the spec-vs-impl divergence on RWMutex and the consequence (caller MUST serialize). Future drop introducing parallel gates needs to add the lock.

**Concrete reproduction:**

```text
WARNING: DATA RACE
Read at 0x00c0000a4d80 by goroutine 10:
  dispatcher.(*gateRunner).Run() gates.go:214
Previous write at 0x00c0000a4d80 by goroutine 29:
  dispatcher.(*gateRunner).Register() gates.go:184
```

### 2.2 ACCEPTED — `gateFunc` panic propagates uncaught

`TestProbeGateFuncPanicPropagates` confirmed: a `gateFunc` that `panic`s sends the panic up through `Run` to the caller. No `defer recover()` in the runner.

**Why ACCEPTED:** the spec is silent on panic handling. A misbehaving gateFunc IS the dev's bug to fix; suppressing it would mask wiring errors. Caller (4b.7 subscriber) will run inside the `till serve` goroutine — if the dispatcher's monitor doesn't already provide top-level recovery, the daemon would crash on a panicking gate. **Surface to 4b.7 planner**: confirm the subscriber goroutine has top-level `defer recover()` or wraps `Run` in one.

### 2.3 ACCEPTED — `Register(name, nil)` silently accepted; Run panics

`TestProbeNilGateFunc` confirmed: `Register("mage_ci", nil)` returns nil; the next `Run` invocation hits a nil function call and panics.

**Why ACCEPTED:** doc-comment lines 176–179 makes the choice explicit ("nil-checking would mask a wiring bug that should be caught at startup smoke-test"). Acceptable design choice. Combined with §2.2's no-recover stance, a nil-fn registration crashes the daemon. **Same surface as §2.2** — 4b.7 needs top-level recover.

### 2.4 ACCEPTED — `Register("", fn)` accepted; `IsValidGateKind` not called

`TestProbeEmptyGateName` confirmed: `Register(GateKind(""))` returns nil; the lookup at `Run` time succeeds for templates referencing the empty kind.

Spec line 145 required `IsValidGateKind` validation in `Register`. Impl skipped with doc-comment justification (lines 168–174 — single source of truth at template-load time). The empty-string and bogus-kind misuse is caught indirectly via `ErrGateNotRegistered` when the template lookup misses. Defensible but weakens the "no malformed registrations" guard from spec.

**Why ACCEPTED:** the doc-comment justification holds — load-time validation in `validateGateKinds` (4b.1) catches malformed templates BEFORE the runner sees them. The vector "code that bypasses templates and registers directly with bogus names" is caller-error and would break elsewhere in the cascade. Not a concrete user-reachable counterexample.

### 2.5 ACCEPTED — `GateName` mismatch in `GateResult` not normalized

`TestProbeGateNameMismatch` confirmed: a gateFunc returning `GateResult{GateName: GateKindMageTestPkg}` while registered under `GateKindMageCI` — the runner emits the result with the WRONG name; downstream consumers see the mismatch.

Doc-comment at struct field (lines 73–76) explicitly cites the invariant ("the runner does not enforce this invariant; downstream consumers rely on it"). **NIT for ergonomics:** the runner could `result.GateName = name` after the gateFunc returns, normalizing the result without trusting the caller. Cheap, defensive, would catch the wiring-bug class. **Surface as Drop 4c refinement candidate** — not a 4b.2 blocker.

### 2.6 NIT — Context cancellation between gates not honored

`TestProbeContextCancelIgnored` confirmed: a pre-cancelled context still drives `Run` to invoke every gate in sequence. The runner does not check `ctx.Err()` between iterations.

Spec (`TestGateRunnerContextCancel`, line 161) required: "register a gate that blocks on `ctx.Done()`; cancel mid-run; assert `Run` returns within 100 ms with `ctx.Err()` wrapped." Implementation has no such test AND no inter-gate cancellation check.

**Why NIT, not CONFIRMED:** real `mage_ci` and `mage_test_pkg` gates DO honor ctx via `exec.CommandContext` (4b.3 / 4b.4 land), so cancellation propagates INSIDE each gate. The case where `ctx` is cancelled BETWEEN two gates and the next gate's `exec.CommandContext` immediately returns `ctx.Err()` is the realistic path — a one-iteration delay before the cancellation surfaces. **However**, a gate like the future `hylla_reingest` (4b.7) might not honor ctx at all, in which case the runner would invoke it post-cancel.

**Mitigation request:** add a single `if err := ctx.Err(); err != nil { return results }` check at the top of the iteration loop in `Run` — defensive, two lines. Or surface as a Drop 4c follow-up.

### 2.7 ACCEPTED — `Skipped` status halts subsequent gates

`TestProbeSkippedHaltsSubsequent` confirmed: a gateFunc returning `Status: Skipped` short-circuits the sequence because `result.Status != GateStatusPassed` evaluates true for Skipped.

Doc-comment lines 22–26 says the 4b.2 runner does not emit Skipped today. A future per-kind skip filter would emit Skipped — but with the current halt-on-non-Passed rule, that filter's "skipped" gate would also halt subsequent gates, which is probably NOT the intended skip semantic.

**Why ACCEPTED:** the future filter doesn't exist yet. When it lands (Drop 4c+), the halt rule needs to differentiate Skipped from Failed (e.g. `if result.Status == GateStatusFailed`). **Surface as Drop 4c refinement** — comment in code already names the future case.

### 2.8 ACCEPTED — `Status: Passed` with non-nil `Err` slips through

`TestProbePassedWithErr` confirmed: a gateFunc returning `{Status: Passed, Err: errors.New(...)}` is appended unchanged AND the runner continues to the next gate (Status==Passed wins the comparison).

Spec doc-comment (lines 95–98) defines the contract: "Nil on GateStatusPassed and GateStatusSkipped." Implementation does not enforce. **Acceptable** because the contract is on the gateFunc author, not the runner — same pattern as §2.5's GateName mismatch.

### 2.9 NIT — Spec-required tests missing from `gates_test.go`

WAVE_A_PLAN.md §4b.2 test scenarios (lines 156–164) + A7 (line 172) explicitly listed:

- `TestGateRunnerContextCancel` — context cancellation propagation.
- `TestGateRunnerNoDeduplication` — `[gates.build] = ["mage_ci", "mage_ci"]` runs twice (resolves Q2).
- `TestGateRunnerConcurrentRuns` — parallel `t.Run` with different fake gates (A7 mitigation).

NONE of those tests exist in `gates_test.go`. The implemented 9 tests cover other valid scenarios but the omitted three were spec-locked. **The behaviors themselves are correct** (verified by inspection: the `Run` loop iterates the slice without dedup; concurrent Run-only without Register IS safe for an established registry; ctx is propagated INTO each gate via gateFunc signature) — but they aren't pinned by tests.

**Why NIT, not FAIL:** the worklog's test summary (lines 75–87) does match the implemented 9 tests, but doesn't acknowledge the spec divergence. **Mitigation request:** add `TestGateRunnerNoDeduplication` (cheap, 15 lines) so the doc-comment-claimed "no deduplication" rule is regression-pinned. ContextCancel + ConcurrentRuns can punt to Drop 4c when the inter-gate-ctx-check (§2.6) and RWMutex (§2.1) decisions get reopened.

### 2.10 NIT — API-shape divergence from spec not flagged in worklog

WAVE_A_PLAN.md §4b.2 acceptance specified:

- `type gateFunc func(ctx context.Context, input GateInput) GateResult` — `GateInput` STRUCT carrying Item/Project/Template.
- `Run` returns `(GateRunOutcome, error)` — single-result struct + infrastructure error split.
- `GateRunOutcome{Success, FailedGate, Output, Err}` — single-row outcome.

Implementation chose:

- `gateFunc func(ctx, item, project) GateResult` — positional params, no `*templates.Template` plumbed in.
- `Run` returns `[]GateResult` — one result per gate, no separate error channel.
- `GateResult{GateName, Status, Output, Duration, Err}` — per-gate row.

The implemented shape is arguably better (per-gate visibility for the subscriber to render), but the divergence is substantial and the worklog §2.7 only addresses the HARD CONSTRAINT (no `dispatcher.go` mutation) — it doesn't acknowledge the API-shape pivot from the spec.

**Why NIT, not FAIL:** PQA-1 in WAVE_A_PLAN line 301 explicitly invited the builder to slim if `Template` proves unused, AND the per-gate `[]GateResult` is more useful downstream than the singular `GateRunOutcome`. Builder exercised author judgment. **Mitigation request:** worklog §2 should add a "spec API-shape divergences" section listing the five (Mutex, GateInput, GateRunOutcome→[]GateResult, Run signature, no `IsValidGateKind` in Register) so the next reviewer (4b.7 planner) knows the contract changed under them.

## 3. Summary

**Verdict:** PASS-WITH-NIT.

**One-line summary:** Code is correct under documented single-goroutine usage; data race exists and is documented; spec-vs-impl API divergences are real but defensible — surface them in the worklog so 4b.7 plans against the actual surface.

**Concrete worklog mitigations requested (none block 4b.2 close):**

1. Add a "spec divergences" section to the worklog naming Mutex, GateInput, GateRunOutcome→[]GateResult, Register-no-IsValidGateKind, no inter-gate ctx check.
2. Add `TestGateRunnerNoDeduplication` (cheap, pins the doc-comment rule).
3. Add a one-line `if err := ctx.Err(); err != nil { return results }` at the top of the `Run` iteration body — two lines, defensive against future ctx-blind gates (e.g. `hylla_reingest` MCP wrapper).
4. (Surface to 4b.7 planner) — subscriber goroutine MUST have top-level `defer recover()` because the runner does not (panic and nil-fn registration both crash the goroutine otherwise).
5. (Drop 4c refinement candidate) — runner could normalize `result.GateName = name` after the gateFunc returns; future `Skipped` filter needs `Status == Failed` halt instead of `Status != Passed`.

**Build verification:** my probe tests ran via `mage testFunc ./internal/app/dispatcher TestProbe*` from this worktree; the data race surfaced under `-race`. Probe file removed after evidence captured.
