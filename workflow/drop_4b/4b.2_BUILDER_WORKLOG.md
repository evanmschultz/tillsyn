## 2. Design choices

### 2.1 Halt-on-first-failure rationale (locked decision L2)

The runner iterates `tpl.Gates[item.Kind]` in template-declared order and stops the moment one gate returns `Status != GateStatusPassed` OR a gate name lookup misses the registry. The failed result is appended last; subsequent gates are not invoked. This matches REVISION_BRIEF L2's "sequential execution, fail-fast" lock.

Two shapes of "failure" both halt:

1. The `gateFunc` returned `GateStatusFailed` (the gate ran and its underlying op rejected).
2. The runner could not resolve the gate name (no `Register` call for it). Surfaced as a synthesized `GateResult{Status: GateStatusFailed, Err: ErrGateNotRegistered}` so the caller's failure-handling path is uniform.

`TestGateRunnerRunHaltsOnFirstFailure` and `TestGateRunnerRunReturnsErrGateNotRegistered` both pin the call-counter on the SECOND gate at zero so the regression bar is unambiguous.

### 2.2 UTF-8 sanitization for binary output (plan-QA-falsification WA-A3)

`mage_ci` and `mage_test_pkg` output is captured as `[]byte` and forwarded into a `string` field for action-item metadata. Two shapes of bad bytes are handled defensively in `sanitizeUTF8`:

- **Null bytes (`0x00`):** dropped via `bytes.ReplaceAll(b, []byte{0}, nil)`. Some downstream consumers (SQLite TEXT columns, JSON encoders that don't strict-validate) treat `\x00` as a string terminator and silently truncate.
- **Invalid UTF-8 sequences:** replaced with U+FFFD via `strings.ToValidUTF8(s, "�")`. A panic dump or a binary-format test-failure dump can include lone continuation bytes; without sanitization, JSON encoding of the result would either error out or produce unreadable garbage.

`TestTailOutputUTF8Sanitization` covers both with a single mixed input (`hello\x00\x80world\n`) and asserts (a) no null bytes survive, (b) U+FFFD appears in place of the lone 0x80.

### 2.3 Closed-enum `GateStatus` is internal-only (acceptance #5)

`GateStatus` is a public type with public constants (`GateStatusPassed` / `GateStatusFailed` / `GateStatusSkipped`) but there is **no** exported `IsValidGateStatus` helper or closed-set membership function. Per the spawn-prompt acceptance #5: "Not exported as a closed-set helper — internal to the dispatcher package." The package's own consumers (subscriber in 4b.7, gate-failure routing in Drop 4c) compare `result.Status` against the constants directly; cross-package validation isn't a use case today.

`GateStatusSkipped` is reserved but unused in the 4b.2 runner (no row is appended for unconfigured gates — they simply aren't iterated). The constant exists so future per-kind disable flags / dev-skip CLI knobs land without an enum widening.

### 2.4 `ErrGateNotRegistered` halts subsequent gates (acceptance #6)

Per the spec: "Result for that gate has `Status: Failed` + `Err: ErrGateNotRegistered`. Halts subsequent gates per the halt-on-first-failure contract." Implementation appends the synthetic failure row, then `return results` immediately — no second-gate invocation.

The error wraps the offending gate name with `fmt.Errorf("%w: %q", ErrGateNotRegistered, name)` so `errors.Is(err, ErrGateNotRegistered)` works AND the dev sees which gate failed lookup in the message. Same idiom as `ErrGateAlreadyRegistered` from `Register`.

### 2.5 Rollback-policy doc-comment (plan-QA-falsification WA-A2)

The `gateRunner` struct's doc-comment carries a load-bearing paragraph noting that **Wave A gates have NO side effects requiring rollback** (`mage_ci` and `mage_test_pkg` are read-only verifications) and that **Drop 4c's `commit` + `push` gates DO produce side effects** (a created commit, a pushed ref) for which rollback design is explicitly out of scope here. This pre-empts the WA-A2 "what happens when push succeeds but reingest fails halfway" question — the answer for Wave A is "doesn't apply"; the answer for Drop 4c will be designed alongside those gates' definitions.

### 2.6 `tailOutput` "shorter of" semantics (REVISION_BRIEF Q7)

Per Q7: "last 100 lines OR last 8KB, whichever is shorter." Implementation runs both `tailByLines` and `tailByBytes` and picks the shorter `[]byte` result. Edge cases:

- One very long line (5000 'A's + '\n'): `tailByLines(maxLines=100)` returns the full 5001 bytes; `tailByBytes(maxBytes=64)` returns 64 bytes; the runner picks 64.
- 10 short lines: `tailByLines(maxLines=3)` returns 12 bytes; `tailByBytes(maxBytes=1024)` returns the full 40 bytes; the runner picks 12.

`maxLines <= 0` falls back to byte-only bounding; `maxBytes <= 0` falls back to line-only bounding; both `<= 0` returns the sanitized full input (degenerate; documented in the doc-comment, not panic). Tests cover lines-shorter, bytes-shorter, and the `maxBytes=0` / `maxLines=0` fallbacks.

### 2.7 No mutation of `dispatcher.go` (HARD CONSTRAINT honored)

Verified by inspection: `gates.go` imports only `bytes`, `context`, `errors`, `fmt`, `strings`, `time`, `internal/domain`, `internal/templates`. No reference to `Options`, `Dispatcher`, `dispatcher` factory, or `RunOnce`. The runner can be exported into the package's existing wiring at 4b.7 by adding one field to `Options` and one method on `Dispatcher` — that's a clean follow-up patch, not a 4b.2 change.

### 2.8 `Register` does not validate `name` against `templates.IsValidGateKind`

Doc-comment justifies: validation lives at template-load time (Drop 3.10's `validateGateKinds`), and re-running it on every `Register` would couple the runner to the templates package's enum more tightly than necessary. A caller that registers an out-of-enum name has a wiring bug; the runner detects it indirectly when `Run()` iterates a template that references the same name and emits `ErrGateNotRegistered`. That's the right surface for the failure mode — template-author typos surface at template load; dispatcher-wiring typos surface at the first dispatch attempt.

### 2.9 `Register` permits `nil` `gateFunc` deliberately

A `nil` `fn` registered against a name will panic when `Run` invokes it. Doc-comment makes the choice explicit: nil-checking would mask a wiring bug that should be caught at startup smoke-test. The smoke-test in 4b.7 will register every gate during dispatcher construction; a nil binding there is a misconfiguration the dev wants to see fail loudly.

## 7. Round 2 — Race Lock + Spec Test Coverage

QA-Falsification Round 1 surfaced one CONFIRMED data race + missing spec-required tests + missing inter-gate ctx check. Round 2 closes all three plus a fourth deliverable (this section). Scope was strictly `gates.go` + `gates_test.go` + this worklog.

### 7.1 Fix 1 — `sync.RWMutex` on the gateRunner registry (CONFIRMED race)

`gateRunner` now carries a `mu sync.RWMutex`:

- `Register` takes `r.mu.Lock()` (exclusive write) wrapping the duplicate check + map write.
- `Run` takes `r.mu.RLock()` (shared read) for each per-iteration `r.gates[name]` lookup. RLock allows concurrent Run goroutines to share the read path; the inner `fn(ctx, item, project)` invocation runs OUTSIDE the lock so a slow gate does not block other Runs from looking up their own gates.

Doc-comment on `gateRunner` updated from "NOT safe for concurrent Register/Run" to "IS safe for concurrent Register and Run calls under a sync.RWMutex." Pin: WAVE_A_PLAN.md §4b.2 spec line 143 explicitly required this; Round 1 implementation diverged and the race reproduced under `-race`.

### 7.2 Fix 2 — Inter-gate `ctx.Err()` check, Skipped (not Failed) on cancel

Two ctx-cancel checks now sit in `Run`:

- **Pre-loop check**: if `ctx.Err()` is non-nil before any gate dispatches, return `nil` (no gate to attribute a Skipped row to; the empty slice already encodes "no work happened").
- **Inter-gate check**: at the top of each iteration, BEFORE the registry lookup, check `ctx.Err()`. On non-nil, append one `GateResult{GateName: name, Status: GateStatusSkipped, Err: ctx.Err()}` naming the next-to-run gate, then halt iteration.

**Skipped vs Failed choice — rationale:**

- Cancellation is **external** (caller's decision: timeout, shutdown, parent ctx teardown), not a verdict from the gate itself.
- A `Failed` row would conflate caller-driven teardown with a gate-driven rejection. Downstream subscribers (4b.7) reading `result.Status == GateStatusFailed` to drive failure-routing would then mis-route a ctx-canceled run as a gate failure, surfacing a false attention item.
- A `Skipped` row preserves the gate-name attribution (which gate did NOT fire because of cancel) AND keeps the status semantics clean. `Err` carries `context.Canceled` / `context.DeadlineExceeded` so the subscriber can distinguish skip-causes when needed.

The choice is documented in the new `GateStatusSkipped` doc-comment (now reads "emitted by the runner when ctx is cancelled between gates" rather than "reserved but unused") AND in the `GateResult.Err` doc-comment (notes ctx.Err() population on Skipped).
