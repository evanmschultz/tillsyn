## 4. ImportSnapshot Single-Bulk-vs-Per-Item Rationale

`Service.ImportSnapshot` writes via direct `s.repo.CreateActionItem` / `s.repo.UpdateActionItem` calls, **not** via the publishing service-layer methods. Without intervention it leaves the dispatcher subscriber unaware that an entire project tree was just imported.

Two design candidates:

1. **Per-item publish.** Wrap each `s.repo.*ActionItem(...)` call in `defer s.publishActionItemChanged(actionItem.ProjectID)`. Conceptually clean but produces N events per project import — for a 100-item snapshot that's 100 `LiveWaitEventActionItemChanged` events fanning out to every dispatcher subscribed to that project. Each wakeup triggers a tree re-walk regardless of which item triggered it, so the additional events are pure waste.
2. **Single bulk publish per project_id.** At end of successful import, collect distinct `project_id` values from `snap.Projects` and `snap.ActionItems` (deduped via `map[string]struct{}`), and publish once per project. Cheap signal; the subscriber re-walks the tree on any wakeup anyway.

Chose **option 2** — explicitly called out in the spec ("single bulk publish per project_id touched at the end of import"). Implementation: deduplicating set built from both `snap.Projects[i].ID` and `snap.ActionItems[i].ProjectID` so that imports that touch only existing projects (no new project rows) still publish, AND imports that introduce new projects without action items still publish (covers the cold-bootstrap-from-snapshot case).

Edge cases:

- **Failed import** — the publish loop sits AFTER the last error-returning call, so a partial-write failure short-circuits before publishing. The repo state may be partially updated; the subscriber will re-discover on the next event. Pre-MVP we don't try to roll back partial imports.
- **Empty snapshot** — `len(projectIDs) == 0`, the loop is a no-op. Correct.
- **Whitespace-only project_id** — defensively trimmed-and-skipped before insertion into the set; this matches `publishActionItemChanged`'s own internal guard.

## 5. Test Strategy

### 5.1 Test Harness Pattern

The publish helper `Service.publishActionItemChanged(projectID)` calls `s.liveWait.Publish(...)` synchronously inside the lifecycle method. Two helpers in `service_test.go` factor the assertion:

- `captureActionItemChangedCursor(t, broker, projectID) int64` — reads `broker.Latest(...)` before the call. Returns 0 for fresh brokers.
- `waitForActionItemChangedSinceCursor(t, broker, projectID, cursor)` — calls `broker.Wait(... afterSequence=cursor)` with a 100ms `context.WithTimeout`. The in-process broker's `register(...)` short-circuits and returns immediately when `latest.Sequence > cursor`, so this is race-free without any goroutine sleep.

This pattern is strictly cleaner than the goroutine + `time.Sleep(5ms)` approach used in `dispatcher/broker_sub_test.go` (which has to race the subscriber-vs-publisher because that test is publishing externally to an already-running subscription loop). For service-layer publish tests where Publish runs synchronously inside the service call, the cursor-capture / Latest-snapshot pattern is deterministic.

### 5.2 New Tests (6 total)

| Test | Asserts |
| ---- | ------- |
| `TestRestoreActionItemPublishesActionItemChanged` | Restore → 1 event after cursor for projectID |
| `TestRenameActionItemPublishesActionItemChanged` | Rename → 1 event after cursor |
| `TestDeleteActionItemArchivePublishesActionItemChanged` | Delete(archive) → 1 event after cursor |
| `TestDeleteActionItemHardPublishesActionItemChanged` | Delete(hard) → 1 event after cursor |
| `TestReparentActionItemPublishesActionItemChanged` | Reparent → 1 event after cursor |
| `TestImportSnapshotPublishesActionItemChangedPerProject` | Import 2 projects × 2 items each → exactly 1 event per project (Sequence == 1, NOT 2 — N-amplification regression gate) |

The ImportSnapshot test specifically pins **`Sequence == 1`** for both `p1` and `p2` after import. If a future regression switched to per-item publishing, the sequence would land at 2 (one per item) and the test would fail loudly.
