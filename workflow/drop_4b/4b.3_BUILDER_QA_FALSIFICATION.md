### T1 — `commandRunner` sufficiency for 4b.4 — **REFUTED**

**Attack:** 4b.4's `mage_test_pkg` iterates per-package and aggregates output. The `commandRunner.Run` signature returns `(stdout, stderr, exitCode, err)` per call. Does iteration work?

**Evidence:** `commandRunner.Run(ctx, dir, name, args...)` (gate_mage_ci.go:45-47) is called once per package by 4b.4 — separate stdout/stderr/exitCode/err per invocation, output ordered by call sequence. The seam is freshly invoked per iteration; per-call buffers in `execCommandRunner.Run` (gate_mage_ci.go:74) are not shared across calls. 4b.4 reuses `combineGateOutput` (gate_mage_ci.go:210-224) per package, prefixes with `"package <pkg> failed:\n"` per WAVE_A_PLAN.md §4b.4 line 246, halts on first non-zero, returns the bounded tail of THAT package's output.

**Verdict:** Seam is sufficient. The four-tuple-per-call design composes naturally over `for _, pkg := range item.Packages`. No counterexample.

### T2 — Ctx-cancel during `cmd.Start()` misclassification — **REFUTED**

**Attack:** Spec says ctx-cancel and start-failure produce distinct Err prefixes (`"mage ci context cancelled:"` vs `"mage ci start failed:"`). What if ctx fires DURING `cmd.Start()` — before `cmd.Wait()` — could the gate misclassify?

**Trace:**
1. `cmd.Start()` (gate_mage_ci.go:78) errors → `execCommandRunner.Run` returns `(nil, nil, 0, err)`.
2. Back in `gateMageCI`, ctx.Err() is checked **first** (gate_mage_ci.go:163) BEFORE runErr (gate_mage_ci.go:172).
3. So if ctx fired during Start, ctx.Err() returns non-nil → ctx-branch wins → Err prefix is `"mage ci context cancelled:"`.
4. If ctx is intact and Start failed for other reasons (PATH miss, fork-EXEC errno) → ctx.Err() returns nil → falls to runErr branch → Err prefix is `"mage ci start failed:"`.

**Counterexample attempt:** Could `exec.CommandContext` complete `Start()` successfully and then have ctx fire before any read happens? Yes — but in that case `cmd.Wait()` returns ctx-killed-process error (not `*exec.ExitError`); `execCommandRunner.Run` returns `(stdout, stderr, 0, err)` per gate_mage_ci.go:87; ctx.Err() check at line 163 still fires first → ctx-branch.

**Verdict:** ctx.Err() priority ordering is the load-bearing line. No misclassification path.

### T3 — Output capture order — interleaved stdout/stderr — **MITIGATED-AS-ACCEPTED**

**Attack:** `combineGateOutput` (gate_mage_ci.go:210-224) does stdout-then-stderr concat. What if mage's failure prints INTERLEAVED between stdout and stderr? E.g., 50 stdout lines, 1 critical stderr line, 50 more stdout lines. The combined output puts ALL stderr (1 line) AFTER all stdout (100 lines). The 1 critical stderr line stays in the bounded window (8KB / 100 lines is ample), but the temporal interleaving is destroyed — a dev grepping the output might mis-attribute cause-and-effect.

**Worse counterexample:** 200 stdout lines + 1 critical stderr line at the start. Combined: `[stdout 0..199, stderr 0]`. Last 100 lines: `[stdout 100..199, stderr 0]`. The critical stderr line is RETAINED at the tail (good — it's the failure summary). But its temporal context (printed BEFORE stdout 100..199) is lost.

**Mitigation:** WAVE_A_PLAN.md §4b.2 A2 explicitly accepts this: "Combined stdout+stderr buffer interleaves writes from concurrent goroutines inside `mage`. Mitigation: `exec.Cmd` provides one-writer-per-stream serialization at the syscall level; sharing the same `bytes.Buffer` via `cmd.Stdout = buf; cmd.Stderr = buf` is the **idiomatic combined-output pattern**." The shipped code went the OTHER direction — separate buffers per stream, then combined-after-the-fact — which is **worse for temporal interleaving** but **better for keeping stderr in the failure-tail window** under byte bounding.

The trade-off is explicit (gate_mage_ci.go:50-54 doc-comment + WAVE_A_PLAN.md §4b.3 line 198): "stderr last keeps the failure-summary tail inside the bounded window when `tailOutput` byte-bounds at 8KB." The dev's failure-loop instinct — find the failure summary — is preserved at the cost of perfect temporal ordering.

**Verdict:** ACCEPTED. The trade-off is documented and intentional. The alternative (single-buffer combined output, like `exec.Cmd.CombinedOutput`) was deliberately rejected per worklog "Output Capture Aggregation Order" section. A future refinement (e.g., interleaved single-buffer with stream-tag prefixes) is a Drop 4c-or-later concern.

### T4 — `tailOutput` corner cases — **REFUTED**

**Attack 1:** 1000 lines, all very short (e.g., `"x\n"`) → 1000 lines = 2000 bytes. Line bound applies (last 100 lines = 200 bytes); byte bound also applies (last 8192 bytes = full input = 2000 bytes). "Shorter" wins → 200 bytes via line bound. ✓

**Attack 2:** 100 lines, EACH 1MB. Line bound: last 100 lines = full input = 100MB. Byte bound: last 8KB = 8KB slice. "Shorter" wins → 8KB via byte bound. ✓

**Attack 3:** Empty input → returns `""` per gates.go:302-304. ✓

**Attack 4:** No trailing newline (e.g., `"abc"`) → `tailByLines` walks backwards looking for `\n`, finds none, count never reaches `maxLines`, falls through to `return b` (gates.go:347). Byte bound separately applies. Both bounds correct.

**Attack 5:** Exactly `maxLines` newlines → walk counts up to `maxLines` but condition `count > maxLines` (gates.go:343) is strict greater-than, never triggers, returns full `b`. ✓ (You want last 100 lines = whole input when there are exactly 100 lines.)

**Attack 6:** `maxLines + 1` newlines → walk hits `count > maxLines` on the (maxLines+1)th newline, returns `b[i+1:]`. Slice starts AFTER that newline, so the first `maxLines` lines are dropped. Last `maxLines` lines retained. ✓

**Attack 7:** Concurrent dimension test — **TestGateMageCITailsLongOutput** (gate_mage_ci_test.go:303-346) explicitly asserts both bounds: `len(Output) <= MaxGateOutputBytes` AND `strings.Count(Output, "\n") <= MaxGateOutputLines`. Note the test's stdout is 1000 lines × ~10 bytes each = ~10000 bytes; line-bound at 100 lines = ~1000 bytes; byte-bound at 8KB = 8192 bytes. Line-bound is shorter → wins. The test's both-bounds assertion is correct.

**Verdict:** All edge cases handled. `tailOutput` is correct.

### T5 — Duration via wall-clock — system clock jump — **MITIGATED-AS-ACCEPTED**

**Attack:** `time.Since(start)` (gate_mage_ci.go:149, 168, 177, 185, 197) uses wall-clock. NTP step or DST change during gate run could produce nonsensical Duration (negative, or wildly inflated).

**Mitigation:** Go's `time.Now()` returns a monotonic-clock-bearing `Time` value since Go 1.9 (per stdlib `time` package docs). `time.Since(t)` subtracts using the monotonic component if both Times have it (the case here — both come from `time.Now()`). System clock jumps do NOT affect monotonic readings. Per Go stdlib: "Because the monotonic clock reading has no meaning outside the current process, serializing a Time and then deserializing it ... loses the monotonic clock reading." We're not serializing across process boundaries, so monotonic survives. NTP step → no effect. DST → no effect (DST doesn't even affect monotonic by design).

**Verdict:** REFUTED. Go's monotonic-clock semantics are correct here. The implementation is safe.

### T6 — `errors.As(*exec.ExitError)` for signal-killed processes — **REFUTED**

**Attack:** What if mage is signal-killed (SIGKILL, SIGTERM)? Does `errors.As(err, &exitErr)` (gate_mage_ci.go:84) capture it?

**Evidence:** From Go stdlib `os/exec` docs: "An ExitError reports an unsuccessful exit by a command. ... ExitError contains a ProcessState which embeds an os.ProcessState." A signal-killed process produces an `*exec.ExitError` wrapping `*os.ProcessState` whose `ExitCode()` returns -1 (per `os.ProcessState.ExitCode` doc: "Returns -1 if the process hasn't exited or was terminated by a signal.").

**Counterexample attempt:** signal-killed → `errors.As(err, &exitErr)` succeeds → `exitErr.ExitCode()` returns -1 → `execCommandRunner.Run` returns `(stdout, stderr, -1, nil)`. Gate's `if exitCode == 0` (gate_mage_ci.go:181) fails, falls through to non-zero-exit branch, Err is `"mage ci failed: exit code -1"`. ✓

What about a SIGKILL from the OS (OOM killer)? Same path — `*exec.ExitError` with exit code -1. Distinguishable in logs by the `-1` value vs. mage's typical 1/2 codes.

What about a process that ctx kills? `exec.CommandContext` kills the child when ctx fires. `cmd.Wait()` returns `*exec.ExitError` wrapping ProcessState with exit-code-from-signal. BUT — ctx.Err() check fires FIRST in `gateMageCI` (gate_mage_ci.go:163). So ctx-killed is reported as ctx-cancel, not as exit-code-failure. That's the right semantics.

**Verdict:** REFUTED. Signal-kill → exit code -1 in non-zero branch. ctx-kill → ctx-cancel branch. OS-kill → exit code -1 in non-zero branch with no ctx.Err(). All distinguishable.

### T7 — Production wrapper untested — **NIT (acceptable)**

**Attack:** `execCommandRunner.Run` (gate_mage_ci.go:70-91) is the production code path. Tests inject a `fakeCommandRunner` swap (gate_mage_ci_test.go:44-51). **Zero tests exercise `execCommandRunner` directly.** I confirmed via `rg "execCommandRunner"`: only doc-comment + the type def + the `defaultCommandRunner` var assignment; no test reference.

**Implication:** The four code paths in `execCommandRunner.Run` (Start error, Wait error of `*exec.ExitError` type, Wait error of other type, Wait nil) are unverified by behavior assertions. A regression in `errors.As` line 84 — say a developer typo'd `errors.Is` instead — would not be caught by 4b.3's test suite alone. The fake's `err` field directly returns whatever the test sets, bypassing the real Start/Wait split.

**Mitigation in spec:** WAVE_A_PLAN.md §4b.3 acceptance line 202 explicitly says "Avoids hitting the actual `mage` binary in CI." This is intentional — the spec accepts that production wrapper testing falls outside this droplet's CI guarantees.

**Why this is a NIT not a CONFIRMED counterexample:**
- The spec acceptance bar (WAVE_A_PLAN.md §4b.3) explicitly draws the test-coverage line at the seam.
- `mage ci` itself runs `mage ci` recursively — meta-circular, hard to test in-CI without infinite recursion guards. The "use the real binary" path is impractical.
- The 4-outcome contract IS pinned by `commandRunner.Run`'s doc-comment (gate_mage_ci.go:16-44) and the gate's branching tests against the fake.
- Detection of an `errors.As` typo would surface immediately in dogfooding (the gate would never produce the right exit-code-bearing branch).

**Routing:** Surface as a Drop 4c refinement candidate — when 4b.7's subscriber wires `gateMageCI` into a real dispatcher path, an integration test could shell out to a no-op binary (`true` / `false` / a tiny scripted test fixture) to exercise `execCommandRunner` end-to-end. NOT a 4b.3 blocker.

**Verdict:** NIT.

**Attack:** Spec says "mage must be on PATH" (gate_mage_ci.go:124-127). Does the gate fail-LOUD or silently when mage is missing?

**Trace:** mage missing → `cmd.Start()` returns error like `"exec: \"mage\": executable file not found in $PATH"` → `execCommandRunner.Run` returns `(nil, nil, 0, err)` → ctx.Err() is nil (assume normal ctx) → runErr is non-nil → Err is `"mage ci start failed: exec: \"mage\": executable file not found in $PATH"`. **Loud error string** matching the test's expected substring (gate_mage_ci_test.go:174). The gate returns Failed status. The 4b.7 subscriber will route to action-item `failed` lifecycle with `BlockedReason = "mage ci start failed: ..."`.

**Counterexample attempt:** Could the gate silently treat "mage missing" as "no gate to run, skip"? No — there is no PATH-existence check; the only short-circuit is empty-worktree (gate_mage_ci.go:144-151). Missing-mage flows through the runner branch and produces a loud failure. ✓

**Verdict:** Fail-loud. REFUTED.

### T10 — Concurrent `Run` safety — **REFUTED**

**Attack:** `gates.go:156-163` claims `gateRunner` is safe for concurrent Run. But `gateMageCI` reads `defaultCommandRunner` (gate_mage_ci.go:153) — a package-level var. Could concurrent test swaps via `withFakeCommandRunner` race?

**Evidence:** The doc-comment (gate_mage_ci_test.go:42-43) explicitly states: "Tests do NOT call `t.Parallel()` because the swap mutates package state." The test file does NOT call `t.Parallel()` anywhere — confirmed via `rg "t.Parallel\(\)" gate_mage_ci_test.go` (no hits). So serial execution is enforced at the test level.

In production, `defaultCommandRunner` is set ONCE at package init (gate_mage_ci.go:96 — `var defaultCommandRunner commandRunner = execCommandRunner{}`) and never reassigned. Concurrent Run calls all read the same value → no race.

**Verdict:** REFUTED. Test-file convention prevents parallelism; production-side var is init-only.

### T11 — Unused `domain.ActionItem` parameter — **NIT**

**Attack:** `gateMageCI(ctx context.Context, _ domain.ActionItem, project domain.Project)` (gate_mage_ci.go:141) — the ActionItem parameter is `_`-discarded. The `gateFunc` type signature (gates.go:67) requires it. But why does THIS gate accept it?

**Mitigation:** Required by `gateFunc` (gates.go:67) — uniform contract. 4b.4 reads `item.Packages`. No way to slim the contract for one gate without forking the type.

**However:** WAVE_A_PLAN.md §4b.3 PQA-1 line 303 raises this exact YAGNI watch: "does the gate need `templates.Template` or just `Item` + `Project`? Builder may slim if `Template` proves unused." The shipped impl already slimmed away `Template` (gate signature is 3-arg, not 4-arg with template). Good — that surface is closed. But the `Item` parameter for `mage_ci` truly is unused — and will stay unused (mage_ci runs the whole CI build, no per-item read).

**Why this is a NIT not a counterexample:** Conformance to a uniform `gateFunc` type is more valuable than narrowly-typed gate signatures. The `_` discard is idiomatic Go for "I implement the contract, this slot doesn't apply." The cost is one ignored parameter; the benefit is the runner uses one type everywhere. ACCEPTED.

**Verdict:** NIT. No action needed.
