## Design Choices

### Empty `item.Packages` → fail-loud (per WAVE_A_PLAN.md WA-A5)

The gate guards `len(item.Packages) == 0` ahead of the worktree guard and returns
`GateStatusFailed` with an explicit `Err` naming the misconfiguration:

```
mage_test_pkg: action item declares no packages — planner must populate item.Packages or remove this gate from kind
```

Silent-pass would mask a real QA gap: a planner that bound `mage_test_pkg` to a
kind without populating `Packages` on the action item has produced unverified
build output. Failing the gate forces the planner / template author to fix
the binding. The dedicated test (`TestGateMageTestPkgRejectsEmptyPackages`)
configures the fake runner to *would-pass* and asserts (a) Failed verdict,
(b) zero runner calls, (c) `Err` contains "no packages" — proving the guard
short-circuits before any process work happens.

Guard-order rationale: **packages-first, worktree-second.** A misconfigured
template (gate bound but no packages) is a project-shape bug worth surfacing
even when the worktree is also empty. Surfacing the most-actionable error
first (planner action vs. dispatcher-config action) keeps the failure-loop
tight.

### Aggregated output ordering — per-package `stdout+stderr`, joined in declared order

Per-iteration the gate calls `combineGateOutput(stdout, stderr)` (the 4b.3
helper, reused verbatim — no copy-paste of the implementation) to fold one
package's streams into a `pkgN_stdout\n+pkgN_stderr` slice, then folds that
into the running `aggregated` buffer with the same helper. Net ordering:
`pkg1_stdout, pkg1_stderr, pkg2_stdout, pkg2_stderr, …` — declared order.

`tailOutput(aggregated, MaxGateOutputLines, MaxGateOutputBytes)` runs **once
at the end**, on the failed result only. This was a falsification target
during design: applying `tailOutput` per-iteration would lose pkg1's lines
when pkg2 fails after them, defeating the "surrounding context" intent. The
single-tail-at-end pattern preserves the run-so-far while still bounding
under 100 lines / 8KB.

Aggregation also runs on the start-error and ctx-cancel paths so any
partial bytes the runner captured before the abort are surfaced, not
silently dropped. Tests assert ordering and bound:

- `TestGateMageTestPkgAggregatesOutputAcrossPackages` — pkg1 (50 lines) + pkg2 (50 lines, fails) + 1 stderr line = 101 lines; line bound (100) drops `PKG1_LINE_00\n` only; pkg1 marker precedes pkg2 marker; pkg2 stdout precedes pkg2 stderr.

### Halt-on-first-failure across packages

The for-loop returns on the first non-zero exit, start-error, or ctx-cancel.
The next package is NOT invoked. Tests prove this via the
`scriptedCommandRunner`'s overflow panic: the script lists only the responses
the gate is allowed to consume; if the gate reaches a missing entry the test
fails loudly rather than silently returning a default response.

- `TestGateMageTestPkgFailsOnFirstPackageNonZero` — pkg1 fails, pkg2/pkg3 absent from script (overflow panics if reached). Asserts `runner.calls == 1`.
- `TestGateMageTestPkgFailsOnSecondPackageNonZero` — pkg1 passes, pkg2 fails, pkg3 absent. Asserts `runner.calls == 2` and last call's `args[1] == "pkg2"`.
- `TestGateMageTestPkgFailsOnStartError` — pkg1 passes, pkg2 start-errors, pkg3 absent. Asserts `runner.calls == 2`.

### `commandRunner` reuse — no copy-paste, package-private import

The gate file imports nothing new from 4b.3: it calls `defaultCommandRunner.Run`
directly (the package-level var declared in `gate_mage_ci.go`) and references
`combineGateOutput` (also from `gate_mage_ci.go`) and `tailOutput` /
`MaxGateOutputLines` / `MaxGateOutputBytes` (from `gates.go`). All four
symbols are package-private — same package, direct reference, no shim.

The 4b.3 test file's `withFakeCommandRunner` t.Cleanup-restored swap helper
is reused verbatim by the new tests; that helper is generic over any
`commandRunner` implementation, so it accepts the new `scriptedCommandRunner`
without modification.

### `scriptedCommandRunner` — distinct test double, sibling of `fakeCommandRunner`

`fakeCommandRunner` from 4b.3's test file returns the same canned tuple on
every call, which doesn't fit gate_mage_test_pkg's per-package iteration
(pkg1 passes, pkg2 fails). I introduced `scriptedCommandRunner` as a
**sibling** rather than a generalization: the existing 4b.3 fake stays
unchanged (HARD CONSTRAINT — no edits to `gate_mage_ci_test.go`), and the
new fake panics on overflow so halt-on-first-failure violations surface
loudly instead of silently consuming default responses.

Per-call invocation recording uses a slice of `scriptedInvocation` rather
than overwriting the most-recent fields: tests need to assert the args of
every call (forwarded packages in declared order), not just the last.

### Doc-comments on the gate

Three load-bearing semantics flagged in the gate function's doc-comment:

1. **Halt-on-first-failure** across packages, mirroring the gate-runner's halt-on-first-failure semantic at the gate level.
2. **Empty `item.Packages` is fail-loud** with the rationale: silent-pass would mask a real QA gap.
3. **Aggregated output strategy**: per-package `stdout+stderr` joined in declared order; failed-package output retained in the bounded tail; surrounding successful-package output retained for failure-loop context.

`commandRunner` reuse is documented in the gate's doc-comment ("commandRunner reuse: this gate calls defaultCommandRunner.Run (defined in gate_mage_ci.go) directly. No new commandRunner type is introduced; the 4b.3 seam is the single extension point for mage-shelling gates.").
