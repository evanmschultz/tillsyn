## Design Choices

### Helper lives in `internal/app`, not `internal/domain`

WAVE_C_PLAN §3.4 acceptance criterion 1 suggested `internal/domain/git_status.go`. The spawn-prompt scope explicitly narrows my `paths` to `internal/app/service.go` + tests and locks the package to `internal/app`. I honored the narrower scope: helper + tests live entirely in `internal/app`. The function-type seam (`type GitStatusChecker func(...) ([]string, error)`) is internal to the package today; a future refactor can promote it to `internal/domain` if a non-`app` caller needs it. Leaving it here avoids creating a domain dependency the spawn prompt forbids.

### `GitStatusChecker` as a function type, not an interface

Spec acceptance criteria 4 says "Choose the lowest-disruption surface — likely a new field on `*app.Service`." I chose `func(ctx, repoRoot, paths) ([]string, error)` over an `interface{ CheckPathsClean(...) error }` because:

1. The seam has one operation. An interface adds a name (`CheckPathsClean`) without earning its keep.
2. Function-type stubs are trivially constructable (`recordingGitChecker.check` is method-bound to a struct with `dirtyPaths` + `err` config; assignment is `svc.gitStatusChecker = checker.check`).
3. The signature returns `(dirty []string, err error)` rather than `error` so the caller (`runGitStatusPreCheck`) can format the wrapped `ErrPathsDirty` message with the dirty-path list once, in one place — instead of every implementation reformatting the message itself.

### Per-path invocation, not batched

Per REVISION_BRIEF Q6 + WAVE_C_PLAN §3.4 acceptance 1: typical droplet path counts are <10. Each invocation is `exec.CommandContext(ctx, "git", "status", "--porcelain", "--", path)` with `cmd.Dir = repoRoot`. The `--` is critical: it disambiguates `path` from `<commit>` in case the path starts with `-` or matches a refname. The for-loop accumulates dirty paths in input order, so the wrapped `ErrPathsDirty` message is deterministic.

Future profiling refinement (documented as Unknown): `git status --porcelain --pathspec-from-file -` collapses N invocations into one when path counts grow to triple-digits. Not in 4b.6 scope.

### Env-isolation reuse from gitdiff (round 3)

`internal/tui/gitdiff/exec_differ_test.go` documents the round-1 → round-3 evolution for env isolation. I adopted the round-3 pattern verbatim:

1. `filteredGitEnv()` strips ALL `GIT_*=...` keys from `os.Environ()` before any append. This is load-bearing: under a `git push` pre-push hook, git leaks `GIT_DIR`, `GIT_INDEX_FILE`, `GIT_WORK_TREE`, `GIT_PREFIX` to child processes. `GIT_DIR` overrides repository-discovery entirely — `GIT_CEILING_DIRECTORIES` does NOT undo it. Only stripping the inherited keys keeps the per-call invocation operating against `repoRoot`.
2. `GIT_CEILING_DIRECTORIES=<repoRoot>` halts the discovery walk so git never escapes upward into a parent bare repo.
3. `GIT_CONFIG_NOSYSTEM=1`, `GIT_CONFIG_GLOBAL=/dev/null`, `HOME=<repoRoot>`, `XDG_CONFIG_HOME=<repoRoot>` quarantine config reads from any system / global / per-user file the dev's machine might have locked under a concurrent push.
4. `GIT_TERMINAL_PROMPT=0` + `GIT_PAGER=cat` keep the subprocess non-interactive so a git invocation cannot block on credential or pager prompts.

The `TestCreateActionItemEnvIsolatesFromBareRoot` + `TestDefaultGitStatusCheckerIgnoresInheritedGitDir` tests prove the isolation by setting `GIT_DIR` to a bogus path via `t.Setenv` and asserting the pre-check still operates against the fixture worktree. These two tests cannot use `t.Parallel()` because `t.Setenv` panics under parallel execution; the rest of the production-checker tests use `t.Parallel()` because they thread isolation through `cmd.Env` exclusively.

### `ErrPathsDirty` sentinel + wrapping pattern

Sentinel lives in `internal/app/errors.go` next to existing `ErrNotFound` etc. The wrap pattern is:

```go
return fmt.Errorf("%w: %s", ErrPathsDirty, strings.Join(dirty, ", "))
```

so callers detect via `errors.Is(err, ErrPathsDirty)` and the dirty-path list is human-readable in the error message. Multiple-dirty-paths are joined with ", " in input order — `TestCreateActionItemRejectsMultipleDirtyPaths` pins the deterministic ordering.

`ErrGitNotFound` is separate from `ErrPathsDirty` so callers can distinguish "fix your tree" from "install git." It wraps `exec.ErrNotFound` so `errors.Is(err, exec.ErrNotFound)` also holds — useful when callers want to handle the env-misconfig case differently.

### Insertion-point ordering inside `CreateActionItem`

Spec called for the pre-check between line 841 (parent-lineage validation) and 907 (repo write). I placed it at line ~915 (post-`mage format` numbering), AFTER the position computation and BEFORE `domain.NewActionItem(...)`. Rationale:

1. After mutation-guard, template-nesting, and kind-resolution checks: a well-formed-but-dirty input still gets the dirty rejection (don't mask it behind another error class), but a malformed input still rejects on its own structural error first (don't run the expensive git pre-check on input we'd reject anyway).
2. Before any state-mutating call — `domain.NewActionItem` is pure, and the actual `s.repo.CreateActionItem` is at line 943. The guarantee "dirty-paths reject doesn't mutate state" holds.
3. The check fires only when `len(in.Paths) > 0` — the existing zero-path code path is byte-for-byte identical post-edit (verified by `TestCreateActionItemHandlesEmptyPaths` asserting zero checker calls).

### Skip semantics for empty `RepoPrimaryWorktree`

Per WAVE_C_PLAN §3.4 acceptance 2: a project with empty `RepoPrimaryWorktree` skips the check rather than rejecting. The rationale is the pre-MVP escape valve — Drop 4a-pre-Drop-4b legacy projects + every existing test fixture leave `RepoPrimaryWorktree` empty. Rejecting would brick those creators. `runGitStatusPreCheck` short-circuits on `strings.TrimSpace(project.RepoPrimaryWorktree) == ""` and returns nil; `TestCreateActionItemSkipsCheckOnEmptyWorktree` pins the behavior with a dirty-reporting stub that nonetheless never gets invoked.

The stub-checker call-recording in `recordingGitChecker.calls` is the proof: zero entries = "the pre-check correctly skipped." That makes "skip" a positive-assertion test rather than a passive observation.

### Skip semantics for project lookup failure

`runGitStatusPreCheck` calls `s.repo.GetProject(ctx, projectID)`. If that returns an error (project deleted between call setup and pre-check, race, etc.), I treat it as a soft skip: return nil and let the downstream `s.repo.CreateActionItem` surface its own typed error. Rationale: the pre-check is best-effort; making it a second source of "project not found" errors would create two rejection paths for the same condition, which is harder to test and harder to surface clearly. The downstream write IS the authoritative source.

### Why a function-type seam, not an interface, in `ServiceConfig`

The spawn prompt called for "lowest-disruption surface." Adding `GitStatusChecker` as a function-type field on `ServiceConfig` matches the existing pattern (see `LiveWaitBroker` at line 94 — also injected via config field, with default constructor in `NewService` if nil). No new `interface` declaration, no new constructor function, no new `WithX` option setter. Production callers leave it nil and pick up `defaultGitStatusChecker` automatically.

### Stub design — `recordingGitChecker`

The stub's two configurable knobs (`dirtyPaths []string` + `err error`) cover every shape the production checker can return: clean (both empty), dirty (paths set), env failure (err set). The recorded `calls []recordingGitCheckerCall` slice supports zero-call assertions (skip-path tests) and exact-args assertions (happy-path tests). Lives in `service_test.go` near the new tests so it stays scoped to the droplet's needs without polluting unrelated tests.

## Round 2 — C1 Ordering Fix + NITs

QA-Falsification surfaced one CONFIRMED ordering bug + 4 NITs. Round 2 applied 4 fixes (C1, NIT-1, NIT-2, NIT-3, NIT-6) and added 3 regression tests. NIT-4/5/7 routed to refinements memory by the orchestrator (out of round-2 scope).

### C1 — Ordering bug: pre-check ran before domain validation

**Problem:** `runGitStatusPreCheck` was called at `service.go:930-934` BEFORE `domain.NewActionItem` (line 936). `domain.NewActionItem` invokes `normalizeActionItemPaths` (`action_item.go:413`) which rejects empty / whitespace / backslash entries with `domain.ErrInvalidPaths` per `action_item.go:713-718`. So malformed inputs like `Paths=[""]` reached `git status --porcelain -- ""` instead of rejecting cleanly at the domain layer.

**Fix:** Adopted Option (a) from the spawn prompt — moved the pre-check call AFTER `domain.NewActionItem` succeeds, BEFORE `s.repo.CreateActionItem`. The pre-check now reads the post-validation `actionItem.Paths` slice (which `normalizeActionItemPaths` has already trimmed, deduped, and validated) instead of raw `in.Paths`. State-mutation guarantee preserved: dirty-paths reject still does not write to the repo (the pre-check sits between domain construction and `s.repo.CreateActionItem`).

**Doc-comment update:** the inline rationale on the pre-check block now explicitly notes that domain validation runs first, so the pre-check can trust `actionItem.Paths` to be well-formed.

### NIT-1 — `ErrGitNotFound` was untested

Added `TestOsGitStatusCheckerHandlesGitBinaryMissing` in `git_status_test.go`. The fixture sets `PATH` to a freshly-created empty tmpdir via `t.Setenv`, so `exec.LookPath("git")` cannot resolve. Asserts `errors.Is(err, ErrGitNotFound)` AND `!errors.Is(err, ErrPathsDirty)` — callers must distinguish "fix your env" from "fix your tree."

Sequential test (no `t.Parallel`) because `t.Setenv` requires it. Mirrors the same constraint as `TestDefaultGitStatusCheckerIgnoresInheritedGitDir` and `TestCreateActionItemEnvIsolatesFromBareRoot`.

### NIT-2 — Nonexistent-path semantics were untested

Added `TestOsGitStatusCheckerHandlesNonexistentPath` in `git_status_test.go`. Fixture commits `alpha.go` then probes a `does/not/exist.go` path that was never tracked AND does not exist on disk. Empirical observation: `git status --porcelain -- <missing>` returns exit 0 with empty stdout, so the checker reports the path as clean.

Test pins this behavior with a doc-comment explaining the rationale: pre-check is best-effort "don't start work on a dirty file," NOT "validate every path exists." Path-existence enforcement is the domain layer's job at the ref-attachment / package-coverage layer. If a future git version changes the behavior (some versions warn on pathspec-not-matched), the test surfaces the change as a refinement candidate rather than silent breakage.

### NIT-3 — HOME asymmetry vs gitdiff fixture

Production sets `HOME=<repoRoot>`; fixture sets `HOME=<separate tmpdir>`. Functionally equivalent because `GIT_CONFIG_GLOBAL=/dev/null` pins the global-config read regardless of where HOME points. Picked the doc-comment route (per spawn-prompt guidance "If that's heavyweight, just document the asymmetry") rather than `os.MkdirTemp`-per-call: a per-invocation tmpdir would require an `os.RemoveAll` cleanup on every call, adding cost to the production hot path with zero behavioral gain.

Doc-comment added to `defaultGitStatusChecker` explaining: (1) functional equivalence via `GIT_CONFIG_GLOBAL=/dev/null`, (2) production avoids the tmpdir cycle, (3) fixture composes more isolation helpers (write-tracking, multi-commit timeline) that benefit from a dedicated home directory.

### NIT-6 — ctx-cancel over-conditioning

`runGitStatusPath` had `if ctx.Err() != nil && (errors.Is(err, context.Canceled) || errors.Is(err, context.DeadlineExceeded) || errors.Is(ctx.Err(), context.Canceled) || errors.Is(ctx.Err(), context.DeadlineExceeded))`. Simplified to `if ctx.Err() != nil { return "", ctx.Err() }`. Rationale: `ctx.Err()` returning non-nil is itself the canonical "context was canceled or deadline exceeded" signal — re-checking which sentinel it is via `errors.Is` is redundant. Moving the ctx-cancel check ahead of the stderr-extraction also fixes a subtle pre-existing bug: a ctx-cancel that left non-empty stderr (e.g. `signal: killed`) would have been wrapped as a stderr error rather than propagating `ctx.Err()`. Now ctx-cancel always wins over stderr.

Removed the now-unused `errors` import from `git_status.go` (only remaining reference was in a comment).

### Regression test for C1

Added `TestCreateActionItemRejectsMalformedPathsBeforePreCheck` in `service_test.go`. Fixture: `Paths=["internal/foo/bar.go", ""]`, `recordingGitChecker{}` with no dirty paths configured. Asserts:

1. `errors.Is(err, domain.ErrInvalidPaths)` — domain validation fires.
2. `len(checker.calls) == 0` — pre-check never reached.
3. `len(repo.tasks) == 0` — no row persisted.

The empty-entry input is the malformed case; `normalizeActionItemPaths` short-circuits at `action_item.go:713`. Without the C1 fix, this test would fail because `runGitStatusPreCheck` would invoke the checker with `Paths=["internal/foo/bar.go", ""]` BEFORE the domain rejects.
