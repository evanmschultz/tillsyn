- **Design notes (judgment calls):**
  - **SSR guard pattern.** Initial first-pass build (before the guard) produced a non-fatal SSR `ReferenceError` in the resumability stream ‚Äî Astro's `client:idle` defers _hydration_ on the client, but the SSG/SSR pass STILL renders the island server-side to produce initial markup, and `createResource`'s loader fires synchronously during that pass. The build exited 0 (Astro serializes the error to the `_$HY` resumability payload for client replay) and the user-visible behavior would be correct (client mount refires the resource), but the dev console + dist artefact noise is undesirable. Added `if (typeof window === 'undefined') return [];` to the loader so SSR returns the empty array, and the client refires on hydration. The SSR'd markup then shows the empty-state literal `No projects yet` ‚Äî which is a UX win: the user sees the empty-state string immediately during the hydration delay, then either an unchanged page (if the DB is empty) or the populated list (if not). Documented inline with a comment-block explaining why.
  - **Triple-`<Show>` nesting.** SolidJS `createResource` exposes three states: loading (resource is undefined, `.loading=true`), error (`.error` is set), and resolved (`resource()` returns the array). The `<Show>` triple cleanly discriminates: outermost gates on loading, middle gates on error, innermost gates on `length > 0`. Each fallback path is a single short JSX expression. Alternative: a single `Switch/Match` block ‚Äî but `Show` reads better for three sequential gates and minimizes per-state markup branching.
  - **`projects() ?? []` defensive default.** `createResource`'s resolved value is `T | undefined`. The `Show when={!projects.loading}` should guarantee `projects()` is `T`, but the TypeScript inference doesn't track that proof so the inner `<Show when={(projects() ?? []).length > 0}` uses a defensive `?? []` to satisfy the type system without an unsafe non-null assertion. Zero runtime cost.
  - **`<For>` over `Index`.** Project rows have stable IDs; `<For>` keyed by reference identity is the correct primitive (re-renders only when an item is added/removed/reordered; `<Index>` is for stable-position keyed-by-index lists). When the resource refetches, the same `{ID, Name}` shapes should produce identical references ‚Äî Solid's reactive primitives memoize the array reference shape across resource refetches.
  - **`role="alert"` on error path.** Single a11y nicety: screen readers announce error states. The other two paths (loading, empty) are visible-state changes that don't need ARIA live-region semantics.
