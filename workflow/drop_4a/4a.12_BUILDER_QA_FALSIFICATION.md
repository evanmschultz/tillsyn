### 1.1 Vector 1 — `NewProject` thin-wrapper faithfulness — REFUTED

### 1.5 Vector 5 — `RepoBareRoot` validation including tilde — REFUTED

`internal/domain/project.go:171-176`: `repoBareRoot != "" && !filepath.IsAbs(repoBareRoot)` rejects relative paths. Verified counterexamples:

- `"./relative"` → `filepath.IsAbs` → false → REJECTED with `ErrInvalidRepoPath`.
- `""` (empty) → short-circuits, ACCEPTED.
- `"/Users/evan/code"` → `filepath.IsAbs` → true → ACCEPTED.
- `"~/expanded"` → `filepath.IsAbs` → false on Unix (Go does NOT expand tildes; verified via Context7 `path/filepath` docs) → REJECTED.

Tilde rejection is consistent with Go's standard handling (`go env -w GOMODCACHE=~/test` rejects with "must be absolute path"). The plan §1.8 did not specify tilde behavior; rejecting is the conservative interpretation. No counterexample.

### 1.7 Vector 7 — Legacy `till.update_project` MCP shim silent-clobber — CONFIRMED

`internal/adapters/server/mcpapi/extended_tools.go:856-862` (legacy `till.update_project` shim):

```go
project, err := projects.UpdateProject(ctx, common.UpdateProjectRequest{
    ProjectID:   args.ProjectID,
    Name:        args.Name,
    Description: args.Description,
    Metadata:    args.Metadata,
    Actor:       actor,
})
```

The legacy shim's `args` struct (lines 813-822) does NOT include `HyllaArtifactRef`, `RepoBareRoot`, `RepoPrimaryWorktree`, `Language`, `BuildTool`, `DevMcpServerName`. It builds a `UpdateProjectRequest` with all 6 new fields = `""` (Go zero value).

Trace through the chain:

1. `common.UpdateProjectRequest{HyllaArtifactRef: "", ...}` → adapter at `app_service_adapter_mcp.go:570-583` calls `strings.TrimSpace("")` → `""` → passes into `app.UpdateProjectInput`.
2. `Service.UpdateProject` at `service.go:449` calls `project.UpdateDetails(in.Name, in.Description, "", "", "", "", "", "", in.Metadata, ...)`.
3. `UpdateDetails` at `project.go:260-265` writes `p.HyllaArtifactRef = ""` etc. — **clobbers any existing values**.

Reproduction: dev creates a project via canonical `till.project(operation=create, ..., hylla_artifact_ref="github.com/evanmschultz/tillsyn@main", language="go", ...)`. Project lands in DB with all fields populated. Dev (or stale agent) then calls legacy `till.update_project(project_id=..., name="Renamed")` — succeeds, but project's `HyllaArtifactRef`, `Language`, etc. are wiped to empty strings.

**Spawn prompt context:** the falsification prompt's Vector 7 explicitly anticipated this: "If a legacy-shim caller now sends a partial payload that the canonical till.project would reject, what happens? (Should: legacy-shim path takes empty-default values, no breakage.)" That guidance focuses on COMPILE / authentication breakage. Behaviorally, the legacy shim now produces silent data loss — empty Project zero-value semantics, by design, mean "absence of intent" but `UpdateDetails` treats them as "explicit empty."

The fix space:

- (a) Read-merge-write inside the legacy shim: read current project, override only the explicitly-supplied fields, pass full struct through. Matches the doc-comment intent on `UpdateProjectRequest`: "MCP callers that omit a field must pass the existing value through (same shape as Metadata)." But `Metadata` itself has the same problem — the legacy shim does not read-merge `Metadata` either, so this is a pre-existing pattern, not a 4a.12 regression.
- (b) Pointer-sentinel the 6 new fields on `UpdateProjectRequest` (matches Drop 3.21 / 4a.5 precedent for action-item update fields).
- (c) Document the legacy shim as "deprecated; partial-update will clobber first-class fields, use `till.project` instead."

The wave plan §1.8 explicitly chose value-typed (option b ruled out). The builder correctly implemented per the plan. **The counterexample is on the plan's design decision, not the build.** Routing: orchestrator decides whether to accept the legacy-shim regression for now (path c — document) or kick back to planner for pointer-sentinel revision.

## 2. Counterexamples

### 2.1 CONFIRMED — Legacy `till.update_project` shim wipes new project fields on every call

See §1.7. Reproduction is direct: the legacy MCP shim's args struct does not carry the 6 new fields, so it forwards empty strings into `UpdateProjectRequest`, which `UpdateDetails` writes through to the DB unconditionally.

The plan explicitly chose value-typed (no pointer-sentinel) project update fields, and the builder followed that choice faithfully. The data-loss surface is a consequence of the design decision, not a builder error.

**Severity:** MEDIUM. Affects only callers using the legacy `till.update_project` tool — agents are guided to use canonical `till.project` per existing tool descriptions. The legacy shim is documented as "alias for till.project operation=update" elsewhere.

**Recommended remediation (orchestrator decides):**

- **Path A (accept):** add a doc-comment warning to the legacy `till.update_project` tool: "deprecated; this alias does NOT plumb Drop 4a L4 first-class fields and will clear them to empty on every update. Use `till.project` instead." Defer real fix to a future drop.
- **Path B (fix now):** thread the 6 fields through the legacy shim's args + adapter call. ~10 LOC change. Preserves the value-typed design but closes the data-loss hole. This is a builder follow-up, not a re-plan.
- **Path C (replan):** pointer-sentinel the 6 update fields (matches 4a.5 precedent). Larger blast radius — touches `UpdateProjectInput`, `UpdateProjectRequest`, `Service.UpdateProject`, `UpdateDetails` signature, all callers.
