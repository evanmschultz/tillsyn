## 2. Counterexamples

### 2.1 CONFIRMED — STEWARD cross-subtree exception is non-functional (attack vector 4)

**Claim under attack.** `Service.ApproveAuthRequest`'s gate allows STEWARD to approve subagent auth requests rooted under a STEWARD persistent-owned ancestor, when the request was created by a different orchestrator (cross-orch path).

**Plan spec verbatim** (`workflow/drop_4a/WAVE_3_PLAN.md:57`):
> walks up to find any ancestor whose `metadata.persistent = true && metadata.owner = STEWARD`; if one exists AND the approver's `AuthRequestPrincipalType == "steward"`, the cross-subtree exception fires.

**Code under attack** (`internal/app/auth_requests.go:447-449`):
```go
if strings.TrimSpace(session.PrincipalType) != "steward" {
    return fmt.Errorf("cross-orch approval requires steward approver ...
}
```

**Counterexample reproduction.**

1. STEWARD session is issued via `autentauth.Service.IssueAuthSession` with `PrincipalType: "steward"`.
2. `IssueSession` (`internal/adapters/auth/autentauth/service.go:194-240`) explicitly collapses the steward axis at autent's boundary:
   ```go
   if principalType == autentdomain.PrincipalType("steward") {
       principalType = autentdomain.PrincipalTypeAgent
   }
   ```
   The autent `principals` table stores `agent`. The "steward" distinction is preserved ONLY in autent's session-metadata blob under key `"auth_request_principal_type"`.
3. The W3.1 gate calls `s.authBackend.ListAuthSessions(ctx, AuthSessionFilter{SessionID: approverSessionID, Limit: 1})`.
4. `ListAuthSessions` → `mapSessionView` (`autentauth/service.go:1392-1416`) constructs `app.AuthSession` with `PrincipalType: strings.TrimSpace(principalType)` where `principalType` is sourced at `:672` as `string(principal.Type)` — i.e., the collapsed `"agent"` from the autent table.
5. The `app.AuthSession` struct (`internal/app/auth_requests.go:54-71`) has only `PrincipalType` — no `AuthRequestPrincipalType` field.
6. The gate's STEWARD check sees `session.PrincipalType == "agent"`, NOT `"steward"`.
7. Every legitimate STEWARD cross-subtree approval is rejected with `cross-orch approval requires steward approver`.

**Pre-existing precedent showing the correct mechanism.** `autentauth/service.go:751-767` `Authorize` recovers the steward distinction via session-metadata read:
```go
authRequestPrincipalType := strings.TrimSpace(validated.Session.Metadata["auth_request_principal_type"])
...
result.Caller = domain.NormalizeAuthenticatedCaller(domain.AuthenticatedCaller{
    ...
    AuthRequestPrincipalType: authRequestPrincipalType,
    ...
})
```
The W3.1 gate uses the wrong code path — it should have used `Authorize`-style validation that preserves `AuthRequestPrincipalType`, OR `mapSessionView` (and `app.AuthSession`) should have been extended with the steward-axis field. Neither change ships in W3.1.

**Pre-existing reference.** `internal/domain/authenticated_caller.go:11-23` documents this exact failure mode:
> `PrincipalType` carries the actor-class axis (user|agent|system) … `AuthRequestPrincipalType` carries the auth-request principal-class axis (user|agent|service|steward) … Distinct from `PrincipalType` (the actor-class axis) so the STEWARD owner-state-lock can key on the "steward" value without collapsing it into "agent". Drop 3 droplet 3.19 added this field …

W3.1 should have re-used the same axis. Instead it reads `session.PrincipalType` directly.

**Test that exposes the bug.** None ships in W3.1 because no positive STEWARD-approve test was written. `TestApproveAuthRequestRejectsCrossOrchWithoutSteward` passes only because the `PrincipalType != "steward"` reject path fires, which is the SAME line that rejects every legitimate STEWARD attempt — false positive on the "feature works".

**Reproduction sketch (informal Go pseudocode, would land in `internal/app/auth_requests_test.go`):**
```go
// Set up STEWARD session with PrincipalType: "steward" via IssueSession.
// Stamp metadata: principal_role=orchestrator, approved_path=project/p1.
// Create persistent STEWARD-owned action_item (Persistent=true, Owner="STEWARD") at branch root.
// Create pending builder request whose path roots under that action_item AND whose RequestedByActor is a DIFFERENT orch (cross-orch).
// Call Service.ApproveAuthRequest with the STEWARD's approver-identity tuple.
// Expected per spec: state=approved, session secret issued.
// Actual: ErrAuthorizationDenied "cross-orch approval requires steward approver" because session.PrincipalType=="agent".
```

**Severity.** Blocks W3.4 case (d) — "STEWARD approves a subagent under one of its persistent parents" — that test cannot pass with the current gate. Also breaks the brief's stated capability ("orch-self-approval rule + STEWARD cross-subtree exception"). The auth-cascade for STEWARD's persistent parents is non-functional.

**Mitigation paths the next builder can take (one of):**

1. Replace the `ListAuthSessions` lookup with a `ValidateAuthSession` (or new helper) that returns `AuthRequestPrincipalType` from session metadata.
2. Extend `app.AuthSession` with `AuthRequestPrincipalType string` and update `mapSessionView` to copy `view.Metadata["auth_request_principal_type"]` into it.
3. Have the gate read the metadata key directly via a new `authBackend` method.

Option (2) parallels the Drop 3 droplet 3.19 precedent for `AuthenticatedCaller` and is the smallest delta.
