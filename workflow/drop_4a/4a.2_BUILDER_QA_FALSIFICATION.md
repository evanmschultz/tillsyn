### 1.1 NIT — Bare `mage` invocation depends on dev's `PATH` at hook-execution time

**Severity:** NIT (operational trade-off, not a builder error).

**Reproduction (hypothetical).**

If a dev's interactive shell `PATH` includes `mage` (e.g. via `~/go/bin` exported in `~/.zshrc`) but git's hook-execution environment does not, `git commit` would emit:

```
.githooks/pre-commit: line 5: mage: command not found
```

The dev sees the raw shell error rather than the spec-mandated `format-check failed — ...` hint. The `set -e` trap fires on the failed lookup; the `if ! mage ...; then ... fi` construct never branches into the hint path because shell `command not found` is exit 127, which `if !` would normally catch — but `set -e` semantics around `command not found` in an `if` condition are POSIX-compliant: failures inside `if` conditions do NOT trigger `set -e`. So the script continues into the `then` branch, the printf fires with the format-check-failed hint, and `exit 1` runs.

Wait — re-tracing: when `mage` is missing, `mage format-check` fails with shell exit 127 + "command not found" stderr. The `if !` inverts to true, the printf runs, `exit 1` runs. So the dev DOES see the `format-check failed — ...` hint, but ALSO sees the upstream `command not found` error from the shell first. The hint is technically misleading (`mage format` in the hint also won't work), but the script does not crash silently.

**Why this is a NIT, not a CONFIRMED counterexample.**

W0.2 acceptance does not require the script to detect `mage` on `PATH` and emit a clearer error. The plan's risk register (`workflow/drop_4a/WAVE_0_PLAN.md` §0.2) does not flag this. The plan's stated philosophy is "scripts must respect the dev's configured environment per `feedback_mage_precommit_ci_parity.md`" — which means trusting the dev's `mage` install. Adding a `command -v mage` pre-check would be a defensive enhancement, but is not spec-required.

**Routing recommendation.**

Surface as a Drop 4a Wave 0 closeout NIT for W0.4's CONTRIBUTING.md docs — the "Local Git Hooks" section (per W0.4 spec lines 109-113) should mention "ensure `mage` is on your `PATH` before running `mage install-hooks`, since git hooks inherit a minimal environment." Non-blocking for 4a.2.

### 2.7 `set -eu` + non-zero exit interaction — REFUTED (attack vector #7)

The script structure:

```sh
set -eu

if ! mage format-check; then
	printf '%s\n' "format-check failed — run 'mage format' to auto-fix, or '--no-verify' to bypass with explicit reason."
	exit 1
fi
```

POSIX `set -e` semantics (verified per `man sh` / Open Group POSIX.1-2017 §2.5.2 errexit): "The -e setting shall be ignored when executing the compound list following the while, until, if, or elif reserved word, a pipeline beginning with the ! reserved word, or any command of an AND-OR list other than the last."

So when `mage format-check` fails with non-zero exit:
1. The `if !` construct inverts the exit code; `set -e` is **disabled** for this command per POSIX rule above.
2. The `if` branch is taken (because `!` of non-zero is "true").
3. `printf` emits the failure-hint to stdout. ← **The dev sees this.**
4. `exit 1` terminates the script with the spec-mandated non-zero code, which git interprets as "abort the commit."

If the builder had used the bare invocation `mage format-check` (no `if !`), `set -e` WOULD trigger early exit on failure, the printf would never run, and the dev would see only `mage`'s own stderr, not the spec-mandated hint. The `if ! ...; then ...; fi` construct is the **only** way to coexist with `set -e` and still emit the hint.

Builder used the correct construct. REFUTED.

### 2.8 Hook fire ordering — REFUTED (attack vector #8)

Pre-commit fires BEFORE the commit lands (git semantic, not script-controlled). When `mage format-check` fails:
1. `if !` branch fires.
2. `printf` writes the hint.
3. `exit 1` exits with code 1.
4. Git sees non-zero exit, **aborts the commit** — the commit never lands, the working tree is untouched, the index is untouched.

The script does NOT have any post-`exit 1` logic that could fire (the `fi` is followed by EOF, no further commands). Even if the script CONTINUED past the `fi` (it doesn't), there is nothing after the `fi` block, so no risk.

Pre-push has identical structure with `mage ci`. Same analysis. REFUTED.

### 2.9 `--no-verify` bypass — REFUTED (attack vector #9)

The hint mentions `--no-verify` as informational guidance only. Git's `--no-verify` flag is honored at the git-level: when `git commit --no-verify` runs, git skips the entire pre-commit hook (the script never executes). No script logic is needed.

Could the script accidentally fail at script-load time even with `--no-verify`? Only if the script has a syntax error that prevents the shebang + interpreter from parsing it. Hand-trace of the 8 lines:

```sh
#!/bin/sh                                                  # shebang, valid
# Activated by 'mage install-hooks' (sets core.hooksPath = .githooks).  # comment
set -eu                                                    # POSIX builtin
                                                           # blank line
if ! mage format-check; then                               # POSIX if-not
	printf '%s\n' "format-check failed — run 'mage format' to auto-fix, or '--no-verify' to bypass with explicit reason."
	exit 1                                             # POSIX builtin
fi                                                         # close if
```

No unmatched quote, no unclosed `if`, no unterminated string, no unmatched `then`/`fi`. Script-load syntax is clean. With `--no-verify`, git skips the script entirely; without `--no-verify`, the script loads, runs, exits cleanly on success or with code 1 on failure. REFUTED.

### 2.11 `mage ci` cold-cache parity — REFUTED (attack vector #11)

Memory `feedback_mage_precommit_ci_parity.md` says: "Local `mage ci` must return the identical verdict as GH Actions `mage ci`. Cold-cache equivalent (`GOMODCACHE=tmp`) before push on tooling/CI changes."

The hook invokes `mage ci` bare — no `GOMODCACHE=tmp` override. Counter-argument analysis:

1. **Memory rule scope.** The rule is for "tooling/CI changes specifically" — i.e. when the dev changes `magefile.go` or `.github/workflows/*` and needs to verify the CI logic itself. The hook's job is "match what GH Actions runs as a smoke gate before push," not "always re-verify tooling correctness."

2. **Forcing `GOMODCACHE=tmp` in the hook would make every push 5-10x slower.** Devs would `--no-verify` constantly, defeating the hook's purpose.

3. **Spec line 73-74:** "**No** Go-toolchain version pinning, `GOPROXY` overrides, or `GOMODCACHE` overrides — scripts must respect the dev's configured environment per `feedback_mage_precommit_ci_parity.md`." Spec EXPLICITLY forbids `GOMODCACHE` overrides in the hook. The memory rule is a dev-discipline rule for tooling changes specifically, not a script-enforcement rule for every push.

4. **Risk register entry §0.2:** plan flagged POSIX portability + `pipefail` only, not cold-cache. The decision to skip cold-cache enforcement is intentional.

The hook's `mage ci` is the dev-discipline smoke gate; cold-cache verification is a separate dev step run before pushing tooling/CI changes. Acceptance is "match the rule," and the spec explicitly aligns. REFUTED.
