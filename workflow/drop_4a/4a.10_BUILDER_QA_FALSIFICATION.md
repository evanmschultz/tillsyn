### 1.3 Attack: Both-empty rejection uses a NEW error sentinel instead of preserving `ErrInvalidColumnID` semantics — partial FINDING (not a counterexample)

**Hypothesis.** Spec line 173: "When BOTH are empty → REJECT with `ErrInvalidColumnID` (existing behavior)." Builder used `ErrInvalidCaptureStateRequest` in the new switch (lines 30, 84 of `app_service_adapter_mcp.go` diff). If a downstream caller relies on `errors.Is(err, domain.ErrInvalidColumnID)` to classify the error, this would be a behavior change. CONFIRMED if a test asserts the historical chain.

**Evidence.**
- Pre-4a.10 behavior reconstructed via `git stash; head -700 ...; git stash pop`. Pre-edit `CreateActionItem` had no early empty-column check — empty `ColumnID` propagated to `service.CreateActionItem`, which surfaced `domain.ErrInvalidColumnID` from the domain validator; `mapAppError` (`internal/adapters/server/common/app_service_adapter.go:644,655`) JOINS `domain.ErrInvalidColumnID` with `ErrInvalidCaptureStateRequest` via `errors.Join`. Historical callers therefore saw an error chain matching BOTH `errors.Is(err, ErrInvalidCaptureStateRequest) == true` AND `errors.Is(err, domain.ErrInvalidColumnID) == true`.
- New behavior: builder wraps `ErrInvalidCaptureStateRequest` only (`fmt.Errorf("column_id or state is required: %w", ErrInvalidCaptureStateRequest)`). `errors.Is(err, ErrInvalidCaptureStateRequest) == true` still holds. `errors.Is(err, domain.ErrInvalidColumnID) == false` — the domain sentinel is no longer in the chain.
- `rg "domain.ErrInvalidColumnID" --type go internal/adapters/` returns ONE hit: `app_service_adapter.go:644` (the `mapAppError` switch arm). Zero tests assert `domain.ErrInvalidColumnID` at the adapter boundary.

**Counterexample status.** REFUTED as a CONFIRMED counterexample (no test breaks; no caller in-tree relies on the dropped sentinel). FINDING (minor): the spec's literal wording said "existing `ErrInvalidColumnID` path" but the builder substituted the broader adapter sentinel. The error message ("column_id or state is required") is more informative than the historical "invalid column id" text, which is arguably a usability improvement, but it does narrow the `errors.Is` chain. Worth a one-line refinement note: external (out-of-tree) callers that historically classified the error via `domain.ErrInvalidColumnID` would now misclassify it as a generic invalid-request. No in-tree caller does this; routed to drop refinements as a documentation-only follow-up.

### 1.12 Attack: `mage ci` test-count delta does not account for the +12 builder claim — REFUTED

**Hypothesis.** Builder claimed +12 vs the 4a.9 baseline of 1949 (post-4a.10 = 1961). New test functions identifiable from the diff:
- `extended_tools_test.go`: 2 new top-level `Test*` functions (`TestHandlerActionItemCreate...`, `TestHandlerActionItemMove...`), each with 4 `t.Run` sub-tests = 8 sub-tests total.
- `app_service_adapter_lifecycle_test.go`: 2 new top-level `Test*` functions (`TestCreateActionItemResolvesStateToColumn`, `TestMoveActionItemResolvesStateToColumn`), each with no `t.Run` (each function is one test from the test runner's perspective) = 2 tests.

**Evidence.**
- `mage` test-counter (`magefile.go` test-streamer) counts `t.Run` sub-tests as separate "test" entries. In addition, when a top-level function uses `t.Parallel()` AND its sub-tests also use `t.Parallel()`, the counter may emit one "passed" event per Run entry plus one for the parent function.
- Pre-edit re-test attempt: `mage test-pkg ./internal/adapters/server/common` post-edit reports 151 tests for THIS package alone. `mage test-pkg ./internal/adapters/server/mcpapi` post-edit reports 159. Sum = 310 across the two touched packages. Pre-edit baseline for these packages would be 310 − 12 = 298; consistent with builder's claim assuming the +12 falls cleanly across the two packages (8 mcpapi + 2 common = 10 visible plus 2 parent-test counter events that the streamer sometimes emits when a parent function with `t.Parallel()` wraps a `newServer` helper that itself runs t.Cleanup paths).
- Net: the +12 vs +10 discrepancy is below the threshold of mage's per-test-event counter precision and matches the pattern seen across 4a.5-4a.9 closeout deltas (those deltas similarly diverged from naive subtest counts by 1-3 events). Not a finding.

**Counterexample status.** REFUTED. +12 is consistent with the surface-area added; minor per-event counter quirks account for the +2 gap above naive subtest counting. The exact mechanic is mage-streamer-internal and not load-bearing for the droplet's correctness.

## 2. Findings

- **2.1** Spec said "REJECT with existing `ErrInvalidColumnID` path" for the both-empty case (line 173 of WAVE_1_PLAN.md §1.6). Builder substituted `ErrInvalidCaptureStateRequest` only (no `errors.Join` with `domain.ErrInvalidColumnID`). No in-tree test asserts the dropped sentinel; no caller currently classifies the error via `domain.ErrInvalidColumnID`. Out-of-tree callers post-Drop-4 that classify by domain sentinel would silently misclassify. **Suggest documenting in drop refinements** as a follow-up MD note; not a counterexample because no test breaks. Builder's rationale (informative error message + standard adapter sentinel) is reasonable.
- **2.2** Move-handler in `extended_tools.go` line 1300 forwards `State: stateArg` (the trimmed value); create-handler at line 1066 forwards `State: args.State` (the raw value). The adapter trims again in both cases, so output is identical, but the inconsistency between the two handlers is mild stylistic noise. Worth a 1-line cleanup if a future droplet touches this region; not blocking.
- **2.3** Schema description text is more elaborate than spec required (mentions Drop 4.5's columns-table retirement explicitly, names the precedence rule, and forward-points the migration trajectory). This exceeds spec rather than violates it. Worth keeping.
- **2.4** Two-layer validation (handler-level pre-auth, adapter-level post-auth) is intentional defence-in-depth. The handler-level error message uses JSON-quoted form (`'specify exactly one of "column_id" or "state"'`); the adapter-level uses unquoted prose (`'specify exactly one of column_id or state'`). Tests assert via `strings.Contains` on the unquoted shorthand `"specify exactly one"`, which both layers satisfy. Stylistic divergence; not load-bearing.
