### 1.1 Attack: `failed` child does not actually block parent close ‚Äî REFUTED

**Hypothesis.** Spec (`WAVE_1_PLAN.md` ¬ß1.7 line 199): "a non-archived child in any state OTHER than `StateComplete` adds an unmet-criterion. Decision: `failed` children also block parent close." A speculative builder, conflating "failed" with a kind of "terminal-success-adjacent" state (since `IsTerminalState` returns true for both `StateComplete` and `StateFailed` per `internal/domain/workitem.go:181-182`), could have written the children-walk gate as `IsTerminalState(child.LifecycleState)` ‚Äî which would treat `failed` children as non-blocking. That would silently violate the spec while still passing every test that uses `StateInProgress` / `StateTodo`. CONFIRMED if the gate predicate exempted `StateFailed`.

**Evidence.**
- `internal/domain/action_item.go:604-616` (Read, 12 lines) ‚Äî full `CompletionCriteriaUnmet` body. The gate predicate at line 611 is `normalizeLifecycleState(child.LifecycleState) != StateComplete` ‚Äî explicit equality against `StateComplete` only, NOT `IsTerminalState`. `StateFailed` is NOT exempted.
- `internal/domain/workitem.go:179-183` ‚Äî `IsTerminalState` returns true for `StateComplete || StateFailed`. The builder did NOT use this predicate; explicit `!= StateComplete` is the chosen gate. Confirms the spec's "failed children block" decision is faithfully implemented.
- `internal/domain/domain_test.go:662-667` ‚Äî the `failed_child_blocks` test case constructs a child with `LifecycleState: StateFailed` and asserts `wantUnmet: true`. The reviewer re-ran `mage test-pkg ./internal/domain` (220 tests, all green) ‚Äî the failed_child_blocks sub-case passed.

**Counterexample status.** REFUTED. The gate is `!= StateComplete`, not `!IsTerminalState`. `StateFailed` correctly blocks parent close.

## 2. Findings

### 2.1 (Observational; not a counterexample) The pre-removal `Policy` JSON tag silently disappears from any newly-written DB row ‚Äî clean break, not a regression

The post-4a.11 `CompletionContract` struct has no `Policy` field. New rows written by the post-4a.11 binary will have `metadata_json.completion_contract` carrying ONLY the five surviving fields (StartCriteria, CompletionCriteria, CompletionChecklist, CompletionEvidence, CompletionNotes). Old rows that pre-existed in stored DB will keep their vestigial `policy` key (per pre-MVP "dev fresh-DBs" rule, this is the dev's responsibility to clean by deleting the DB). New-row JSON shape is the desired clean break. Routed as observational note rather than a counterexample.

### 2.2 (Routed to drop refinements; forward-compat tripwire) Wire-level CompletionContract bag would collide with `DisallowUnknownFields` if any future drop adds it

Today's wire surface does NOT structurally expose `CompletionContract` ‚Äî metadata is pre-flattened at the wire layer. If a future drop (Drop 4b's commit-agent metadata-passthrough wave is the leading candidate) adds a wire-level `completion_contract` JSON object, ANY legacy MCP/HTTP client carrying the now-removed `policy` key would be hard-rejected by `decodeJSONBody`'s `DisallowUnknownFields`. This is correct strict-shape behavior, but it means clients across the version cliff need to drop the `policy` key. Recommend Drop 4b planner annotates this.

**Severity:** documentation/forward-compat. Not a 4a.11 blocker.
