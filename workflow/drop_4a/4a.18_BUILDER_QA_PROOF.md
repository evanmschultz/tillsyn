### 3.2 Eligibility predicate covers four clauses

**PROVEN.** `walker.go:167-200` (`isEligible` helper) implements all four:

- Clause 1 (LifecycleState=todo) — `walker.go:168-170` (`if item.LifecycleState != domain.StateTodo { return false }`).
- Clause 2 (every BlockedBy resolves to complete) — `walker.go:171-182` iterates `item.Metadata.BlockedBy`, looks up each in the by-ID index, returns false on missing reference OR on non-complete blocker.
- Clause 3 (children-complete NOT required, asymmetry) — package doc-comment `walker.go:54-62` documents the asymmetry; `isEligible` does NOT walk children, confirming behavioral compliance.
- Clause 4 (parent in_progress OR Persistent OR root) — `walker.go:183-199`: empty `parentID` returns true (root); `parent.Persistent` returns true; otherwise `parent.LifecycleState == StateInProgress`.

### 3.3 `Promote(ctx, item)` separate method, resolves in_progress column ID, calls `Service.MoveActionItem`, wraps `ErrTransitionBlocked` as `ErrPromotionBlocked`

**PROVEN.**

- Separate method — `walker.go:220` declares `func (w *treeWalker) Promote(ctx context.Context, item domain.ActionItem) (domain.ActionItem, error)` (note: returns `domain.ActionItem` in addition to `error` — strictly stronger than spec; not a finding).

### 3.6 Children-complete asymmetry documented

**PROVEN.** Package doc-comment `walker.go:54-62`:

> Children-complete is NOT required for promotion to in_progress. The children-complete invariant is enforced by Service.MoveActionItem when the destination state is complete (see ensureActionItemCompletionBlockersClear in internal/app/service.go); auto-promotion to in_progress on a parent only requires the parent's own gating, not its descendants.

Plus design-rationale paragraph explaining why (in_progress is a working state; cascade ordering is captured by blocked_by edges).

### 3.7 Broker-after-write read ordering documented in package doc-comment

**PROVEN.** Package doc-comment `walker.go:31-43`:

> The walker is invoked by the broker subscriber loop in broker_sub.go AFTER the repo write that triggered the LiveWaitEventActionItemChanged event. The in-process LiveWaitBroker stamps every Publish into its latest map BEFORE waking waiters, and Service.MoveActionItem / UpdateActionItem call Publish only after the repo write returns. By the time the walker reads the tree, the latest write is durable.

Cross-referenced by `broker_sub.go:11-24` which describes the same monotonic-cursor + coalesce-on-burst contract from the publisher side. Wave 2.10 manual-trigger CLI semantic also documented.
