## Falsification Attack Matrix

### F1. Missing publish call sites in `service.go` (write paths beyond Create/Move/Update)

**Scenario:** Spec narrows publishing to `CreateActionItem` /
`MoveActionItem` / `UpdateActionItem`. Five other Service methods write
to action items and would, under a "dispatcher sees all action-item
state changes" claim, also need to publish.

**Evidence:** `rg -n "func \(s \*Service\)" internal/app/service.go`
plus per-method read of returns. Action-item write paths that DO NOT
call `publishActionItemChanged`:

| Method | Line range | Write call | Publishes? |
|---|---|---|---|
| `CreateActionItem` | 813–964 | `repo.CreateActionItem` (943) | **YES** (962) |
| `MoveActionItem` | 967–1051 | `repo.UpdateActionItem` (1030) | **YES** (1049) |
| `UpdateActionItem` | 1124–1310 | `repo.UpdateActionItem` (1295) | **YES** (1308) |
| `RestoreActionItem` | 1054–1088 | `repo.UpdateActionItem` (1081) | **NO** |
| `RenameActionItem` | 1091–1121 | `repo.UpdateActionItem` (1107) | **NO** |
| `DeleteActionItem(Archive)` | 1313–1383 | `repo.UpdateActionItem` (1333) | **NO** |
| `DeleteActionItem(Hard)` | 1313–1383 | `repo.DeleteActionItem` (1346) | **NO** |
| `ReparentActionItem` | 1591–1640 | `repo.UpdateActionItem` (1636) | **NO** |
| `ImportSnapshot` (snapshot.go) | 276–349 | direct `repo.CreateActionItem` / `repo.UpdateActionItem` (335 / 328) | **NO** |

`RestoreActionItem` is dispatch-relevant: line 1077 explicitly calls
`SetLifecycleState(restoredState, ...)` so a restore CAN move an item
back into `todo` — a state the dispatcher cares about. `Archive` removes
an item from dispatch eligibility (the walker excludes archived items).
A continuous-mode dispatcher running across a Restore or Archive would
NOT wake.

**Verdict:** **ACCEPTED**, narrowly. The droplet's acceptance criteria
explicitly enumerated `MoveActionItem` / `CreateActionItem` /
`UpdateActionItem` and no others. Builder honored the spec
verbatim. Not a counterexample to *this* droplet.

**Refinement raised:** the broader claim "dispatcher sees every
action-item change that affects dispatch decisions" is not delivered by
this droplet. Restore + Archive (Delete-archive) at minimum should
publish before the walker (Wave 2.5) ships — flag for plan-QA on the
walker droplet, or as a Drop 4b refinement. ImportSnapshot is bulk
restoration semantics; the dispatcher will re-walk on its next
state-change event anyway, so it is the lowest-priority gap of the
five.

### F2. Doc-comment drift on `DeadlineExceeded` handling

**Scenario:** `broker_sub.go` lines 36–40 doc-comment claims:

> context.DeadlineExceeded while ctx.Err() is still nil indicates an
> internal Wait timeout (the service layer wraps Wait with WithTimeout
> for list surfaces) and is treated as a benign re-subscribe trigger.

But the code at lines 76–78:

```go
if errors.Is(err, context.Canceled) || errors.Is(err, context.DeadlineExceeded) {
    return
}
```

treats `DeadlineExceeded` as a TERMINATION trigger, not a re-subscribe
trigger. The doc-comment lies about the code's behavior.

**Evidence:** Direct read of broker_sub.go lines 36–40 (worklog wording)
vs lines 76–78 (actual code).

**Verdict:** **CONFIRMED-NIT (doc-only)**. The actual production
behavior is fine: `subscribeBroker` is called with the dispatcher's
parent context, never with a `WithTimeout` wrapper at the dispatcher
layer; if a caller intentionally passes a deadline-bearing ctx, the
deadline IS the termination signal and returning is correct. So no
runtime bug, but the doc-comment misleads future readers.

**Suggested fix:** rewrite the paragraph as:

> Both `context.Canceled` and `context.DeadlineExceeded` are treated as
> termination signals and exit the loop. `subscribeBroker` does not
> wrap Wait in its own timeout; if a caller passes a deadline-bearing
> ctx, the deadline IS the intended subscription lifetime.

### F3. Burst-coalescing under rapid publishes

**Scenario:** Three publishes A (Seq=1), B (Seq=2), C (Seq=3) land in
quick succession while the subscriber goroutine is between Wait
returning event A and re-registering for the next Wait. Trace:

1. Subscriber registered, cursor=0.
2. Publish A: nextSequence=1, latest={Seq=1}, take() removes waiter,
   sends event{Seq=1} to ch.
3. Subscriber receives event{Seq=1}, advances cursor=1.
4. Publish B: latest={Seq=2}, no waiters → no channel send.
5. Publish C: latest={Seq=3}, no waiters → no channel send.
6. Subscriber sends event{Seq=1} to `out`.
7. Subscriber re-registers with afterSequence=1: register sees
   latest={Seq=3}, 3 > 1, returns event{Seq=3} immediately.
8. Subscriber sends event{Seq=3} to `out`. **Event{Seq=2} is never
   delivered.**

**Evidence:** `internal/app/live_wait.go` lines 109–130 (Publish stamps
latest and wakes waiters via non-blocking send) plus lines 137–148
(register checks latest first, returns immediately on match).

**Verdict:** **MITIGATED-with-doc-nit**. The dispatcher's *purpose* is
wake-up not per-event delivery — Wave 2.5's tree walker re-reads the
project tree on every wake, so a coalesced burst still triggers exactly
one walk that picks up all changed items. The dispatch contract holds.

But:

- The worklog claim "every Sequence observed corresponds to exactly
  one repo-write committed by the service layer" is true *in the
  observed direction* (every observed event is a real write) but its
  obverse ("every repo-write produces an observed event") is FALSE
  under burst conditions.
- `broker_sub.go` lines 19–22 inherits the same wording. Similarly
  imprecise.
- `TestSubscribeBrokerForwardsMultipleEventsInOrder` masks the
  coalescing path: 5ms between publishes is enough for the subscriber
  to re-register, so the test observes 3-of-3. A back-to-back-publish
  variant would observe 2-of-3 (events 1 and 3) and would invalidate
  a strict per-write delivery claim.

**Suggested fix:** rewrite both doc passages as:

> Each wake corresponds to *at least one* repo write since the previous
> wake. Under burst conditions the broker coalesces consecutive
> publishes into the latest event observed; the walker (Wave 2.5)
> re-reads the project tree on each wake so coalescing is invisible to
> the dispatch contract.

Optionally, add a back-to-back burst test to pin the coalescing
behavior explicitly.

### F4. Goroutine leak via no-cancel callers

**Scenario:** Caller passes `context.Background()` and forgets to
cancel. Goroutine sits in Wait forever.

**Evidence:** `broker_sub.go` lines 24–27 documents the contract
("callers MUST cancel ctx"). No timeout fallback.

**Verdict:** **ACCEPTED**. Documented contract; symmetric with all
other goroutine-spawning APIs in the codebase. The dispatcher's actual
caller (Drop 4b's `Start`) will own the parent context and cancel it on
`Stop`.

### F5. Race on Latest seed → Wait register window

**Scenario:** Goroutine calls `Latest` at line 63, gets Sequence=N. A
publish lands BEFORE line 71 (Wait registration), bumping latest to
Sequence=N+1. The subscriber then calls Wait with afterSequence=N.

**Evidence:** `internal/app/live_wait.go` lines 137–139:

```go
if event, ok := b.latest[key]; ok && event.Sequence > afterSequence {
    return 0, nil, event, true
}
```

Wait's register checks `latest[key]` BEFORE adding the waiter and
returns immediately if latest is newer than afterSequence.

**Verdict:** **MITIGATED**. The broker's register has a built-in
double-check that closes the seed-vs-register race window. Subscriber
sees event N+1 immediately on the next Wait call.

### F6. Channel-close race on context cancellation mid-send

**Scenario:** Subscriber blocks on `out <- event` (line 91) because
consumer is slow; ctx cancels concurrently.

**Evidence:** `broker_sub.go` lines 88–92:

```go
select {
case <-ctx.Done():
    return
case out <- event:
}
```

**Verdict:** **MITIGATED**. The select picks ctx.Done, goroutine
returns, `defer close(out)` fires. The abandoned-`out <- event` case
never completed (select chose ctx.Done), so no panic. Buffered events
already in `out` remain receivable until the consumer drains them; ok=false
appears only after the buffer empties.

### F7. nil-receiver / nil-broker panic

**Scenario:** `subscribeBroker` called on `var d *dispatcher` (nil) or
on a dispatcher constructed with nil broker.

**Evidence:** `broker_sub.go` line 47: `if d == nil || d.broker == nil
|| projectID == ""` — short-circuit eval prevents nil dereference.
`TestSubscribeBrokerNilDispatcherReturnsClosedChannel` and
`TestSubscribeBrokerEmptyProjectIDReturnsClosedChannel` pin both
short-circuits.

**Verdict:** **MITIGATED**.

### F8. Re-subscribe loop spinning on Sequence=0 forever after broker close

**Scenario:** The plan-QA-falsification round-1 attack the worklog
explicitly addresses. A misbehaving broker returns `(zeroEvent, nil)`
from Wait, the loop never advances cursor, spins forever.

**Evidence:** Two layers of mitigation:

1. `internal/app/live_wait.go` line 113: `if event.Sequence <= 0 { event.Sequence = b.nextSequence(...) }` — Publish in the
   in-process broker FORCES a positive Sequence, so a zero-Sequence
   event cannot land in `latest` and cannot be returned to a waiter.
2. The in-process broker has NO `Close` method (verified by
   `rg "func.*Close" internal/app/live_wait.go` returning empty).
   The broker is process-lifetime; ctx cancellation is the only
   termination signal.

**Verdict:** **MITIGATED**. The hypothetical (zeroEvent, nil) return
cannot occur in the in-tree broker. If a future broker implementation
violated this, the cursor guard at `broker_sub.go` line 85 plus the
non-context error fallthrough at lines 79–81 would prevent infinite
spin (loop exits on any non-context error).

### F9. Burst-coalescing test gap (test rigor attack)

**Scenario:** `TestSubscribeBrokerForwardsMultipleEventsInOrder` claims
to verify that "successive publishes are each observed exactly once"
but spaces publishes 5ms apart. The 5ms gap is large enough for the
subscriber to receive, advance cursor, deliver, and re-register before
the next publish — so the test never exercises the burst-coalesce
path.

**Evidence:** `broker_sub_test.go` lines 167–177 (the goroutine sleeps
5ms between each publish).

**Verdict:** **MITIGATED-with-test-nit**. The test passes for the
correct reason (cursor advancement is monotonic and each event in
isolation does deliver), but the wording in the test doc-comment
(lines 146–148: "successive publishes are each observed exactly once")
is broader than what the test actually pins. Same fix as F3:
clarify that bursts may coalesce.

### F10. publish-on-error paths (publishing on failure)

**Scenario:** `publishActionItemChanged` fires after a partial failure
(e.g., embedding enqueue fails AFTER repo write succeeds), waking the
dispatcher when the action item is in an inconsistent state.

**Evidence:** `service.go` lines 943–963 (CreateActionItem),
1030–1049 (MoveActionItem), 1295–1308 (UpdateActionItem). In all
three, repo write is FIRST; subsequent operations
(`enqueueActionItemEmbedding`, `seedDropFindingsAndGate`,
`raiseRefinementsGateForgottenAttention`,
`enqueueThreadContextEmbedding`) early-return their error WITHOUT
calling publish. Publish only fires on the full-success path.

**Verdict:** **MITIGATED**. Builder's "publish only on full success"
placement convention (worklog §5) is correctly applied. If `embed
enqueue` fails, the repo-written-but-not-embedded state is also not
published; the dispatcher misses the wake but the item also has not
been fully promoted, which is consistent with "treat partial-failure
items as unfinished writes."

Symmetric edge: same pattern as `publishCommentChanged` at
`service.go:1477–1478`, established by Drop 4a Wave 1. Convention-
preserving.

### F11. Concurrent subscribers for same project

**Scenario:** Two subscribers for `proj-1` register simultaneously.
Publish wakes both. Each goroutine must independently advance cursor
without interfering with the other.

**Evidence:** `internal/app/live_wait.go` lines 181–196 (`take` removes
ALL waiters atomically and returns the slice; each waiter gets its
own send via the slice iteration). Each subscriber goroutine maintains
its own local `cursor` variable.

**Verdict:** **MITIGATED**. No shared mutable state between
subscribers other than the broker's own mutex-protected maps.

### F12. cursor never advances if event.Sequence ≤ cursor (defensive guard at line 85)

**Scenario:** Wait returns an event with Sequence ≤ current cursor;
the guard at line 85 skips the cursor advance, next Wait reuses same
cursor, loop spins on the same event.

**Evidence:** Broker `register` (live_wait.go line 137) only returns
the event-resolved-immediately path when `event.Sequence > afterSequence`
(strict greater-than). Channel-delivered events have Sequence assigned
by `nextSequence` (always > previous latest). So `event.Sequence ≤
cursor` cannot occur in the in-tree broker. The line-85 guard is
defensive against a hypothetical future broker.

**Verdict:** **ACCEPTED**. Defensive code; not exercisable today.

### F13. Test 100ms bounds reliability under CI load

**Scenario:** `TestDispatcherSubscribesToActionItemChanges` and
`TestDispatcherStopsOnContextCancel` use 100ms timeouts. On a slow CI
runner (esp. with `-race` overhead and shared Mac runners), 100ms
might be tight enough to flake.

**Evidence:** Worklog reports baseline + post-edit `mage ci` green.
Local `mage test-pkg ./internal/app/dispatcher` (with `-race` per
mage default) passes in <1 second wall.

**Verdict:** **ACCEPTED**. No flakes observed; 100ms is the spec-
required bound from WAVE_2_PLAN.md §2.2 acceptance criteria. If CI
flakes appear, raise the bound or add retry — both are post-hoc
refinements, not pre-merge blockers.

### F14. ImportSnapshot bypasses publish (out-of-scope but worth flagging)

**Scenario:** `ImportSnapshot` calls `s.repo.UpdateActionItem` and
`s.repo.CreateActionItem` directly (snapshot.go lines 328, 335) for
each action item in the snapshot, bypassing the Service-layer
publishers entirely.

**Evidence:** Direct read of `internal/app/snapshot.go` lines 325–338.

**Verdict:** **ACCEPTED-with-refinement**. ImportSnapshot is bulk
restoration semantics typically used at startup, DR, or migration —
contexts where a dispatcher subscriber would not be running yet (or
where re-walking on the next state change would naturally pick up the
imported state). Not in droplet scope, but worth flagging as a
walker-droplet (Wave 2.5) plan-QA refinement: if continuous-mode
dispatcher (Drop 4b) runs across an ImportSnapshot, the walker should
either re-walk on a startup signal or `ImportSnapshot` should publish
once after each project's items are landed.

### F15. Race on shared `dispatcher` fields

**Scenario:** Multiple goroutines read `d.broker` concurrently with
some other goroutine writing to it.

**Evidence:** `dispatcher` struct fields are set in `NewDispatcher`
(constructor) and never re-assigned. Tests construct via `&dispatcher{...}`
literal, also one-shot.

**Verdict:** **ACCEPTED**. No assignment after construction; reads
are race-free.
