# Droplet 4a.26 — QA Falsification Review (Round 1)

Audit-trail end-to-end plumbing of `ApprovingPrincipalID` /
`ApprovingAgentInstanceID` / `ApprovingLeaseToken` from the
orch-self-approval cascade path through domain → SQLite → MCP record.

Active counterexample search across migration safety, bind drift,
scanner alignment, JSON omitempty, defense-in-depth assertion shape,
4a.24/4a.25 collision, and existing-call-site impact.

## Round 1 — Attack Matrix

### 1. ALTER TABLE migration on freshly-created DB AND existing DB

- **Hypothesis:** the new columns might miss the `CREATE TABLE IF NOT
  EXISTS` block, causing them to be added only on first ALTER but
  failing on a brand-new DB if any schema-validation step ran before
  the ALTER block.
- **Evidence:** `internal/adapters/auth/autentauth/service.go:889-922` —
  `CREATE TABLE` block does NOT list the new columns. `:927-934` —
  ALTER block adds them via `ADD COLUMN ... TEXT NOT NULL DEFAULT ''`.
  `:935-939` — loop runs each ALTER, swallowing duplicate-column
  errors via `isDuplicateColumnErr`.
- **Trace:** Fresh DB → `CREATE TABLE` succeeds without the columns →
  ALTER block adds all three → DB is in target shape. Existing DB →
  `CREATE TABLE IF NOT EXISTS` is no-op → ALTER block runs; if columns
  already exist, `isDuplicateColumnErr` skips silently. Pre-existing
  precedent at `:928-930` (`principal_role`, `approved_path`,
  `approved_session_ttl_seconds`) uses the same pattern and has been
  in production through several drops.
- **Verdict:** **MITIGATED.** The post-CREATE ALTER pattern works for
  both shapes. Test
  `TestApproveAuthRequestDevTUIPathLeavesAuditEmpty`
  (`service_test.go:1564-1575`) exercises the re-run case explicitly.

### 2. `isDuplicateColumnErr` SQLite-driver portability

- **Hypothesis:** the substring match `"duplicate column name"` might
  be wording the modernc.org/sqlite driver doesn't emit — could be
  driver-specific or version-specific.
- **Evidence:** `service.go:943-949` — case-insensitive substring
  match. The error string is the upstream SQLite C-source canonical
  wording (`sqlite/src/build.c`); modernc.org/sqlite is a faithful
  port. The pre-existing precedent (3 prior columns) already
  exercised the same pattern.
- **Verdict:** **MITIGATED-ON-PRECEDENT.** No new portability surface
  introduced; identical to the W3.1-precursor migration shape that
  is in CI green today.

### 3. INSERT vs UPDATE bind drift / off-by-one

- **Hypothesis:** the new columns might land in INSERT but not UPDATE
  (or vice-versa), or bind-arg count might mismatch column count.
- **Evidence:**
  - INSERT (`service.go:323-335`): 34 columns listed, 34 `?`
    placeholders, 34 bind args (counted manually). Audit fields are
    columns 32/33/34 → bind args
    `req.ApprovingPrincipalID/AgentInstanceID/LeaseToken` at
    `:367-369`.
  - UPDATE (`service.go:1000-1046`): `SET` clause lists all 33 mutable
    columns + the WHERE `id = ?` placeholder = 34 placeholders. Audit
    field SET clause at `:1009`; bind args at `:1042-1044` immediately
    before `req.ID` at `:1045`.
  - SELECT in `ListAuthRequests` (`:399-407`) lists all 34 columns
    matching INSERT order.
- **Verdict:** **MITIGATED.** Column / placeholder / bind-arg counts
  align across all three statements. The
  `TestApproveAuthRequestPersistsApprovingOrchAudit` round-trip
  (`service_test.go:1466-1503`) catches drift end-to-end —
  approve → INSERT → UPDATE (via `Approve` transition) → SELECT (via
  `GetAuthRequest`) all run.

### 4. Scanner column-order vs SELECT column-order drift

- **Hypothesis:** `scanAuthRequest` `Scan(...)` arg order might not
  match `SELECT` column order, leading to silent value-into-wrong-
  field assignment.
- **Evidence:** `service.go:399-407` (SELECT column list) vs
  `service.go:1068-1102` (Scan arg list). Both end with the same
  trailing trio: `approving_principal_id`,
  `approving_agent_instance_id`, `approving_lease_token` →
  `&req.ApprovingPrincipalID`, `&req.ApprovingAgentInstanceID`,
  `&req.ApprovingLeaseToken`. All preceding 31 columns/scans match
  positionally and were unchanged from the W3.1-precursor baseline.
- **Verdict:** **MITIGATED.** Round-trip test
  (`TestApproveAuthRequestPersistsApprovingOrchAudit`) would fail
  loudly on any positional drift — the assertion checks for
  `"orch-approving"` / `"instance-7"` / `"lease-token-xyz"` exact
  values.

### 5. JSON `omitempty` round-trip preservation

- **Hypothesis:** `omitempty` for `string` elides only the empty
  string, but a marshaller with a populated value followed by
  unmarshal-into-empty might round-trip as nil — worth checking.
- **Evidence:** Go's `encoding/json` `omitempty` for `string` elides
  if-and-only-if `value == ""`. A populated `"X"` always emits as
  `"approving_principal_id":"X"`. Counter-case test
  `TestAuthRequestRecordSurfacesApprovingOrchIdentity`
  (`app_service_adapter_auth_requests_test.go:1346-1358`) verifies
  the populated branch emits all three keys with their values.
  `:1394-1404` verifies the dev-TUI counter-case emits NONE of the
  three keys (full omitempty elision).
- **Verdict:** **MITIGATED.** Both branches asserted. No
  nil-vs-empty-string ambiguity at the SQL layer either — the schema
  defaults are `NOT NULL DEFAULT ''`, so the column is always a Go
  empty string, never `NULL`, so the scanner doesn't need
  `sql.NullString` (which would have been a separate landmine).

### 6. Defense-in-depth assertion shape — two-of-three empty?

- **Hypothesis:** the test table covers all-three-empty,
  missing-instance-only, missing-lease-only, all-populated, and
  user-empty. It does NOT exercise missing-principal-only — could the
  assertion's logical shape be wrong?
- **Evidence:** `auth_request.go:608-612`:
  `if approvingPrincipalID == "" || approvingAgentInstanceID == ""
  || approvingLeaseToken == ""`. The OR-shape is symmetric across
  all three fields, so missing-principal-only would also reject. The
  test's coverage gap is a TEST RIGOR nit (not a code bug): a fourth
  rejection case would tighten the table to "all three single-missing
  cases asserted" but the OR-symmetry guarantees behavior.
- **Verdict:** **PASS-WITH-NIT.** Code is correct; the test could add
  the missing-principal-only case for symmetric coverage. Non-
  blocking.

### 7. Validation order — assertion reachable?

- **Hypothesis:** earlier validation steps in `Approve` might short-
  circuit before the audit assertion runs, making the assertion
  unreachable in some configurations.
- **Evidence:** `auth_request.go:583-612` validation order:
  1. Nil receiver → `ErrInvalidID`.
  2. `ensurePending` → state check.
  3. `sessionID == "" || sessionSecret == ""` → `ErrInvalidID`.
  4. `!isValidActorType(resolvedByType)` → `ErrInvalidActorType`.
  5. **Audit assertion fires HERE** for `ActorTypeAgent` only.
  Test fixture at `:852-863` passes `"sess-1"` / `"secret-1"` (non-
  empty) so step (3) passes, step (4) passes (`ActorTypeAgent` is
  valid), and step (5) is the rejection point — failure mode
  isolated.
- **Verdict:** **MITIGATED.** Assertion is reachable and isolated in
  the test.

### 8. 4a.24 plumbing assumption — does 4a.26 re-plumb or assume?

- **Hypothesis:** if 4a.24's `ApproveAuthRequestGatewayInput` shape
  drifted, 4a.26 might be passing wrong fields or duplicating
  plumbing.
- **Evidence:** `internal/app/auth_requests.go:210-221` —
  `ApproveAuthRequestGatewayInput` carries the four approver fields
  unchanged from 4a.24. `service.go:508-510` reads three of them into
  `req.Approve(...)`; `ApproverSessionID` is intentionally NOT
  persisted (gate uses it; audit row doesn't need it).
- **Verdict:** **MITIGATED.** 4a.26 consumes 4a.24's plumbed input
  shape; no duplicated plumbing. Spec line W3.3:114 confirms the
  intent.

### 9. Dev-TUI vs orch-self-approval discriminator path

- **Hypothesis:** the adapter must somehow distinguish dev-TUI
  approvals (no audit fields) from orch self-approvals (must have
  audit fields). If the discriminator path is wrong, audit fields
  get stored on dev-TUI rows or not stored on orch rows.
- **Evidence:** Two approve entrypoints exist:
  - **Dev-TUI**: `app.Service.ApproveAuthRequest` called directly
    with `ResolvedType=user` and four approver fields empty
    (`ApproveAuthRequestInput.Approver*` zero values). The
    "all-or-none" check at `auth_requests.go:373-377` passes
    (none-shape). `ApprovingPrincipalID/AgentInstanceID/LeaseToken`
    forwarded as empty strings to the gateway.
  - **Orch self-approval**: `AppServiceAdapter.ApproveAuthRequest`
    (`app_service_adapter_mcp.go:241+`) sets
    `ResolvedType: domain.ActorTypeAgent` (line 294) and forwards
    `actingSession.PrincipalID` + `agentInstanceID` + `leaseToken` +
    `actingSession.SessionID` (lines 296-299) to the app layer.
  - The `Approve` domain method's audit assertion gates on
    `resolvedByType == ActorTypeAgent` — exactly the discriminator.
- **Verdict:** **MITIGATED.** Discriminator is `ResolvedType`; both
  paths are covered by tests
  (`TestApproveAuthRequestPersistsApprovingOrchAudit` for agent,
  `TestApproveAuthRequestDevTUIPathLeavesAuditEmpty` for user, and
  `TestAuthRequestRecordSurfacesApprovingOrchIdentity` exercises both
  branches via the adapter).

### 12. `ensureAuthRequestSchema` concurrent-invocation race

- **Hypothesis:** if two `NewSharedDB` constructors race on the same
  DB file, two ALTER blocks might fight over the new columns.
- **Evidence:** `service.go:123` — `ensureAuthRequestSchema(cfg.DB)`
  runs once during the constructor. SQLite ALTER TABLE serializes
  through the SQLite write lock at the storage engine level; modernc
  `database/sql` connection pool further serializes via the
  `*sql.DB` mutex. Even concurrent constructors interleave at the
  ALTER granularity — a duplicate-column error is converted to no-op
  by `isDuplicateColumnErr`.
- **Verdict:** **ACCEPTED.** Tillsyn ships single-process; no
  concurrent constructor invocation in production. Even if a future
  consumer constructed two services on the same DB, the SQLite
  write-lock + duplicate-column-skip pattern handles it.

### 13. Audit columns visible via `till.auth_request(operation=get|list)`

- **Hypothesis:** the MCP layer might serialize `AuthRequestRecord`
  through some path that drops the new fields (e.g., a DTO mapper
  that wasn't updated).
- **Evidence:** `mapAuthRequestRecord` (`app_service_adapter_mcp.go:
  1653-1655`) plumbs all three fields. `AuthRequestRecord` is the
  exclusive transport-facing record returned by `GetAuthRequest`,
  `ListAuthRequests`, `CreateAuthRequest`, `ClaimAuthRequest`,
  `ApproveAuthRequest`, `CancelAuthRequest`. No alternative DTO
  mapper exists for auth-request rows.
- **Verdict:** **MITIGATED.** Single mapper, all three fields
  plumbed.

### 14. `cloneAuthRequest` value-copy correctness

- **Hypothesis:** if the new fields were nested types
  (slices/maps/pointers), `cloneAuthRequest` would need explicit
  deep-copy. Strings are value types, but worth confirming the clone
  doesn't drop them.
- **Evidence:** `service.go:1383-1388`:
  ```go
  func cloneAuthRequest(req domain.AuthRequest) domain.AuthRequest {
      req.PhaseIDs = append([]string(nil), req.PhaseIDs...)
      req.Continuation = cloneJSONMap(req.Continuation)
      return req
  }
  ```
  Go pass-by-value copies the entire struct including all string
  fields; explicit deep-copy is only needed for the slice + map.
  String fields ride along correctly.
- **Verdict:** **MITIGATED.** No clone shape change required.

### 15. Round-trip via `GetAuthRequest` after re-`ensureAuthRequestSchema`

- **Hypothesis:** running `ensureAuthRequestSchema` a second time
  might leave the DB in a state where `GetAuthRequest` fails to
  scan.
- **Evidence:** `service_test.go:1564-1593` —
  `TestApproveAuthRequestDevTUIPathLeavesAuditEmpty` re-runs
  `ensureAuthRequestSchema(repo.DB())` and then re-issues
  `GetAuthRequest(created.ID)`, asserting both `state == approved`
  and all three audit columns are empty. The test passes per worklog.
- **Verdict:** **MITIGATED.** Re-run is a no-op for the schema and
  doesn't perturb the row state.
