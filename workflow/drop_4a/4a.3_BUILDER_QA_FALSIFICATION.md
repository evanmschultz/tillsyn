### 2.5 Pre-flight ordering / accumulate-vs-fail-fast — REFUTED (attack vector #5)

Source lines 238-242:

```go
for _, path := range []string{".githooks/pre-commit", ".githooks/pre-push"} {
    if _, err := os.Stat(path); err != nil {
        return fmt.Errorf("install-hooks: %s missing — run from the worktree root and ensure .githooks/ files are tracked", path)
    }
}
```

The loop iterates pre-commit first, then pre-push. On the first missing file, it returns immediately — fail-fast semantics, not accumulate.

**Counterexample attempt: should it accumulate both errors instead?**

Constructed counterexample: rename both `.githooks/pre-commit` AND `.githooks/pre-push` simultaneously, run `mage install-hooks`. Expected (under accumulate semantics): "install-hooks: .githooks/pre-commit, .githooks/pre-push missing — ...". Actual: only `pre-commit missing` would surface; the dev re-runs after restoring pre-commit, then sees `pre-push missing`, has to fix again. Two cycles instead of one.

**Why fail-fast is correct here.**

Spec line 89 mandates the exact format string `"install-hooks: %s missing — ..."` with single `%s`, not a join of multiple paths. An accumulate-and-join implementation would have to use a different format string ("missing files: %s" or similar), which would violate the spec's exact-string requirement. Spec line 96 verification step says: "With `.githooks/pre-commit` deleted in a temp branch, `mage install-hooks` fails with the pre-flight error message before touching `git config`." — singular file, singular error. The plan deliberately chose fail-fast at the format-string level.

Additionally: in practice, both files are added in the same W0.2 commit; if one is missing, the other will also be missing. The "two-cycle" scenario above requires intentional deletion of one but not the other, which is not a plausible dev workflow.

REFUTED — fail-fast matches spec; accumulate would force a spec violation.

### 2.9 Working-directory dependency — REFUTED (attack vector #9)

`git config core.hooksPath .githooks` writes a literal `.githooks` string to `.git/config`. Git resolves `core.hooksPath` against the **repo root** at hook-execution time, not against the cwd of the `git config` command — so the relative-path value `.githooks` always points at `<worktree>/.githooks/` regardless of where the dev was when they ran `mage install-hooks`.

Live test: attempted `cd /Users/evanschultz/.../main/internal && mage install-hooks` — mage rejected with `No .go files marked with the mage build tag in this directory.` (exit 1). Mage discovers magefiles in the cwd; since the magefile only lives at worktree root, running mage from any subdir fails at the magefile-discovery layer before the `os.Stat` pre-flight or `git config` step ever runs. The "from a subdir" attack is therefore prevented by mage itself; the relative-path concern is moot.

For completeness: even if mage permitted invocation from a subdir, the pre-flight `os.Stat(".githooks/pre-commit")` would be evaluated against the process cwd (Go's default), not the repo root — so `os.Stat` would fail in the subdir case, and the spec-mandated error string ("run from the worktree root and ensure .githooks/ files are tracked") would fire. The error message itself instructs the dev to fix this. REFUTED.

### 2.10 Pre-flight `os.Stat` mode-bit gap (chmod regression) — REFUTED (attack vector #10)

Source line 239: `if _, err := os.Stat(path); err != nil { return ... }` — only checks for existence (file-not-found surfaces as `err != nil`). It does NOT verify the executable bit (`info.Mode() & 0o111 != 0`).

**Counterexample attempt: chmod `.githooks/pre-commit` to `0644` (lose +x), run `mage install-hooks`.** `os.Stat` succeeds (file exists), no err returned, pre-flight passes, `git config` runs, `core.hooksPath = .githooks` set. But on next `git commit`, the hook file is non-executable; git either silently skips the hook (older git) or errors out (newer git with `core.hooksPath` strict enforcement). The dev's "format-check" gate is silently disabled.

**Why this is a REFUTED attack, not a CONFIRMED counterexample.**

Spec line 89 mandates exactly: "before the `git config` call, `os.Stat(".githooks/pre-commit")` and `os.Stat(".githooks/pre-push")`. If either is missing, return..." — the spec only requires existence check, not mode check. Adding `info.Mode() & 0o111 != 0` would be over-spec.

W0.2's acceptance line 71 + verification line 76 require `git ls-files -s` to show mode `100755` for both files at commit time — meaning the executable bit is enforced at the W0.2 layer (committed mode), not at W0.3's runtime layer. Once committed at `100755`, the only way to lose +x post-checkout is for the dev to manually `chmod -x` (intentional override, dev-discipline domain) or for a clone on a filesystem that strips exec bits (Windows VFAT, certain network shares). Neither is W0.3's responsibility.

Additionally, `core.fileMode = false` clones (Windows defaults) handle this at the git layer — git tracks the bit via the index, not the filesystem mode, so `git ls-files -s` shows `100755` regardless of working-tree mode. The W0.3 check is for "the file exists in the working tree"; the W0.2 check is for "the file is committed with +x." Both layers do their job.

YAGNI: a mode check in W0.3 would be defensive coding for an out-of-scope failure mode (deliberate dev override or cross-platform file-mode stripping). Spec mandates existence; YAGNI accepts. REFUTED at acceptance bar; mode check is over-spec.
