## Key Design Choices

### 1. Two-map representation (path → holder + holder → pathset)

The spec calls for `sync.Mutex` + `map[string]string` (path → holder). The
implementation extends this with one auxiliary map (`itemPaths map[string]map[string]struct{}`)
indexed by holder so `Release(actionItemID)` is O(holder-set), not O(all
paths). Without the index, a Release of one item would have to walk every
held path in the manager — Wave 2.7's conflict detector calls Release every
time a spawn finishes, so the linear cost would be visible at scale.

Both maps are kept in sync inside `Acquire` and `Release` under the same
mutex. The doc comment names `holders` as the single source of truth and
`itemPaths` as a derived index.

### 2. Partial-acquire semantics

Spec quote: "returns the subset of paths now held + a map of conflicting
paths → holding action-item ID."

`Acquire(item, [a, b])` against an existing holder of `b` returns
`acquired=[a]`, `conflicts={b: <holder>}`. The contended path is reported
without rolling back the un-contended one. This matches the spec's
`TestFileLockAcquirePartialConflictReturnsConflicts` scenario.

A retry by the same item against `[a, c]` then idempotently re-acquires `a`
(see same-holder rule below) and adds `c`. Pinned by an extra assertion at
the bottom of the partial-conflict test.

### 3. Same-holder idempotent re-acquire

If `actionItemID` already holds path `p`, calling `Acquire(actionItemID, [p])`
again is a success: `p` appears in `acquired`, NOT in `conflicts`. Reasoning:

- The lock is held by the requester. Reporting it as a conflict would force
  the caller to track its own held paths to avoid a false-positive on retry.
- A spawn whose process is briefly partitioned and then re-attempts spawn
  should not double-fail itself on an already-owned lock.

Documented in the `Acquire` doc comment under "Same-holder semantics" and
exercised by `TestFileLockEmptyInputsAreNoOps`.

### 4. Path opacity (matches spec)

Spec falsification attack quote: *"what if `paths` contains relative paths
and one item normalizes differently than another?"* Mitigation:

- `./a` and `a` are distinct map keys. The manager does not call
  `filepath.Clean` or any normalization helper.
- The doc comment names the conflict detector (Wave 2.7) and tree walker
  (Wave 2.5) as the normalization owners.
- `TestFileLockPathsAreOpaque` asserts the contract directly.

### 5. Deadlock-free flat API (matches spec)

Spec falsification attack quote: *"deadlock if Acquire is called from inside
a Release callback."* Mitigation: there are no callbacks. `Acquire` and
`Release` are flat synchronous calls. Documented in the struct doc comment
under "Deadlock-free contract."

### 6. Conflict-map allocation per call

Each `Acquire` call returns a freshly-allocated `conflicts map`. Callers may
mutate it freely; the manager retains no reference. This is the simpler
half of the alias-prevention rule (the alternative — sharing a manager-side
map — would require `defer copy(out, internal)` plumbing for every call).

### 7. Zero-value validity

Per the existing dispatcher style (see `broker_sub.go`'s `if d == nil ||
d.broker == nil` defensive check), a zero-value `fileLockManager{}` is
documented as usable. Both `Acquire` and `Release` lazy-init the inner maps
under the mutex. `TestFileLockZeroValueIsUsable` covers both code paths.

### 8. Constructor-but-not-wired

`newFileLockManager()` exists but no exported wiring is added to
`dispatcher.go` — that's 4a.23's job per the hard constraint. The
constructor pre-allocates inner maps so the first `Acquire` skips the
lazy-init branch in the production path; the lazy branch only fires for the
zero-value test.

### 9. `err` always nil today

`Acquire` returns `(acquired, conflicts, err)`. The `err` slot is wired but
always nil in this droplet. Keeping it in the signature lets Drop 4b add a
SQLite-mirror without a breaking API change. Documented inline.

### 10. Removed defensive guard in `Release`

An earlier draft had `if m.holders[path] == actionItemID { delete(...) }`
inside the Release loop. The current code cannot produce divergence between
`holders` and `itemPaths`, so the false branch was dead from a coverage
standpoint. Removed for YAGNI cleanliness. If a future refactor needs the
guard, it should arrive with a test that exercises the false branch.

## Test Summary

Eight tests added, all in `internal/app/dispatcher/locks_file_test.go`:

| Test                                                          | Spec? | Notes                                                                |
| ------------------------------------------------------------- | ----- | -------------------------------------------------------------------- |
| `TestFileLockAcquireSinglePathSucceeds`                       | yes   | Happy-path baseline.                                                 |
| `TestFileLockAcquireSamePathTwiceByDifferentItemsConflicts`   | yes   | Conflicts map populated, acquired empty.                             |
| `TestFileLockReleaseFreesAllPathsHeldByItem`                  | yes   | Multi-path Release frees the full set.                               |
| `TestFileLockAcquirePartialConflictReturnsConflicts`          | yes   | `[a, b]` against held `b` → acquired=[a], conflicts={b}; plus retry. |
| `TestFileLockConcurrentAcquireRaceFree`                       | yes   | 32 goroutines, exactly one winner, all losers see the same holder.   |
| `TestFileLockPathsAreOpaque`                                  | added | `./a` vs `a` are distinct keys.                                      |
| `TestFileLockEmptyInputsAreNoOps`                             | added | nil/empty paths, Release on unknown ID, same-holder idempotent.      |
| `TestFileLockZeroValueIsUsable`                               | added | `var mgr fileLockManager` works without `newFileLockManager`.        |

Race detection: `TestFileLockConcurrentAcquireRaceFree` runs 32 goroutines
on a shared `mgr`. The mage CI runs without `-race` by default but
`testFunc` runs `-race` per-test invocation; verified clean via
`mage testFunc ./internal/app/dispatcher/ TestFileLockConcurrentAcquireRaceFree`.
