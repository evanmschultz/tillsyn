## 4. Falsification Attempts

### 4.1 Could the loop variable `path` leak the wrong filename into the error message?

The `for _, path := range []string{".githooks/pre-commit", ".githooks/pre-push"}` loop binds `path` to each slice element in order. The `if err := os.Stat(path); err != nil { return fmt.Errorf("‚Ä¶%s‚Ä¶", path) }` closes over the same iteration's `path` because the `return` is inside the same iteration. No goroutine, no closure capture, no defer ‚Äî direct-evaluated format string with the current `path` value. The pre-Go-1.22 loop-variable-aliasing footgun does not apply here because there's no closure capture.

**No counterexample.**

### 4.2 Could `os.Stat` succeed on a missing file via stale FS cache or symlink chicanery?

`os.Stat` follows symlinks and returns `nil, err` if the final target is unreachable. For the canonical case (`.githooks/pre-commit` deleted from the working tree), `os.Stat` returns `&fs.PathError{Op: "stat", Path: ".githooks/pre-commit", Err: syscall.ENOENT}` ‚Äî non-nil err ‚Äî triggering the pre-flight fail path. There's no kernel-level cache that would lie about file existence on a path that's been `git rm`-ed in the same session. The pre-flight is robust to the documented attack mode.

**No counterexample.**

### 4.3 Could `runCommand` swallow the `git config` exit code on a malformed config?

`runCommand` at `magefile.go:395-398` delegates to `runCommandWithEnv` which uses `exec.Command(name, args...)` with `cmd.Stdout = os.Stdout` / `cmd.Stderr = os.Stderr` (streaming, not capturing) and returns `cmd.Run()`'s error. `cmd.Run()` returns a non-nil `*exec.ExitError` when the subprocess exits non-zero. `git config core.hooksPath .githooks` exits 0 on success and non-zero on permissions / locked-config / disk-full failures. `runCommand`'s error propagates up; `InstallHooks` returns it directly without wrapping. The dev sees the error verbatim on stderr (because of streaming) AND mage emits a non-zero exit. No silent failure path.

**No counterexample.**

### 4.4 Could `mage ci` invoke `InstallHooks` indirectly via a chain we missed?

Call-graph reachability check: `CI()` (line 151) ‚Üí stage table ‚Üí `verifySources`, `formatCheck`, `coverage`, `Build`. `Build()` (line 130) ‚Üí `runCommand("go", "build", ‚Ä¶)`. `formatCheck()` (line 247) ‚Üí `trackedGoFiles()` + `runGofumptList()`. `coverage()` and `verifySources()` (not shown in this QA but unchanged from baseline `mage ci` green) do not reach `InstallHooks`. None of the called functions appear in a `git grep "InstallHooks"` result; the entire reachable call graph from `CI()` excludes `InstallHooks`. Combined with the live `mage ci` run leaving `core.hooksPath` unset, the spec is operationally satisfied.

**No counterexample.**

### 4.5 Could the doc comment's backticks confuse `go doc`?

The doc comment uses Markdown-style backticks (e.g. `` `core.hooksPath = .githooks` ``) inside Go `//` line comments. `go doc` and `gopls` render comments as-is ‚Äî backticks are not interpreted; they appear literally in `go doc -all` output and in IDE tooltips. The Go style-guide convention since gofmt 1.19 explicitly accepts Markdown-styled doc comments. No semantic surprise.

**No counterexample.**

### 4.6 Could the `Aliases` map ordering matter for kebab-case dispatch?

Mage's alias resolution iterates the map (which has stable lookup but no stable iteration order in Go), but lookup is by key, not by index. `mage install-hooks` triggers a `map[string]interface{}` lookup of `"install-hooks"`, which Go's hash table resolves in O(1) regardless of insertion order. No ordering bug possible.

**No counterexample.**
