### 2.6 CI-parity claim ("strong predictor of green CI") — REFUTED (attack vector #6)

CONTRIBUTING.md line 62: "`.githooks/pre-push` runs `mage ci` — mirrors what GitHub Actions runs, so a green pre-push run is a strong predictor of green CI."

Cross-reference to existing `## GitHub Actions Model` section (CONTRIBUTING.md lines 78-79):

> CI runs:
> - `mage ci` on all OSes (`ubuntu-latest`, `macos-latest`, `windows-latest`)

The local hook runs `mage ci` on the dev's single OS (one of the three matrix OSes, typically macOS). CI runs `mage ci` across all three. So a green local `mage ci` does NOT prove a green CI run — windows-latest can flake on path-separator/EOL/Go-toolchain edge cases that the local mac/linux dev won't hit; macos-latest can flake on different macOS versions, and so on.

The prose hedge — "**strong predictor**", not "**guarantees**" — is honest. "Predictor" is the right word: a green local `mage ci` is high-confidence signal that CI will pass, but it's not a proof. The phrasing does NOT promise CI parity; it claims correlation. The existing CONTRIBUTING.md line 96 already calibrates this: "Local gates greatly reduce CI surprises, but they do not fully replace GitHub-hosted OS/environment validation." The new section's "strong predictor" phrasing is consistent with that calibration.

The spawn-prompt's own analysis flagged this: "the 'strong predictor' hedge is honest. No misclaim, but flag if 'predictor' is too strong." After cross-reading the existing CONTRIBUTING.md GitHub Actions Model section + the existing CI-surprises hedge: "predictor" is the correct word, not too strong. REFUTED.

### 2.11 Worktree safety claim — REFUTED (attack vector #11)

The new section does NOT claim that `core.hooksPath` "works in any clone including future worktrees" — re-reading lines 46-72 carefully, that exact phrasing is absent. The closest claim is line 46-47:

> This repo ships two POSIX `sh` hooks at `.githooks/` that gate every commit and push against the same `mage` targets CI runs. Activation is one-time-per-clone via a tracked mage target

"One-time-per-clone" is technically correct: `mage install-hooks` runs `git config core.hooksPath .githooks`, which writes to `.git/config` for that clone. In git's worktree model:

- A primary worktree's `.git/config` is the shared config.
- A linked worktree (`git worktree add ...`) shares the primary's `.git/config` by default — so `core.hooksPath` set in the primary applies to linked worktrees too.
- A linked worktree CAN override with `git config --worktree core.hooksPath ...` if `extensions.worktreeConfig` is set, but that's opt-in.

So the "one-time-per-clone" framing is honest: a clone's `core.hooksPath` setting propagates to its linked worktrees by default. The docs do not over-promise — they do not claim "works in every future worktree even with --worktree config overrides," they claim "one-time-per-clone." Correct framing.

REFUTED.

### 2.12 Local-hooks-overrides note — REFUTED (attack vector #12)

CONTRIBUTING.md line 66:

> If you have local hooks in `.git/hooks/`, copy them into `.githooks/` before running `mage install-hooks` — `core.hooksPath` overrides the default lookup, so untracked local hooks would otherwise stop firing.

This matches WAVE_0_PLAN.md §0.3 risk-register text (line 134): "If you have local hooks in `.git/hooks/`, copy them into `.githooks/` before running `mage install-hooks`, since `core.hooksPath` overrides the default lookup."

Git behavior: when `core.hooksPath = .githooks`, git looks for hooks ONLY in `.githooks/`, never in the default `.git/hooks/`. Any pre-existing untracked hook in `.git/hooks/` (e.g. a dev's personal `commit-msg` template) silently stops firing. The doc note correctly warns about this — and tells the dev the right remediation (copy them into `.githooks/` first).

REFUTED.
