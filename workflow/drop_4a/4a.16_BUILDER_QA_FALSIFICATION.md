## Detailed Findings

### Attack 1 — Same-holder reacquire (REFUTED)

Trace: `Acquire("item-1", ["a"])` then `Acquire("item-1", ["a"])`.

- First call: `m.holders["a"] = "item-1"`, `m.itemPaths["item-1"]["a"] = {}`,
  returns `acquired=[a]`, `conflicts={}`.
- Second call: loop reads `m.holders["a"]` → `("item-1", true)`, branch
  `taken && holder != actionItemID` is **false** (same holder), falls
  to record branch. Map writes are idempotent (`m.holders["a"] = "item-1"`
  re-write; set-add of "a" to `owned`). Returns `acquired=[a]`,
  `conflicts={}`.

Maps are sets, not counters. No double-count. `Release("item-1")` frees
the path once cleanly. WAI per spec.

### Attack 2 — Partial-acquire atomicity (REFUTED)

Spec is **explicit** that partial acquisition without rollback is the
contract:

- WAVE_2_PLAN.md line 125: `paths [a, b] against existing holder of b
  returns acquired=[a], conflicts={b: <holder>}`.
- PLAN.md line 258: `Acceptance: ... fileLockManager with Acquire ...
  Release ...` matches the WAVE_2_PLAN expansion.

Builder implements partial-hold. `TestFileLockAcquirePartialConflictReturnsConflicts`
pins it directly. No spec violation.

### Attack 3 — Returned `conflicts` map aliasing (REFUTED)

`locks_file.go` line 83: `conflicts = make(map[string]string)` — fresh
allocation per call. Populated only via `conflicts[path] = holder`
where `holder` is a copy of a string from `m.holders`. Caller mutating
the returned map cannot reach manager state. Same is true for the
fresh `acquired` slice (line 98).

### Attack 4 — Two-map divergence (REFUTED)

Walked every state-mutating branch in `Acquire`:

- Empty paths: returns at line 84-86 before the lock; no state mutation.
- Conflict branch (line 101-104): writes only to `conflicts` (caller's
  return value). No `holders` / `itemPaths` mutation.
- Free-or-same-holder branch (line 105-114): writes `m.holders[path] =
  actionItemID` AND `owned[path] = struct{}{}` AND ensures
  `m.itemPaths[actionItemID]` exists. Always paired under the same
  mutex.

`Release` (line 127-139): for every path in `owned`, deletes from
`m.holders`; then deletes the whole `owned` entry from `m.itemPaths`.
Both maps drop in sync.

### Attack 4a — Same-holder reacquire of own path divergence (REFUTED)

After `Acquire("item-1", ["a"])`, `m.itemPaths["item-1"]` exists. A
second `Acquire("item-1", ["a"])` falls through to the record branch
(per attack 1), re-writes `m.holders["a"] = "item-1"`, then on line
108 `owned, ok := m.itemPaths["item-1"]` returns `(<existing-set>, true)`,
skips the lazy-init at line 109-112, adds "a" to the set (set-add of
existing key is no-op). Both maps still consistent.

### Attack 5 — Empty `actionItemID` collision (ACCEPTED)

Calling `Acquire("", ["a"])` records `m.holders["a"] = ""` and
`m.itemPaths[""]["a"] = {}`. A second `Acquire("", ["b"])` then shares
the empty-string holder bucket. `Release("")` frees both.

This is **internally consistent** — empty string behaves as a single
shared holder. Builder docs name this explicitly (locks_file.go line
72-73): `an empty actionItemID is treated as opaque (no validation in
this droplet)`.

Risk surface: any caller that accidentally passes the zero string
collides with every other empty-string caller. The conflict detector
(Wave 2.7) and walker (Wave 2.5) are the normalization owners and
must guard. Worth a single-line guard in 4a.20 or 4a.18 — not in 4a.16.
ACCEPTED.

### Attack 13 — `equalStringSlices` masks input-order regression (NIT)

The doc comment on `Acquire` (line 77) claims `The acquired slice
preserves input order for caller-side determinism.` The test helper
`equalStringSlices` (line 307-321) sort-compares, so a future code
change that broke input-order preservation would not be caught by any
existing test.

Test rigor mismatch: builder accepted this (locks_file_test.go line
303-306 explicitly defends the choice as readable), but the test-suite
no longer pins the docstring claim. Either:

- Drop the input-order claim from the docstring (consistent with
  test rigor), or
- Add one ordered-compare assertion alongside the sort-compare in
  `TestFileLockReleaseFreesAllPathsHeldByItem` or
  `TestFileLockAcquirePartialConflictReturnsConflicts`.

Not blocking — the input-order property currently holds via straight
`for _, path := range paths`. Carry forward as a 4a.16 nit or fold into
a Wave 2.5 walker test that depends on input-order.

### Attack 16 — Lazy-init in `Acquire` dead under `newFileLockManager` (ACCEPTED)

Builder worklog point 10 removed a defensive `if m.holders[path] ==
actionItemID` guard from Release on the YAGNI argument that the false
branch was unreachable. The same argument applies to the lazy-init
nil-checks in Acquire (line 91-96) — production callers go through
`newFileLockManager` which pre-allocates. The lazy branch is reached
only by `TestFileLockZeroValueIsUsable`'s `var mgr fileLockManager`.

Builder defends this in worklog point 7 via parity with `broker_sub.go`'s
defensive shape. Either choice is internally consistent — accept the
asymmetry between Acquire (defensive) and Release (non-defensive) as
the builder's call.

### Attack 19 — Duplicate input path semantics (NIT)

`Acquire("item-1", ["a", "a"])`:

- First "a": records, `acquired = [a]`.
- Second "a": `m.holders["a"] == "item-1"` already, same-holder branch,
  records again. `m.holders["a"] = "item-1"` (no-op), `owned["a"] = {}`
  (no-op set-add), `acquired = [a, a]`.

Returned `acquired` slice contains duplicate "a". No test exercises
this. The spec is silent on duplicates in input. The internal map
state is consistent (set-add + re-write are both no-ops on duplicate),
but the returned slice's duplicate may surprise callers.

Mitigation options:

- Document explicitly: `the acquired slice preserves input order
  including duplicates; callers that care should de-dupe before
  calling.`
- De-dupe the input loop (small cost, not worth it for opaque API).
- Add a one-line test pinning the current behavior.

Not blocking — internal state is sound. Consider a single-line test
in Wave 2.5 walker / 2.7 conflict-detector to pin which side dedupes.
