## NIT-1 — Multi-project auth-request paths bypass the toggle (non-blocking)

**Surface:** `internal/app/auth_requests.go:425-445` (Reject 0).

**Issue:** When `req.Path` parses to `AuthRequestPath{Kind: AuthRequestPathKindProjects, ProjectIDs: [...]}` (multi-project shape, see `auth_request.go:292-307`), `path.ProjectID` is empty (the per-project IDs live in `path.ProjectIDs`). The fallback at `:433-435` reads `path.ProjectID` → empty. The check at `:437` (`if projectID != "" && projectID != AuthRequestGlobalProjectID`) skips. **Result:** any orch-self-approval cascade against a multi-project path silently bypasses every project's toggle.

**Why non-blocking today:** The orch-self-approval cascade is per-subagent / per-project by design (W3.1 happy path uses `Path: "project/<id>"`). No production code path constructs multi-project subagent auth requests today. The W3.2 spec implicitly assumes single-project paths.

**Surfaces if/when broken:** If a future cascade introduces multi-project subagent delegation (e.g. cross-project STEWARD subagent), and ANY of those projects has `OrchSelfApprovalEnabled = *false`, the toggle would silently fail to backstop.

**Suggested mitigation (future drop):** When `path.Kind == AuthRequestPathKindProjects`, iterate `path.ProjectIDs` and require ALL toggles enabled (most-restrictive wins). Or reject multi-project subagent auth requests entirely at gate-entry, since per-project semantics dominate the cascade design.

**Routing:** Drop 4a refinements list (orchestrator captures via `project_drop_4a_refinements_raised.md` MEMORY.md entry). Not a 4a.25 blocker.
