# Droplet 4a.21 — PROCESS MONITORING — QA Falsification

**Verdict:** PASS-WITH-NIT
**Mode:** filesystem-MD only
**Round:** 1

## Summary

The 4a.21 process monitor implementation is correct, race-safe, and goroutine-leak-free under nominal flow. No CONFIRMED unmitigated counterexample. Three substantive NITs surfaced — the most load-bearing being a TOCTOU window on the state-conflict guard whose impact is bounded by the service-layer terminal-state guard at `internal/app/service.go:1003`. Three modernization NITs (LSP `unusedfunc` on `newProcessMonitor`, LSP `rangeint` hints in tests, tight 100ms timing margin on duration test). The `BlockedReason` vs `failure_reason` carrier choice is **correct** for Drop 4a — `ActionItemMetadata` has no `FailureReason` field today; `BlockedReason` is the only typed free-form failure-context slot. Drop 4b restructures.

## Attack Matrix

### 1. State-conflict guard TOCTOU window

**Attack:** Race between guard read at `monitor.go:305` (`GetActionItem`) and `MoveActionItem(failed)` at `monitor.go:327`. If item moves to `complete` AFTER guard read but BEFORE `MoveActionItem`, the monitor would issue a complete→failed downgrade.

**Reasoning:** Sequence
1. T1 (monitor goroutine): `GetActionItem` returns item with `LifecycleState=in_progress`.
2. T2 (parallel orch / agent): completes the item (LifecycleState→complete).
3. T1: skips the guard (`current.LifecycleState != StateComplete`), proceeds to `MoveActionItem(failed)`.

**Verdict:** CONFIRMED race-as-described, MITIGATED.

The service-layer transition guard at `internal/app/service.go:1003` rejects transitions FROM terminal states with `ErrTransitionBlocked`. Builder's doc-comment at `monitor.go:36-40` acknowledges this is the actual safety net; the in-monitor guard is a clean-signal short-circuit, not the correctness boundary. Net effect on a lost race: `MoveActionItem` returns `ErrTransitionBlocked`, monitor wraps and stores in `h.waitErr` (line 295), `Wait()` callers see it. No corruption, no spurious failed-state.

**NIT (logged for Drop 4b):** Add explicit re-check + structured warning log on `ErrTransitionBlocked` so the operator can distinguish "monitor lost the race" from "service mutation broke." Builder's prefix-pinning ("agent process crashed:") is good but the lost-race outcome is currently surfaced as a generic wrapped error.

### 2. `BlockedReason` vs `metadata.failure_reason` carrier choice

**Attack:** Spec at `WAVE_2_PLAN.md:236` and `PLAN.md:298` calls for `metadata.failure_reason`. Builder used `BlockedReason` field. Drift?

**Reasoning:** Verified `internal/domain/workitem.go:142-164`:

- Fields: `Objective`, `ImplementationNotesUser`, `ImplementationNotesAgent`, `AcceptanceCriteria`, `DefinitionOfDone`, `ValidationPlan`, **`BlockedReason`** (line 150), `RiskNotes`, `CommandSnippets`, `ExpectedOutputs`, `DecisionLog`, `RelatedItems`, `TransitionNotes`, `DependsOn`, `BlockedBy`, `ContextBlocks`, `ResourceRefs`, `KindPayload`, `CompletionContract`, **`Outcome`** (line 163).
- **No `FailureReason` field exists.** No JSON tag `failure_reason` either.

Spec language `metadata.failure_reason` is **legacy informal phrasing pre-dating Drop 1.75's typed-completion-fields landing.** `BlockedReason` is the only free-form failure-context slot on the closed struct. Builder doc-comment at `monitor.go:331-337` explicitly calls this out: "Drop 4b refactors the failure shape into a structured type per PLAN.md §17.3.Q5; until then we use the existing free-form BlockedReason slot as the carrier." PLAN.md row 4a.21 line 300 itself says: **"`metadata.failure_reason` shape: free-form string for 4a. Drop 4b refactors to structured type."**

**Verdict:** REFUTED as drift. CORRECT carrier choice for Drop 4a.

**NIT (spec-language alignment):** WAVE_2_PLAN.md and PLAN.md should be updated post-Drop-4a to say `BlockedReason` (or remove the `metadata.` prefix since it's no longer JSON-blob-shaped). Logged for Drop 4b refinement.

### 3. Goroutine leak — caller never calls Wait or Close

**Attack:** `Track` starts a goroutine via `go m.runHandle(ctx, h)` at `monitor.go:261`. If caller drops the Handle without calling Wait or Close, does the goroutine leak?

**Reasoning:** The goroutine's lifecycle is bound to `cmd.Wait()` (line 277) returning, NOT the Handle's GC. Sequence:
1. Goroutine calls `h.cmd.Wait()` — blocks until process terminates.
2. If process exits naturally (any reason): `Wait` returns, goroutine proceeds to outcome construction, optional crash transition, then deferred close+delete (lines 270-275).
3. If process never exits AND caller drops Handle: goroutine blocks forever on `cmd.Wait()`. **This is a leak iff the process is unbounded.**

**Verdict:** CONFIRMED edge-case leak only when caller drops Handle AND process is genuinely unbounded.

In practice, agent processes always terminate (claude subprocess exits, signal, OOM, etc.). The "leak" requires both (a) caller forgets Handle (programming bug) AND (b) process never exits (unbounded child). Standard contract violation, not a monitor bug.

**MITIGATED** by the documented contract (`monitor.go:120-127`, `145-152`): caller MUST call Wait or Close. `Close` issues kill, forcing process termination, forcing goroutine exit. If caller obeys the contract, no leak.

**NIT (doc-comment polish):** Add a one-line "Calling neither Wait nor Close on a Handle whose process never exits leaks the per-Handle goroutine — always defer Close on the returned Handle" guidance to Track's doc-comment. Today this is implicit in the leak-free claim.

### 4. Race between concurrent Wait and Close

**Attack:** Caller goroutines: G1 calls `Wait`, G2 calls `Close` simultaneously. Race? Outcome inconsistency?

**Reasoning:**
- `Wait` (line 161): blocks on `<-h.done` (line 165), then reads outcome under `resultMu`.
- `Close` (line 180): `closeOnce.Do(...)` issues kill, then blocks on `<-h.done`.
- Both wait on the same channel `h.done`, closed exactly once by the goroutine at line 274 under defer.
- The goroutine writes outcome (line 282-284) BEFORE closing done (line 274). All Wait readers observe the cached value through happens-before via the channel close.

Concurrent Close + Wait safe: kill propagates to cmd, cmd.Wait returns (likely with `*exec.ExitError`), goroutine builds outcome with `Crashed=true`, closes done. Both callers see same outcome.

**Verdict:** REFUTED. Race-free. Memory ordering correct.

### 5. Multiple Wait calls on same Handle

**Attack:** Wait called twice. Second call: blocks forever? Panics? Returns same value?

**Reasoning:** `done` channel closed by goroutine. Closed channels return immediately on receive. Second Wait does `<-h.done` (returns immediately on closed channel), reads cached outcome under `resultMu`. Returns same TerminationOutcome and waitErr.

**Verdict:** REFUTED. Idempotent by construction (channel-close semantics + cached outcome). Verified by `TestHandleWaitIsIdempotent` (test line 507).

### 6. Concurrent Track calls — monitor internal state

**Attack:** Worklog claims tracked-PID map. Race-safe?

**Reasoning:** `processMonitor.tracked map[string]*Handle` guarded by `processMonitor.mu` (line 203). All accesses:
- Track insert: `m.mu.Lock(); m.tracked[trimmed] = h; m.mu.Unlock()` (lines 257-259).
- runHandle delete (deferred): `m.mu.Lock(); delete(m.tracked, h.actionItemID); m.mu.Unlock()` (lines 271-273).

Per-Handle state is owned by that Handle; no cross-Handle sharing. `TestMonitorConcurrentTrackHandlesAreIndependent` (test line 458) exercises 5 concurrent Tracks with mixed exit/crash modes; passes under `-race`.

**Verdict:** REFUTED. Concurrent-safe.

### 7. Process kill signal classification — SIGTERM vs SIGKILL

**Attack:** Does `Signal` string match observed `os.Signal` type? Cross-platform?

**Reasoning:** `signalNameFromState` (line 393):
- Unix: casts `state.Sys()` to `syscall.WaitStatus`; if `Signaled()`, returns `Signal().String()` — which yields names like `"killed"` (SIGKILL), `"terminated"` (SIGTERM), `"interrupt"` (SIGINT).
- Fallback: parses `state.String()` for `signal: <name>` prefix.
- Empty when `ExitCode() >= 0` (clean or non-zero exit without signal).

`TestMonitorSignalKilledMarksFailed` (test line 288) skips on Windows explicitly (`runtime.GOOS == "windows"`); test invokes `cmd.Process.Kill()` (which is SIGKILL on Unix); asserts `outcome.Signal != ""` and `BlockedReason` carries prefix `"agent process crashed: signal:"`.

**Verdict:** REFUTED. Cross-platform handling correct. Windows skip is appropriate (Drop 4a is Unix-only per worklog).

### 8. Atomicity — error path on `MoveActionItem` failure

**Attack:** If `MoveActionItem` fails (line 327), does `UpdateActionItem` still fire? Should it?

**Reasoning:** Sequence in `applyCrashTransition` (lines 304-348):
1. `GetActionItem` — error returned, no further calls.
2. State-conflict guard short-circuit.
3. `ListColumns` — error returned, no further calls.
4. Failed-column resolution — error returned, no further calls.
5. `MoveActionItem` — **on error, returns immediately at line 328. UpdateActionItem NOT called.** Correct atomicity: a half-applied state (still in `in_progress` but with `BlockedReason` set) is worse than no metadata change.
6. `UpdateActionItem` — on error, returned, item is in `failed` column without populated metadata. Surfaced via `Wait`'s waitErr.

**Verdict:** REFUTED. Atomicity correct: state-change failure aborts metadata write. Edge case: state moved but metadata write fails → operator sees a `failed` item with stale metadata. Acceptable degradation since the state is the load-bearing signal.

**NIT (Drop 4b):** Consider compensating reverse-transition or attention-item raise on Move-success-but-Update-failure path. Not blocking; metadata can be reconstructed from process logs if needed.

### 10. LSP `unusedfunc` warning on `newProcessMonitor` (line 214)

**Attack:** Genuinely unused? Dead code?

**Reasoning:** `newProcessMonitor` is called by all 11 monitor tests (e.g., `monitor_test.go:186`, `228`, `300`, etc.). It is **NOT** unused — LSP `unusedfunc` is a false positive because it doesn't resolve the test-file callers cross-referencing the production constructor (test files in same package can call unexported symbols). The constructor is also the production-path API ramp for 4a.23 (`till dispatcher run`) wiring.

**Verdict:** REFUTED. False-positive LSP hint. Constructor is live (test consumers + 4a.23 wiring target).

**NIT (LSP-noise):** None — false positive on the LSP side; ignore.

### 11. LSP `rangeint` hints at lines 464, 470

**Attack:** Modernization missed?

**Reasoning:** Two `for i := 0; i < n; i++` loops in `TestMonitorConcurrentTrackHandlesAreIndependent` (test lines 464 and 470). Go 1.22+ supports `for i := range n`. This is a stylistic modernization, not a correctness issue.

**Verdict:** ACCEPTED as NIT. Not a counterexample. Modernize in a Drop 4b sweep alongside other rangeint cleanups.

### 12. Test rigor — `TestMonitorTracksDurationAccurately` 100ms margin flake risk

**Attack:** Is 100ms enough margin to be flaky on slow CI?

**Reasoning:** Test (line 349) asserts `outcome.Duration >= 100*time.Millisecond`. The fake agent sleeps 100ms then exits. The assertion is **lower-bound only** — process startup overhead can only ADD to wall-clock duration, never subtract. Race between `m.clock()` capture in Track (line 245) and `m.clock()` capture in runHandle (line 278): startedAt happens BEFORE cmd.Start; endedAt happens AFTER cmd.Wait returns; difference is monotonic ≥ sleep duration.

**Verdict:** REFUTED. The lower-bound-only assertion cannot flake from slow CI; only from a system clock running backwards (impossible with `time.Now`). The test is robust.

**Note:** If the test asserted an **upper bound** (e.g., `< 200ms`), that would be flaky. It does not.

### 13. Goroutine count audit

**Attack:** Are there hidden goroutine leaks across the 11-test suite?

**Reasoning:** Worklog claims 95 dispatcher tests pass under `-race`. Confirmed via `mage testPkg ./internal/app/dispatcher` (95/95 pass). `-race` would flag any concurrent unsynchronized access. No `goleak`-style explicit goroutine-count assertion in the test suite, but the per-Handle goroutine exits via deferred close+delete pattern bound to `cmd.Wait()` return.

**Verdict:** REFUTED for nominal flow. ACCEPTED with NIT.

**NIT (Drop 4b):** Add `goleak.VerifyTestMain` or equivalent to harden the goroutine-leak-free claim against future drift. Not blocking — the explicit teardown pattern is robust today.
