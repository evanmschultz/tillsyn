## Hylla Feedback

Two misses + one ergonomic gripe.

### Miss 1 — `hylla_search_keyword` returned empty for `UpdateActionItem` and `ErrInvalidDispatcherConfig`

- **Query**: `hylla_search_keyword` with `query="UpdateActionItem"` (later `ErrInvalidDispatcherConfig`), `fields=["content"]`, `node_type="block"`, `id_search_mode="tail_symbol"` (default), `limit=10`.
- **Missed because**: tail_symbol mode appears to mismatch on top-level method names that share their name with a struct (`UpdateActionItem` is a method on `*Service`; tail_symbol resolution returned no block). `RaiseAttentionItem` worked because its tail-symbol form (`Service.RaiseAttentionItem`) is unique in the index. `ErrInvalidDispatcherConfig` returned empty — likely because it's a `var` declaration grouped with other dispatcher errors and the keyword indexer didn't catch the grouped-decl tail symbol.
- **Worked via**: ripgrep against `internal/app/*.go` and `internal/domain/*.go` to locate `func (s *Service) UpdateActionItem`, `type UpdateActionItemInput`, `type RaiseAttentionItemInput`, and the `ErrInvalidDispatcherConfig =` declaration line.
- **Suggestion**: tail-symbol mode should accept method-only names by walking the method-receiver index, not just the package-symbol index. Grouped `var ( ... )` declarations should index each name individually.

### Miss 2 — `hylla_search` errored mid-session: `enrichment still running for github.com/evanmschultz/tillsyn@main`

- **Query**: hybrid `hylla_search` with `query="Service UpdateActionItem app input struct"` and `query="ErrInvalidDispatcherConfig dispatcher errors"`.
- **Missed because**: the artifact ref's enrichment was still running when these queries fired (snapshot 5 was being re-enriched). Both calls returned the explicit error message rather than partial results.
- **Worked via**: same ripgrep fallback above.
- **Suggestion**: if enrichment is still running, the search response could fall back to keyword-only results (which are already indexed) rather than erroring out — or the artifact ref endpoint could expose enrichment state pre-query so the caller knows to wait or to use keyword mode.

### Ergonomic — `hylla_search` `fields` (plural) vs `field` (singular) parameter clash

When passing only `fields=["summary","content"]` (plural — search target fields) to `hylla_search`, the validator rejects the call with `field must be summary, content, or docstring` — looking for the singular `field` (the vector embedding target). That field-vs-fields naming clash forces every hybrid call to specify both. Suggested fix: auto-default `field` to the first valid entry of `fields`, OR rename one to disambiguate (e.g. `embed_field` vs `search_fields`).
