## 2. Extra File Beyond Spec — `app_service_adapter_lifecycle_test.go` (+196)

Builder added an extra file beyond the §1.6 paths list: two new tests appended to the pre-existing `app_service_adapter_lifecycle_test.go`:

- `TestCreateActionItemResolvesStateToColumn` (`:1014-1108`).
- `TestMoveActionItemResolvesStateToColumn` (`:1110-1207`).

Builder's worklog rationale (per spawn prompt): "locks in load-bearing dispatcher behavior."

### 2.1 Soundness Assessment

**Architecture symmetry check:** the existing 8 mcpapi tests use `stubExpandedService` — they verify the wire layer forwards the right fields into the right request slots. They cannot verify the **resolver actually maps `"todo"` → the column titled `"To Do"`** because `lastCreateActionItemReq` is a passthrough capture, not a real service.

The new two tests use `commonLifecycleFixture` (real `sqlite.Repository` + real `app.Service` + real `AppServiceAdapter` + real columns created via `fixture.svc.CreateColumn`). They assert that `CreateActionItem` with `State: "todo"` lands the action item in `todo.ID` — the column whose name `"To Do"` maps to `domain.StateTodo` via `actionItemLifecycleStateForColumnName`. Symmetric assertion for `MoveActionItem` with `State: "in_progress"` landing in the `In Progress` column.

**This is the layer the mcpapi stubs cannot reach.** A bug in `actionItemLifecycleStateForColumnName` (e.g. case-sensitivity drift, whitespace handling, ordering) would slip past the wire-level stubs but trip these integration tests immediately. Asymmetric coverage, not redundant.

### 2.2 Non-Redundancy Check

| Layer | Test file | What it verifies |
|---|---|---|
| Wire (handler dispatch) | `extended_tools_test.go` | JSON-RPC call → handler → `common.CreateActionItemRequest{State, ColumnID}` correctly populated; handler-level reject for both-empty / both-supplied. |
| Adapter (validation + resolver) | `app_service_adapter_lifecycle_test.go` | Adapter runs four-corner switch, normalizes state, calls `resolveActionItemColumnIDForState` against real columns, lands the action item in the correct column. Verifies `LifecycleState` field on the returned domain item. |

Each layer attacks a different failure mode. A wire test passing while the integration test fails would mean: "handler forwards state correctly but the resolver is broken." A wire test failing while the integration test cannot run would mean: "handler is broken so the adapter never gets the state." Two independent guards.

### 2.3 Implementation Quality

- Both tests use `errors.Is(err, ErrInvalidCaptureStateRequest)` to verify sentinel wrapping — matches the adapter's `fmt.Errorf("...: %w", ErrInvalidCaptureStateRequest)` pattern. Without this assertion, a future refactor that drops the `%w` wrap would slip through.
- Both tests assert the human-readable error substring (`"column_id or state is required"`, `"specify exactly one of column_id or state"`) — defends against silent message drift that would harm CLI / agent diagnostic UX.
- The test seed uses `seedOrphanKindsForTest` (existing helper at `:57`) — no new seed introduced. Reuses existing fixture infrastructure.
- The test names follow the project's `TestX...` naming convention with descriptive verb-suffix pattern (`ResolvesStateToColumn`).

### 2.4 Verdict on Extra File

**SOUND + NON-REDUNDANT.** Lock-in for the integration seam (real-resolver-against-real-columns) the mcpapi stubs cannot exercise. The +196 LOC investment is proportional to the load-bearing surface — Wave 2 dispatcher will route every state-driven create/move through these code paths, and the integration tests pin the column-name → state mapping behavior so a Drop 4.5 columns-table refactor cannot silently drift.

**Recommended:** keep the file. The asymmetric coverage justifies the extra LOC.
