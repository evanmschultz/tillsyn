#### A6 — Path normalization (`./a` vs `a` vs `../proj/a`): ACCEPTED-WITH-DOC

The detector treats path strings as opaque (map equality). `domain.normalizeActionItemPaths` (`/Users/evanschultz/Documents/Code/hylla/tillsyn/main/internal/domain/action_item.go:705-726`) only trims surrounding whitespace and rejects backslashes — it does NOT canonicalize `./` prefixes or `..`. Two siblings declaring `./internal/foo.go` and `internal/foo.go` would NOT overlap.

**Why ACCEPTED**: `conflict.go:88-92` doc-comment is explicit: *"Treated as opaque by the detector — no canonicalization beyond the trim/dedupe domain.NewActionItem already applies on create."* The contract pushes path-style discipline upstream (planner-QA). The dispatcher cannot canonicalize without touching domain semantics that 4a.20 is explicitly out of scope for. Surface this as a refinement candidate (route via planner-QA), not a 4a.20 counterexample.

#### A7 — Tie-break determinism on equal Position + equal ID: ACCEPTED

`TieBreakSibling` returns `b` when `a.ID < b.ID` is false (i.e. equal IDs). Identical IDs are a domain-layer impossibility (UUID enforced via `domain.ErrInvalidID` at `action_item.go:301`), so this is unreachable in practice. `TestDetectorTieBreakByPositionThenID` covers position-wins, position-tie-id-tiebreak, and reverse-id-tie-break. **Cannot break in production.**

#### A8 — `InsertRuntimeBlockedBy` non-atomicity (Update succeeds, Attention fails): ACCEPTED

If `UpdateActionItem` succeeds and `RaiseAttentionItem` fails, the runtime blocker is in place but no attention raised. `conflict.go:280-287` doc-comment explicitly states this trade: *"the runtime blocker is still in place (the dependency edge is the load-bearing part) and the error is returned wrapped... the orchestrator already polls attention items, so a missed raise surfaces on the next sweep."* **Documented, not hidden.** ACCEPTED.

The wrapped-error return ensures callers see the failure (no swallow). Atomicity violation surfaces; idempotency on retry (next sweep finds blocker present, skips Update, attempts Attention again) recovers cleanly.

#### A11 — Sibling-only limitation (cousin overlap): ACCEPTED

Cross-subtree overlap is filtered out (`conflict.go:190` skips different ParentID). `TestDetectorIgnoresNonSiblings` covers it. The `conflict.go:24-33` doc-comment is explicit: *"cross-subtree overlap does not race agents within a single promotion sweep"* because the runtime `fileLockManager` / `packageLockManager` already serialize at acquire time. **Documented, deferred to Drop 4b.** ACCEPTED.

#### A12 — Permanent runtime blocker: ACCEPTED

Once `InsertRuntimeBlockedBy` succeeds, the new BlockedBy entry is permanent. `conflict.go:46-57` doc-comment lays out the rationale: (a) runtime blocker IS the dependency edge, (b) cleanup-on-completion would race the eligibility walk, (c) eligibility checks treat completed blockers as cleared so a stale row is harmless. Mirrors `domain.ActionItemMetadata.BlockedBy` append-only semantics. **Cannot break — by design.**

#### A13 — Idempotency under concurrent goroutines: ACCEPTED

`conflict.go:305-310` reads `item.Metadata.BlockedBy` from the input `domain.ActionItem` (passed by value — Go copies the slice header, not the backing array). Two concurrent calls with the same `item` can both pass the idempotency check and both call `UpdateActionItem`. The race is closed at the service layer: `Service.UpdateActionItem` (`service.go:1124`) does `s.repo.GetActionItem` first, then `enforceMutationGuardAcrossScopes`, then writes — the second writer reads the freshly-augmented `BlockedBy` and the service-layer normalization (`normalizeStringList` in `domain/workitem.go:608-623`) dedupes via trim-equality.

**Trace**: Caller A and B both pass `item` with `BlockedBy=[]`, both pass the in-memory check, both call `Update` with `BlockedBy=[sibling]`. Service applies update; `normalizeStringList` dedupes within a single update (no help cross-update), but the second call's update overwrites the first with the same value — idempotent terminal state. The DUPLICATE attention item is the actual hazard: both callers raise attention. Not a correctness bug, but noise.

**MITIGATED but documented as a refinement candidate** — single-flight is a 4b dispatcher concern (the dispatcher is the only caller per spec, and runs single-threaded per RunOnce sweep today). For 4a.20, ACCEPTED.

#### A14 — `OverlapValue` shape when both file AND package overlap: REFUTED with NIT

When both `item.Paths` and `item.Packages` overlap with the SAME sibling, `conflict.go` emits two entries: one `kind=file` and one `kind=package`. `TestDetectorFindsFileOverlapBetweenSiblings` (lines 56-100) sets up this exact case (item has `Paths=["walker.go"]`, `Packages=["dispatcher"]`, sibling identical), but only asserts the file overlap is present (the assertion loop scans for first `OverlapKind == SiblingOverlapFile` and stops). It does not assert the package overlap is ALSO emitted in the same call.

**NIT**: a future regression that drops the package emission in the file branch would not be caught by either of the two existing happy-path tests in isolation — the file test stops at file, the package test (line 104) excludes file overlap by construction (disjoint Paths). A combined-overlap regression would need to break BOTH tests, which it would, but a more direct assertion would be a tighter regression net. Recommendation: extend `TestDetectorFindsFileOverlapBetweenSiblings` to assert `len(overlaps) == 2` AND that one is `kind=file` and one is `kind=package`. **Not blocking — production code is correct (verified by inspection).**

1. **NIT (A14)**: `TestDetectorFindsFileOverlapBetweenSiblings` could additionally assert the simultaneous package overlap is present in the same call (`len(overlaps) == 2`), tightening the regression net for combined-overlap behavior. **Non-blocking.**
