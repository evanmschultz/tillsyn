## Key Design Choices

### 1. Re-subscribe loop with `afterSequence` cursor

The broker's `Wait(ctx, type, key, afterSequence)` is single-shot — it returns
the next event newer than `afterSequence` and unregisters the waiter. To
deliver every event in order, the goroutine wraps `Wait` in a re-subscribe
loop:

```
cursor = broker.Latest(...).Sequence  // baseline so we don't replay history
loop:
    event = broker.Wait(ctx, type, key, cursor)
    cursor = event.Sequence            // advance before delivery
    out <- event                        // (or ctx.Done -> exit)
```

The cursor advances **before** delivery so a slow consumer cannot stall the
broker's monotonic ordering. This pins the contract Wave 2.5 (the tree
walker) relies on: every Sequence observed corresponds to exactly one
service-layer write, in commit order.

### 2. Broker-close handling (mitigates plan-QA falsification attack)

Plan-QA falsification raised: *"broker re-subscribe loop spins on Sequence=0
forever after broker close."* Mitigation in code + comment:

- The in-process broker has no `Close` method; the broker is process-lifetime
  and ctx cancellation is the **only** supported termination signal for a
  subscriber. Documented as the broker-close contract in the doc-comment on
  `subscribeBroker`.
- Defensively, the loop treats every non-context error from `Wait` as a fatal
  subscription error and returns. If a future broker implementation closes
  its underlying channel without a non-nil error contract, the loop still
  exits because the in-process broker's `Wait` returns either an event with a
  positive Sequence or ctx.Err(); a hypothetical (zero-event, nil-error)
  return would deliver a zero-Sequence event downstream, but the cursor
  guard (`if event.Sequence > cursor`) would freeze and the next Wait would
  block on the same cursor — which is the desired no-spin behavior if the
  broker is misbehaving but not yet failing outright.
- Tests pin both the cancellation-during-Wait path
  (`TestDispatcherStopsOnContextCancel`) and the pre-cancelled-context path
  (`TestSubscribeBrokerCancelledContextExitsImmediately`).

### 3. Channel semantics

- Output channel is **buffered with capacity 1** so a single-event-in-flight
  delivery does not require the consumer to be ready before the goroutine
  can advance to the next Wait registration.
- Send is wrapped in a `select { ctx.Done() | out <- event }` so a stalled
  consumer cannot trap the goroutine if ctx is cancelled mid-delivery.
- The goroutine `defer close(out)` so consumers iterating with
  `for ev := range ch` see clean termination.

### 4. Empty / nil short-circuits

- `subscribeBroker(ctx, "")` returns an immediately-closed channel without
  starting a goroutine — no goroutine leak from misconfigured callers.
- `subscribeBroker` on a nil `*dispatcher` receiver also returns a closed
  channel rather than panicking.

### 5. Publisher placement at end of write paths

`publishActionItemChanged` is called as the **last** action before each
method's success `return`, after all repo writes, embedding enqueues, and
auxiliary side effects (drop-findings seeding in `CreateActionItem`,
refinements-gate audit in `MoveActionItem`, thread-context embedding in
`UpdateActionItem`). This mirrors the placement convention established by
`publishCommentChanged` + `publishAttentionChanged` in
`Service.CreateComment` (`service.go` lines 1477–1478): publish only on full
success, never on a partially-applied write.

### F2 — DeadlineExceeded doc/code drift

The Round 1 doc claimed `context.DeadlineExceeded` was a "benign re-subscribe trigger," but the code at `runBrokerSubscriber` lines 76–78 returns on it (treating it as termination). The doc was rewritten to match the code's actual behavior: both `context.Canceled` and `context.DeadlineExceeded` exit the loop, and `subscribeBroker` does not wrap `Wait` in its own timeout — a deadline-bearing ctx carries the intended subscription lifetime.

**Before (Round 1, lines 37–40):**

```
// returning (zero-event, nil-error) in a tight loop. context.DeadlineExceeded
// while ctx.Err() is still nil indicates an internal Wait timeout (the
// service layer wraps Wait with WithTimeout for list surfaces) and is treated
// as a benign re-subscribe trigger.
```

**After (Round 2, lines 39–43):**

```
// returning (zero-event, nil-error) in a tight loop. Both context.Canceled
// and context.DeadlineExceeded are treated as termination signals and exit
// the loop. subscribeBroker does not wrap Wait in its own timeout; if a
// caller passes a deadline-bearing ctx, the deadline IS the intended
// subscription lifetime.
```

### F3 — Per-event delivery overclaim

The Round 1 doc claimed "every Sequence observed here corresponds to exactly one repo-write committed by the service layer." Under burst conditions the broker coalesces consecutive publishes (per `internal/app/live_wait.go:109–148`); the obverse "every repo-write produces exactly one observed event" is false. The doc was rewritten to be precise about coalescing: each wake corresponds to *at least one* repo write since the previous wake, and the walker re-reads the project tree on each wake so coalescing is invisible to the dispatch contract.

**Before (Round 1, lines 19–22):**

```
// Wave 2.5's tree walker reads the same event stream and relies on this
// monotonic ordering to walk the project tree consistently — every Sequence
// observed here corresponds to exactly one repo-write committed by the
// service layer.
```

**After (Round 2, lines 19–24):**

```
// Wave 2.5's tree walker reads the same event stream and relies on this
// monotonic ordering. Each wake corresponds to *at least one* repo write
// since the previous wake. Under burst conditions the broker coalesces
// consecutive publishes into the latest event observed; the walker
// re-reads the project tree on each wake so coalescing is invisible to the
// dispatch contract.
```
