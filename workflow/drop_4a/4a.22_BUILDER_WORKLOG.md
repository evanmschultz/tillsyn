## Design Choices

### 1. Function-typed seams over interface seams

`cleanupHook` carries four function-typed fields (`releaseFileLocks`, `releasePackageLocks`, `revokeAuthBundle`, `unsubscribeMonitor`) rather than four interface fields. Two reasons:

1. **The acceptance test `TestCleanupContinuesPastIndividualFailure` requires injectable error returns** for file-lock and package-lock release. The committed `fileLockManager.Release(actionItemID string)` and `packageLockManager.Release(actionItemID string)` methods return NO error (locks_file.go:127, locks_package.go:142). Tests need to drive errors from those steps; closures are the cleanest seam.
2. **Lifts the Release signature to error-returning** so Drop 4b's SQLite-mirror-backed Release can surface persistence failures without breaking the cleanup contract. Documented in cleanup.go's package doc-comment as the Drop 4b extension point.

Production wiring (`newCleanupHook`) closes over the lock managers concretely; tests inject erroring closures via direct struct-literal construction (same package, exported field is unnecessary).

### 2. Error aggregation with `errors.Join`

All four steps run regardless of individual failures. `errors.Join(errs...)` collapses the per-step error slice into one return value. The `unsubscribeMonitor` step does NOT contribute to the error chain (its signature is fire-and-forget per `monitorUnsubscriber` doc) — the tracked-PID map is a dashboard-facing convenience, not a state-machine input, so losing a delete on a transient failure is recoverable and never blocks an action-item state transition.

Pipeline order is fixed: file-lock → package-lock → auth-revoke → monitor-unsub. Documented but not load-bearing today; all four steps always run before `OnTerminalState` returns.

### 3. Idempotency via mutex-guarded set

`cleanupHook.cleaned map[string]struct{}` records every action-item ID that's been processed. Second `OnTerminalState(item)` call short-circuits before any per-step closure fires. Critical section is bounded by one map lookup + write per call.

Tradeoff documented in cleanup.go: the set grows unbounded over the dispatcher's lifetime. Acceptable in Wave 2 (manual-trigger CLI process exits between runs); Drop 4b will either bound with LRU or wire cleanup off a per-spawn arena.

### 4. Wave 3 auth-revoke seam

Per acceptance criterion 2.iii, the auth-bundle revoke step is a documented stub. Implemented as the package-level `revokeAuthBundleStub(actionItemID string) error` whose body is `// Wave 3 fills this in` returning `nil`. Wired into `newCleanupHook` so production picks up the stub; Wave 3 swaps it for a `till.auth_request(operation=revoke)` call without touching the call site or `cleanupHook` shape.

Cross-references in doc-comments:
- Drop 4c Theme F.7 (eventual wiring).
- spawn.go's `AuthBundle{}` stub (matching seam on the spawn side, droplet 4a.19).
- WAVE_2_PLAN.md §2.9 mitigation paragraph (dev manually revokes via `till auth_request revoke` until Wave 3).

### 5. `monitorUnsubscriber` interface

Defined in cleanup.go for forward compatibility with the eventual `processMonitor.Unsubscribe(actionItemID)` method. Today no concrete production type satisfies it — droplet 4a.23 (CLI bootstrap) will add the method to `*processMonitor` (or wrap in a small adapter) during constructor wiring. Out of scope for this droplet because monitor.go is committed and PLAN row 4a.22 says "(NEW only)" for paths.

The cleanupHook's `unsubscribeMonitor` field is wired through this interface in `newCleanupHook`; tests inject `&stubMonitorUnsubscriber{}`.

### 6. `newCleanupHook` validates dependencies

Mirrors the wrap shape used by `NewDispatcher` (dispatcher.go:152) and `processMonitor.Track` (monitor.go:234): non-nil dependency checks emit "dispatcher: invalid configuration: <dep> is nil" via `errInvalidCleanupDep`. Used `errors.New` rather than `fmt.Errorf("%w", ErrInvalidDispatcherConfig)` to keep the file dependency-light; wrap-vs-string is a refinement that doesn't block this droplet's contract. Tested via `TestNewCleanupHookValidatesDependencies` with three subtests.

### 7. Empty-ID short-circuit

`OnTerminalState(ctx, item)` returns nil without firing any step when `item.ID == ""`. Prevents the cleanup hook from accidentally nuking the empty-string keyhole in the lock managers (which IS a valid holder ID per the managers' opacity contract — locks_file.go:71-73). Documented in `OnTerminalState`'s doc-comment; tested via `TestCleanupEmptyActionItemIDIsNoop`.

### NIT 2 (A14) — `errors.Is` doc-vs-code drift

Round-1 `errInvalidCleanupDep` used `errors.New(...)` so the doc-comment claim that callers can detect misconfiguration via `errors.Is(err, ErrInvalidDispatcherConfig)` was FALSE — `errors.Is` only matches identity, not substring.

**Fix:** wrap with `fmt.Errorf("%w: %s is nil", ErrInvalidDispatcherConfig, name)`, mirroring the precedents in the dispatcher package:
- `dispatcher.go:152-158` (`NewDispatcher`): `fmt.Errorf("%w: svc is nil", ErrInvalidDispatcherConfig)` and `fmt.Errorf("%w: broker is nil", ErrInvalidDispatcherConfig)`.
- `monitor.go:235` (`processMonitor.Track`): `fmt.Errorf("%w: process monitor service is nil", ErrInvalidDispatcherConfig)`.

**Before:**
```go
import (
    "context"
    "errors"
    "sync"
    ...
)

func errInvalidCleanupDep(name string) error {
    return errors.New("dispatcher: invalid configuration: " + name + " is nil")
}
```

**After:**
```go
import (
    "context"
    "errors"
    "fmt"
    "sync"
    ...
)

func errInvalidCleanupDep(name string) error {
    return fmt.Errorf("%w: %s is nil", ErrInvalidDispatcherConfig, name)
}
```

The rendered string is identical (`"dispatcher: invalid configuration: <dep> is nil"` because `ErrInvalidDispatcherConfig`'s message is `"dispatcher: invalid configuration"`), but `errors.Is(err, ErrInvalidDispatcherConfig)` now returns true — making the package-level claim true.

**Test tightened:** `TestNewCleanupHookValidatesDependencies` previously string-matched only the per-dependency tail ("fileLocks", "pkgLocks", "monitor"), with a comment explicitly noting that a future shift to `fmt.Errorf` with `%w` should stay detectable. Round 2 added an explicit `errors.Is(err, ErrInvalidDispatcherConfig)` assertion to every subtest while preserving the substring tail-message guard. All three subtests pass — confirming the wrap is real.

No new test function added (R2 directive: "if it asserts only on string content, tighten it" — tightened in place). Total dispatcher tests unchanged at 105.
