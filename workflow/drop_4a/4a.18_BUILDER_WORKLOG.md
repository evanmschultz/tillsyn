## Design Choices

### 1. Walker is read-only on the tree (acceptance §5)

`treeWalker.EligibleForPromotion(ctx, projectID) ([]domain.ActionItem, error)` returns a snapshot of the eligible set without mutating any state. `Promote(ctx, item)` is a separate method on the same struct so the Wave 2.7 conflict detector can intercede between "eligible" and "promoted" — for example, to insert a runtime blocker when two siblings hit the same file lock. This was non-negotiable per the plan; the split is reflected in the package-doc comment as the load-bearing architectural invariant.

### 2. Asymmetry doc — children-complete vs blocked_by

The package overview in `walker.go` explicitly calls out that children-complete is NOT required for promotion to in_progress. Only `blocked_by` clearance is checked at promotion time. The children-complete invariant is enforced separately by `Service.MoveActionItem` when the destination state is complete (via `ensureActionItemCompletionBlockersClear`). The asymmetry exists because in_progress is a working state — children may legitimately be todo / in_progress concurrently, and "parent-before-children" cascade ordering is captured by `blocked_by` edges, not by children-complete.

### 3. Broker-ordering doc

Walker is invoked from broker events that fire AFTER the repo write (per 4a.15 publisher contract). The package overview documents this, explains how the broker's coalescing works, and notes that Wave 2.10's manual-trigger CLI invocation has the same read-after-write semantics because CLI invocations come from outside any in-flight repo transaction.

### 4. Eligibility predicate (acceptance §2)

Four-clause predicate, each documented:
1. `LifecycleState == StateTodo`.
2. Every entry in `Metadata.BlockedBy` resolves to an item in `StateComplete`.
3. Children-complete NOT required (asymmetry — see above).
4. Parent (if any) is in `StateInProgress` OR has `Persistent=true`. Project-root (`ParentID == ""`) is treated as having an in_progress parent per cascade design.

Defensive behaviors: missing BlockedBy reference → not-clear (skip); detached child (parent missing from listing) → not-clear (skip); self-reference in BlockedBy → naturally skips because the item's own state is StateTodo.

### 5. ErrPromotionBlocked typed error

`Promote` rewraps `domain.ErrTransitionBlocked` as `ErrPromotionBlocked` using `errors.Join`, so `errors.Is(err, ErrPromotionBlocked)` AND `errors.Is(err, domain.ErrTransitionBlocked)` both match. The double-target match is what lets the conflict detector distinguish "planner-side blocker" (recoverable, walk again next tick) from "infrastructure failure" (non-recoverable, surface up). Other service errors propagate WITHOUT the ErrPromotionBlocked sentinel — verified by `TestWalkerPromoteSurfacesNonTransitionErrors`.

### 6. Column resolution — dispatcher-local helper

`columnIDForLifecycleState` + `canonicalStateSlug` mirror the canonical normalizer used by `internal/adapters/server/common.normalizeStateLikeID`. The walker cannot import the adapter helper (would invert the package layering), and the existing `internal/app.lifecycleStateForColumnID` is package-private and goes the wrong direction (column ID → state, not state → column ID). The local helper is small, fully tested, and rejects legacy aliases (done, completed, progress, doing, in-progress) by returning empty — matching the strict-canonical behavior of the adapter version.

### 7. Walker service interface

`walkerService` declares only the three methods walker actually uses (`ListColumns`, `ListActionItems`, `MoveActionItem`). `*app.Service` will satisfy it once 4a.23 wires the constructor; the test stub satisfies it directly. A compile-time assertion via `walkerServiceCompileCheck` pins the interface to a concrete type so future drift in service signatures fails at compile rather than at the constructor seam.

### 8. Constructor

`newTreeWalker(svc walkerService) *treeWalker` is intentionally unexported. The dispatcher constructor wiring lives in 4a.23 — when that lands, `dispatcher.go` will have a `NewDispatcher` extension that constructs and stores `*treeWalker`. Today the symbol is package-internal so other dispatcher droplets can read it without crossing an export boundary.

## Test Summary

19 distinct test functions in `walker_test.go`; 30 total test cases including subtests. Five required scenarios delivered by name:

- `TestWalkerFindsTodoItemWithClearedBlockers` — happy path: blocker complete, parent in_progress → eligible.
- `TestWalkerSkipsTodoItemWithUnmetBlockedBy` — table-driven over `todo`/`in_progress`/`failed` blocker states; assertion is "candidate not in eligible set" so unrelated eligible siblings (parents that pass) don't false-fail the test.
- `TestWalkerSkipsTodoItemWhoseParentIsTodo` — pins parent under in_progress grandparent so the parent itself cleanly cascades; only the candidate child's exclusion is asserted.
- `TestWalkerPromotesEligibleItem` — verifies the dispatcher → service contract: ID forwarded, in_progress column resolved by canonical slug match (not hardcoded), position preserved.
- `TestWalkerPropagatesTransitionBlocked` — `errors.Is(err, ErrPromotionBlocked)` AND `errors.Is(err, domain.ErrTransitionBlocked)` both true.

Additional coverage tests (defensive corners, contracts):

- `TestWalkerPromotesPersistentParentChild` — Persistent=true parent must not gate descendants regardless of state.
- `TestWalkerPromotesRootItem` — `ParentID == ""` treated as in_progress parent.
- `TestWalkerSkipsItemWithMissingBlockedByReference` — phantom BlockedBy → conservative skip.
- `TestWalkerEligibleForPromotionSkipsEmptyProjectID` — defensive no-op.
- `TestWalkerEligibleForPromotionPropagatesListError` — infra errors not swallowed.
- `TestWalkerPromoteRejectsMissingInProgressColumn` — descriptive error, no MoveActionItem call.
- `TestWalkerPromoteRejectsEmptyProjectID` — guard.
- `TestWalkerPromoteSurfacesNonTransitionErrors` — non-transition errors propagate without ErrPromotionBlocked sentinel.
- `TestNewTreeWalkerNilServiceCallSafe` — nil-receiver guards return ErrInvalidDispatcherConfig.
- `TestColumnIDForLifecycleStateMatchesCanonicalNames` — column lookup over all five states.
- `TestCanonicalStateSlugRejectsLegacyAliases` — strict-canonical alias rejection.
