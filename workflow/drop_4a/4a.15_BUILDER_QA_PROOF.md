  Receiver, parameter list, and return type match the spec exactly.
- **Re-subscribe loop** (lines 67–94): `for { ... }` calls

### AI-6 — Broker-close-yields-error contract documented

**Verdict: PROVEN.**

Evidence: `broker_sub.go` lines 29–40 — multi-paragraph doc-comment paragraph
explicitly titled "Broker-close contract (mitigates the 'spin on Sequence=0
forever' attack raised by plan-QA falsification)". The block:

1. Pins the current contract: in-process broker has no `Close`; ctx.cancel is
   the only supported termination signal (lines 30–33).
2. Specifies the future-implementation requirement: "any future broker
   implementation that closes its underlying channel MUST surface a non-nil,
   non-context error from Wait" (lines 33–36).
3. Codifies the loop's defensive behaviour: any non-context error from Wait
   exits the loop, so a buggy `(zero-event, nil-error)` return cannot trap
   the goroutine (lines 36–38).
4. Cross-references the test pin: `context.DeadlineExceeded`-while-`ctx.Err`-
   nil is treated as a benign re-subscribe trigger (lines 38–40).

This satisfies the spec's "Plan-QA falsification attack to address: 'broker
re-subscribe loop spins on Sequence=0 forever after broker close.' Mitigation:
document broker-close-yields-error contract, test exercises it" requirement
verbatim.

### AI-7 — `afterSequence` cursor handling documented

**Verdict: PROVEN.**

Evidence: `broker_sub.go` lines 14–22 — paragraph beginning "Cursor handling
(afterSequence): the loop seeds its cursor from `LiveWaitBroker.Latest` so it
does not replay history seen before the dispatcher started." The doc-comment
also forward-references "Wave 2.5's tree walker reads the same event stream
and relies on this monotonic ordering" — pinning the cross-droplet contract
the spec calls out.

The implementation (lines 62–65 + 85–87) matches the documentation exactly:

- Seed: `cursor = latest.Sequence` if `Latest` returned `(_, true, nil)`;
  otherwise `cursor = 0`.
- Advance: `if event.Sequence > cursor { cursor = event.Sequence }` BEFORE
  the delivery select (so a slow consumer cannot stall the broker's
  monotonic ordering).
