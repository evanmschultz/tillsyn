### 2.1 NIT carry-over from 4a.1 falsification §1.1 — `mage -h <alias>` returns "Unknown target"

**Source:** `workflow/drop_4a/4a.1_BUILDER_QA_FALSIFICATION.md` §1.1 — Mage's `-h` flag iterates declared targets (camelCase), not the alias map. So `mage -h fmt` / `mage -h format-check` / `mage -h format-path` all return "Unknown target", but `mage -h format` works.

**Routing recommendation from that NIT:** "Surface as a Drop 4a Wave 0 closeout note for W0.4's CONTRIBUTING.md docs — the 'command list' should explicitly cite `mage format-check` and `mage format-path` (kebab-case) rather than the camelCase target names, so devs reading docs hit the alias surface."

**Evidence:** `CONTRIBUTING.md:74`

```
For `mage -h <target>` help lookup use the canonical camelCase name (e.g. `mage -h format`); the kebab-case forms (`format-check`, `install-hooks`) work for invocation but Mage's help resolver doesn't follow aliases.
```

This explicitly documents:
- The execution-vs-help split: kebab-case works for invocation, camelCase needed for help.
- A worked example (`mage -h format`).
- The two specific kebab-case targets that exhibit the quirk (`format-check`, `install-hooks`).
- The root cause ("Mage's help resolver doesn't follow aliases").

NIT-routing recommendation #1 from 4a.1 (the implicit "doc the camelCase-help-vs-kebab-execution split") fully addressed.

**Conclusion:** PASS.

### 2.2 NIT carry-over from 4a.2 falsification §1.1 — bare `mage` PATH dependency

**Source:** `workflow/drop_4a/4a.2_BUILDER_QA_FALSIFICATION.md` §1.1 — POSIX `sh` hooks invoke `mage` directly, so dev's hook-execution `PATH` must include the `mage` binary location (typically `~/go/bin`).

**Routing recommendation from that NIT:** "Surface as a Drop 4a Wave 0 closeout NIT for W0.4's CONTRIBUTING.md docs — the 'Local Git Hooks' section should mention 'ensure `mage` is on your `PATH` before running `mage install-hooks`, since git hooks inherit a minimal environment.'"

**Evidence:** `CONTRIBUTING.md:68-72`

```
Ensure `mage` is on your `PATH` so the hooks find it. Install Mage locally with:

```bash
go install github.com/magefile/mage@v1.17.0
```
```

This explicitly documents:
- The PATH requirement ("Ensure `mage` is on your `PATH` so the hooks find it").
- The install command (`go install github.com/magefile/mage@v1.17.0`) — cited verbatim from the legacy "Install Mage locally with:" content, preserved across the section rewrite.

The phrasing is slightly tighter than the routing recommendation suggested ("hooks inherit a minimal environment" was elided), but the actionable content — ensure `mage` is on PATH — is preserved. **PASS.**
