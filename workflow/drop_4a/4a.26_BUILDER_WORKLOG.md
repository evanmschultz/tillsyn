## Design Choices

### Defense-in-Depth Domain Assertion

`(*AuthRequest).Approve` rejects with an `ErrInvalidID`-wrapped error when `resolvedByType == ActorTypeAgent` AND any of the three approver-identity strings is empty. User (dev-TUI) approvals stay exempt — empty audit fields are the dominant case for the legacy path. Rationale: the W3.1 gate already enforces "all four or none" at the app-service boundary; the domain assertion is a second-layer guard against future plumbing bugs that would silently strand an audit-trail row with a partial identity. The test `TestApproveRejectsAgentApprovalWithEmptyAuditFields` covers four cases (all-empty, missing-instance-id, missing-lease, all-populated) plus the user-empty exemption.

### ALTER Idempotency

Three new `ALTER TABLE auth_requests ADD COLUMN ... TEXT NOT NULL DEFAULT ''` statements appended to the existing `alterStatements` block in `ensureAuthRequestSchema`. The pre-existing `isDuplicateColumnErr` helper (matches "duplicate column name" in the error string) already catches re-runs and skips silently. `TestApproveAuthRequestDevTUIPathLeavesAuditEmpty` verifies idempotency by running `ensureAuthRequestSchema` a second time on a sqlite DB that already has the columns; the function must return `nil`.

### JSON `omitempty` Semantics

The three new `AuthRequestRecord` fields carry `json:"approving_*,omitempty"`. Effect:

- **Orch-self-approval path**: all three keys appear in the JSON surface with their populated values. The MCP client / TUI inventory can read them as audit-trail evidence.
- **Dev-TUI / system path**: all three values are empty strings; `omitempty` elides them entirely from the JSON surface. The dev-TUI inventory is byte-identical to its pre-W3.3 shape, preserving the additive promise from W3.1's docstring.
- **Pending / denied / canceled / expired states**: same as dev-TUI — empty string + `omitempty` elision.

`TestAuthRequestRecordSurfacesApprovingOrchIdentity` verifies both branches: the orch-approved branch asserts the JSON encoding contains all three keys with the populated values; the dev-TUI counter-case asserts the JSON encoding contains none of the three keys (omitempty elision).
