## 3. Falsification Attempts

### 3.1 Could a builder commit gofumpt-dirty code with this hook installed?

The hook calls `mage format-check`, which calls the private `formatCheck()` (4a.1 verified), which lists tracked Go files and runs `gofumpt -l` on them. If any file is dirty, `formatCheck()` returns `fmt.Errorf("gofumpt required for:\n%s", out)`, which mage propagates as a non-zero exit. The `if ! mage format-check; then ... exit 1` block then fires the hint and propagates the failure to git, blocking the commit.

The only path to a dirty commit is `git commit --no-verify`, which is git-native bypass and explicitly mentioned in the hint as the deliberate escape hatch. Acceptable.

**No counterexample.**

### 3.2 Could `set -eu` cause the script to exit before the hint prints?

`set -e` aborts on any command exiting non-zero, **except** commands inside an `if`/`while`/`until`/`||`/`&&` construct (POSIX rule). The hook's `if ! mage format-check` puts the failing command inside an `if`, so `set -e` does NOT abort the script when `mage format-check` exits non-zero — control flows to the `then` branch, prints the hint, then `exit 1` propagates failure.

`set -u` aborts on unset variable expansion. The hook references no shell variables. Safe.

**No counterexample.**

### 3.3 Could the printf string contain shell metacharacters that misbehave?

The hint string is wrapped in `"..."` (double quotes), inside which `"`, `$`, `` ` ``, and `\` would be special. The actual content uses single quotes (`'mage format'`, `'--no-verify'`) and an em-dash — none of which are shell-special inside double quotes. `printf '%s\n' "..."` is the safe form (format string is a literal, content goes through `%s`), so even if the content contained a `%` it would not be interpreted as a format directive.

**No counterexample.**

### 3.4 Could the hook fail silently if `mage` is not on `$PATH`?

If `mage` is not installed, `if ! mage format-check` would route the missing-binary error (typically `command not found`, exit 127) into the `then` branch, which prints the format-check hint. The hint would be misleading (it'd suggest `mage format`, but the dev doesn't have `mage` at all), but the commit would still be blocked — fail-safe direction.

This is a real but acceptable edge case: a dev who installs the hooks via `mage install-hooks` (W0.3) necessarily has mage available; a dev without mage would never reach the hook in normal flow. The plan does not require defensive `command -v mage` checking, and adding it would be out of scope. Routed for awareness, not as a finding.

**No counterexample relative to spec.**

### 3.5 Could the install comment placement (line 2, after shebang) violate "top-of-file"?

POSIX shebang interpretation requires `#!` to be at byte 0 of the file. Any comment before the shebang would break shebang resolution on stricter kernels. "Top-of-file" in the spec must therefore mean "the first comment in the file" — which is line 2 in both hooks. No reasonable alternative placement exists.

**No counterexample.**

### 3.6 Could `core.hooksPath = .githooks` fail to find the hook on Windows?

Windows git honors `core.hooksPath` and resolves relative paths against the worktree root. POSIX shebang on Windows requires git-bash (`bash.exe`) to be on PATH, which is the standard git-for-windows install. The Wave 0 plan and the existing `CONTRIBUTING.md` already note Windows as a supported dev platform via WSL or git-bash; the hook will run there unmodified. Out of scope for QA — relevant for W0.4 docs.

**No counterexample.**

### 3.7 Could the staged-but-uncommitted state drift before commit?

The QA review reads the working-tree files directly. If the builder amends the staged content between QA and commit, the QA artifact's blob hashes (`55d182b1...` and `59dc051c...`) would no longer match — but that's an orchestrator-side commit-discipline concern, not a QA finding. The blob hashes recorded in §2.1 above are the audit anchor.

**No counterexample.**

**No unmitigated counterexamples found.**
