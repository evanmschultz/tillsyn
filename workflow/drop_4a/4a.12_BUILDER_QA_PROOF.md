## 2. Design-Choice Reviews

### 2.1 Choice A — `NewProject` co-existence with `NewProjectFromInput`

**Spec said:** "or extended argument list — builder picks; `ProjectInput` struct is more readable per `ActionItemInput` precedent."

**Builder did:** Both. New `NewProjectFromInput` is the canonical surface; legacy 4-arg `NewProject(id, name, description, now)` survives as a thin wrapper.

**Faithfulness audit:**

- Wrapper at `internal/domain/project.go:136-142` builds `ProjectInput{ID: id, Name: name, Description: description}` and calls `NewProjectFromInput`. All 6 new fields default to `""` in the struct literal.
- Validation behavior on empty inputs: `Language=""` is in the closed enum (passes); empty `RepoBareRoot` / `RepoPrimaryWorktree` bypass the abs-check (`if repoBareRoot != ""` guard); free-form trim-only fields trim empty to empty.
- **No silent data loss:** wrapper accepts only `id` / `name` / `description` and forwards them verbatim. Legacy callers that didn't carry the 6 new fields couldn't lose data they never had.
- **No silent validation rejection:** every default-empty value passes the new validation paths.
- Caller migration completeness: only one non-test caller (`internal/app/service.go:288`) was migrated to `NewProjectFromInput`; the 27+ test-file callers stay on `NewProject` and continue to work. Future test code that needs the new fields can opt-in via `NewProjectFromInput` (already happening at e.g. `internal/domain/domain_test.go:280`, `internal/app/snapshot_test.go:1268`, `internal/adapters/storage/sqlite/repo_test.go:875`).

**Verdict:** PASS. Wrapper is faithful, canonical surface is correctly identified for new-field-aware callers, no data loss, no validation surprise. The choice respects YAGNI on test churn (avoids 27+ unrelated test edits to pass `ProjectInput{}`-shaped args that all default to identical legacy behavior).

### 2.2 Choice B — Idempotent `ALTER TABLE projects` migration extension

**Pre-MVP rule:** "no migration logic in Go, no `till migrate` CLI, no `ALTER TABLE`" — the orchestrator's intent is fresh-DB-on-schema-change.

**Builder claim:** the addition is **consistent extension** of an existing `ALTER TABLE projects` precedent (`metadata_json`), NOT net-new infrastructure.

**Audit of the precedent:**

- `internal/adapters/storage/sqlite/repo.go` HEAD (pre-builder-edit) line 470 contains: `r.db.ExecContext(ctx, \`ALTER TABLE projects ADD COLUMN metadata_json TEXT NOT NULL DEFAULT '{}'\`)` with `isDuplicateColumnErr(err)` idempotency. The exact same pattern exists for `attention_items.target_role` (`:473`), and a 13-statement `workItemAlterStatements` slice at `:476-490` for `action_items` columns. **This is the established pre-Drop-4a precedent.**
- Builder's addition at `:479-495` (post-edit) is a 6-statement `projectsAlterStatements` slice with the same `isDuplicateColumnErr` idempotency, placed adjacent to the existing `metadata_json` ALTER (line 476 post-edit). The pattern is mechanically identical to the already-shipped `workItemAlterStatements`.
- **Net-new vs extension classification:** this is **extension**. No new function, no new infrastructure — six statements appended to an existing migration block that already runs on every startup.

**Counterfactual — what would the builder have done if they'd skipped the ALTER?**

- Fresh DBs (the `mage ci` test path uses `t.TempDir()` databases) would still pick up the 6 new columns via the `CREATE TABLE IF NOT EXISTS` path. **`mage ci` would still pass.** The plan's "DB action: dev DELETEs `~/.tillsyn/tillsyn.db` BEFORE `mage ci`" instruction confirms this is the expected baseline.
- Legacy DBs in dev's hands (any pre-existing `~/.tillsyn/tillsyn.db`) would silently miss the columns and fail at INSERT-time on `CreateProject`. The dev would need to fresh-DB.
- Builder's ALTER block is **defense-in-depth for legacy DBs in the wild** — it doesn't violate the plan, it just exceeds the minimum.

**Risk assessment:**

- On a fresh DB, the ALTER block runs against a CREATE TABLE that already has the columns → returns `duplicate column` error → caught by `isDuplicateColumnErr` → no-op.
- On a legacy DB (pre-Wave-1.8), the ALTER block adds the 6 columns successfully.
- On a partially-migrated DB (some ALTERs ran, others didn't), each ALTER succeeds or no-ops idempotently.
- No data corruption path identified.

**Verdict:** PASS. The precedent is real (multiple ALTER blocks already exist at HEAD; the `metadata_json` ALTER cited by the builder is on `projects` specifically); the addition is mechanically identical to existing patterns; the idempotency is preserved; the pre-MVP rule is bent consistently with how it has already been bent for legacy schema rotation. **Bordeline accepted as consistent extension.**

**Note for closeout:** if the orchestrator wants to enforce strict pre-MVP rule in the future, the cleanup would be a refinement that DELETES all `ALTER TABLE` blocks once the dogfood DB stabilizes — applies uniformly to `metadata_json`, `target_role`, `action_items` (13 statements), and the new `projects` (6 statements). That refinement is out of scope for this droplet.

## 4. Possible Gaps Considered (And Dismissed)

### 4.1 Could `UpdateProject` clobber existing project values?

**Concern:** value-typed `UpdateProjectInput` means a caller sending `Language=""` would CLEAR a previously-set `Language="go"`.

**Mitigation:** the spec EXPLICITLY chose this shape. Doc comments at `internal/app/service.go:415-417` and `internal/adapters/server/common/mcp_surface.go:71-73` document the contract: "Callers that want to preserve existing values must read the project first and pass them through unchanged." This is intentional admin-driven design, not a bug. Pointer-sentinels were considered and rejected by the planner per "the Project surface is admin-driven (not agent-driven), so explicit-empty intent is rare."

**Verdict:** Acceptable as designed. Documented contract.

### 4.2 Could `filepath.IsAbs` be platform-incorrect?

**Concern:** `filepath.IsAbs` uses OS-specific semantics. On Windows it accepts `C:\...` but rejects `/foo`. A multi-platform Tillsyn deployment could see different validation behavior.

**Mitigation:** Tillsyn is currently macOS/Linux dev-focused (per project README and Hylla artifact ref pointing at a personal GitHub clone). Windows support is not in scope for Drop 4a. The spec's "absolute path required" rule deferring to `filepath.IsAbs` is the canonical Go idiom for this check.

**Verdict:** Acceptable. Note: if Tillsyn ever ships Windows support, the path validation would be platform-aware — that's correct behavior, not a bug.

### 4.3 Could the MCP `WithString` schema accept `null` and panic?

**Concern:** `mcp.WithString` declares the JSON schema; client-supplied `null` values could break the `args` decode.

**Mitigation:** I read `extended_tools.go:455-475` — the `args` struct uses plain `string` fields (not `*string`). `encoding/json` decodes `null` to the zero value (empty string) for non-pointer fields. No panic path.

**Verdict:** Acceptable. Same as the existing `Name` / `Description` handling.

### 4.4 Could the legacy snapshot test miss a new field?

**Concern:** `TestSnapshotProjectFirstClassFieldsLegacyFormatCompatibility` deserializes a legacy JSON without the 6 fields. If the test only checks 6 fields are empty but the round-trip silently injects nonsense, the test would still pass.

**Mitigation:** the test asserts BOTH the snapshot fields AND the hydrated domain fields are empty. The decode path (`snapshot.go:1280-1299`) calls `strings.TrimSpace(p.HyllaArtifactRef)` — `strings.TrimSpace("") == ""`, so the path is mechanically clean. The test exercises both output points.

**Verdict:** Acceptable.

### 4.5 Could the new `projects` columns interact badly with `.tillsyn/template.toml` baking?

**Concern:** Drop 3 added `KindCatalogJSON` baked-at-creation behavior. The new fields are INSERT-time but `bakeProjectKindCatalog` runs after `UpdateDetails` in `CreateProjectWithMetadata` (`service.go:323`). Could this re-overwrite the new fields?

**Mitigation:** `bakeProjectKindCatalog` mutates only `project.KindCatalogJSON` (verified via Hylla `hylla_search_keyword` "bakeProjectKindCatalog"). It does not touch the 6 new fields.

**Verdict:** Acceptable. No interaction.
