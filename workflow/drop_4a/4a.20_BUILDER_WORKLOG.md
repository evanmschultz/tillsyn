# 4a.20 — CONFLICT DETECTOR (SIBLING OVERLAP) — Builder Worklog

## Scope

Wave 2.7 of Drop 4a Dispatcher Core. Add the sibling-overlap conflict detector that intercedes between the tree walker's eligibility decision (4a.18) and `Promote`. The detector reports every (sibling, kind, value) overlap on `paths` / `packages` for items sharing the same `ParentID`, plus an `InsertRuntimeBlockedBy` writer that mutates `Metadata.BlockedBy` and raises an `AttentionKindBlocker` attention row.

New files only:

- `internal/app/dispatcher/conflict.go`
- `internal/app/dispatcher/conflict_test.go`

`internal/app/dispatcher/dispatcher.go` was NOT touched (parallel droplets 4a.17 + 4a.21 are editing it; constructor wiring lands in 4a.23).

## Design Choices

### Sibling-only scope (acceptance §7)

`DetectSiblingOverlap` enforces same-`ParentID` at the boundary as a defensive filter rather than relying on caller pre-filtering. Cross-subtree cousin overlap is explicitly out of scope for Wave 2.7 — it's already covered by the runtime `fileLockManager` / `packageLockManager` at acquire time, so cross-subtree overlap cannot race agents within a single promotion sweep. Doc comment makes the limitation explicit and points at Drop 4b for the eventual cross-subtree expansion. The reserved `ctx` parameter on `DetectSiblingOverlap` is intentional — it's where Drop 4b will plug a service lookup once cousin sets need to be resolved beyond the caller-provided slice.

### Tie-break (acceptance §8)

Exposed as a free function `TieBreakSibling(a, b domain.ActionItem) domain.ActionItem` rather than a method, because the caller (4a.10's manual-trigger CLI; future continuous-mode dispatcher) needs to pick between two equally-eligible siblings without needing an instance. Order: `Position` ascending wins; on ties, ID lexically ascending wins. Both inputs assumed same-`ParentID`; the function does NOT validate that — it's a leaf comparator and the detector already filters by `ParentID` before any caller would reach this point.

### Permanent runtime blocker (acceptance §9)

Once `InsertRuntimeBlockedBy` succeeds, the new `BlockedBy` entry is permanent. Doc comment lays out the reasoning: (a) the runtime blocker IS the dependency edge, mirroring the planner's append-only static `blocked_by`; (b) any cleanup-on-completion hook would race the next eligibility walk and briefly re-open the conflict window; (c) eligibility checks treat completed blockers as cleared, so a stale row is harmless. This is consistent with the walker's eligibility predicate documented in 4a.18.

### Idempotency (acceptance §4)

Idempotent insert is a TRUE no-op: when `siblingID` is already in `item.Metadata.BlockedBy`, neither `UpdateActionItem` nor `RaiseAttentionItem` is invoked. This matches `fileLockManager.Acquire`'s same-holder semantics — a caller can retry the conflict-detection sweep without accumulating duplicate attention rows.

### Update + attention coupling (intentional non-atomicity)

The `BlockedBy` update lands first; the attention raise follows on success. If `RaiseAttentionItem` fails the runtime blocker is still in place (the dependency edge is the load-bearing part) and the wrapped error surfaces to the caller. Doc comment states this trade explicitly: simpler call shape vs strict atomicity, and the orchestrator already polls attention items so a missed raise surfaces on the next sweep.

### Dedupe key

`(siblingID, kind, value)` triples are deduped via a string key with `\x00` separators. Avoids an O(N²) scan of the result slice on every append. Keys are deterministic and the final slice is sorted by `(SiblingID, OverlapKind, OverlapValue)` so test assertions can compare slices without re-sorting.

### Narrow service interface

`conflictDetectorService` declares only the two methods the detector calls (`UpdateActionItem`, `RaiseAttentionItem`). `*app.Service` satisfies it implicitly; the test stub injects deterministic results. This mirrors `walkerService` (4a.18) and `dispatcher`'s `actionItemReader` for the same package-internal patterns.

## Hylla Feedback

Two relevant calls; one miss + one tooling-side rough edge.

### Miss 1 — keyword search returned empty for `UpdateActionItem` and `RaiseAttention`

- **Query**: `hylla_search_keyword` with `query="UpdateActionItem"`, `fields=["content"]`, `node_type="block"`, `limit=10`. Repeated for `RaiseAttention`, `CreateAttention`.
- **Missed because**: First two queries returned zero results; subsequent queries returned an explicit `enrichment still running for github.com/evanmschultz/tillsyn@main` error. The artifact ref was being re-enriched mid-session, so keyword search couldn't see the symbols. Hybrid `hylla_search` returned a `field must be summary, content, or docstring` error when I tried passing `fields` plus `field` together — schema confusion (one of those is singular vs plural and they conflict).
- **Worked via**: `/usr/bin/grep -n "func (s \*Service) UpdateActionItem\|func (s \*Service) RaiseAttentionItem" internal/app/*.go` to find both signatures. `Read` for the input-struct definitions at the resolved line numbers.
- **Suggestion**: Hylla should expose enrichment-state on artifact list/info endpoints so callers can pre-check rather than discover mid-search. Also worth distinguishing "no results" from "enrichment running" in the keyword response shape — the first two of three identical-shape calls returned empty rather than the enrichment-running error, which is a transient inconsistency.

### Ergonomic — `hylla_search` `fields` vs `field` naming clash

The hybrid search tool has both a `fields []string` (search-targets) and an embedded `field` (vector-embedding-target) parameter. Passing only `fields` returns `field must be summary, content, or docstring` — the validator is checking the singular `field` even when the user only supplied `fields`. Either auto-select a default `field` from the first valid `fields` entry, or rename one of them to disambiguate.

### Mage `test-pkg` argument-handling

Not Hylla, but worth raising as a Drop 4a refinement: `mage test-pkg internal/app/dispatcher` (relative path without `./`) silently fails with `[PKG FAIL]` because `go test` swallows the build-resolution error before the json stream starts. The mage helper could either prepend `./` when the argument is a directory, or surface stderr more visibly. (Logged as a refinement candidate; not blocking 4a.20.)
