## Design Choices

### Concurrent-safe `Handle` + `processMonitor`

- One `processMonitor` services many concurrent `Track` calls. The monitor's only shared state is a `tracked map[string]*Handle` (used by Drop 4b's dashboard); guarded by `sync.Mutex`. Per-Handle state — `*exec.Cmd`, the cached outcome, the close signal — is owned by that Handle alone.
- Each `Handle` runs **exactly one** `cmd.Wait` goroutine, started inside `Track` after `cmd.Start` succeeds. The goroutine closes `done` on exit so any number of `Wait()` callers observe the same cached outcome.
- `Wait()` and `Close()` are both idempotent. `Wait` reads the cached outcome under `resultMu`. `Close` uses `sync.Once` to send a single best-effort `cmd.Process.Kill` and then waits on `done`. Calling `Close` after natural process exit is a no-op (the process is gone; the kill error is intentionally swallowed).
- **Goroutine-leak-free contract:** once any caller has observed `Wait` returning OR `Close` returning, the per-Handle goroutine is gone. The map entry is also gone — the goroutine `delete`s itself before closing `done` (deferred under `mu`).

### State-conflict guard

The crash-handling path refetches the action item via `Service.GetActionItem` BEFORE applying the failed transition. If `current.LifecycleState == StateComplete`, the monitor short-circuits — no `MoveActionItem`, no `UpdateActionItem`. Two reasons:

1. `Service.MoveActionItem` already rejects transitions FROM terminal states (`internal/app/service.go:1003`); calling it would surface `ErrTransitionBlocked` as line noise.
2. The semantic is "the agent already self-reported success before its process exited" — the right answer is no-op, not a service-level failure. Handled in-monitor for clean test signal.

`TestMonitorStateConflictGuardSkipsCompleteItem` pins this: pre-seed item in `StateComplete`, run an `exit1` fake agent, assert `moveCalls == 0 && updateCalls == 0`.

### `metadata.failure_reason` carrier choice

`ActionItemMetadata` (closed struct in `internal/domain/workitem.go:143`) has NO `FailureReason` field. The spec says `metadata.failure_reason = "agent process crashed: <signal/exit code>"` — and PLAN.md §7 line 300 explicitly defers structured `failure` typing to Drop 4b ("free-form string for 4a").

Picked `BlockedReason` as the 4a carrier — the only free-form failure-context string slot on the closed metadata struct. The "agent process crashed:" prefix is load-bearing for tests so future Drop 4b refactor (renaming to a proper `FailureReason` field) finds them and migrates atomically. Documented loudly in `monitor.go:applyCrashTransition`.

`metadata.outcome = "failure"` lands cleanly because `ActionItemMetadata.Outcome` (`workitem.go:163`) is exactly the right field.

Production code in `monitor.go` does NOT shell out to `go`. `mage test` already shells out to the `go` toolchain underneath, so the spirit of "no agent / no production code path bypasses mage" is preserved.

### Cross-platform signal extraction

`signalNameFromState` extracts a human-readable signal name:
- Unix: cast `state.Sys()` to `syscall.WaitStatus`; if `Signaled()`, return `Signal().String()` (e.g. "killed", "terminated").
- Windows / cast-fail fallback: parse `state.String()` for a `signal: …` prefix.

The signal-kill test (`TestMonitorSignalKilledMarksFailed`) skips on Windows because the cascade dispatcher targets Unix in Drop 4a.

### Local `updateActionItemInput`

Defined a tiny package-local `updateActionItemInput` struct (just `ActionItemID` + `Metadata *ActionItemMetadata`) on the `monitorService` interface rather than importing `app.UpdateActionItemInput`. Two reasons:

1. The dispatcher package will be wired into `*app.Service` in 4a.23; an `app` import here would be premature and forces test stubs to instantiate the full `UpdateActionItemInput` shape.
2. The narrow shape names exactly the fields the monitor populates — `Metadata` only — so anyone reading the monitor's interface does not misread it as "monitor can also tweak title/description/paths."

When 4a.23 wires `*app.Service`, the binding adapter is one line: take the local input, splat it into the full `app.UpdateActionItemInput`. Same pattern walker+conflict use today (narrow `walkerService` view of `*app.Service`).
