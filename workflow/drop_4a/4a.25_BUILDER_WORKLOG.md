### Non-obvious design decisions

1. **Pointer-bool with nil-means-enabled (NOT plain bool with default-true semantics).** The spec mandates this; the rationale is the W3.2 falsification-attack-3 hedge: a plain `bool` zero-values to `false`, so any legacy project's metadata round-trip through JSON-decoded SQLite storage would silently flip the toggle off and break every existing orch-self-approval cascade on the next ingest. The pointer preserves the absent-vs-explicit-false distinction across JSON encode/decode and across the SQLite `metadata_json` column (verified by the round-trip test's pointer-nil-ness assertion).

2. **Toggle check is Reject 0, BEFORE all W3.1 rejects.** The check sits at the very top of `checkOrchSelfApprovalGate` so a disabled project rejects with `ErrOrchSelfApprovalDisabled` rather than `ErrAuthorizationDenied`. This keeps observability sharp — operators monitoring auth-request rejections can distinguish "project opted out" from "role/path mismatch" without re-running the request through different fixture conditions. The `non_steward_orch` sub-test asserts the rejection does NOT also wrap `ErrAuthorizationDenied`.

3. **Total backstop including STEWARD cross-subtree path.** STEWARD is the canonical cross-subtree-exception holder per W3.1, but the W3.2 toggle is meant as a hard project-level opt-out — including STEWARD. The `steward_cross_subtree` sub-test sets up the full Persistent + Owner=="STEWARD" ancestor fixture from W3.1, flips the toggle, and asserts the cascade still rejects. This is the "total" semantics the dev called out.

4. **Project-id resolution: req.ProjectID with path-fallback, skip the global sentinel.** Auth requests carry both a `ProjectID` field and a parsed `Path`. The gate tries `req.ProjectID` first (cheapest), falls back to parsing `req.Path` (the W3.1 gate uses the same fallback for path resolution), and skips the lookup entirely when the resolved id is the global-auth sentinel `domain.AuthRequestGlobalProjectID`. The sentinel project is internal-only (created by `ensureGlobalAuthProject`) and never carries user-facing metadata; trying to gate cross-project orch-self-approval against it would surface confusing error paths.

5. **No mutation-pipeline rewrite on the new field.** The MCP surface (`CreateProjectRequest`/`UpdateProjectRequest.Metadata`) is already typed `domain.ProjectMetadata`. The SQLite layer round-trips metadata via `json.Marshal/Unmarshal` over the `metadata_json` column, so the new pointer-bool field flows through transport→adapter→service→repository with zero touch. This is exactly the tasteful "additive change" that the W3.2 spec was set up to enable — proven by the adapter round-trip test driving the toggle through `repo.UpdateProject` and observing it on the next `repo.GetProject`.

6. **Test scaffolding flips the toggle via repo.UpdateProject, not via adapter.UpdateProject.** The adapter's `UpdateProject` runs through `withMutationGuardContext` which requires a valid lease tuple — overhead we don't need to exercise in a gate-focused test. Both new test families (service-level + adapter-level) use the repo path with a comment noting the production callers go through `till.project(operation=update)`. The point of the adapter test is to verify the gate honors the persisted toggle; mutation-pipeline coverage is out of W3.2's scope.

7. **DB action note for the dev (per spec §8 + Pre-MVP rule).** No SQL migration is shipped — the new field rides as JSON inside the existing `metadata_json` column. However, per the project's "No Migration Logic Pre-MVP" memory + "Pre-MVP fresh-DB on schema/state-vocab change" rule, the dev should fresh-DB `~/.tillsyn/tillsyn.db` before testing W3.2 if their DB carries pre-W3.2 rows whose autent-side enum or auth-request schema changed in 4a.24's enum-widening drop. Pure W3.2 doesn't strictly require a fresh DB (the JSON field decodes with `nil` on absence, which is exactly the post-W3.2 enabled-default), but coupling the test to the W3.1+W3.2 stack means the dev's local DB needs to be in sync with the post-4a.24 enum.
