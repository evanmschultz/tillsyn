### 1.1 Attack: Coverage rule bypass via direct `repo.UpdateActionItem` — REFUTED (BY DESIGN)

**Hypothesis.** Builder enforces the coverage rule (`len(Paths)>0 → len(Packages)>0`) at `domain.NewActionItem` (create) and `Service.UpdateActionItem` (post-apply), but the repo layer accepts coverage-violating writes. A caller could persist a coverage-violating state via `repo.UpdateActionItem` directly.

**Evidence.**
- `internal/adapters/storage/sqlite/repo.go:1277-1336` — `Repository.UpdateActionItem` does NOT call domain validation; it writes the raw `domain.ActionItem` struct fields.
- `internal/adapters/storage/sqlite/repo_test.go:2912-2926` — the new test EXPLICITLY exercises this path: `clearTarget.Packages = nil` while `Paths` was populated, then `repo.UpdateActionItem(ctx, clearTarget)` succeeds. Reload confirms the coverage-violating row persists.
- The test comment documents this as intentional: *"The repo layer does not re-validate the domain coverage invariant on update (UpdateActionItem writes raw fields), so clearing Packages while Paths remains populated is permitted at the storage layer — domain invariant enforcement happens in Service.UpdateActionItem."*

**Trace.**
1. Production callers go through `Service.UpdateActionItem` (`internal/app/service.go:1071-1104`) — the post-apply re-check at `:1102` rejects coverage-violating updates with `domain.ErrInvalidPackages`.
2. MCP adapters route via `tasks.UpdateActionItem` (which is the service method) — see `extended_tools.go:1137-1167` — never directly via the repo.
3. The repo is only called directly from internal seeding paths and tests, none of which write coverage-violating data in production code.
4. The coverage rule is documented as a "domain-light" invariant per `WAVE_1_PLAN.md §1.2`: *"Coverage strict-check deferred to Wave 2."*

**Counterexample status.** REFUTED. Architectural choice (domain/service-layer invariant, not SQL-layer) matches existing precedent — other domain invariants (lifecycle state machine, kind-capability checks) are also enforced at domain/service, not at SQL. The repo accepting coverage-violating writes is acceptable because no production-code path reaches the repo bypassing the service. Pre-MVP dev-fresh-DB rule means no migration code can persist coverage-violating data either.

### 1.3 Attack: Pointer-sentinel `null` JSON ergonomics — REFUTED (with one minor coverage gap)

**Hypothesis.** MCP JSON-RPC clients may send `"packages": null` (explicit-null) vs omit the key entirely. Both should be treated as "preserve existing." The `*[]string` shape with no `,omitempty` tag means encoding/json applies its standard rule: JSON `null` deserializes to a nil pointer, identical to omit.

**Evidence.**
- `internal/adapters/server/mcpapi/extended_tools.go:877` — args struct has `Packages *[]string \`json:"packages"\`` (no omitempty on read side; that's normal).
- Standard `encoding/json` behavior:
  - omit → field stays at zero value (nil `*[]string`).
  - `"packages": null` → field set to nil pointer (json explicitly handles JSON null → pointer-nil).
  - `"packages": []` → field set to non-nil pointer to empty slice.
  - `"packages": ["x"]` → field set to non-nil pointer to populated slice.
- `internal/adapters/server/mcpapi/extended_tools.go:1089-1091` — adapter-side gate: `if args.Packages != nil { createReq.Packages = ... }` — correctly treats the nil pointer (whether from omit or JSON null) as "not supplied."

**Coverage gap.** `TestActionItemMCPPackagesRoundTrip` does NOT explicitly cover the `"packages": null` case — only the omit case ("update without packages preserves prior") and the empty-array case (`[]any{}`). The null case follows the same code path as omit (both produce nil `*[]string`), but the test does not pin it explicitly.

**Counterexample status.** REFUTED on behavior — the pointer-sentinel correctly distinguishes "absent / null" (preserve) from "explicit empty" (clear) from "populated" (replace). The minor coverage gap on JSON null is documented as N1 below; it is not a counterexample because the standard library guarantees the behavior.

## 3. NITs

**N1 — Test gap on JSON `null` for Packages.**

`TestActionItemMCPPackagesRoundTrip` covers omit (preserve), `[]any{}` (explicit clear), and populated (replace) — but NOT `nil` (JSON null). Standard `encoding/json` semantics guarantee the null case follows the same code path as omit (both produce nil `*[]string`), so behavior is correct, but the test does not pin the null case explicitly. Future Wave 1 droplets (4a.7 files, 4a.8 start_commit, 4a.9 end_commit) will likely repeat the same omission. Adding one extra `t.Run` per droplet that submits `"packages": nil` (or in raw JSON, `null`) as the JSON RPC value would close the gap. Not a blocker for 4a.6.

**N2 — `pathsJSON, packagesJSON` storage layer accepts coverage violations.**

`internal/adapters/storage/sqlite/repo_test.go:2912-2926` deliberately exercises a coverage-violating UPDATE (Packages cleared while Paths populated). The test passes because the repo writes raw fields. The choice is documented inline. However, this opens the door for any future direct-repo caller (e.g., a migration script, a seeding helper) to silently persist a coverage-violating row that would then fail validation on the next domain pass. A defensive `domain.ValidateActionItemCoverage` called at the repo-write boundary would close this loophole at no real cost (the validation is O(1) given the lengths involved). Not in scope for 4a.6 — flagging for a future hardening drop.

**N3 — No `4a.6_BUILDER_WORKLOG.md` exists.**

Same observation as 4a.5 N2. Continuing the lapsed practice. Not a blocker.
