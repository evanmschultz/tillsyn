### 1.1 Attack: Disjoint-axis enforcement leak (`Files↔Paths` cross-check accidentally added) — REFUTED

**Hypothesis.** 4a.6 added a coverage rule (`len(Paths) > 0 → len(Packages) > 0`) at both `domain.NewActionItem` (create) and `Service.UpdateActionItem` (post-apply re-check). Builder for 4a.7, working in the same file under same-pattern pressure, may have accidentally generalized the cross-axis check pattern to Files — either as a coverage rule (`Files ⊆ Paths`) or a disjointness rule (`Files ∩ Paths == ∅`). Either rejects the legitimate read-then-edit overlap case the spec calls out at `WAVE_1_PLAN.md` line 122.

**Evidence.**
- `internal/domain/action_item.go:380-387` — Files normalization in `NewActionItem`:
  ```
  // Files is the reference-attachment slice (read attention). Normalized
  // independently of Paths/Packages — disjoint-axis rule per
  // WAVE_1_PLAN.md §1.3 means legitimate overlap with Paths is permitted
  // and no cross-axis coverage check applies.
  files, err := normalizeActionItemFiles(in.Files)
  if err != nil {
      return ActionItem{}, err
  }
  ```
  No reference to `paths` or `packages` after the call. `normalizeActionItemFiles` itself (`action_item.go:740-754`) operates on the `files` slice in isolation — no second-arg, no closure-captured paths.
- `internal/app/service.go:1126-1139` — `Service.UpdateActionItem` Files block sits AFTER the Paths/Packages coverage re-check at line 1123 and contains no `actionItem.Paths` reference:
  ```
  if in.Files != nil {
      normalized, err := domain.NormalizeActionItemFiles(*in.Files)
      ...
      actionItem.Files = normalized
      actionItem.UpdatedAt = s.clock().UTC()
  }
  ```
- `rg -n 'actionItem\.Files|t\.Files' internal/domain/ internal/app/` returns 9 hits across 4 files; every hit is a literal field read/write or test assertion. No conditional combining `Files` with `Paths` exists.
- `rg -n "Files.*Paths|Paths.*Files" internal/domain/ internal/app/ | rg -v "test|comment|//|disjoint|overlap"` returns zero hits — no production-code line co-references both fields.

**Counterexample attempt (constructive).** I attempted to construct an `ActionItem{Paths: ["a"], Files: ["a"]}` via `NewActionItem` and verify it round-trips. The test `TestNewActionItemFilesAllowsOverlapWithPaths` at `domain_test.go:1264-1291` does exactly this:
```
shared := "internal/domain/action_item.go"
actionItem, err := NewActionItem(ActionItemInput{
    ...
    Paths:    []string{shared},
    Packages: []string{"internal/domain"},
    Files:    []string{shared},
}, now)
if err != nil { t.Fatalf(...) }
if len(actionItem.Paths) != 1 || actionItem.Paths[0] != shared { ... }
if len(actionItem.Files) != 1 || actionItem.Files[0] != shared { ... }
```
`mage ci` green confirms this test passes — overlap construct doesn't error, both slices retained.

**Counterexample status.** REFUTED. Disjoint-axis rule is correctly implemented: Files validation runs in isolation, no Files↔Paths cross-check exists at any layer (domain, service, adapter, snapshot), and an explicit overlap test pins the behaviour. Spec line 122 honored.

**Evidence.** `snapshot_test.go:990-1086` (`TestSnapshotActionItemFilesLegacyFormatCompatibility`):

**Counterexample status.** REFUTED.

### 1.11 Attack: `app_service_adapter_mcp.go` create-path uses defensive copy, update-path doesn't — REFUTED (BY DESIGN)

**Hypothesis.** `CreateActionItem` adapter at `:660-664` uses `Files: append([]string(nil), in.Files...)` (defensive copy), but `UpdateActionItem` at `:732-736` uses `Files: in.Files` (pointer pass-through). A caller mutating its own `*[]string` after the call could observe state corruption.

**Evidence.**
- `app_service_adapter_mcp.go:663` (Create): `Files: append([]string(nil), in.Files...)` — defensive copy of the slice header AND elements (since `[]string` element type is value-type-of-string, a slice copy is sufficient).
- `app_service_adapter_mcp.go:735` (Update): `Files: in.Files` — pointer threaded directly. The `*[]string` IS the pointer-sentinel; the service's `domain.NormalizeActionItemFiles(*in.Files)` at `service.go:1133` then performs its own copy/transform pass.
- This pattern is identical to Paths (`:660` vs `:732`) and Packages (`:661` vs `:733`). The asymmetry is deliberate: Create takes a value-type `[]string` (no caller-mutation worry — copy preserves caller's data integrity), Update takes a `*[]string` (the pointer IS the sentinel; copying the pointer would lose the nil-vs-non-nil distinction).

**Counterexample status.** REFUTED. The asymmetry is a feature of the pointer-sentinel pattern, not a bug. Same shape as 4a.5 + 4a.6.

### 1.12 Attack: Repo round-trip test only exercises 3 of 5 SELECT sites — REFUTED (acceptable coverage)

**Hypothesis.** `TestRepository_PersistsActionItemFiles` exercises `GetActionItem` (`getActionItemByID`), `ListActionItems`, `ListActionItemsByParent`, and the post-update reload path. It does NOT explicitly exercise `FindActionItemByOwnerAndTitle` or `ListActionItemsByDropNumber`. An ordinal mismatch in those two SELECTs could persist undetected.

**Evidence.** `rg -n "files_json" repo.go` confirms all 5 SELECTs use the identical column projection: `... paths_json, packages_json, files_json, lifecycle_state, column_id, position, title, description, priority, due_at, labels_json, metadata_json, ...`. All 5 SELECTs feed into `scanActionItem` (the only scan helper for this row shape). If `getActionItemByID` decodes `files_json` correctly into `t.Files` (which the test asserts), the same `scanActionItem` decoding the same projection from `FindActionItemByOwnerAndTitle` and `ListActionItemsByDropNumber` will succeed identically.

The 4a.5 (Paths) and 4a.6 (Packages) repo tests adopted the same coverage scope (3-of-5 SELECTs), and both passed under post-droplet `mage ci` runs. Architectural consistency.

**Counterexample status.** REFUTED. Coverage is acceptable given identical projection across all 5 SELECTs and a single shared scan helper.
