### 1.1 Attack: Slice-validation pattern accidentally adopted on a single-string field — REFUTED

**Hypothesis.** Builders 4a.5 / 4a.6 / 4a.7 each shipped slice fields (`Paths`, `Packages`, `Files`) that funneled through an exposed `domain.NormalizeActionItem<X>` helper running trim → dedupe → reject-on-empty / reject-on-backslash. Working under same-pattern pressure, the 4a.8 builder may have accidentally generalized the slice-validator pattern to the new single-string `StartCommit` field — invented an `ErrInvalidStartCommit` sentinel, wrote a `NormalizeActionItemStartCommit` exposed helper, and rejected whitespace-only / non-hex / slash-bearing inputs. The spec is explicit (WAVE_1_PLAN.md §1.4 lines 138–143): "Hex-format check NOT enforced … short-SHAs (7-char), full-SHAs (40-char), and untracked-state empty all need to round-trip. Trim-only matches `Owner` + `BlockedReason` precedent."

**Evidence.**
- `internal/domain/action_item.go:411-420` — the entire StartCommit validation block in `NewActionItem`:
  ```
  // StartCommit is a free-form opaque-domain string. Trim surrounding
  // whitespace; empty is the legitimate zero value ("not yet captured").
  // No format check — git short-SHAs (7-char), full-SHAs (40-char), and
  // caller-supplied identifiers all round-trip. The wrapper inline trim
  // is kept here rather than behind a NormalizeActionItemStartCommit
  // helper because a single TrimSpace call is too thin to warrant the
  // indirection — matches Owner's inline-trim precedent (line 320 above).
  // Domain primitive per Drop 4a L3 / WAVE_1_PLAN.md §1.4.
  in.StartCommit = strings.TrimSpace(in.StartCommit)
  ```
  ONE line of code. No call to a `Normalize*` helper. No conditional branching. No error return path. The doc-comment explicitly cites why it stays inline (Owner precedent at `:320`).
- `rg -n "NormalizeActionItemStartCommit|NormalizeStartCommit" internal/` returns one hit, and it is the comment at `action_item.go:416` that *negatively* asserts no such helper exists. No exposed function, no internal helper.
- `internal/domain/errors.go` is unchanged in this commit (`git diff internal/domain/errors.go` empty). The 11-sentinel block at `errors.go:30-33` (`ErrInvalidPaths`, `ErrInvalidPackages`, `ErrInvalidFiles`, `ErrInvalidLifecycleState`) is untouched — no `ErrInvalidStartCommit` added.

**Counterexample attempts.**
1. *Whitespace-only StartCommit should reject (slice-pattern would).* Looked at `domain_test.go:1305` `{name: "whitespace-only collapses to empty", input: "   ", want: ""}` — explicit test case asserts whitespace-only rounds to empty, NOT rejected. Test passes (`mage ci` green).
2. *Slash-bearing identifier should reject (slice-pattern would, mirroring `Paths` backslash check).* Test at `domain_test.go:1308` `{name: "non-SHA identifier round-trips", input: "branch/feature@head", want: "branch/feature@head"}` — slash-bearing free-form input round-trips. Test passes.
3. *Empty StartCommit should reject (slice-pattern's "non-empty entry" check would).* Test at `domain_test.go:1303` `{name: "empty round-trips empty", input: "", want: ""}` — empty round-trips. Test passes.

**Counterexample status.** REFUTED. Validation is exactly the spec-mandated single TrimSpace. No sentinel. No helper. No reject path. Three table-driven cases pin the absence of the slice-pattern leak.

### 1.3 Attack: Ordinal mis-alignment between SCHEMA, INSERT, UPDATE, and 5 SELECTs — REFUTED

**Hypothesis.** A new column inserted in the middle of an action_items SELECT list (NOT at the end) gives 8 distinct call sites that must all stay in lock-step: CREATE TABLE, INSERT column-list + values, UPDATE SET clause, 5 SELECTs (`ListActionItems`, `ListActionItemsByParent`, `FindActionItemByOwnerAndTitle`, `ListActionItemsByDropNumber`, `getActionItemByID`), and `scanActionItem`'s scan-targets. An off-by-one in any of these silently corrupts data — `lifecycle_state` text would land in `start_commit` and vice versa. Compile would still pass; tests covering only the SELECT path with default-empty data could mask the bug. The empty-default test case in particular would never exercise the corruption (both fields would stringify empty).

**Evidence.**
- CREATE TABLE column ordering at `repo.go:181-185`:
  ```
  paths_json TEXT NOT NULL DEFAULT '[]',
  packages_json TEXT NOT NULL DEFAULT '[]',
  files_json TEXT NOT NULL DEFAULT '[]',
  start_commit TEXT NOT NULL DEFAULT '',
  lifecycle_state TEXT NOT NULL DEFAULT 'todo',
  ```
  `start_commit` sits after `files_json`, before `lifecycle_state`. Spec line 138 demands exactly this position.
- INSERT at `repo.go:1203` lists 36 columns with `start_commit` between `files_json` and `lifecycle_state`. Bound parameters at `repo.go:1208-1244` count 36 (manual count, includes `t.StartCommit` at `:1223` between `filesJSON` `:1222` and `string(t.LifecycleState)` `:1224`). Placeholders in `VALUES (...)` at `:1206` count 36. All three counts match.
- UPDATE at `repo.go:1318` SET clause includes `..., files_json = ?, start_commit = ?, lifecycle_state = ?, ...`. Bound parameters at `:1335-1337` mirror this order: `filesJSON`, `t.StartCommit`, `string(t.LifecycleState)`.
- 5 SELECTs (`repo.go:1395, :1433, :1467, :1488, :2529`) — all 5 list `..., files_json, start_commit, lifecycle_state, ...` in identical order. Verified visually in the diff.
- `scanActionItem` at `repo.go:2854-2877` — declared variables AND `s.Scan` argument list both place `startCommitRaw` between `filesRaw` and `state`. The variable declaration order at `:2854-2858` and the scan call at `:2874-2878` are identical.
- The empty-default case alone is a weak test (it can't detect a swap between two empty fields), so I followed up the test cases:
  - `domain_test.go:1306` short-SHA `"0cf5194"`,
  - `repo_test.go:3134` short-SHA `"0cf5194"`,
  - `repo_test.go:3135` full-SHA `"0cf5194d4cb6c8d4f9b9b1d7e1f9d3c2b4e5a6f7"`.
  These are non-empty values that would NOT match `lifecycle_state` (which is `"todo"`/`"in_progress"`/`"complete"`/`"failed"`) — a SELECT-side ordinal swap would surface as `t.LifecycleState = "0cf5194"` and `t.StartCommit = "todo"`, blowing up either `LifecycleState` validation or the `loaded.StartCommit != tc.want` assertion.

**Counterexample attempt.** Construct a row where `start_commit = "todo"` and `lifecycle_state = "complete"` — an ordinal swap would scan them backwards. The repo round-trip test at `repo_test.go:3092-3226` does exactly this asymmetric matching: writes `StartCommit="0cf5194d4..."` (40-char), reads it back, and asserts the 40-char value lands in `StartCommit` not `LifecycleState`. `mage ci` green confirms no swap.

**Counterexample status.** REFUTED. All 8 call sites are in lock-step ordering. Non-empty SHA test cases provide the asymmetric anchor that would catch a swap.

### 1.8 Attack: Pointer-sentinel asymmetry (create takes `string`, update takes `*string` — fine; but adapter forgets to pass through) — REFUTED

**Hypothesis.** The pointer-sentinel pattern requires a clean three-tier passthrough: MCP wire (`*string`) → adapter request (`*string`) → app input (`*string`) → service applies inline. A break anywhere — e.g. adapter dereferencing the pointer at the wire boundary and passing a `string` zero value down — silently turns "preserve" semantics into "clear" semantics. Predecessor 4a.5 had this exact category of bug attacked in `4a.5_BUILDER_QA_FALSIFICATION.md`; the same risk applies on every new pointer-sentinel field.

**Evidence.**
- Wire schema: `extended_tools.go:879` `StartCommit *string \`json:"start_commit"\`` (anonymous request struct). Pointer.
- Wire→common: `extended_tools.go:1199` `StartCommit: args.StartCommit` (direct pointer pass-through; type `*string` ⇒ `*string`). No deref.
- Common→app: `app_service_adapter_mcp.go:737` `StartCommit: in.StartCommit` (direct pointer pass-through). No deref.
- App→service apply: `service.go:1171-1173`:
  ```
  if in.StartCommit != nil {
      actionItem.StartCommit = strings.TrimSpace(*in.StartCommit)
      actionItem.UpdatedAt = s.clock().UTC()
  }
  ```
  nil-guard before deref. Trim applied at the application boundary (matching create-time domain trim). Not-supplied (nil) → no field touched ⇒ "preserve."
- Tests at `extended_tools_test.go:4310-4366` cover four cases:
  - update with `"start_commit": "0cf5194"` ⇒ pointer non-nil with `"0cf5194"` ⇒ apply.
  - update with `"start_commit": ""` ⇒ pointer non-nil with `""` ⇒ explicit clear.
  - update without the field ⇒ pointer nil ⇒ preserve.
  - update with the field set on create ⇒ string flows directly (`"0cf5194"` plumbs through to `lastCreateActionItemReq.StartCommit`).

**Counterexample attempt.** Wire payload `{"start_commit": ""}` on update. Spec demands explicit-clear (pointer non-nil with empty string), NOT preserve (which would be absent field). Test at `:4332-4348` pins exactly this distinction:
```
"start_commit": "",
...
if service.lastUpdateActionItemReq.StartCommit == nil {
    t.Fatalf(... "want non-nil pointer to empty string (explicit clear)")
}
if got := *service.lastUpdateActionItemReq.StartCommit; got != "" {
    t.Fatalf(... "want empty string (explicit clear)")
}
```
Test passes (`mage ci` green) — the JSON null vs JSON empty-string distinction survives the unmarshal.

**Counterexample status.** REFUTED. Pointer-sentinel symmetry holds end-to-end. Empty-string explicit-clear distinguishable from nil-preserve at every layer.
