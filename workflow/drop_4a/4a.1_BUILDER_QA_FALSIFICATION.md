### 1.1 NIT — `mage -h fmt` / `mage -h format-check` / `mage -h format-path` return "Unknown target"

**Severity:** NIT (Mage CLI ergonomic quirk, not a builder error).

**Reproduction.**

```
$ mage -h fmt
Unknown target: "fmt"

$ mage -h format-check
Unknown target: "format-check"

$ mage -h format-path
Unknown target: "format-path"

$ mage -h format
Format rewrites every tracked Go file with 'go tool gofumpt -w'. Use FormatPath for a single file or directory.

Usage:

	mage format

Aliases: fmt
```

The aliases EXECUTE correctly:

```
$ mage fmt              # works, runs Format()
$ mage format-check     # works, runs FormatCheck() → formatCheck()
$ mage format-path ./magefile.go   # works, runs FormatPath("./magefile.go")
```

But `mage -h <alias>` doesn't resolve to canonical-name help. This is a known Mage CLI behavior — the `-h` flag iterates declared targets (camelCase), not the alias map.

**Why this is a NIT, not a CONFIRMED counterexample.**

W0.1 acceptance does not require `mage -h <alias>` to work. It requires:
- `mage` (Default = CI) green ✓
- `mage format` (no arg) succeeds ✓
- `mage format-check` succeeds clean ✓
- `mage format-path ./magefile.go` succeeds (alias dispatch works at execution time) ✓
- `mage ci` green ✓

Plan author actually anticipated the dispatch quirk in WAVE_0_PLAN.md line 56: *"Builder must verify mage's CLI-name resolution for `FormatPath` — if `mage format ./path` does NOT route to `FormatPath` automatically, expose it as `mage format-path ./path` via the alias map."* The builder picked the explicit-alias route, which is the safer landing.

**Routing recommendation.**

Surface as a Drop 4a Wave 0 closeout note for W0.4's CONTRIBUTING.md docs — the "command list" should explicitly cite `mage format-check` and `mage format-path` (kebab-case) rather than the camelCase target names, so devs reading docs hit the alias surface, not the camelCase targets that `mage -h <alias>` won't help with. Non-blocking for 4a.1.

### 2.7 `FormatPath` validation correctness — REFUTED (attack vector #7)

func FormatPath(path string) error {
	path = strings.TrimSpace(path)

### 2.12 `format-path` alias YAGNI / scope creep — REFUTED (attack vector #12)

The builder added `"format-path": FormatPath` to the alias map. This was not strictly required by W0.1 — spec line 56 says: *"if `mage format ./path` does NOT route to `FormatPath` automatically, expose it as `mage format-path ./path` via the alias map (`"format-path": FormatPath`) and document. Either form is acceptable."*

Why it's not YAGNI:
1. **Spec-anticipated.** WAVE_0_PLAN.md explicitly authorized this alias as one of two acceptable paths.
2. **Mage's positional-arg routing on aliases is fragile.** `mage format ./path` would have required Mage to (a) recognize `format` as an alias for `Format` (which now takes zero args), then (b) bounce `./path` to a different target. Mage doesn't do that — it would error `Format() doesn't accept arguments`. The builder's empirical test confirms this: `mage format-path ./magefile.go` works, but `mage format ./magefile.go` would fail because `Format()` is now zero-arg.
3. **Symmetry with existing kebab-case aliases.** `test-golden`, `test-pkg`, `test-func`, `format-check` all map kebab-case CLI names to camelCase Go identifiers. `format-path` follows the same precedent.
4. **CONTRIBUTING.md (W0.4) will document this.** Spec line 114 already lists `mage format-path` as a documented target — so the alias must exist for the docs to be honest.

The alias is defensible and spec-anticipated. Not YAGNI.
