## 2. Counterexamples (CONFIRMED)

### 2.1 Stage 8 (monitor.Track) failure leaks a phantom in_progress action item

**Surface:** `internal/app/dispatcher/dispatcher.go:481-486`

```go
handle, err := d.monitor.Track(ctx, item.ID, cmd)
if err != nil {
    d.fileLocks.Release(item.ID)
    d.pkgLocks.Release(item.ID)
    return DispatchOutcome{}, fmt.Errorf("dispatcher: track spawn for %q: %w", item.ID, err)
}
```

**Counterexample:** When `cmd.Start()` fails (binary not on PATH, exec format error, EACCES, ENOENT — all real production cases when "claude" is not staged), `monitor.Track` returns `ErrMonitorNotStarted` (monitor.go:246-247). The dispatcher correctly releases file + package locks. **But Stage 7 (`walker.Promote`, line 465-474) ALREADY moved the action item to `in_progress`.** The action item is now:

- LifecycleState = `in_progress`
- No subprocess running
- No monitor goroutine watching
- Locks released
- CleanupHook never fires (only fires on terminal state — `complete` / `failed` / `archived`)

The next dispatcher tick will skip this item (Stage walker.isEligible filters non-todo). The dev sees a hard error from the CLI but doesn't know the action-item state is corrupted. Manual recovery: query the row, demote to todo via direct DB or service mutation.

**Reproduction (conceptual — not run):**
1. Eligible item with binding pointing at a nonexistent agent path.
2. `till dispatcher run --action-item <id>`.
3. Stages 1-7 succeed; Stage 8's `cmd.Start()` returns ENOENT.
4. CLI exits non-zero with "track spawn for X: monitor not started: ...".
5. `till action_item get X` shows lifecycle=in_progress.

**Severity:** CONFIRMED design gap. Mitigation paths:
1. **Roll back Stage 7's promote** when Stage 8 fails (move back to todo). Symmetric with stage-5/6/7 lock-rollback pattern.
2. **OR:** treat Track-failure as a `ResultFailed` outcome; have the cleanup hook fire on the spawn-fail path (extend cleanup to fire on "failed-to-spawn" pseudo-terminal).
3. **OR:** explicitly document the leak in the worklog and define a recovery CLI (`till dispatcher reset --action-item X`).

The worklog does NOT mention this case. Stage 8 is a real spawn-fail blast-radius gap.

**Classification:** CONFIRMED counterexample. Recommend 4a-refinement entry for Stage 7 rollback symmetry on Stage 8 failure.

### 2.2 `--project` flag is plumbed end-to-end but never read

**Surface:** `cmd/till/dispatcher_cli.go:37`, `cmd/till/main.go:838`, `cmd/till/dispatcher_cli.go:61-132`

The `--project` flag is:
- declared on `dispatcherRunCommandOptions.projectID` (dispatcher_cli.go:37)
- registered via `cobra.Flags().StringVar(&dispatcherRunOpts.projectID, "project", ...)` (main.go:838)
- threaded into `runDispatcherRun(ctx, svc, broker, opts, ...)` (main.go:2501)
- **never read** inside `runDispatcherRun` (verified via `rg projectID dispatcher_cli.go` — only declaration + struct-comment, no consumer)

The struct doc-comment claims: *"projectID overrides the action-item's ProjectID for the project resolution step. Optional — when empty, the dispatcher resolves the project from the action item itself. Reserved for the case where the dev wants to validate dispatch against a sibling project's catalog without re-parenting the item."*

Spec WAVE_2_PLAN.md:279: *"`--project <UUID>` (optional; resolved from action-item if omitted)"* — implies functional override.

**Implementation reality:** The dispatcher resolves project via `item.ProjectID` (dispatcher.go:325) ALWAYS. The CLI silently accepts and discards the flag.

**Severity:** CONFIRMED spec drift + YAGNI. Two fixes:
1. **Honor the override:** when `opts.projectID != ""`, look up THAT project and pass it through (requires PreviewSpawn / RunOnce signature widening or a service-level override).
2. **Remove the flag:** if no use case justifies the override pre-Drop-4b, remove the flag declaration + struct field. Keep the surface minimal (YAGNI).

Either resolution is valid. The current state is the worst of both — surface that lies.

**Classification:** CONFIRMED counterexample. Recommend 4a-refinement entry.

### 2.3 `--dry-run` does NOT walk eligibility check (spec drift)

**Surface:** `internal/app/dispatcher/dispatcher.go:506-534` (`PreviewSpawn`)

Spec WAVE_2_PLAN.md:281: *"On `--dry-run`: walks the eligibility check + builds the spawn descriptor + prints it; does NOT call the spawner. Exits 0."*

**Implementation reality:** PreviewSpawn does:
1. GetActionItem
2. GetProject
3. decodeProjectCatalog
4. BuildSpawnCommand

It does NOT call:
- `walker.isEligible` (eligibility predicate)
- `conflict.DetectSiblingOverlap` (sibling overlap check)
- `fileLocks.Acquire` / `pkgLocks.Acquire` (lock contention check)

**Counterexample:** A dev runs `till dispatcher run --action-item X --dry-run` against an item that:
- has a sibling-overlap on `paths` without explicit `blocked_by`, OR
- has its file/package locks held by another in-flight item, OR
- has unresolved blocked_by referents

`--dry-run` returns a successful descriptor JSON. The dev pipes it to a script. The dev then runs the same command without `--dry-run`. Live RunOnce returns ResultBlocked. The descriptor JSON the dev cached is now stale or misleading.

**Severity:** CONFIRMED spec drift. Two fixes:
1. **Make PreviewSpawn walk eligibility:** call walker.isEligible + conflict detection (no lock acquire — dry-run must remain side-effect-free). Surface block-state in the JSON or via stderr.
2. **OR amend the spec:** if the implementation is the desired behaviour, update WAVE_2_PLAN.md §2.10 acceptance to drop "walks the eligibility check" and document the simpler PreviewSpawn semantics.

**Classification:** CONFIRMED counterexample. Recommend either spec-amend or implementation-extend; track as 4a-refinement entry.

## 3. NIT-grade observations

### 3.1 Dead code in stage 2

`internal/app/dispatcher/dispatcher.go:343-346`:

```go
catalog, ok, err := decodeProjectCatalog(project)
if err != nil {
    return DispatchOutcome{}, fmt.Errorf("dispatcher: decode kind catalog for project %q: %w", project.ID, err)
}
```

`decodeProjectCatalog` (lines 561-575) NEVER returns a non-nil error — both empty-string and json.Unmarshal-error paths return `(zero, false, nil)`. The error branch at line 343 is unreachable. Defensive, but worth either documenting ("`err` reserved for future hard-failure paths") or simplifying to `catalog, ok := decodeProjectCatalog(project)`.

### 3.2 Vestigial discards

- `dispatcher.go:424` — `_ = acqFiles` after `acqFiles, fileConflicts, err := d.fileLocks.Acquire(...)`. `acqFiles` is never used.
- `dispatcher.go:439` — same with `_ = acqPkgs`.
- `dispatcher.go:458` — `_ = binding // binding's fields are read indirectly through descriptor`, but the comment is wrong: `binding` is constructed at Stage 2 (line 351) and never used after that — the comment should say "Stage 6's BuildSpawnCommand re-resolves binding from catalog".

These can be cleaned by switching to `_, fileConflicts, err :=` form and removing the dangling `binding` variable + its discard.

### 3.3 `spawnDescriptorJSON` field-set drift risk

`cmd/till/dispatcher_cli.go:168-187`'s anonymous `wire` struct hardcodes 7 fields. Adding a field to `dispatcher.SpawnDescriptor` (spawn.go:155-163) silently drops it from the JSON. Suggested mitigation: add a compile-time check (e.g. `var _ = func() { _ = dispatcher.SpawnDescriptor{} == dispatcher.SpawnDescriptor(spawnDescriptorJSON{}) }()` — won't compile cleanly but the alias `type spawnDescriptorJSON dispatcher.SpawnDescriptor` already gives a structural guarantee that fields match. Verify: does adding a field to SpawnDescriptor cause MarshalJSON's `wire` type to lose data? Yes — silent drop. **NIT.**

A test that asserts `len(reflect.TypeOf(SpawnDescriptor{}).NumField()) == len(reflect.TypeOf(wire{}).NumField())` would lock the alignment.
