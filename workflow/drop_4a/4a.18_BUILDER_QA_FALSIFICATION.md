## Falsification Attack Matrix

### F1. Archived ANCESTOR (parent or grandparent) eligibility

**Scenario:** A child whose parent has been archived (e.g., the parent
was a `build` action item that got Archive-deleted before the child
finished). The walker calls `ListActionItems(projectID, false)` —
`includeArchived=false` (`walker.go` line 138) — which filters out
archived items at the SQL layer (`internal/adapters/storage/sqlite/repo.go`
lines 1431–1433: `if !includeArchived { query += " AND archived_at
IS NULL" }`). The archived parent therefore never enters `byID`. The
child then hits the `byID[parentID]` lookup at `walker.go` line 189,
falls into the `if !ok` branch at line 190 ("detached child"), and is
skipped. **CORRECT** — but the test suite does not pin this branch
explicitly.

**Evidence:**
- `walker.go` line 138 (`ListActionItems(ctx, projectID, false)`).
- `repo.go` lines 1431–1433 (archive filtering at the SQL layer).
- `walker.go` lines 189–195 (the "detached child" guard).
- `walker_test.go` exercises detached-via-missing only via
  `TestWalkerSkipsItemWithMissingBlockedByReference` for the *blocker*
  side; there is no equivalent for the *parent* side.

**Verdict:** **MITIGATED** at the code layer; **CONFIRMED-NIT** at
the test-rigor layer. Behavior is correct (archived parent → child
skipped, conservative) but it relies on an SQL-level filter and a
guard that has no direct test. A regression to
`ListActionItems(ctx, projectID, true)` in walker.go would silently
flip archived parents into "not Persistent, not StateInProgress" →
skipped also, so the production behavior is double-protected —
**however**, if the SQL filter changed and the walker started loading
archived items, an archived item with `LifecycleState=StateArchived`
that was at one point `StateInProgress` would still be skipped because
StateArchived ≠ StateInProgress. So the layered defense holds, but
only by accident; a `TestWalkerSkipsItemUnderArchivedParent` would pin
the contract.

**Refinement raised:** add a `TestWalkerSkipsItemUnderArchivedParent`
test fixture before 4a.20 lands (the conflict detector also reads the
project tree and benefits from the same defensive coverage).

**Evidence:** `walker.go` `isEligible` (lines 167–200) checks only:

`ensureActionItemCompletionBlockersClear` as the place where
children-complete IS enforced (on the to-complete transition, not the
to-in_progress transition).

**Verdict:** **MITIGATED**. Asymmetry correctly implemented and
documented.

### F4. Promote race vs walker snapshot

**Scenario:** The walker returns an eligible-set snapshot at time T0;
by the time the caller invokes `Promote(item)` at time T1, the item's
state in the DB may have changed (e.g., another agent already moved
it to in_progress, or a Restore moved it to todo from archived).
Does Promote re-read or trust the snapshot?

**Evidence:** `walker.go` `Promote` (lines 220–244):
1. Validates `w.svc != nil`, `projectID != ""`.
2. Calls `ListColumns` to resolve the in_progress column ID.
3. Calls `Service.MoveActionItem(ctx, item.ID, columnID, item.Position)`.

The walker does NOT re-read the item before calling MoveActionItem —
it trusts the caller-supplied snapshot. **However, this is correct:**
`Service.MoveActionItem` is the authoritative gate. Per WAVE_2_PLAN.md
§2.5 mitigation note (line 177): "Service.MoveActionItem already
enforces start-criteria + completion-criteria invariants atomically.
Walker is best-effort eligibility; final gate is in Service." If the
item has already moved, MoveActionItem will either succeed
idempotently (already in_progress) or return an
`ErrTransitionBlocked` that the walker re-wraps as
`ErrPromotionBlocked` — both outcomes are recoverable for the
conflict detector (Wave 2.7).

**Verdict:** **MITIGATED**. The race is real but the contract is
explicit (worklog §1 + spec line 177): walker is best-effort, Service
is authoritative. No defensive re-read is needed inside the walker.

### F5. Concurrent walks (two parallel `EligibleForPromotion` calls)

**Scenario:** Two goroutines call `EligibleForPromotion` on the same
walker simultaneously. Race on tree read?

**Evidence:** `walker.go` line 106–108: `treeWalker` struct holds only
`svc walkerService`; no mutable state. Each invocation builds its own
local `byID` map (line 148) and its own local `eligible` slice (line
153) on the goroutine's stack. The underlying `Service.ListActionItems`
is read-only and serialized by SQLite's read lock.

`walker.go` lines 76–79: package-doc explicitly documents
"`treeWalker` holds no mutable state. Concurrent calls are safe to the
extent the underlying service permits".

**Verdict:** **MITIGATED**. No shared mutable state in `treeWalker`.
Concurrency safety is documented and structurally enforced.

### F6. Stale state: broker fires after repo write but consumer is delayed

**Scenario:** Broker publishes an event after Service write at time
T0; consumer goroutine doesn't drain it until T2. By T2, additional
state changes have happened (T1 events were coalesced). What does the
walker observe?

**Evidence:** `walker.go` lines 31–43 (broker-ordering doc):
"Coalesced bursts collapse into a single wake; the walker re-reads
the project tree on each wake so coalescing is invisible to the
eligibility decision." The walker on every wake calls
`ListActionItems(...)` which goes to the repo (current state) and
re-evaluates the predicate against fresh data.

This is also the mitigation for 4a.15 F3 (burst coalescing) — the
walker's "always re-read" semantics make per-event delivery
unnecessary.

**Verdict:** **MITIGATED**. Walker is tolerant of missed publishes
because it re-reads the tree on every wake. This is the design
contract; coalescing is invisible to dispatch.

### F7. F1-from-4a.15 publish-gap on Restore / Rename / Archive / Reparent

**Scenario:** Per `4a.15_BUILDER_QA_FALSIFICATION.md` F1, five Service
methods write to action items WITHOUT calling
`publishActionItemChanged`: `RestoreActionItem`, `RenameActionItem`,
`DeleteActionItem(Archive)`, `DeleteActionItem(Hard)`,
`ReparentActionItem`, plus `ImportSnapshot` in snapshot.go. Does the
walker close this gap?

**Evidence — Restore:** `RestoreActionItem` calls
`SetLifecycleState(restoredState)` which can put an item back into
todo. A continuous-mode dispatcher running across a Restore would NOT
wake on the Restore itself (no publish), so the walker would not
re-evaluate eligibility until some OTHER write triggers a wake.
**The walker does NOT close this gap.**

**Evidence — Archive:** `DeleteActionItem(Archive)` removes an item
from dispatch eligibility (the walker excludes archived items via
`includeArchived=false`). If a walker is mid-evaluation and an Archive
lands, the next wake (via some OTHER write) re-reads the tree and
correctly excludes the archived item. So Archive's missing-publish is
recoverable on the next event — but if no other event ever arrives,
the walker's last view of the tree is stale until either CLI manual
trigger (Wave 2.10) or a real publishable write lands.

**Evidence — Reparent:** `ReparentActionItem` changes parent_id. A
parent change can flip a child from "ineligible" (parent was todo) to
"eligible" (parent now in_progress) or vice versa. Without a publish,
the walker doesn't wake.

**Evidence — Rename:** Title-only mutation; doesn't affect
eligibility. The missing publish is a transparency gap, not an
eligibility gap.

**Evidence — Hard delete:** Removes the row entirely. If the row was
in someone's `BlockedBy`, that BlockedBy reference becomes phantom →
walker treats as not-clear (correct conservative behavior, per `walker.go`
lines 173–177). But the walker doesn't wake on the delete itself.

**Verdict:** **CONFIRMED — but ACCEPTED as a 4a.15 carry-forward
refinement**. The publish-gap is real and the walker does NOT close
it. Two options to close:

1. **Wave 2.10 manual-trigger path** — once the CLI manual-trigger
   ships, an operator can force a walker re-evaluation from outside
   the broker. This is sufficient for cascade-pre-Drop-4b
   correctness (no continuous-mode dispatcher exists yet).
2. **Drop 4b refinement** — add `publishActionItemChanged` calls to
   `RestoreActionItem`, `DeleteActionItem(Archive)`,
   `ReparentActionItem` before continuous-mode dispatcher ships.
   `RenameActionItem` and `DeleteActionItem(Hard)` are
   eligibility-irrelevant — `Hard` removes the item entirely (the
   walker won't see it on the next list, so the next wake handles
   the BlockedBy phantom-ref defensively); `Rename` is a title-only
   mutation that has no eligibility consequence.

**The 4a.18 droplet does not need to close this gap** — the walker is
imperatively-callable in Wave 2.10, and Wave 2.10 + Drop 4b can
together complete the publish coverage. Flagging here so plan-QA on
Wave 2.10 / Drop 4b carries the refinement forward.

### F10. Self-reference in BlockedBy

**Scenario:** An action item lists its own ID in `BlockedBy`. Walker
behavior?

**Evidence:** `walker.go` lines 71–75 (package-doc): "BlockedBy
entries that point back at the item itself are treated as not-clear
(the item's own state is StateTodo, not StateComplete) so the walker
skips it. The walker does NOT detect or break cycles in BlockedBy —
that is a planner / domain validation concern."

The mechanism: `byID[itemID].LifecycleState == StateTodo` (because the
item under evaluation is in todo, otherwise it wouldn't be checked at
predicate clause 1). Predicate clause 2 returns false at line 180
(`blocker.LifecycleState != domain.StateComplete`).

**Verdict:** **MITIGATED**. Self-reference naturally short-circuits
because the item under evaluation must be in StateTodo to even reach
the BlockedBy check, and StateTodo ≠ StateComplete.

### F17. `errors.Join` order reversal probe

**Verdict:** **MITIGATED**. Join order is irrelevant to `errors.Is`
contract; tests pass.
