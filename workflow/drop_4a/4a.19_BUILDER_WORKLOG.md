## Design Choices

### Auth-Bundle Stub Seam (Wave 3 hand-off)

`AuthBundle` is an empty struct with a `_ struct{}` placeholder field. The placeholder forces positional struct-literal callers to break loudly when Wave 3 adds real fields (session/lease IDs, MCP-config materialization, capability scope). All call sites today pass `AuthBundle{}` via named-zero-value composition, which keeps Wave-3 forward-compat clean.

The Wave-3 seam is documented with `Wave 3:` doc-comments at three load-bearing places:

1. `AuthBundle` type-doc — names what Wave 3 fills in.
2. `mcpConfigPlaceholderPath` — names what Wave 3 will write to the returned path before the monitor (4a.21) executes the cmd.
3. `assemblePrompt` — names where Wave 3 folds session/lease IDs into the prompt.

The placeholder MCP-config path shape is `<worktree>/.tillsyn/dispatcher-spawn-<action_item_id>.json`. This shape is pinned by `TestBuildSpawnCommandPropagatesAuthBundleStubPath` so Wave 3 can't accidentally break path discoverability.

### Agent-Variant Resolution Division of Labor

Per WAVE_2_PLAN.md §2.6 and the brief: `BuildSpawnCommand` reads `binding.AgentName` **verbatim** from the catalog. It does NOT synthesize `{lang}-{role}` agent names from `project.Language` — that synthesis is the planner's responsibility (Wave 1 gave the planner `project.Language`; the template gives the binding; combining them happens at template-load time, not at dispatch time). This keeps the dispatcher one-input-one-output and avoids the "two sources of truth for agent name" failure mode.

The spawn function defensively re-runs `binding.Validate()` even though `templates.Load` already runs it at template-load. The re-run catches the case where a corrupted binding entered the in-memory catalog through some path other than `Load` (today: only test mutation; tomorrow: any future `KindCatalog` mutator). `TestBuildSpawnCommandRejectsCorruptedAgentBinding` pins this path.

### Prompt-Assembler Contract

`assemblePrompt(item, project, _ AuthBundle) string` is the single producer. The prompt body is **opaque to the dispatcher contract** — only the structural tokens are asserted by tests: `task_id:`, `project_id:`, `project_dir:`, `hylla_artifact_ref:`, `kind:`, and `move-state directive:`. Wave 4's CLAUDE.md updates document the prompt-body contract for agent authors. The body intentionally uses simple `key: value` lines rather than YAML / JSON / TOML so the agent-side prompt parser stays trivial.

The function ignores the `AuthBundle` argument today (named `_`) but takes it positionally so Wave 3 can fold real session/lease fields in without changing the signature.

### Input Validation

Three early-return guards reject malformed inputs with `ErrInvalidSpawnInput`:

1. Empty `item.ID` — would yield a non-deterministic MCP-config path with a missing segment.
2. Empty `item.Kind` — would silently surface as `ErrNoAgentBinding` (less precise routing).
3. Empty `project.RepoPrimaryWorktree` — would default `cmd.Dir` to caller cwd, violating the dispatcher contract.

Each guard has a dedicated test (`TestBuildSpawnCommandRejectsEmpty{ActionItemID,Kind,Worktree}`).

### Closed-Enum Errors

Three sentinels exposed:

- `ErrNoAgentBinding` — kind has no entry in `catalog.AgentBindings`.
- `ErrInvalidSpawnInput` — input field empty / malformed.
- `templates.ErrInvalidAgentBinding` — propagated via `binding.Validate()`.

Callers route via `errors.Is`. No new sentinel for "Validate corruption" — the templates-package one is the right level.

  - **Missed because:** Either snapshot 5 predates 4a.14's reingest, or `LookupAgentBinding` lives in a method-receiver doc that the keyword index doesn't tokenize as a tail-symbol search target. The schema returned `null` blocks/results despite the symbol being present at `internal/templates/catalog.go:92`.
  - **Worked via:** `Read` on `internal/templates/catalog.go` directly (path was supplied in the brief).
  - **Suggestion:** Either re-ingest after 4a.14 lands (committed at `1812e80`) or surface method-receiver bindings in the keyword index even when only the receiver type is named.

- **Query:** `mcp__hylla__hylla_search_keyword` with `query: "AgentBinding Validate"`.
  - **Missed because:** The query returned `Config.Validate`, `Service.ValidateSession`, etc. — none of them `AgentBinding.Validate` from `internal/templates/schema.go:264`. The ranker is matching the bare token "Validate" in any package and not biasing toward the receiver-type pair.
  - **Worked via:** `Read` on `internal/templates/schema.go` directly.
  - **Suggestion:** When a keyword query contains two tokens that look like a Go receiver-type pair (capitalized noun + capitalized verb), prefer matches whose `parent_id` ends in the noun.

- **Ergonomic gripe:** `mcp__hylla__hylla_search` rejected the call with "field must be summary, content, or docstring" when I passed `fields: ["content"]` (which IS one of the valid options) AND `search_types: ["keyword", "vector"]`. The error message is misleading — it implies a bad `fields` value when the actual problem is the combination with `vector` search (which uses `field` singular, not `fields`). Suggest: surface the precise param name in the error (`vector search uses 'field' singular; got 'fields'`).

No bugs to file beyond the index/schema mismatch.
