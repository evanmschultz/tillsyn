## Scope

Drop 4a Wave 3.4 — MCP-layer golden tests for the `till.auth_request(operation=approve)` JSON-RPC endpoint. Five test cases (a/b/c/d/e) per the W3.4 brief plus one tool-schema-shape assertion form the regression net for Wave 3's full behavior:

- (a) orch-in-subtree approves non-orch in same subtree → success.

## Design Choices

### Integration vs Unit Split

The W3.4 spec calls cases (a)/(b)/(c)/(e) unit-style and case (d) integration-style. Concrete interpretation:

- **Unit-style** (a/b/c/e): drive `NewHandler(Config{}, &stubAuthRequestService{...}, nil)` over `httptest.NewServer`, JSON-RPC the approve op, assert on the wire format. The stub stages either a success `ApproveAuthRequestResult` (case a, with audit fields populated) or a wrapped-sentinel error (cases b/c/e). The stub does NOT validate the caller's tuple; it merely records what was passed in (`lastApprove`) so the test can assert the tuple round-tripped intact. **Justification**: the gate logic itself is exhaustively tested at the service layer (`internal/app/auth_requests_test.go`) and adapter layer (`internal/adapters/server/common/app_service_adapter_auth_requests_test.go`); the unit cases here verify the JSON-RPC handler's mapping from the gate's wrapped errors to MCP `isError=true` responses with the correct error-text fragments and `auth_denied:` prefix.

- **Integration-style** (d): real `AppServiceAdapter` wired to real sqlite + real autent service via `NewAppServiceAdapter(svc, auth)`, fed into a real `NewHandler`. The STEWARD-typed orch session is minted via `auth.IssueSession(ctx, autentauth.IssueSessionInput{PrincipalType: "steward", ...})` exactly like `stewardCrossSubtreeFixture` in `internal/app/auth_requests_test.go` does. The persistent STEWARD-owned ancestor uses `domain.NewActionItemForTest({Persistent: true, Owner: "STEWARD"})`. The cross-orch subagent request roots under that ancestor's ID via path `project/<id>/branch/<persistentID>`. Approving from STEWARD's tuple over JSON-RPC drives the live gate's `requireStewardPersistentAncestor` ancestry walk end-to-end. **Justification**: the metadata-driven cross-subtree exception (Persistent + Owner==STEWARD) is the single piece of Wave 3 logic that mostly lives in the live gate; stubbing it would amount to asserting a stub-state-machine. Integration is the only way to confirm the full chain (MCP wire → adapter → service-layer gate → ancestry walk → autent-issued subagent session) works.

### Fixture Helpers (Falsification Attack 7)

The W3.4 brief calls for `setupOrchSession`, `setupStewardSession`, `createPendingSubagentRequest` helpers. I split them by mode:

- **Unit-mode helpers** (`unitOrchSession`, `unitStewardSession`, `stubPendingSubagentRequest`, `approvedAuthRequestRecord`, `approveArgs`) — return deterministic value tuples / records the stub-driven cases share. They do not allocate sessions in any DB; the stub just receives the values back from `lastApprove` so the test can assert round-trip equality.
- **Integration-mode helpers** (`stewardApproveFixture` + `newStewardApproveFixture`) — the real-stack equivalent. One factory is enough for case (d) since the wave's only integration case is the STEWARD persistent-parent path.

The struct `orchSessionTuple` is reused across both modes — case (d) plugs the integration-minted STEWARD session into the same tuple shape the unit cases use, keeping `approveArgs(...)` shared.

### Tool-Schema Assertion Mechanism

The existing `TestAuthRequestApproveToolSchemaHasNoConfigurabilityKnobs` (added in 4a.24) already covers most of the W3.4-mandated forbidden vocabulary. The brief explicitly lists `auto_approve_threshold`, `max_per_hour`, `auto_approve_roles`, `dev_required_roles` — three of those (`auto_approve_threshold`, `max_per_hour`, plus aliases `max_approvals_per_hour`, `rate_limit`) were already covered. The brief's two new ones (`auto_approve_roles`, `dev_required_roles`) were not. Rather than mutating the existing 4a.24 test, I added a sibling `TestAuthRequestToolSchemaApproveAcceptsOnlyDocumentedArgs` that pins the brief's literal vocabulary verbatim. Two tests with overlapping forbidden lists is intentional: the 4a.24 test guards aliases, the W3.4 test guards the brief's exact wording. Drift in either direction trips a test.

The schema-shape mechanism: `tools/list` JSON-RPC response → `findToolSchemaByName(t, toolsRaw, "till.auth_request")` → `properties` map probe + `operation` enum probe. Both `properties[forbidden]` and `operation` enum traversal must NOT contain any of the brief's four forbidden values, AND the five documented approve args (`request_id`, `acting_session_id`, `acting_session_secret`, `agent_instance_id`, `lease_token`) must remain present.

### Audit-Fields-Populated Assertion (Cases a + d)

Both case (a) and case (d) assert the response carries the orch's `approving_principal_id`, `approving_agent_instance_id`, `approving_lease_token`. Mechanism differs:

- Case (a) — stub-driven. The test sets `approveResult.Request.ApprovingPrincipalID = orch.principalID` (etc.) on the stub's response, then asserts the JSON-RPC structuredContent surfaces those values intact. Verifies the wire format / `omitempty` semantics for populated values.
- Case (d) — DB round-trip. After the JSON-RPC approve succeeds, the test calls `adapter.GetAuthRequest(ctx, fixture.pendingReq.ID)` to read the persisted row directly, then asserts `persisted.ApprovingPrincipalID == "STEWARD"` etc. Verifies 4a.26's audit columns persisted through the cross-subtree exception path AND that `json.Marshal(persisted)` emits the `approving_*` keys (omitempty does NOT elide populated values).

### `ErrOrchSelfApprovalDisabled` Mapping Refinement (Case e Note)

The handler's `mapToolError` switch (handler.go:893-966) currently has no case for `domain.ErrOrchSelfApprovalDisabled`. Case (e)'s wrapped sentinel falls through to the default `internal_error:` prefix. The error-message TEXT propagates faithfully ("orch self-approval disabled by project metadata" + "opted out of orch self-approval" wrap fragment), so the test asserts on message content not the prefix. **Refinement candidate (added to project_drop_4a_refinements_raised memory)**: map `ErrOrchSelfApprovalDisabled` to a sharper code (e.g. `auth_disabled` or extend the `auth_denied` branch) for client observability symmetry with the `auth_denied` path. Out of scope for 4a.27; routed for the drop-end refinements rollup.

### NIT-3 — Case-(e) Spec-Faithful Integration Pair

**Problem.** WAVE_3_PLAN.md:139 prescribes driving the toggle via `till.project(operation=update)` on a real project. The Round-1 unit test stubs `approveErr` to simulate the wrapped sentinel — proves wire-format-to-error mapping but not the toggle-gate-to-handler integration end-to-end.

**Fix.** Added a paired integration test alongside case (d) in `handler_steward_integration_test.go`:

`TestAuthRequestApproveProjectToggleDisabledRejectedIntegration` reuses `newStewardApproveFixture` (the case-(d) real-stack fixture: sqlite + autent + AppServiceAdapter + httptest handler + STEWARD session + persistent ancestor + pending cross-orch request). After fixture setup it flips the project's `Metadata.OrchSelfApprovalEnabled` to `*false` via `repo.UpdateProject`, then drives `till.auth_request(operation=approve)` over JSON-RPC with the STEWARD session, and asserts:

- HTTP-level: `isError=true`.
- Error-text: `Contains "orch self-approval disabled by project metadata"` (the `ErrOrchSelfApprovalDisabled` sentinel) AND `Contains "opted out of orch self-approval"` (the wrap fragment from `auth_requests.go:443`). Substring match (not prefix) — the test stays robust regardless of any future `mapToolError` refinement (NIT-1) that sharpens the error code from `internal_error:` to e.g. `auth_disabled:`.
- DB-level backstop: `persisted.State == "pending"` (toggle gate fires BEFORE the approve mutation persists; rejected approval must NOT have leaked an audit trail). Also asserts `persisted.ApprovingPrincipalID == ""` to guard against partial-write regressions.

The **why** behind the choice to flip the toggle via `repo.UpdateProject` rather than the `till.project(operation=update)` MCP wire format: the `till.project(operation=update)` MCP layer is already exercised end-to-end by `TestProjectMCPFirstClassFieldsRoundTrip` (extended_tools_test.go) and the metadata-typed pointer round-trip is covered at the adapter layer by `TestApproveAuthRequestAdapterToggleDisabled` (app_service_adapter_auth_requests_test.go:1135-1230, which uses the same `repo.UpdateProject` pattern this test follows). The new test owns the **toggle-gate-to-handler integration** specifically — flipping via the repo and approving via JSON-RPC isolates that contract without redundant coverage of the project-update MCP wire format. The orchestrator's NIT-3 spec explicitly accepted this route ("creates a project with `OrchSelfApprovalEnabled = *false` (or flips an existing project's toggle via real `till.project(operation=update)` MCP call)").

KEPT: the existing unit-style `TestAuthRequestApproveProjectToggleDisabledRejected` (handler_test.go) — covers the wire-format-to-stub-error path. The two tests are complementary.

The test asserts the toggle as a **TOTAL backstop** — it rejects even the STEWARD cross-subtree path that case (d) proves works under default-enabled metadata. This mirrors the service-layer `steward_cross_subtree` sub-case in `TestApproveAuthRequestRejectsWhenProjectToggleDisabled` (auth_requests_test.go:1411-1452) but at the MCP-handler integration level.
