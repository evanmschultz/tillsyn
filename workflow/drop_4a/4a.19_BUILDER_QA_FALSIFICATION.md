## Nits (Non-Blocking)

### N1 — Empty `project.HyllaArtifactRef` propagates as blank token

`BuildSpawnCommand` rejects empty `project.RepoPrimaryWorktree` with `ErrInvalidSpawnInput`, but does NOT reject empty `project.HyllaArtifactRef` or empty `project.ID`. When either is empty, `assemblePrompt` emits a line with the key and no value (e.g. `hylla_artifact_ref: \n`). The agent receives a meaningless field. Per `internal/domain/project.go:18-25` the field is "free-form trimmed string — empty string is the meaningful zero value (project not yet wired to a Hylla artifact)" — so empty IS legal at the domain layer. But Wave 2 dispatcher consumption (per `WAVE_2_PLAN.md §2.6` line 198) lists `HyllaArtifactRef` as a required cross-wave field.

**Recommendation:** either (a) add an early-reject guard mirroring the worktree check (`ErrInvalidSpawnInput` if empty), OR (b) accept empty as valid and document that agents must handle the empty-ref case. The dispatcher contract should be explicit either way.

### N2 — `project.ID` empty propagation (same shape as N1)

Same gap on `project.ID`. The function reads `project.ID` for two purposes: the `ErrInvalidSpawnInput` error message (`project=%s`) AND the prompt body (`project_id: <ID>`). Empty `project.ID` produces a useless prompt token without rejecting upstream.

### N3 — `MaxBudgetUSD = 0` doc-vs-emit drift

`AgentBinding.Validate` documents `MaxBudgetUSD: 0` as "permitted; means unlimited at dispatcher's choice." `BuildSpawnCommand` emits `--max-budget-usd 0` verbatim. Claude CLI's interpretation of `--max-budget-usd 0` is "zero budget" (immediate abort), not "unlimited." Either the doc or the dispatcher needs to align. **Lowest-friction fix:** narrow the doc to "zero permitted; will be passed through verbatim" — matches actual behavior. **Higher-friction fix:** the dispatcher omits the flag entirely when `MaxBudgetUSD == 0`, materializing the "unlimited at dispatcher's choice" semantic.

### N4 — `*exec.Cmd` mutability for 4a.21 monitor

`BuildSpawnCommand` returns `*exec.Cmd` by pointer; `SpawnDescriptor` is by value. A caller between `BuildSpawnCommand` and 4a.21's monitor that mutates `cmd.Args` or `cmd.Dir` would diverge from `descriptor.Prompt` / `descriptor.WorkingDir`. Today the dispatcher is the single caller (per worklog `Coordination Risk Surfaced` section, no other package consumes these), but 4a.21 should document that the cmd is treated as immutable post-construction.
