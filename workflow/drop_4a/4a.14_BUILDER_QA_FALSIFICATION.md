### 1.1 Spec literal vs. struct reality — placeholder fields for future droplets (SOFT FINDING)

> Concrete impl `dispatcher` struct holds a `*app.Service` reference, a `LiveWaitBroker`, and zero-value lock-manager / walker / spawner / monitor fields (filled in by later droplets).

The builder's `dispatcher` struct (`internal/app/dispatcher/dispatcher.go:125-142`) carries four fields only: `svc`, `broker`, `opts`, `clock`. No zero-value `fileLockManager`, `packageLockManager`, `treeWalker`, `spawner`, or `processMonitor` fields exist on the struct.

**Verdict: SOFT — defensible YAGNI choice with explicit worklog justification + future-field map in doc-comments. Not a CONFIRMED counterexample. Surface for orchestrator awareness; if dev wants the literal spec shape, the fix is a follow-on builder pass adding placeholder fields.**
