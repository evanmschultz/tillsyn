## 3. Design Notes

### 3.1 Argv-0 hard-coding + test wrapper script

`BuildSpawnCommand` (4a.19) hard-codes argv[0] = `claude`. To run a fake-agent binary in `TestDispatcherRunCmdSpawnsAndReports`, the test stages a small shell wrapper named `claude` in a temp directory, prepends the directory to `PATH` via `t.Setenv`, and exec.Command's PATH lookup resolves "claude" to the wrapper. The wrapper delegates to `dispatcher/testdata/fakeagent.go exit0`. This:
- Reuses 4a.21's `testdata/fakeagent.go` — the documented `go build` carve-out applies to the test fixture compile (same rationale as 4a.21).
- Avoids exposing a `cmd.Path` override on `BuildSpawnCommand` (which would widen the dispatcher's API surface for a test-only concern).

The trade-off: `t.Setenv` is incompatible with `t.Parallel`. The spawn test runs serially; runtime is dominated by the one-shot fakeagent compile (~1s), so the loss is bounded.

### 3.2 `--dry-run` JSON shape

The CLI emits `SpawnDescriptor` as snake_case JSON (`agent_name`, `model`, `max_budget_usd`, `max_turns`, `mcp_config_path`, `prompt`, `working_dir`). Snake_case is the dev-script-friendly form (jq, python, shell) and isolates the dispatcher package's Pascal-case Go shape from CLI-surface drift. Implemented via a `spawnDescriptorJSON` named-type alias with a `MarshalJSON` method — the alias keeps the JSON tags out of the dispatcher package.

When `--dry-run` hits `ErrNoAgentBinding`, the CLI emits an empty JSON object on stdout + the reason on stderr, then exits 0. This keeps machine-driven callers from crashing on a non-JSON stream when the catalog is missing.

### 3.3 RunOnce 8-stage pipeline ordering

The pipeline is intentionally linear with no stage-skip branches. Each stage either short-circuits (skip / block / error) or proceeds. The order is:

1. Project resolution — needs item.ProjectID; skip on `app.ErrNotFound`.
2. Catalog decode + binding lookup — soft-fallback per Drop 3 droplet 3.12; missing catalog or missing kind binding both skip.
3. Walker eligibility predicate — uses sibling list to evaluate blocked_by + parent state; non-eligible items skip.
4. Sibling-overlap detection — same-parent overlap on paths/packages without explicit blocked_by inserts a runtime blocker + raises an attention row → ResultBlocked.
5. File + package lock acquire — cross-subtree holders also block; partial acquires roll back.
6. BuildSpawnCommand — defensive ErrNoAgentBinding fallback (stage 2 already filtered).
7. Walker.Promote — moves item to in_progress; ErrPromotionBlocked → ResultBlocked.
8. Monitor.Track — launches subprocess; ErrMonitorNotStarted → hard error.

Any error in stages 5-8 rolls back the locks acquired in stage 5 to keep manager state symmetric with the action item's state.

### 3.4 GetActionItem exposure check

Verified `Service.GetActionItem` was already public at `internal/app/auth_scope.go:21` (pre-existing — the 4a.21 QA-Proof S1 forward-pointer's stated case did not need to fire). Added `Service.GetProject` to mirror the same shape for the dispatcher's project-row needs.

### 3.5 Documented limitation: parallel-CLI race on lock managers

Per the spec's plan-QA falsification mitigation: each `till dispatcher run` invocation bootstraps its own dispatcher (broker, locks, monitor). Two parallel invocations DO race on the in-process lock managers because each CLI process owns its own copy. The walker's eligibility predicate still holds across processes (it reads sqlite state), so the cross-process race only matters for the same-tick same-overlap case. Documented in:
- `dispatcher_cli.go` doc-comment on `dispatcherRunCommandOptions`.
- `cmd/till/main.go` `dispatcherCmd.Long` text.
- `cmd/till/help.go` long-form help spec.

Recommendation: dev runs invocations serially during the manual-trigger milestone. Drop 4b lands the daemon variant with a long-running shared-broker dispatcher.

### R2.1 §2.1 fix — Stage 8 monitor.Track failure no longer leaks phantom in_progress

Adopted **Option A** per spawn brief: when Stage 8 (`monitor.Track`) fails after Stage 7 already promoted the action item to `in_progress`, the dispatcher now transitions the item to `failed` (with `metadata.Outcome="failure"` + `metadata.BlockedReason="dispatcher: agent process failed to start: ..."`) and fires the cleanup hook so locks are released and the monitor's tracked-PID map is scrubbed. The originating Track error is still wrapped and returned so the dev sees both the failure surface and the reason.

**Implementation:**
- New `failureTransitioner` interface (`internal/app/dispatcher/dispatcher.go`) with `ListColumns` + `MoveActionItem` + `UpdateActionItem`. `*app.Service` satisfies it via `monitorServiceAdapter`; tests inject a stub.
- New `mutator` field on `*dispatcher`, wired in `NewDispatcher` to the adapter.
- New helper `transitionToFailed(ctx, item, reason)` mirrors the metadata shape `monitor.applyCrashTransition` writes (canonical "agent failure" surface across both spawn-fail-at-start and crash-mid-run paths).
- Stage 8 error path: `transitionToFailed` first, then `cleanup.OnTerminalState(ctx, failed)` with `LifecycleState=StateFailed` to drive the hook's terminal-state filter. If `transitionToFailed` itself errors, fall open with explicit `Release(item.ID)` on both lock managers.

**Test:** `TestRunOnceStage8FailureMarksActionItemFailed` (`internal/app/dispatcher/dispatcher_test.go`). Builds a rich fixture (`buildRichFixture` + `richDispatchService`), forces `cmd.Start` ENOENT by pointing PATH at an empty dir, asserts `errors.Is(err, ErrMonitorNotStarted)`, asserts `LifecycleState==StateFailed`, asserts `movedTo` order is `[col-inprogress, col-failed]`, asserts metadata `Outcome="failure"` + non-empty `BlockedReason`.
