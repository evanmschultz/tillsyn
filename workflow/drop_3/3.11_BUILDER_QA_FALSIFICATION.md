## 2. Refuted Attacks

### 2.1. Map-iteration nondeterminism — REFUTED

`ChildRulesFor` iterates `t.ChildRules` (a `[]ChildRule` slice — declaration order is deterministic), not `t.Kinds` (a map). The map is only consulted via single-key lookup `t.Kinds[rule.CreateChildKind]` per rule, which is deterministic for any given key. `sort.SliceStable` finalizes order on `(StructuralType, Kind)`. `TestChildRulesFor_DeterministicOrder` exercises two back-to-back calls and asserts `reflect.DeepEqual`. Cannot construct a flake.

### 2.2. Sort key total ordering — REFUTED

The comparator chain is `StructuralType < StructuralType` → tiebreak on `Kind < Kind`. Both axes are string-typed, so `<` is byte-lexicographic and total. `SliceStable` preserves declaration order on full ties (identical `(StructuralType, Kind)` from different `WhenParentKind` parents). The fixture has no full ties, but the algorithm tolerates them. Note: a resolution with empty `StructuralType` (missing-from-`Kinds` lookup) sorts BEFORE all non-empty values — surprising but deterministic; flagged in nit 4.1 below.

### 2.3. Filter precedence on empty `WhenParentStructuralType` — REFUTED

`child_rules.go:102` reads `if rule.WhenParentStructuralType != "" && rule.WhenParentStructuralType != parentType`. Empty `WhenParentStructuralType` short-circuits the filter — the rule fires for any `parentType`. `TestChildRulesFor_BuildSpawnsBuildQA` (parent=build, type=droplet) and `TestChildRulesFor_PlanSpawnsPlanQA` (parent=plan, type=droplet) confirm this: both rule-pairs declared with empty `WhenParentStructuralType` fire correctly.

### 2.4. Empty `Kinds` resolution — REFUTED

When `t.Kinds[rule.CreateChildKind]` misses (zero-valued or nil-map lookup), Go returns the zero `KindRule` — no panic. The resolution carries empty `StructuralType` + empty `Owner`. `TestChildRulesFor_EmptyTemplateReturnsEmpty` exercises a fully-zero `Template{}` value across the 12-kind × 5-structural-type cartesian product (60 combos), all expected to return empty slices. The narrower case "non-empty `ChildRules` with nil `Kinds`" is not exercised explicitly; covered indirectly because no rule fires when `ChildRules` is nil and any single missing-kind lookup just produces an empty resolution row. Not a counterexample — flagged as a coverage micro-nit.

### 2.5. PLAN.md line 1635 fidelity — REFUTED

Line 1635 specifies (a) `build` → `build-qa-proof` + `build-qa-falsification`, (b) `plan` → `plan-qa-proof` + `plan-qa-falsification`, (c) `structural_type=drop` → planner droplet + qa-proof droplet + qa-falsification droplet. The fixture encodes (a) and (b) as universal `WhenParentKind=…` rules with empty `WhenParentStructuralType`, and (c) as `WhenParentKind=Plan` AND `WhenParentStructuralType=Drop` adding ONE extra `KindPlan` resolution (the "planner droplet") on top of the unconditional plan-QA pair. `TestChildRulesFor_DropStructuralTypeSpawnsThreeChildren` asserts exactly three resolutions: `KindPlan` (planner droplet) + `KindPlanQAFalsification` + `KindPlanQAProof`. The "planner droplet IS itself kind=plan" interpretation is reasonable — the planner agent runs on plan-kind nodes, and Drop 1.75 did NOT add a separate `drop` kind. Encoding is faithful to line 1635.

### 2.6. One-level-only — REFUTED

`ChildRulesFor` body is iteration + sort + return. No recursion, no self-call, no expansion of children-of-children. The doc comment at lines 87–90 explicitly defers recursive expansion to the dispatcher (Drop 4). A chain `A → B → C` declared as two `[child_rules]` rows: calling `ChildRulesFor(A)` returns only `{B}`, never `{B, C}`. Construction confirmed by code reading; no test specifically exercises a chain, but the absence of recursion is structural.

### 2.7. Pure-function — REFUTED

`t Template` is a value receiver — even if the body mutated `t`, the mutation could not escape. The body reads `t.ChildRules` + `t.Kinds[…]`; allocates a fresh `out` slice each call (`make([]ChildRuleResolution, 0, len(t.ChildRules))`); appends locally; sorts the local slice; returns. No I/O, no logging, no DB access, no global state, no goroutine spawn. `TestChildRulesFor_DeterministicOrder` confirms idempotence by calling twice on the same template and asserting `reflect.DeepEqual`. The returned slice is freshly-backed each call — no shared backing array between successive invocations.

### 2.8. `errors.AsType[T]` correctness — REFUTED

Original: `var strictErr *toml.StrictMissingError; errors.As(err, &strictErr)`. New: `strictErr, ok := errors.AsType[*toml.StrictMissingError](err)`. Verified `go doc errors.AsType` (Go 1.26+ stdlib): "AsType finds the first error in err's tree that matches the type E, and if one is found, returns that error value and true. Otherwise, it returns the zero value of E and false." Same DFS-unwrap traversal as `errors.As`, same target shape (target type is the type parameter `E` rather than a pointer-target to a typed local — equivalent matching semantics). The wrapped fmt.Errorf call sites use `strictErr.String()` identically. No semantic delta.

### 2.9. Hardcoded `Persistent=false`, `DevGated=false` — REFUTED (with nit 4.2)

`child_rules.go:111-112` hardcodes both axes. Verified the schema doesn't expose them: `ChildRule` (`schema.go:118-140`) has 5 fields — `WhenParentKind`, `CreateChildKind`, `Title`, `BlockedByParent`, `WhenParentStructuralType`. `KindRule` (`schema.go:91-112`) has 4 fields — `Owner`, `AllowedParentKinds`, `AllowedChildKinds`, `StructuralType`. Neither carries `Persistent` or `DevGated`. The hardcoded `false` is therefore the only legal source value today. Confirmed not reading from `KindRule` or `ChildRule`. Documentation at lines 17-20 + 41-50 calls this out as a forward-compat axis. Nit 4.2 below: the docstring's reference to "Drop 3.13's expanded validator" is incorrect — 3.13 is `AgentBinding` validation, NOT `KindRule` schema expansion. Doc-drift nit only.

### 2.11. YAGNI — REFUTED

The function is 27 lines including the docstring. No premature abstraction: no recursion, no DB hooks, no event emission, no memoization, no parameterized comparators, no plugin interfaces. `ChildRuleResolution` carries exactly the 7 fields the auto-gen consumer (3.20) needs. Nothing speculative.

### 2.12. Returned-slice mutation safety — REFUTED

Each call freshly allocates `out := make([]ChildRuleResolution, 0, len(t.ChildRules))`. Caller mutating the returned slice cannot affect the next call's result. Confirmed by `DeterministicOrder` test — two calls return separately-allocated identical slices.

### 2.13. Pre-allocation correctness — REFUTED

`make([]…, 0, len(t.ChildRules))` pre-allocates capacity equal to the worst-case match count. If most rules don't match, cap exceeds len — wasted ~24 bytes per unmatched rule. Negligible at template scale (≤ a few dozen rules). Correctness unaffected.

## 4. Nits

### 4.1. Sort with empty `StructuralType` is silently order-bias-positive

When `t.Kinds[rule.CreateChildKind]` misses, the resolution carries `StructuralType=""`. Empty string sorts BEFORE all non-empty `StructuralType` values — so a missing-Kinds-lookup resolution lands at index 0 of the result. Today `validateMapKeys` + load-time gates prevent this in TOML-loaded templates, but `ChildRulesFor` is also reachable from hand-built templates (test fixtures, in-memory construction). No broken invariant — just a sort-ordering footgun if a test author ever omits a `t.Kinds` row matching a `CreateChildKind`. Defer; documenting only.

### 4.2. Docstring drift on Drop 3.13

`child_rules.go:17-20` says "the methodology calls them out as forward-compat axes the auto-generator reads from KindRule once Drop 3.13's expanded validator lands." Verified droplet 3.13 (PLAN.md:325-337) is `AgentBinding` validation only — it does NOT add `Persistent` / `DevGated` to `KindRule`. The eventual landing point is unspecified in current PLAN.md. Recommend rewording to "until a future droplet expands the schema" or naming the actual landing droplet (currently TBD). Non-blocking — doc-only, no functional impact.

### 4.3. Empty-Kinds-with-non-empty-ChildRules path uncovered

`TestChildRulesFor_EmptyTemplateReturnsEmpty` exercises a wholly zero `Template{}`. The case "ChildRules populated, Kinds map nil or missing the matched CreateChildKind" is not exercised explicitly. Behavior is correct (empty `KindRule{}` zero value, no panic, just empty `StructuralType`+`Owner` on the resolution), but a one-line table row asserting "missing-Kinds resolution carries empty StructuralType" would close the gap. Pure coverage micro-nit; coverage gate already at 96.0%.

## 5. Verdict Summary

**PASS-WITH-NIT.**

`Template.ChildRulesFor` correctly implements droplet 3.11's acceptance criteria:

- Pure function on `Template` value receiver, no I/O, no side effects.
- Iterates the deterministic `ChildRules` slice; `Kinds` is single-key-lookup only.
- Filter logic on `WhenParentKind` + optional `WhenParentStructuralType` is correct, including the empty-string universal-allow short-circuit.
- Sort is `SliceStable` on `(StructuralType, Kind)` with declaration-order tiebreaker — total ordering, deterministic.
- `Persistent`+`DevGated` hardcoded `false` is consistent with the current schema's axes — not a regression, deferred to a later schema expansion.
- One-level-only — no recursion.
- 7 tests + 60 cartesian rows; coverage 96.0%; `mage ci` 1753/1753 green.
- `errors.AsType[T]` rewrite in `load.go` preserves original semantics.

Three nits are all non-blocking (sort ordering footgun on missing-Kinds-lookup, docstring drift on Drop 3.13 reference, an uncovered narrow coverage row). None constitute a counterexample.

Recommend tightening the docstring (nit 4.2) before merge if the orchestrator wants doc-cleanliness; otherwise ship.
