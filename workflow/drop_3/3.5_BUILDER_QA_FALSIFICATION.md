## 2. Refuted Attacks

### 2.1 Empty `StructuralType` deserialize attack — REFUTED

**Hypothesis:** A pre-3.5 snapshot lacks `structural_type`. After unmarshal `t.StructuralType == ""`. If `toDomain` silently default-filled (like `Kind`'s `KindPlan` fallback at `snapshot.go:1278-1282`), schema drift would be masked and 3.2's `NewActionItem` validator would never see it.

**Evidence:**
- `snapshot.go:1319-1324` carries an explicit doc-comment plus `StructuralType: t.StructuralType` with NO `if t.StructuralType == ""` fallback. Empty rides through to the hydrated `domain.ActionItem`.
- `snapshot.go:402-409` (legacy-`Kind`-import block in `Validate`) does NOT touch `t.StructuralType`. The validator is intentionally permissive on this field per the legacy-compat contract.
- 3.2's `domain.NewActionItem` (`internal/domain/action_item.go:243-249`) rejects empty `StructuralType` with `ErrInvalidStructuralType` on the next mutation. The handoff to 3.2's gate is intact.

**Trace:**
- Legacy snapshot JSON → `t.StructuralType = ""` (omitempty key missing from input) → `Validate` skips → `toDomain` returns `domain.ActionItem{StructuralType: ""}` → caller next mutation → `NewActionItem` rejects.

**Conclusion:** REFUTED. Behaviour matches contract literally.

### 2.2 `omitempty` semantic attack on typed-string `StructuralType` — REFUTED

**Hypothesis:** `omitempty` on a `domain.StructuralType` typed string omits when underlying string is empty. If Go's `encoding/json` had any subtlety around named-string-types, the wire format could deviate.

**Evidence:**
- Go's `encoding/json` treats named-string types identically to `string` for omitempty: zero value is `""` for both. Confirmed via Drop 2.2 `Role` precedent at `snapshot.go:63` (`json:"role,omitempty"` on `domain.Role`) — same machinery, demonstrated working in `TestSnapshotActionItemRoleJSONShape:517-573`.
- The new tag `json:"structural_type,omitempty"` (`snapshot.go:64`) is structurally identical.

**Conclusion:** REFUTED. Same idiom that already works for `Role`.

### 2.3 `omitempty` semantic attack on `bool Irreducible` — REFUTED with NIT

**Hypothesis:** `omitempty` on `bool` omits when `false`. The dominant case is `false`. Legacy snapshots without the key deserialize to `false` regardless. Could a true-value loss be masked by the test fixture matching the implicit zero?

**Evidence:**
- `snapshot.go:65` — `Irreducible bool \`json:"irreducible,omitempty"\``. Standard Go bool omitempty: omitted when `false`, present when `true`. Correct legacy behaviour: missing key → `false`, which matches pre-3.5 reality.
- `TestSnapshotActionItemIrreducibleRoundTripPreservesBothStates` (`snapshot_test.go:626-657`) covers `irreducible=true` (the discriminating case) — if any layer drops the bit, the assertion at `:649` (snap) or `:653` (hydrated) fires.
- Builder's own docstring at `snapshot_test.go:611-617` calls out the limitation explicitly: the in-memory path doesn't pass through JSON; the false case can't distinguish "preserved" from "dropped + re-zero." This is honest and the true case carries the load.

**Conclusion:** REFUTED. NIT (§ 4.1): no JSON-level wire test for `Irreducible` parallel to `TestSnapshotActionItemRoleJSONShape`.

### 2.5 Snapshot version bump — REFUTED (correctly NOT bumped)

### 2.6 Cross-field interference — REFUTED

**Hypothesis:** Inserting two fields between `Role` (line 63) and `LifecycleState` (line 66) could shift JSON ordering, struct-literal keying, or accidental binding to adjacent fields.

**Evidence:**
- Go struct field order does NOT affect `json.Marshal` output beyond the key/value pairs themselves; `json.Unmarshal` is key-driven, not positional.
- `snapshotActionItemFromDomain` (`snapshot.go:1062-1093`) and `toDomain` (`snapshot.go:1271-1347`) use struct literals with named keys — order-independent.
- `domain.ActionItem` field order is `Role, StructuralType, Irreducible, Owner, DropNumber, Persistent, DevGated, ...` (`internal/domain/action_item.go:33-90`). Snapshot order is `Role, StructuralType, Irreducible, LifecycleState, ...`. The skipped fields (`Owner / DropNumber / Persistent / DevGated`) are 3.17 work — correctly NOT in 3.5 scope.
- The Role assertion in the new tests (where `Role` is unset) confirms no clobbering: `TestSnapshotActionItemStructuralTypeRoundTripPreservesAllValues` runs under `NewActionItemForTest` which doesn't touch `Role`, and `original.Role == ""` propagates through cleanly (otherwise the existing `TestSnapshotActionItemRoleEmptyRoundTripsEmpty` would now fail — and `mage test-pkg ./internal/app` reports 287/287 green).

**Conclusion:** REFUTED. No interference.

### 2.8 `NewActionItemForTest` test-fixture interaction — REFUTED

**Hypothesis:** The test helper at `internal/domain/action_item_test_helpers.go:30-35` defaults empty `StructuralType` to `StructuralTypeDroplet`. In `TestSnapshotActionItemIrreducibleRoundTripPreservesBothStates`, neither test case sets `StructuralType`, so both rows construct with `StructuralType=StructuralTypeDroplet` and pass through snapshot. Could this mask a bug?

**Evidence:**
- The Irreducible test asserts on `snap.Irreducible` and `hydrated.Irreducible` — orthogonal axes. Whatever `StructuralType` is set to (droplet by default), it survives orthogonally.
- Concretely: `original.StructuralType == StructuralTypeDroplet`, `original.Irreducible == tc.irreducible`. Snapshot preserves both. Hydrate preserves both. The Irreducible test doesn't assert on `StructuralType`, but it doesn't need to — that's the StructuralType test's job.
- Cross-coverage: `TestSnapshotActionItemStructuralTypeRoundTripPreservesAllValues` sets `Kind=KindBuild` and varies `StructuralType` — `Irreducible` defaults to `false` there. Both axes are covered, and a regression in either copy assignment would break at least one of the four StructuralType cases or the `irreducible=true` Irreducible case.

**Conclusion:** REFUTED. Test orthogonality holds; helper-default doesn't mask either axis.

## 3. Most Damaging Counterexample (Hypothetical)

The strongest attack I could mount is **the one the builder pre-empted in their own docstring**: the new tests do NOT cross the JSON wire boundary. A typo'd JSON tag (e.g. `json:"structuraltype,omitempty"` vs `json:"structural_type,omitempty"`) would PASS every new test (in-memory copy unaffected) yet silently corrupt every snapshot written to disk.

**Why this is a NIT, not a CONFIRMED counterexample:**

1. The tag in the diff (`snapshot.go:64`) literally reads `json:"structural_type,omitempty"` — the correct snake_case form. Manually inspected.
2. The `Role` precedent's `TestSnapshotActionItemRoleJSONShape` (`snapshot_test.go:517-573`) is the canonical guard against this class of bug, but the contract for 3.5 only required the all-values round-trip mirror, not the JSON-shape mirror. Builder met the literal letter.
3. No CONFIRMED breakage — just thinner regression coverage than the Role precedent.

**Recommended NIT (defer-able):** in a follow-up droplet (or 3.6/3.7 sweep), add `TestSnapshotActionItemStructuralTypeJSONShape` mirroring `TestSnapshotActionItemRoleJSONShape:517-573` so a future tag typo is caught at unit-test time rather than during a downstream snapshot import. Same for `Irreducible`. Optional, not blocking.
