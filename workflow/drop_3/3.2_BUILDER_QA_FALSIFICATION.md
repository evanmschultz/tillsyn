## 1. Counterexamples

### 1.1 [LOW — coverage gap, not contract break] No test exercises `Irreducible: true` round-trip

- **Hit:** Attack vector #13 — `Irreducible` field landing.
- **Claim:** "`Irreducible` is a `bool` with no validation, defaults `false`. Every behaviour is covered."
- **Counterexample:** `git grep -n "Irreducible" -- "*.go"` returns six hits: 4 in `internal/domain/action_item.go` (struct definitions + assignment in the constructor return literal at `:238`) + the schema-coverage read-only entry in `internal/tui/model_test.go:14868`. **Zero test fixtures and zero assertions construct an `ActionItem` with `Irreducible: true`.** A regression that drops the `Irreducible: in.Irreducible` assignment in `NewActionItem` (action_item.go:238) would silently flip every input's `Irreducible: true` back to `false` and `mage ci` would still report 1525/1525 green.
- **Repro (hypothetical):** delete line 238 (`Irreducible: in.Irreducible,`). Run `mage test-pkg ./internal/domain` — passes. Run `mage ci` — passes. The bug ships.
- **Mitigation:** PLAN.md line 144 explicitly says `Irreducible` is a bool with no validation; PLAN does not REQUIRE a round-trip test. The droplet's spelled-out acceptance criteria do not include "exercise `Irreducible: true`." Future droplets (3.3 SQLite round-trip; 3.5 snapshot round-trip) will exercise persistence and likely catch this incidentally. Recommend: add one sub-case to `TestNewActionItemStructuralTypeValidation` (or a new `TestNewActionItemIrreducibleRoundTrip`) that constructs an `ActionItem` with `Irreducible: true` and asserts `actionItem.Irreducible == true`. ~6 LOC.
- **Severity:** LOW.

### 1.3 [LOW — documented limitation, no contract break] `NewActionItemForTest` is exported from a regular `.go` file (not `_test.go`)

- **Hit:** Attack vector #2 — helper visibility.
- **Claim:** The helper is test-only.
- **Counterexample:** `ls internal/domain/action_item_test_helpers.go` — file is `*.go`, NOT `*_test.go`. Production code in any package can import `domain.NewActionItemForTest` to bypass `ErrInvalidStructuralType` enforcement of the empty case (helper defaults empty `StructuralType` to `StructuralTypeDroplet` then delegates).
- **Repro:** A hypothetical attacker writes `domain.NewActionItemForTest(domain.ActionItemInput{...empty StructuralType...}, time.Now())` in a non-test file — compiles, runs, returns a valid `ActionItem` with `StructuralType: StructuralTypeDroplet`. The empty-input rejection is silently overridden.
- **Mitigation:** PLAN.md line 135 explicitly anticipates this: "recommended `internal/domain/action_item_test_helpers.go` mirroring the Drop 1.75 `Kind` precedent." Cross-package fixture helpers cannot live in `_test.go` files because Go's test-only build constraint prevents cross-package import. The doc comment at lines 14-29 of the helper file documents the intent and explicitly cites a future hardening: "Future hardening could move this helper to a dedicated `domaintest` sub-package so it is unimportable from non-test files." Today, every actual call site IS in `_test.go` files (verified via `git grep -nE "NewActionItemForTest" -- "*.go"` — all callers are `*_test.go`).
- **Severity:** LOW (intentional architectural tradeoff documented in both the PLAN and the helper's doc comment).

### 1.4 [LOW — known limitation flagged by builder] Snapshot + SQLite round-trip silently DROPS `StructuralType` and `Irreducible`

- **Hit:** Attack vectors #8 + #9 — snapshot serialization, SQLite persistence.
- **Claim:** 3.2 is scoped to in-memory `ActionItem` validation only; persistence + serialization are deferred to droplets 3.3 (SQLite) and 3.5 (snapshot).
- **Counterexample:** `internal/app/snapshot.go:1060` (`snapshotActionItemFromDomain`) does NOT propagate `StructuralType` / `Irreducible`; `internal/app/snapshot.go:1267` (`SnapshotActionItem.toDomain`) builds `domain.ActionItem` by direct struct literal, NOT through `NewActionItem`, so the round-trip silently produces zero-value `StructuralType: ""` and `Irreducible: false`. Same pattern in `internal/adapters/storage/sqlite/repo.go` — no `structural_type` / `irreducible` columns yet. A test that constructs an `ActionItem` via `NewActionItemForTest` (default `StructuralTypeDroplet`), persists or serializes it, and reads it back gets back an `ActionItem` whose `StructuralType` is empty.
- **Repro:** Construct `original := NewActionItemForTest({Kind: KindBuild, ...}, now)` — `original.StructuralType == StructuralTypeDroplet`. Run through `snapshotActionItemFromDomain → SnapshotActionItem.toDomain` — got back `roundTripped.StructuralType == ""`. The two values disagree.
- **Mitigation:** PLAN.md line 200 (droplet 3.5) explicitly lands snapshot serialization. PLAN.md line 159+ (droplet 3.3) lands SQLite columns + scanner. The 3.2 builder did NOT modify snapshot.go / repo.go (verified — `git diff --stat` shows neither file changed). NO test in 3.2 asserts post-round-trip `StructuralType` equality, so `mage ci` doesn't catch the silent loss. This is by design: scope-narrowing per the PLAN's droplet split.
- **Severity:** LOW (deferred to 3.3 + 3.5; verified no test asserts the round-trip; no contract break for 3.2's narrow scope).

## 3. Most Damaging Counterexample

**§ 1.1 — No test exercises `Irreducible: true` round-trip.**

A regression that silently drops the `Irreducible: in.Irreducible,` assignment at `internal/domain/action_item.go:238` would not be caught by `mage ci`. Severity is LOW because:

- The PLAN does not REQUIRE an `Irreducible` round-trip test.

But it is a real gap — a one-sub-case test would close it cheaply (~6 LOC under `TestNewActionItemStructuralTypeValidation` or a sibling).
