## 1. Counterexamples

### 1.1 [CONFIRMED] Acceptance bullet "parses from TOML duration strings (`"30s"`, `"5m"`)" is unmet ‚Äî load path cannot decode `blocked_retry_cooldown = "30s"`

**PLAN.md droplet 3.13 acceptance line 333** says:

> `BlockedRetryCooldown` parses from TOML duration strings (`"30s"`, `"5m"`).

**Schema.go line 173** (doc-comment on the field) restates this:

> `// attempts. Parsed from TOML duration strings such as "30s" or "5m".`

But `BlockedRetryCooldown` is `time.Duration`, an alias for `int64`, with **no** `encoding.TextUnmarshaler` method. `pelletier/go-toml/v2`'s `go doc` is explicit:

> *"Types implementing the `encoding.TextUnmarshaler` interface are decoded from a TOML string."*

Live reproduction (verified):

```go
type X struct {
    D time.Duration `toml:"d"`
}
err := toml.Unmarshal([]byte(`d = "30s"`+"\n"), &x)
// err = "toml: cannot decode TOML string into struct field main.X.D of type time.Duration"
// x.D = 0s
```

The test (`agent_binding_test.go:38-54`) round-trips the value as the **integer-nanosecond** form that `pelletier/go-toml/v2` actually emits ‚Äî `blocked_retry_cooldown = 30000000000` ‚Äî not as `"30s"`. Marshal‚ÜíUnmarshal succeeds because both ends use the same int64 wire form. The acceptance bullet's "duration strings" claim is never exercised.

**Severity**: PASS-WITH-NIT. The droplet's *test* claim (round-trip stability for fully-populated AgentBinding) is honestly met; the acceptance-bullet's stronger ergonomic claim (TOML-`"30s"` decode) is **not** met by this droplet's code AND will fail the moment any `default.toml` (3.14) writes the field as a quoted-string TOML value.

**Routing recommendation**: either (a) drop the "duration strings" half of acceptance line 333 + retract the field doc-comment claim ‚Äî accept that templates author the value as integer-nanoseconds ‚Äî or (b) add a string-form decoder hook in 3.14 (custom UnmarshalTOML on a wrapper type, OR replace the field with `string` and parse via `time.ParseDuration` at load). Either choice is a follow-up droplet, not a 3.13 fix; (a) is YAGNI-cleanest because the field is read by the dispatcher (Drop 4) which can do its own parsing.

## 3. Most Damaging Counterexample

**1.1** ‚Äî acceptance bullet "parses from TOML duration strings (`"30s"`, `"5m"`)" is NOT satisfied by 3.13's implementation. `pelletier/go-toml/v2` will `toml.Unmarshal` error on a TOML document that writes `blocked_retry_cooldown = "30s"`, because `time.Duration` does not implement `encoding.TextUnmarshaler`.

This is "most damaging" because:

1. It's an unmet acceptance bullet, not a code defect ‚Äî the test passes by exercising the integer-nanosecond round-trip the package *can* do, not the string-form round-trip the acceptance line claims.
2. It will surface as a real failure the moment 3.14's `default.toml` writes durations as `"30s"` strings ‚Äî which is the natural author-ergonomic TOML form most humans write.
3. The droplet's own field-doc-comment (1.2) repeats the same false claim, propagating the misconception forward.

It is **NOT** a blocker for 3.13 closing because 3.13's test only claims marshal‚Üíunmarshal round-trip stability for the populated struct, which is honestly green. But the acceptance bullet should be either dropped, retracted in the field doc, OR satisfied via a string-decode hook in a follow-up droplet (most likely a 3.14 task or a Drop 4 dispatcher concern). Verdict downgrade: **PASS-WITH-NIT**.
