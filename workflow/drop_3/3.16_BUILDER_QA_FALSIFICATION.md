## 1. Counterexamples

### 1.1 — CONFIRMED — Path-and-scope violation: 3.18 SQLite work bundled into 3.16

**Premises (declared by droplet 3.16):**

- Paths: `internal/app/kind_capability.go`, `internal/app/kind_capability_test.go`, `internal/domain/attention.go`, `internal/app/inbox_attention.go`.
- Packages: `internal/app`, `internal/domain`.

**Evidence:**

- `git diff --stat` reports seven modified files, four of them outside the declared paths:
  - `internal/adapters/storage/sqlite/repo.go` (38 lines changed — schema columns + INSERT/UPDATE/SELECT + scanner + 2 indexes).
  - `internal/adapters/storage/sqlite/repo_test.go` (256 lines added — `TestRepository_PersistsActionItemOwnerAndDropNumber`).
  - `internal/adapters/storage/sqlite/embedding_lifecycle_adapter_test.go` (legacy-migration test updated for new columns).
  - `internal/app/service.go` (CreateActionItem wiring — undeclared but architecturally necessary; see 1.2).
- The four SQLite changes implement four columns on `action_items`: `owner`, `drop_number`, `persistent`, `dev_gated` — verbatim the schema deltas droplet **3.18** declares ("`internal/adapters/storage/sqlite/repo.go`, `internal/adapters/storage/sqlite/repo_test.go`" + four named columns + scanActionItem + INSERT/UPDATE alignment + index design at PLAN.md 3.18 lines 426–429).
- Droplet 3.18 has not yet started (no `3.18_BUILDER_QA_*.md` files exist; commits stop at 3.17 + 3.15 docs).

**Trace:**

- 3.17 already shipped the four domain fields (commit `606241b`). Domain-only — repo persistence not required for compile.
- 3.16's audit-trail tests do NOT exercise `Owner` / `DropNumber` / `Persistent` / `DevGated` on persisted action items; they only exercise comment + attention persistence.
- Therefore the SQLite delta in 3.16's working tree is gratuitous to 3.16 and competes with 3.18's scope.

**Conclusion:** scope creep — 3.16 has accumulated 3.18's work. If committed as-is the 3.16 commit will straddle two droplets, leaving 3.18 with empty scope and a misaligned audit trail. Belongs in 3.18.

**Severity:** material — invalidates path-locking + sequencing discipline. Even though `mage ci` passes, the droplet boundary is wrong.

### 1.2 — NIT — `internal/app/service.go` modification undeclared

**Premises:** droplet 3.16's declared paths do not include `internal/app/service.go`.

**Evidence:** `internal/app/service.go` is modified (the new 30-line audit-fire block in `CreateActionItem` between lines 618 and 646).

**Trace:** the new helpers in `kind_capability.go` are dead code without this wiring — `CreateActionItem` is the only auth-gated create boundary and it lives in `service.go`. This is the **architecturally necessary** wiring site, but the planner failed to declare it. Two valid resolutions: (a) PLAN.md should add `internal/app/service.go` to 3.16's paths; (b) builder should have moved the helpers' caller into kind_capability.go (not viable — the create boundary is in service.go).

**Conclusion:** path-spec gap upstream of the builder, not a builder fault. Flag and continue.

**Severity:** nit — discovered at QA, planner should patch the droplet path list before commit.

### 1.3 — NIT — Half-write window on audit-trail persistence (docstring overstates atomicity)

**Premises:** `recordTemplateRejectionAudit` docstring says "Errors persisting either row bubble up so the create path returns an audit-failed error rather than a silent half-write."

**Evidence:**

- `kind_capability.go:679-727` performs two sequential writes: `s.repo.CreateComment` then `s.repo.CreateAttentionItem`.
- No transaction wrapper exists in `internal/app` (verified: `grep -rn "withTx\|BeginTx" internal/app/` returns 0 hits — consistent with the codebase's pattern, not unique to this builder).

**Trace:**

- If `CreateComment` succeeds and `CreateAttentionItem` fails, the comment row is persisted; the function returns the attention-write error; `CreateActionItem` returns that error (not `ErrKindNotAllowed`).
- A caller that does `errors.Is(err, domain.ErrKindNotAllowed)` (the public contract for nesting rejection) gets `false` despite the parent already having a rejection comment recorded.

**Trace or cases:** unobservable in unit tests because `fakeRepo.CreateComment` / `CreateAttentionItem` don't fail in practice. Theoretical for sqlite — both inserts are independent rows with no shared constraint.

**Conclusion:** docstring overstates the guarantee. Either (a) reword the docstring to say "best-effort: both writes are attempted; an error from either bubbles up", or (b) make the pair atomic via a single transaction. Today's behavior is consistent with the rest of the app layer's non-transactional pattern, so (a) is the smaller change. NIT.

**Severity:** nit — comment-language drift; no functional regression.

## 2. Refuted Attacks

### 2.1 — REFUTED — Empty-catalog audit attack

**Attack:** a project with empty `KindCatalogJSON` should NOT trigger audit on any create.

**Evidence:** `kind_capability.go:781-783`:

```go
if len(project.KindCatalogJSON) == 0 {
    return true, "", nil
}
```

`templateNestingDecision` returns `(allowed=true, reason="", err=nil)` — `CreateActionItem` line 635 `if !allowed` short-circuits to false; no audit fires.

**Conclusion:** behavior is correct — empty-catalog projects do not pay any audit cost.

### 2.2 — REFUTED — Update + Reparent path narrowness

**Attack:** an Update or Reparent that triggers a kind rejection should NOT write comment + attention; the resolver's own gate still fires.

**Evidence:**

- `service.go:1224 ReparentActionItem` does NOT call `templateNestingDecision`. It calls `validateActionItemKind` (line 1262), which routes through `resolveActionItemKindDefinition` — the resolver's catalog gate at line 607-614 still fires.
- `service.go:859 UpdateActionItem` similarly relies on resolver-level gates.

**Conclusion:** matches finding 5.B.15 N2 scope-narrow — audit fires only on auth-gated `CreateActionItem`. Behavior is intentional and correct.

### 2.3 — REFUTED — Project-allowlist rejection should not trigger template-rejection audit

**Attack:** a project-allowlist rejection (`ErrKindNotAllowed` from project-allowlist gate, NOT from catalog) should NOT trigger template-rejection audit.

**Evidence:**

- `templateNestingDecision` (kind_capability.go:769) examines ONLY the catalog's `AllowsNesting` decision.
- Project-allowlist enforcement lives in `resolveActionItemKindDefinition` lines 615-620, downstream of the audit-firing block, with no path back into `recordTemplateRejectionAudit`.

**Conclusion:** the two error sources are correctly differentiated. Project-allowlist rejection is silent (no audit row); catalog `AllowsNesting` rejection captures audit + raises `ErrKindNotAllowed`.

### 2.4 — REFUTED — Zero-id collision attack

**Attack:** `s.idGen()` deterministic IDs must not collide between the comment and the attention insert.

**Evidence:** `newDeterministicService` (kind_capability_test.go:23-26) uses an incrementing `idCounter`. Each call to `s.idGen()` returns a unique `id-<formatted>` string. `recordTemplateRejectionAudit` calls `s.idGen()` twice (once for comment, once for attention) with the counter monotonically incrementing.

**Conclusion:** no collision. The fallback-on-empty-id branches (`fmt.Sprintf("template-rejection-comment-%d", now.UTC().UnixNano())`) are dead in deterministic tests but provide a non-empty fallback if `idGen()` ever returns `""` in production paths.

### 2.5 — REFUTED — Wire-value naming attack

## 3. Most Damaging Counterexample

**1.1 — Path-and-scope violation: 3.18 SQLite work bundled into 3.16's working tree.**

The four sqlite-package changes implement droplet 3.18's exact acceptance criteria (`owner` / `drop_number` / `persistent` / `dev_gated` columns + INSERT/UPDATE/SELECT alignment + scanActionItem reads + the (project_id, drop_number, owner) compound index named in PLAN.md 3.18 line 429a). Droplet 3.18 has not started — there are no `3.18_BUILDER_QA_*.md` files. If 3.16's working tree is committed verbatim, two consequences:

1. **The 3.16 commit will be heterogeneous** — its diff straddles two droplets, breaking the per-droplet commit cadence and per-droplet QA boundaries. Future bisects + audit-trail readers will see "drop_number index added — but in the audit-trail droplet?" and have to reconstruct intent by archaeology.
2. **Droplet 3.18 will spawn into a state where its work is already done** — the builder will either find empty scope (and the QA twins will have no diff to review) or worse, will re-apply the same edits, generating spurious "no-op" commits.

The 3.16 commit must be split: keep `internal/app/kind_capability.go`, `internal/app/kind_capability_test.go`, `internal/domain/attention.go`, **and** `internal/app/service.go` (per nit 1.2 — declared-path patch first, then commit) on 3.16; revert all four sqlite changes locally and route them to 3.18 when its builder spawns. Without this split, 3.18 enters the cascade with corrupted scope.

## 4. Verdict Summary

**Verdict:** **FAIL**

Functional correctness of the audit-trail logic is solid — every targeted attack on the 3.16 acceptance contract refuted (2.1–2.12). The failure is **scope and ordering**, not behavior:

- **1.1 (material):** four SQLite-package files modified that belong to droplet 3.18; bundling will corrupt 3.18's spawn scope and split-history the 3.16 commit.
- **1.2 (nit):** `internal/app/service.go` was modified but not declared in 3.16's paths (architecturally necessary; planner-side path-spec gap).
- **1.3 (nit):** `recordTemplateRejectionAudit` docstring overstates atomicity — two non-transactional writes can leave a partial state surfaced as a non-`ErrKindNotAllowed` error.

**Required action before commit:**

1. Revert the four sqlite-package changes in the working tree (`internal/adapters/storage/sqlite/repo.go`, `internal/adapters/storage/sqlite/repo_test.go`, `internal/adapters/storage/sqlite/embedding_lifecycle_adapter_test.go`). Hold for droplet 3.18.
2. Patch droplet 3.16's PLAN.md path list to include `internal/app/service.go` (planner edit, not builder).
3. (Optional) Soften `recordTemplateRejectionAudit` docstring to match actual best-effort semantics, OR add a transaction wrapper if atomicity is desired.

Re-run QA after 1+2 land. The 3.18 droplet then spawns clean.
