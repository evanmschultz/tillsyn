### 1.1 NIT — STEWARD field-level guard does not cover Persistent or DevGated mutations (consistency gap)

**Severity**: NIT.

**Reproduction analysis.**

`internal/adapters/server/common/app_service_adapter_mcp.go:707-715` gates the STEWARD field-level guard on:

```go
if in.Owner != nil || in.DropNumber != nil {
    existing, fetchErr := a.service.GetActionItem(ctx, ...)
    if err := assertOwnerStateGateUpdateFields(ctx, existing, in.Owner, in.DropNumber); err != nil {
        return domain.ActionItem{}, err
    }
}
```

The gate fires only when Owner or DropNumber is supplied; Persistent and DevGated never trigger the gate's pre-fetch-and-check. Then `internal/app/service.go:1010-1017` unconditionally applies the dereferenced pointer values:

```go
if in.Persistent != nil {
    actionItem.Persistent = *in.Persistent
    actionItem.UpdatedAt = s.clock().UTC()
}
if in.DevGated != nil {
    actionItem.DevGated = *in.DevGated
    actionItem.UpdatedAt = s.clock().UTC()
}
```

So a non-steward (`agent`) caller can issue an update against a STEWARD-owned anchor with body `{"persistent": false, "dev_gated": false}` and silently flip both flags. The pointer-sentinel design — explicitly motivated in the wire-shape doc-comment at `mcp_surface.go:139-149` (`"a description-only update with a value-typed bool would silently clobber [Persistent=true] to false"`) — protects only against IMPLICIT clobbering (caller omits the field). EXPLICIT clobbering is unguarded.

**Why this is a NIT, not a CONFIRMED counterexample.**

PLAN.md line 447 (the L1 field-level guard requirement that 3.19 implements) reads literally:

> "REJECT any update whose `Owner` or `DropNumber` differ from existing values."

The plan scopes the gate to those two fields exclusively. 3.21 is the wire-plumbing droplet for all four fields; gating Persistent/DevGated is not part of 3.21's acceptance criteria nor 3.19's. The 3.21 acceptance criteria (PLAN.md lines 496-504) require only schema correctness, threading, and snapshot round-trip — none mention a write-guard for Persistent/DevGated.

**Routing recommendation.**

Surface as a Drop 3 refinement (`DROP_3_REFINEMENTS_RAISED`) rather than a 3.21 build-back. The widening question — should the STEWARD field-level guard cover all four primitives or only the two cited in the plan? — is a design decision that crosses 3.19's territory, not a builder error in 3.21.

### 1.3 Anti-counterexample — gate test re-fetch is genuine (attack vector #8 confirmed sound)

The 3.19 NIT-tightening at `app_service_adapter_steward_gate_test.go:217-223` reads back the action item via the adapter's `GetActionItem` and asserts `after.Owner == mutatedOwner`:

```go
after, getErr := fixture.adapter.GetActionItem(ctx, stewardGated.ID)
if getErr != nil {
    t.Fatalf("GetActionItem() after steward Owner mutation error = %v", getErr)
}
if after.Owner != mutatedOwner {
    t.Fatalf("UpdateActionItem(steward) did not persist Owner: got %q, want %q", after.Owner, mutatedOwner)
}
```

Pre-3.21, the test only asserted `err == nil` from the `UpdateActionItem` call — the field could have stayed at "STEWARD" (because the input wasn't wired through the service layer) and the test would pass. With 3.21's service-side `if in.Owner != nil { actionItem.Owner = strings.TrimSpace(*in.Owner); ... }` and the adapter's pointer-sentinel forwarding, the re-fetch now genuinely proves the field changed. The tightening is real.

## 2. Refuted Attacks

### 2.1 Pointer-sentinel false positive — REFUTED (attack vector #1)

A JSON request without an `owner` field leaves `args.Owner == nil` because `*string`-typed JSON fields default to nil when the key is absent. The MCP boundary at `extended_tools.go:1126-1135` forwards `args.Owner` (the pointer) verbatim into `UpdateActionItemRequest.Owner`. The `extended_tools_test.go` test "update without the four fields preserves prior via nil pointer-sentinels" (lines 3811-3833) exercises exactly this case:

```go
"operation":      "update",
"action_item_id": testActionItemUUID,
"title":          "ActionItem One Updated",
// owner / drop_number / persistent / dev_gated all omitted
```

then asserts `service.lastUpdateActionItemReq.Owner != nil` is false (i.e., nil, preserve). Same for the other three pointers. PASSES locally per `mage test-pkg ./internal/adapters/server/mcpapi`.

### 2.2 Pointer-sentinel false negative — REFUTED (attack vector #2)

Standard `encoding/json` unmarshal of `{"owner": null}` into `*string` leaves the pointer at nil; unmarshal of `{"owner": ""}` produces a non-nil pointer to the empty string. The boundary forwards either case verbatim:
- `null` → `nil` → no-op preserve at the service.
- `""` → `&""` → service applies `strings.TrimSpace("") = ""`. On a STEWARD-owned item by a non-steward caller, the adapter gate at line 712 fires (`assertOwnerStateGateUpdateFields` with `wantOwner = &""` and `existing.Owner = "STEWARD"` → trimmed compare differs → REJECTED with `ErrAuthorizationDenied`). On a non-STEWARD-owned item (or a steward-principal caller), the empty string applies cleanly.

Behavior is consistent with the documented contract on `mcp_surface.go:139-149` and `extended_tools.go:1411` (`"supplying any value (including empty string) triggers the L1 STEWARD field-level guard"`).

### 2.3 Legacy snapshot round-trip — REFUTED (attack vector #3)

`snapshot_test.go:760+` (TestSnapshotActionItemOwnerLegacyFormatCompatibility) parses a literal pre-3.21 JSON snapshot whose body lacks the four new fields and asserts:
- `snap.Owner == ""`
- `snap.DropNumber == 0`
- `snap.Persistent == false`
- `snap.DevGated == false`
- `snap.toDomain()` produces `domain.ActionItem` with the same zero values.

The `omitempty` tags on the four fields (`snapshot.go:66-69`) keep them out of newly-encoded snapshots when zero-valued, so existing fixtures survive forward-compat without a `SnapshotVersion` bump. The doc-comment at `snapshot.go:1336-1341` correctly explains the zero-value identity = legitimate-default invariant. No crash, no panic, no leaked non-zero values. PASSES locally.

### 2.4 Steward gate Owner-mutation — REFUTED (attack vector #4)

Test `TestAssertOwnerStateGateUpdateActionItemOwnerMutationAgentRejected` (`app_service_adapter_steward_gate_test.go:172-224`) directly exercises this:
- Agent-principal caller setting `Owner = ""` → `ErrAuthorizationDenied` (line 184-188).
- Agent-principal caller setting `Owner = "ROGUE"` → `ErrAuthorizationDenied` (line 191-198).
- Steward-principal caller setting `Owner = "ROGUE"` → SUCCESS, with re-fetch asserting `after.Owner == "ROGUE"` (line 209-223).

The gate at `app_service_adapter_mcp.go:707-715` correctly fires the pre-fetch + `assertOwnerStateGateUpdateFields` whenever `in.Owner != nil`. The test re-fetch closes the proof loop the 3.19 falsification flagged.

### 2.5 MCP tool schema correctness — REFUTED (attack vector #5)

`extended_tools.go:1411-1414` registers:
- `mcp.WithString("owner", ...)` → JSON Schema `string` matches Go `*string`.
- `mcp.WithNumber("drop_number", ...)` → JSON Schema `number` (covers ints + floats) matches Go `*int` (Go's `encoding/json` accepts integer-valued floats, e.g. `5.0` → 5; rejects `5.5`).
- `mcp.WithBoolean("persistent", ...)` → matches Go `*bool`.
- `mcp.WithBoolean("dev_gated", ...)` → matches Go `*bool`.

The pattern matches the existing `mcp.WithNumber("position", ...)` against `Position *int` (`extended_tools.go:1403`), which has shipped since pre-3.21. The mcpapi test "create plumbs owner/drop_number/persistent/dev_gated" sends `"drop_number": float64(7)` (Go's default JSON unmarshal of integer literals produces float64) and asserts `lastCreateActionItemReq.DropNumber == 7`. PASSES.

### 2.6 Cross-create-update consistency — REFUTED (attack vector #6)

The args struct in `extended_tools.go:872-875` uses `*string`, `*int`, `*bool`, `*bool` for the four fields, then on:

- **Create** (`extended_tools.go:1058-1075`) collapses pointer-sentinels onto value-typed fields on `CreateActionItemRequest`: nil → leave zero; non-nil → dereference. Service then trims Owner at the adapter boundary (`app_service_adapter_mcp.go:657-660`).
- **Update** (`extended_tools.go:1126-1135`) forwards pointer-sentinels verbatim onto `UpdateActionItemRequest`: nil → preserve; non-nil → mutate. Service trims Owner at line 1000.

Wire-shape asymmetry is intentional and documented (`mcp_surface.go:78-100` for Create plain types vs `:118-160` for Update pointer-sentinels). Both paths trim Owner at exactly one site. Consistent.

### 2.7 Snapshot round-trip exhaustiveness — REFUTED (attack vector #7)

The prompt cites "4 fields × 2 states = 8 cases." The snapshot test exercises 6 cases (all-zero, steward-anchor with all four populated, four single-axis variants) plus a separate JSON-legacy-format test. The 4×2=8 enumeration over individual fields is fully covered: each field is exercised in both its zero-value AND its populated state across the 6 cases. The combinatorial 2^4=16 exhaustive enumeration would catch a copy bug that mixed two specific fields and not others — but the in-memory round-trip (`snapshotActionItemFromDomain` then `snap.toDomain()`) doesn't go through JSON, so per-field copy bugs surface independently regardless of pairing. The Irreducible-round-trip precedent (`snapshot_test.go:625+`) used the same pattern. Adequate.

### 2.8 3.19 gate test tightening — REFUTED (attack vector #8)

Already covered in §1.3 (anti-counterexample). Re-fetch is genuine; the field is wired through; the assertion is meaningful. The PROOF reviewer's §1.7 finding aligned.

### 2.9 YAGNI scope check — REFUTED at acceptance bar (attack vector #9)

Inside the 3.21 declared paths, no YAGNI surfaces:
- `mcp_surface.go` adds exactly 4 fields × 2 structs = 8 wire-shape additions, each with load-bearing doc-comments.
- `app_service_adapter_mcp.go` adds 4 lines × 2 (create + update) = 8 threading lines + 1 gate pre-condition (line 707, scoped to Owner+DropNumber per plan).
- `extended_tools.go` adds 4 lines to args struct + 8 conditional collapses on Create + 4 verbatim forwards on Update + 4 `mcp.WithX` registrations. Mirrors existing `Position` precedent.
- `snapshot.go` adds 4 fields × 3 sites (struct, `from`, `to`).
- Tests: 1 round-trip test + 1 legacy-format test + 1 mcpapi 4-subtest table.

No abstraction without two consumers; no premature generalization; no helper "for future use." (The 3.20 surplus discussed in §1.2 is a scoping question, not a YAGNI question — that work has its own consumer in the auto-generator.)

## 3. Most Damaging Counterexample

None at CONFIRMED severity. The most material NIT-level finding is §1.1 — the STEWARD field-level guard's asymmetric coverage: Owner + DropNumber gated, Persistent + DevGated unguarded. The wire-shape doc-comment at `mcp_surface.go:139-149` and the schema description at `extended_tools.go:1413` both name the Persistent=true clobber risk explicitly, but the runtime gate enforces the protection only for the IMPLICIT-clobber case (caller omits the field) — not the EXPLICIT-clobber case (caller sends `persistent: false`).

Reproduction sketch (would land in Drop 3 refinement, not 3.21 build-back):

```go
// In a non-steward principal-typed session targeting a STEWARD-owned anchor:
adapter.UpdateActionItem(ctx, UpdateActionItemRequest{
    ActionItemID: stewardAnchor.ID,
    Title:        stewardAnchor.Title,
    Persistent:   ptr(false), // explicit clobber
    Actor:        agentActor,
})
// Returns nil error; subsequent GetActionItem shows Persistent = false.
// The STEWARD anchor's load-bearing flag is gone.
```

The plan's L1 line 447 only mandates the gate over Owner + DropNumber, so this is not a 3.21 builder error; it is a design widening question for 3.19's auth-layer scope.
