## 2. Refuted Attacks

### 2.1 REFUTED — Cycle trap (Attack 1)

`internal/domain/project.go` imports only `bytes`, `encoding/json`, `strings`, `time` — no `internal/templates`. `KindCatalogJSON` is `json.RawMessage`, not a typed `KindCatalog`. No methods on `Project` decode the envelope. L5 import direction is satisfied.

### 2.2 REFUTED — Deep-copy correctness (Attack 2)

`Bake` calls `cloneKindRule` (catalog.go:102-116) which `make`+`copy`s every slice (`AllowedParentKinds`, `AllowedChildKinds`); `cloneAgentBinding` (catalog.go:120-127) does the same for `Tools`. `TestKindCatalogBakeDeepCopiesSlices` mutates `tpl.Kinds[KindBuild].AllowedChildKinds[0]` and asserts the catalog's view is unchanged. Pass.

### 2.3 REFUTED — Fallback path safety (Attack 3)

Both empty-envelope and malformed-JSON paths fall back to `s.repo.GetKindDefinition`:

- `lookupKindDefinitionFromCatalog:621-622` — `len(project.KindCatalogJSON) == 0` → `(zero, false, nil)`.
- `lookupKindDefinitionFromCatalog:624-630` — `json.Unmarshal` failure → `(zero, false, nil)` (soft fallback).

Both paths covered by `TestKindCatalogResolutionFallsBackToRepoOnEmpty` and `TestKindCatalogResolutionFallsBackOnMalformedJSON`. Neither path panics.

### 2.4 REFUTED — Catalog-first preference (Attack 4)

`TestKindCatalogResolutionFromBakedCatalog` deletes `repo.kindDefs[KindBuild]` after baking the catalog; the resolver succeeds, proving the catalog path was hit. The synthesized `DisplayName="build"` (lower-case kindID) versus the legacy `DisplayName="Build"` cleanly distinguishes the two paths. Pass.

### 2.5 REFUTED — `Lookup` zero-value safety (Attack 5)

`Lookup` (catalog.go:81-87) checks `c.Kinds == nil` before map indexing. `TestKindCatalogLookupZeroValue` exercises a zero-value `KindCatalog{}` for both `Lookup` and `LookupAgentBinding`. No panic.

### 2.6 REFUTED — JSON round-trip (Attack 6)

`TestKindCatalogJSONRoundTrip` marshals + unmarshals the baked catalog and asserts `reflect.DeepEqual` equality. The fixture covers both populated `Kinds` (with non-empty slices) and a populated `AgentBindings` (with `Duration`-typed `BlockedRetryCooldown` and `Tools` slice). Round-trip clean.

Edge case — empty maps vs nil maps: `Bake`'s body uses `if len(t.Kinds) > 0 { make + copy }` — empty source map produces nil catalog map, which `omitempty`-elides on marshal and unmarshal. `Lookup` zero-checks. No nil panic.

### 2.7 REFUTED — `loadProjectTemplate` stub safety (Attack 7)

`loadProjectTemplate` (service.go:347-349) is a one-line `return templates.Template{}, false, nil`. No I/O, no clock, no shared state. Every project created today gets `KindCatalogJSON == nil`. Pass.

### 2.8 REFUTED-DOCUMENTED — Persistence gap (Attack 8)

`projects` SQLite table (repo.go:147-156) has 8 columns: `id, slug, name, description, metadata_json, created_at, updated_at, archived_at`. No `kind_catalog_json` column. `scanProject` (repo.go:2765-2789) scans 8 fields and never touches `KindCatalogJSON`. The field is dropped on the floor when round-tripping through SQLite.

This is a **documented** gap, not a counterexample to the droplet's claim. `TestKindCatalogResolutionFromBakedCatalog` uses fakeRepo (which stores the full `domain.Project` value in memory) so the test exercises the catalog path despite the SQLite gap. The PLAN.md acceptance criterion does not require SQLite persistence — droplet 3.18 (`Owner` + `DropNumber` + `Persistent` + `DevGated` columns) and a future Project-table migration drop pick up the persistence hook.

### 2.9 REFUTED — Catalog hit but kind absent (Attack 9)

`Lookup` (catalog.go:81-87) returns `(KindRule{}, false)` for a kind not in `c.Kinds`. `lookupKindDefinitionFromCatalog:632-635` checks the bool and returns `(zero, false, nil)` so the legacy repo path fires. `TestKindCatalogLookupMiss` covers the catalog level; the integration test would cover the resolver level if the catalog had a partial kind set.

### 2.10 REFUTED — `synthesizeKindDefinitionFromRule` correctness (Attack 10)

Synthesis maps `AllowedParentKinds` (slice of `domain.Kind`) → `AllowedParentScopes` (slice of `domain.KindAppliesTo`) by direct string coercion (kind_capability.go:665-670). This is correct because `KindAppliesTo` values "mirror the 12 Kind values exactly" (kind.go:59-67), so `KindAppliesTo(string(KindPlan)) == KindAppliesToPlan` by construction.

`AppliesTo` is set to `[KindAppliesTo(kindID)]` — single entry — which matches the same 1:1 closed-enum invariant. `PayloadSchemaJSON` is empty (intentionally — templates v1 doesn't encode schemas).

NIT-level observation: `DisplayName = string(kindID)` produces lower-case `"build"` whereas the legacy repo path returns capitalized `"Build"`. The two paths produce different `DisplayName` for downstream consumers (`kind_capability.go:97-100` sorts on it; `snapshot.go:1099` persists it). This is a *visible behavioral divergence* but not a correctness bug — pre-3.14 nothing actually populates a real catalog, so the divergence is unreachable in production today. Flag for future-droplet attention, not FAIL.

### 2.11 REFUTED — Boot compatibility (Attack 11)

Empty `KindCatalogJSON` (the dominant pre-3.14 case, since `loadProjectTemplate` always returns `(zero, false, nil)`) routes through the legacy `s.repo.GetKindDefinition` path verbatim. The Drop 2.8 universal-nesting default behavior is preserved bit-for-bit when the catalog is empty. `TestKindCatalogResolutionFallsBackToRepoOnEmpty` covers it.

### 2.12 REFUTED — Concurrency (Attack 12)

`KindCatalog` exposes no mutator methods (`Lookup` and `LookupAgentBinding` are value-receiver reads). `Bake` deep-copies every slice and produces fresh maps, so the returned catalog cannot share state with the source `Template`. Concurrent `Lookup` calls on a single catalog instance read disjoint memory — Go's memory model permits unsynchronized concurrent reads on a map that is never written after construction. Caveat: external packages CAN mutate `c.Kinds[k] = ...` through the exported field, but this is the standard Go convention for value types and not a bug for this droplet.

## 3. Most Damaging Counterexample

**§ 1.1 — scope leak / out-of-spec refactor.** The droplet's contract specifies a closed 5-file path list under 3 packages. The builder shipped diffs touching 10 additional files across 4 unauthorized packages (`internal/tui`, `internal/adapters/server/common`, `internal/adapters/server/mcpapi`, `internal/adapters/storage/sqlite`). The cleanup is plausibly valid dead-code removal, but the contract gates exist exactly to prevent this kind of "while I'm in here" expansion — they enforce blast-radius discipline so QA + plan-QA can reason about a droplet's reach.

The builder's prompt explicitly flagged this attack vector (#13). The committed diff fails it.

## 4. Verdict Summary

**FAIL** — droplet contract violation via out-of-scope refactor leak.

In-spec work (catalog.go, catalog_test.go, kind_capability.go diff, service.go diff, project.go diff, kind_capability_catalog_test.go) is **technically sound**: tests pass, deep-copy works, fallback paths are safe, persistence gap is acknowledged. If the cleanup were stripped, the verdict would flip to PASS-WITH-NIT (the `DisplayName` divergence in §2.10 is the residual nit, plus the per-resolution `GetProject` round-trip is a quiet performance regression worth noting).

**Required action before 3.12 lands:**

1. Either: revert the 10 out-of-scope files (preferred — keeps droplet contract intact).
2. Or: dev explicitly widens `Paths` + `Packages` in PLAN.md droplet 3.12 and rationalizes the cleanup, then QA re-runs against the widened contract.

`mage ci` going green in spite of the leak is a known hazard of file-level (not commit-level) gating — the planner's `Paths` is the contract, not whatever passes CI.
