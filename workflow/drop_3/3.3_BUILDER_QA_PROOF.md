## 4. Findings

### NIT-N1 — Cross-product enum × bool coverage is 5 of 8 named cells (severity: NIT)

**Observation.** The spawn brief described coverage as "all 4 `StructuralType` enum values × 2 bool states." The `cases` table at `repo_test.go:2342-2346` covers 4 enum values, but only 4 of the 8 cross-product cells explicitly: `(drop,false)`, `(segment,false)`, `(confluence,false)`, `(droplet,true)`. The UPDATE block at lines 2419-2420 adds a fifth — flipping `t-st-drop` to `(confluence,true)`. The remaining named cells `(drop,true)`, `(segment,true)`, `(droplet,false)`, `(confluence,false-without-update)` are not exercised.

**Impact.** Every distinct *code path* (each of the 4 enum strings round-tripping through TEXT, both bool→int directions for `boolToInt`, both int→bool directions in `irreducibleRaw != 0`) is hit by the existing 5 cells. The missing cells are combinatorial repetitions, not new paths. The test does prove what it needs to prove.

**Why NIT, not BLOCKING.** Acceptance bullet `repo_test.go round-trip per StructuralType value via CreateActionItem → GetActionItem` (PLAN.md §3.3) does not mandate the full 8-cell cross-product. The test demonstrably round-trips every enum value through Create + Get + List + ListByParent + Update, with both bool states represented.

**Optional follow-up (not required for PASS).** A future tightening could add three rows to flip the unrepresented combinations, but doing so adds redundancy without new coverage and is not a blocker.
