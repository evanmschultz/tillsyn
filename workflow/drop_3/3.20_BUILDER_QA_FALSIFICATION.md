## 2. Refuted Attacks

### 2.1 Recursion attack — REFUTED

**Hypothesis:** `seedDropFindingsAndGate` calls `Service.CreateActionItem` for each finding. Findings carry `DropNumber = N > 0`. If the trigger gate in `Service.CreateActionItem` re-fires `seedDropFindingsAndGate` for any item with `DropNumber > 0`, infinite recursion.

**Evidence — `internal/app/service.go` lines 779–791:**

```go
if s.autoSeedSteward && strings.TrimSpace(actionItem.ParentID) == "" && actionItem.DropNumber > 0 {
    if err := s.seedDropFindingsAndGate(ctx, actionItem); err != nil {
        ...
    }
}
```

The trigger gate requires both:

- `ParentID == ""` (level_1 under the project root), AND
- `DropNumber > 0`.

Findings are created with `ParentID = anchor.ID` (a non-empty STEWARD anchor ID). The refinements-gate is created with `ParentID = drop.ID` (the level_1 drop's own ID, non-empty). Neither child path satisfies the recursion gate's `ParentID == ""` clause. `strings.TrimSpace` neutralizes whitespace-only edge cases.

**Conclusion:** the gate's `ParentID == ""` clause is the recursion stop. Refuted.

### 2.2 Idempotent re-seed produces duplicate anchors — REFUTED

**Hypothesis:** running `seedStewardAnchors` twice on the same project creates 12 anchors instead of 6.

**Evidence — `internal/app/auto_generate_steward.go` lines 105–109 + `auto_generate_steward_test.go` `TestAutoGenSeedsIdempotentOnReseed`:**

The seeder calls `repo.FindActionItemByOwnerAndTitle(ctx, project.ID, "STEWARD", title)` BEFORE attempting `CreateActionItem`. A nil error from the lookup means the row exists; the seeder `continue`s without creating a duplicate. The test directly invokes `svc.seedStewardAnchors` a second time after project creation and asserts `stewardCount == 6`.

**Conclusion:** idempotency is enforced by the (project_id, owner, title) lookup. Refuted.

### 2.3 Missing-anchor handling crashes instead of erroring cleanly — REFUTED

**Hypothesis:** when `seedDropFindingsAndGate` runs for a numbered drop on a project whose STEWARD anchors were never seeded, it nil-dereferences or panics.

**Evidence — `auto_generate_steward.go` lines 218–224:**

```go
anchor, anchorErr := s.repo.FindActionItemByOwnerAndTitle(ctx, drop.ProjectID, stewardOwner, plan.parentTitle)
if anchorErr != nil {
    if errors.Is(anchorErr, ErrNotFound) {
        return fmt.Errorf("%w: %q", errStewardParentNotSeeded, plan.parentTitle)
    }
    return fmt.Errorf("seed drop findings: lookup anchor %q: %w", plan.parentTitle, anchorErr)
}
```

The loop returns `errStewardParentNotSeeded` wrapped with the missing parent's title. `TestAutoGenSeedsRejectsMissingAnchor` directly exercises this path with `errors.Is(err, errStewardParentNotSeeded)`. No panic; clean error wrap with `%w`.

**Conclusion:** Refuted.

### 2.4 Refinements-gate `blocked_by` correctness — REFUTED (with NIT)

**Hypothesis:** the gate's `blocked_by` either misses the drop, the findings, or includes the gate itself (self-loop).

**Evidence — `auto_generate_steward.go` lines 296–324 (`assembleRefinementsGateBlockedBy`):**

The helper calls `ListActionItemsByDropNumber(drop.ProjectID, drop.DropNumber)` AFTER all 5 findings have been persisted but BEFORE the gate's own `CreateActionItem` runs. Result: `dropItems` contains the drop + 5 findings. The set is then unioned with `findingIDs`, sorted, returned. The gate's own ID cannot appear because the gate has not been created yet. `TestAutoGenSeedsLevel2FindingsOnNumberedDropCreation` lines 300–316 assert exactly this:

- `blockedSet[drop.ID]` exists.
- Each finding ID appears.
- `blockedSet[gate.ID]` does NOT exist (asserted after gate creation).

**Conclusion:** correct. Refuted.

**NIT (out-of-band, not a counterexample):** the `findingIDs` argument to `assembleRefinementsGateBlockedBy` is dead-code-equivalent — the same IDs already appear in `dropItems` because every finding carries `DropNumber=N`. The map-set collapses the duplicates so behavior is correct. It survives only as defensive-redundancy; could be deleted in a future tidy without behavioral change. Not blocking.

### 2.5 Non-numbered drop skip — REFUTED

**Hypothesis:** creating a level_1 plan with `DropNumber=0` triggers findings seeding under fictitious `DROP_0_*` parents.

**Evidence — `service.go` line 787:**

```go
if s.autoSeedSteward && strings.TrimSpace(actionItem.ParentID) == "" && actionItem.DropNumber > 0 {
```

The `DropNumber > 0` clause is necessary. `seedDropFindingsAndGate` lines 207–210 also short-circuit:


Defense in depth: even if the outer guard were bypassed, the inner guard would no-op. `TestAutoGenSeedsSkipsNonNumberedDrop` walks `repo.tasks` after creating a level_1 plan with `DropNumber=0` and asserts no `DROP_0_*` rows exist.

**Conclusion:** Refuted.

### 2.6 Index usage on the new repo methods — REFUTED

**Hypothesis:** the new SQL queries miss the indexes claimed in their godoc.

**Evidence — `internal/adapters/storage/sqlite/repo.go`:**

- Line 442: `CREATE INDEX IF NOT EXISTS idx_action_items_drop_number ON action_items(project_id, drop_number, owner);`
- Line 443: `CREATE INDEX IF NOT EXISTS idx_action_items_owner_title ON action_items(project_id, owner, title);`
- Line 1428–1437: `FindActionItemByOwnerAndTitle` query: `WHERE project_id = ? AND owner = ? AND title = ?` — left-prefix match for `idx_action_items_owner_title (project_id, owner, title)`. SQLite plans this as an equality lookup on the composite index.
- Line 1449–1457: `ListActionItemsByDropNumber` query: `WHERE project_id = ? AND drop_number = ? ORDER BY created_at ASC, id ASC` — left-prefix match for `idx_action_items_drop_number (project_id, drop_number, owner)`. SQLite plans this as an index range scan; the `ORDER BY` outside the index prefix is acceptable for the small expected result set per PLAN.md line 429 ("small expected row counts; either is fine").

**Conclusion:** column order in both queries matches the index left-prefix. Refuted.

### 2.7 Default-false flag — REFUTED

**Hypothesis:** the `AutoSeedStewardAnchors` flag's default of `false` leaves production code unprotected, or test code accidentally enables it.

**Evidence:**

- `internal/app/service.go` line 82 — field is value-typed `bool` with no default initialization in `NewService`; absent `cfg.AutoSeedStewardAnchors` consumer means `s.autoSeedSteward = false`.
- `cmd/till/main.go` line 2247 — production callsite explicitly sets `AutoSeedStewardAnchors: true`.
- `internal/app/auto_generate_steward_test.go` line 58 — the auto-generator test fixtures explicitly opt in (`AutoSeedStewardAnchors: true`).
- The 306 other tests in `internal/app` (per `mage test-pkg`) inherit the default `false` and continue to work without anchor materialization, preserving fixture stability. `mage test-pkg ./internal/app` returns 307/307 PASS.

**Conclusion:** flag default behavior is correct in both branches. Refuted.

### 2.8 Transaction atomicity — REFUTED-WITH-DOCUMENTED-HAZARD

**Hypothesis:** if `seedStewardAnchors` partially completes (3 of 6) or `seedDropFindingsAndGate` fails after creating the drop, the system is left in an inconsistent state with no documented recovery.

**Evidence:**

- `seedStewardAnchors` godoc lines 83–87 explicitly states: *"The helper does NOT roll back already-created seeds on later failures; partial seeding is preferred to a project with zero anchors. Callers that detect the failure can re-invoke seedStewardAnchors after fixing the root cause and idempotency will pick up where the previous attempt left off."*
- The (project_id, owner, title) lookup ensures re-invocation is idempotent.
- `seedDropFindingsAndGate` follows the same pattern — its own godoc lines 199–202: *"if the helper finds an already-existing (project_id, owner=STEWARD, title) row for any of the 5 findings or the gate, that row is skipped (consistent with seedStewardAnchors)."*

**Caveat (NIT, not blocking):** when `Service.CreateActionItem` runs for a numbered drop, the drop's own `repo.CreateActionItem` succeeds first (line 773) and is NOT rolled back if `seedDropFindingsAndGate` fails (line 788). A caller seeing the error has no surface signal that a partial drop row exists in the DB. Re-invoking `Service.CreateActionItem` with the same `Title="DROP_N"` would create a SECOND drop row (different `id`) with the same `drop_number=N`, and the second pass's findings would still skip-by-idempotency under the STEWARD anchors. End state: two drop rows, one set of findings. Drop 3.22's "missing anchors" test (Test 5 missing) does NOT verify post-failure DB shape. The behavior is documented at the helper level but the failure-recovery semantics for the TWO-LEVEL retry (drop + findings) are not captured. Documenting this is a refinement — not a counterexample because the documented contract permits it.

**Conclusion:** the documented contract is consistent with the code; partial-state hazard is acknowledged in helper godoc. Refuted as a counterexample; flagged as an out-of-band ergonomics nit for Drop 4 dispatcher / 3.22 integration test consideration.

### 2.9 YAGNI on auto_generate_steward.go — REFUTED

**Hypothesis:** the auto-generator carries abstractions / generalizations not required by current consumers.

**Evidence:**

- `loadStewardSeedTemplate` is a function-typed package var so tests can substitute fixtures via `withSeedTemplateFixture` without injecting a constructor argument; one consumer pair (production + test fixture) — meets the "two concrete uses" bar.
- `dropFindingPlan` struct + `canonicalDropFindings` slice are the data-driven backbone of the loop; alternative designs (separate functions per finding, hardcoded literals) would either duplicate the `(parent, suffix)` pairing or sacrifice the deterministic test ordering.
- `firstColumnForProject` is shared between `seedStewardAnchors` and `seedDropFindingsAndGate` — two concrete uses, no generalization.
- `assembleRefinementsGateBlockedBy` collapses two sources (drop-numbered items + just-created findings) into a deterministic sorted list; both sources are real, even if findings are redundant in the union (see § 2.4 NIT).

**Conclusion:** no premature abstractions. Refuted.

## 3. Most Damaging Counterexample

§ 1.1 (file/package gating bypass) is the only CONFIRMED counterexample and is squarely the most damaging. The cascade's promise is that builders edit only what their plan-item declares, that QA twins fire on a stable surface, and that same-file-lock `blocked_by` chains are honored so each droplet inherits a settled file shape from its declared blockers. 3.20 silently completed 3.21's wire-surface work on `mcp_surface.go` / `app_service_adapter_mcp.go` / `extended_tools.go` / `snapshot.go` (plus their test files), unilaterally bypassing 3.21's same-file-lock chain to 3.4 / 3.5 / 3.18 / 3.19. The damage compounds at 3.22, whose "Blocked by: 3.20, 3.21" wiring becomes degenerate. The droplet's behavior on its own scope (auto-gen + idempotency + repo methods + opt-in flag) is solid; the decision to attach 3.21's work to the same commit set is the failure.

## 4. Verdict Summary

**FAIL.** Required action: roll the 3.21-scope edits out of this droplet's commit set, ship 3.20 against its declared `paths`, and let 3.21 land separately with its same-file-lock chain to 3.4 / 3.5 / 3.18 / 3.19 honored. Alternative: re-scope `PLAN.md` 3.20 to absorb 3.21 wholesale, retire 3.21, and re-run plan QA on the merged droplet — but this is the more invasive path and requires re-pricing the same-file-lock chain.

The auto-generator / idempotency / repo-method / default.toml-seed work itself is correct and lock-tight (see § 2 refutations). The verdict is FAIL solely on plan-vs-implementation scope mismatch, not on auto-generator correctness.
