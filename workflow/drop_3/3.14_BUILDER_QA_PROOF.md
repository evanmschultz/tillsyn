## 5. Hylla Feedback

**Decision: bypassed Hylla, used LSP/Read directly.**

The droplet-3.14 review focused on three new Go source files (`embed.go`, `embed_test.go`, `default.toml`) plus reading existing committed sources (`schema.go`, `nesting.go`, `child_rules.go`, `load.go`, `nesting_test.go`, `domain/kind.go`, `domain/structural_type.go`). The new files are uncommitted at review time — Hylla would not see them. The existing Go files I cross-referenced are committed but the questions were file-local symbol lookups (constant declarations, struct field tags, function signatures) that `Read` + `grep` answer faster than a Hylla query.

**Misses:** None to log under the project's Hylla-first rule, because the review was either touching uncommitted source (correctly outside Hylla scope per `feedback_hylla_go_only_today.md`) or doing local symbol lookups that don't benefit from Hylla's vector search.

**Suggestion:** When the review subject is "verify this builder's just-landed files," there's no benefit to Hylla — the artifacts aren't ingested. The QA-PROOF agent definition could note this carve-out explicitly: "for builder-just-landed-files reviews, `Read` and `LSP` are first-line; Hylla is for cross-referencing committed call sites the new code may regress." That's already implicit in the evidence-order rule but worth surfacing for future reviewers.
