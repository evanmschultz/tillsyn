## 1. Counterexamples

### 1.1 N1 — `[kinds.<bogus-key>]` map keys go UNVALIDATED against the closed enum *(NIT — soft counterexample on spec scope)*

**Reproduction.** Loading the TOML below succeeds with **no error**:

```toml
schema_version = "v1"

[kinds.totally-bogus-not-a-real-kind]
owner = "STEWARD"
structural_type = "droplet"
```

The bogus map key lands in `tpl.Kinds["totally-bogus-not-a-real-kind"]` with no rejection. Same hole on `[agent_bindings.<bogus>]`:

```toml
schema_version = "v1"

[agent_bindings.bogus-kind-here]
agent_name = "fake-agent"
model = "opus"
```

Verified end-to-end by adding probe tests `TestFALSIFY_AssertKindsMapKeyUnchecked` and `TestFALSIFY_AssertAgentBindingsMapKeyUnchecked` to `internal/templates/`; both tests asserted "Load returns nil err" and `mage test-pkg ./internal/templates` passed (24/24, including the new probes — counterexample reproduces). Probe file removed; tree restored to 16/16.

**Spec text (PLAN.md droplet 3.9 acceptance line 271):**

> "asserts every referenced `Kind` is in `domain.validKinds` (closed 12-value enum)"

**Implementation (`load.go:80`):**

```go
if err := validateChildRuleKinds(tpl.ChildRules); err != nil { ... }
```

`validateChildRuleKinds` iterates `[]ChildRule` only — never `tpl.Kinds` map keys, never `tpl.AgentBindings` map keys. LSP `findReferences` on `domain.IsValidKind` confirms the only callers in `internal/templates/load.go` are at `:132` (`WhenParentKind`) and `:135` (`CreateChildKind`).

**Why NIT, not FAIL.** The acceptance bullet is grouped under "Load-time validator: builds parent → child kind graph from `[child_rules]`; runs DFS … to detect cycles; asserts every referenced `Kind` is in `domain.validKinds`". A reasonable reading restricts "every referenced Kind" to the kinds that appear inside `[child_rules]` — the same scope the surrounding sentence introduces. Under that reading, the implementation matches the spec.

The counterexample matters anyway because:

1. The user-facing impact is real — a `[kinds.bogus-kind]` row is silently accepted, producing a Template that contains a junk kind no validator complains about. Drop 3.10's `AllowsNesting` queries `Kinds` by parent/child kind; a bogus parent would never match a real query, so the bogus row is functionally inert — but inert ≠ rejected.
2. Drop 3.14 (`default.toml` + STEWARD parent seeds + 12-kind coverage) will rely on `embed_test.go` to assert "every closed-12-kind has a `[kinds.<kind>]` section." That test does NOT today have a partner that asserts "no `[kinds.<unknown>]` section is allowed."
3. `load.go:64-66` doc comment claims strict decode catches "unknown `[kinds.*]` and `[agent_bindings.*]` subkeys". That is true for unknown *fields inside a kind row* (verified — `TestFALSIFY_AssertNestedKindFieldUnknown` rejects), but **NOT** true for unknown *map keys themselves*. The doc comment is misleading.

**Recommended action.** Either:
- **(A) Tighten:** add a `validateMapKeys` helper that iterates `tpl.Kinds` and `tpl.AgentBindings` map keys against `domain.IsValidKind`, returning `ErrUnknownKindReference`. This costs ~10 LOC and one new sentinel-routed test.
- **(B) Spec clarification:** if the spec genuinely meant "child_rules only," fix the misleading doc comment at `load.go:64-66` to say "unknown *fields* inside `[kinds.X]`/`[agent_bindings.X]`" (singular, scoped to fields, not keys).

The dev should pick the path; both are PASS-with-followup.

### 1.2 N2 — `StrictMissingError` is NOT preserved via `%w` *(NIT — soft counterexample on attack vector 7)*

**Reproduction.** `load.go:73`:

```go
return Template{}, fmt.Errorf("%w: %s", ErrUnknownTemplateKey, strictErr.String())
```

The second verb is `%s`, **not `%w`** — `strictErr` is flattened to a string via its `String()` method, and the typed `*toml.StrictMissingError` is lost. Verified by probe `TestFALSIFY_AssertStrictMissingErrorNotPreserved`: `errors.As(err, &strictErr)` returns `false` for the resulting error.

**Attack-vector text:**

> "Verify `errors.Is(err, ErrUnknownTemplateKey)` succeeds AND the original `StrictMissingError` is preserved via `%w` for callers that need detail."

`errors.Is(err, ErrUnknownTemplateKey)` **does** succeed (REFUTED first half). `errors.As(_, *toml.StrictMissingError)` **does not** (CONFIRMED second half).

**Spec text (droplet 3.9):**

> "Sentinel errors at package scope: `ErrUnknownTemplateKey`, `ErrUnsupportedSchemaVersion`, `ErrTemplateCycle`, `ErrUnreachableChildRule`, `ErrUnknownKindReference`."

The spec only mandates sentinel routing via `errors.Is`. It does not require nested unwrapping to the underlying `*toml.StrictMissingError`. Under spec, this is fine.

**Why NIT, not FAIL.** The attack vector assumed `%w` for both verbs. The implementation produces a UX-readable string (the strict err's `String()` carries position-aware filename + line + key path — that's fine for a dev reading the error). What's lost is *programmatic* access to the typed error for callers that want to do something fancier than sentinel routing. Drop 3 has zero such callers today.

**Recommended action.** Two-line change:

```go
return Template{}, fmt.Errorf("%w: %w", ErrUnknownTemplateKey, strictErr)
```

This is Go 1.20+ multi-`%w` syntax (Go 1.26 in go.mod, fully supported). The error message text is the same; `errors.Is` still routes; `errors.As` to `*toml.StrictMissingError` now works.

Same suggestion applies (less critically) to the cycle wrap `load.go:173`: `fmt.Errorf("%w: %s", ErrTemplateCycle, formatCyclePath(...))` — but `formatCyclePath` returns a `string`, not a wrapped error, so `%s` is correct there. Pre-pass error at `:60` correctly uses `%w`. Only the strict-decode wrap is affected.

## 2. Refuted Attacks

The remaining 8 of 10 specified attack vectors REFUTE — no counterexample produced.

### 2.1 R1 — Pre-pass / strict-decode ordering *(REFUTED)*

`load.go:50-61` runs the tolerant pre-pass before line `:67-69`'s strict decoder. `TestLoadRejectionTable[schema_version pre-pass beats unknown-key pass for v2 + bogus table]` (line 172-181) directly asserts: `schema_version = "v2"` + `[bogus_top_level]` → `ErrUnsupportedSchemaVersion`, NOT `ErrUnknownTemplateKey`. Probe `TestFALSIFY_AssertPrePassPrecedesStrict` independently confirms `errors.Is(err, ErrUnsupportedSchemaVersion) == true && errors.Is(err, ErrUnknownTemplateKey) == false`. Ordering correct.

### 2.2 R2 — Cycle detection edge cases *(REFUTED)*

- **Direct (A→A):** `TestLoadSelfCycleSingleRule` (load_test.go:219). Cycle path `"build -> build"` asserted in error text. Probe-confirmed.
- **Two-rule (A→B→A):** `TestLoadRejectionTable[cycle build->plan->build]` (line 108-122). Returns `ErrTemplateCycle` with `"build"` in path.
- **Three-rule (A→B→C→A):** Probe `TestFALSIFY_AssertThreeRuleCycle` constructs `build → plan → research → build`. Returns `ErrTemplateCycle`. Confirmed.

The colored DFS is correct: `colorGray` on the recursion stack triggers cycle detection precisely when a successor's traversal re-enters the stack. `formatCyclePath` slices the stack at the closure point and appends the closing kind, producing `"A -> B -> C -> A"` rendering. Note: `for node := range graph` on line 184 has non-deterministic Go map order, so for a TOML doc with multiple disconnected cycles, the reported cycle is non-deterministic — but `ErrTemplateCycle` always fires. Acceptable.

### 2.3 R3 — `GateRulesRaw` strict-decode *(REFUTED)*

Probe `TestFALSIFY_AssertGateRulesDeepUnknown` constructs `[gate_rules.bogus_attribute]` + nested arbitrary tables. `Load` returns `nil` error. `tpl.GateRulesRaw["bogus_attribute"]` populated. Existing `TestLoadGateRulesReservedTable` (load_test.go:62) covers the basic case. `Template.GateRulesRaw map[string]any` (`schema.go:63`) absorbs arbitrary nested content because `map[string]any` decode does not invoke strict-key checks on its keys. Forward-compat hatch (L6) intact.

### 2.4 R4 — Unknown-kind reference (both paths) *(REFUTED)*

`TestLoadRejectionTable[unreachable when_parent_kind surfaces as unknown-kind]` (line 126-137) asserts `ErrUnknownKindReference` + substring `"when_parent_kind"`. `TestLoadRejectionTable[unknown create_child_kind rejected]` (line 139-150) asserts the same sentinel + substring `"create_child_kind"`. ### 2.5 R5 — Reachability no-op extension point *(REFUTED)*

Spec says (line 271): "asserts no `[child_rules]` entry references an unreachable `WhenParentKind`" — but droplet 3.9's introductory text (per finding 5.B.4 cited in load.go:147) leaves the choice to the implementer. Builder picked "no-op + extension-point comment + reserved sentinel" — defensible. Future drop wires actual reachability without API churn.

## 3. Most Damaging Counterexample

**N1 — `[kinds.<bogus-key>]` map keys go UNVALIDATED.**

Damage assessment: low today, latent tomorrow. The bogus row is functionally inert because no consumer queries `Kinds[bogus]` deliberately. But Drop 3.14 (`default.toml` round-trip + 12-kind coverage assertion) ships a test that says "every kind has a section" — never "no extra section exists." A typo in `default.toml` (`[kinds.bulid]` instead of `[kinds.build]`) would silently accept, with the missing real `build` row caught only by the existence-assertion in `embed_test.go`. Asymmetric: misspellings drop one row AND add one bogus row; the existence check fires; the bogus check does not.

The `load.go:64-66` doc comment compounds the issue by claiming strict decode catches "unknown `[kinds.*]` and `[agent_bindings.*]` subkeys" — that's true for *fields inside* a kind row (verified) but *not* true for *map keys*. Dev/QA reading the doc comment will believe the validator is tighter than it is.

Recommended path: **N1(A) — add the validator** in this droplet (cheap, ~10 LOC + 2 tests), OR **defer to 3.10/3.14** with a tracking note in `default.toml`'s test alongside the 12-kind existence assertion. Either is PASS-with-followup.
