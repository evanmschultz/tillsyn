## 2.12 Deeper attacks (beyond spawn-prompt vectors)

### 2.12.1 Parent-allow-list precedence over child-rejection — TEST GAP, not a defect

The spec resolution order in `nesting.go:16-32` declares step 1.2 (`parent's AllowedChildKinds contains child → allow`) WITHOUT consulting child-side rules. There is no test row with the shape `{ parent.AllowedChildKinds = [child], child.AllowedParentKinds = [other-parent-only] }` — i.e. a case where the parent allow-list explicitly admits the child while the child's parent allow-list explicitly excludes the parent. The implementation handles this correctly (returns `(true, "")` at line 43, never reaching the child-side branch), but the test does not specifically lock this precedence ordering. Coverage instrument doesn't flag this because the line is hit by other rows. NIT-grade testing gap; does not falsify the implementation. Suggest as a refinement, not a blocker.

### 2.12.2 Reason-format escape in `%q` — REFUTED

Kind values like `"build-qa-proof"` contain hyphens, which `%q` Go-escapes by wrapping in literal double-quotes. Both implementation and test use identical `%q` formatting, so the escaped-form strings compare equal. No drift.

### 2.12.3 Nil-map read on `t.Kinds` — REFUTED

`t.Kinds[parent]` on a nil map (the empty-Template case) returns the zero-value `KindRule` per Go map semantics. `isZeroKindRule(KindRule{})` returns true; line 38 returns `(true, "")`. Empty-template test verifies this across 144 pairs.
