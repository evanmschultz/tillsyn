### 1.1 (NIT) `assertOwnerStateGate` comment contradicts code (low-severity comment drift)

`internal/adapters/server/common/app_service_adapter_mcp.go:917-920`:

```go
caller, ok := app.AuthenticatedCallerFromContext(ctx)
if !ok {
    // No authenticated caller in context (e.g. unguarded user tuple) —
    // treat as non-steward. The dominant non-MCP test path won't have

```

The comment says "treat as non-steward" but the code RETURNS `ErrAuthorizationDenied`. Behavior is fail-closed (correct), but the comment phrasing implies a permissive treatment. Real meaning: "a non-steward caller (including the no-caller path) on a STEWARD-owned item is denied." Single-line clarification would resolve it; not a logic bug.

### 1.2 (NIT) "Steward CAN change Owner" test asserts permission, not mutation

`internal/adapters/server/common/app_service_adapter_steward_gate_test.go:200-209`:

```go
// Steward principal CAN change the Owner.
stewardActor := stewardGatedActor("steward")
if _, err := fixture.adapter.UpdateActionItem(ctx, UpdateActionItemRequest{
    ActionItemID: stewardGated.ID,

    t.Fatalf("UpdateActionItem(steward changing Owner) error = %v", err)

```

The test only verifies the permission grant (no error). It does NOT verify Owner actually changed in the persisted row. Look at the adapter:

`internal/adapters/server/common/app_service_adapter_mcp.go:712-726` — the call into `a.service.UpdateActionItem(ctx, app.UpdateActionItemInput{...})` does NOT include `Owner` or `DropNumber` in the `UpdateActionItemInput`. The pointer fields are read by the gate then dropped on the floor. Builder explicitly notes this in `mcp_surface.go:91-110` — "Service-side wiring of the value lands in 3.21 (alongside the rest of the Owner/DropNumber/Persistent/DevGated MCP plumbing); 3.19 reads the pointer in the gate only." So the deferral is documented, but the test phrasing implies behavior the code does not yet exhibit. If a follow-up droplet refactors UpdateActionItem and accidentally drops the gate read, this test would still pass green because it never checked the post-state. NIT-level: tighten the test in 3.21 when wiring lands by adding `if got.Owner != "ROGUE" { t.Fatalf(...) }` after the call.

### 1.3 (Out-of-scope but worth flagging) `DeleteActionItem` is NOT gated

`internal/adapters/server/common/app_service_adapter_mcp.go:803-816` — `DeleteActionItem` calls straight through to `a.service.DeleteActionItem` with no `assertOwnerStateGate` call. An agent-principal session can archive or hard-delete a STEWARD-owned item.

I cross-checked PLAN.md: the spec's gated surface is explicitly enumerated as Move + MoveState + Update field-guard + Reparent + supersede + autent boundary-map. Delete is NOT listed. So this is YAGNI-compliant per planner intent and NOT a counterexample against 3.19's claim. But it is the most state-affecting mutation possible (item is gone). Worth surfacing as a refinement candidate for a future droplet — if reparent (which preserves the row) is in scope, delete (which removes the row) probably should be too. Recording here so the orchestrator can decide whether to file a follow-up.

### 1.4 (Out-of-scope) L1 field-level guard is not reachable through any current MCP/HTTP wire path

`internal/adapters/server/mcpapi/extended_tools.go:1098-1108` — the MCP `update_task` tool constructs `common.UpdateActionItemRequest{}` without populating `Owner` or `DropNumber`. HTTP has no `UpdateActionItem` route at all (verified via `rg`). So the L1 field-level gate is dead-code via every transport surface that exists today. The state-neutral gate (assertOwnerStateGate, used by Move/MoveState/Reparent) IS live.

This is per-spec — the comment on `mcp_surface.go:91-114` explicitly defers Owner/DropNumber wire wiring to 3.21. Not a counterexample. Flagged so 3.21's QA verifies the field-level path actually receives traffic once wiring lands, otherwise the gate's protection is theoretical.

## 2. Refuted Attacks

### 2.1 REFUTED — Silent gate bypass via empty `AuthRequestPrincipalType`

`assertOwnerStateGate` checks `caller.AuthRequestPrincipalType` via `strings.EqualFold(strings.TrimSpace(...), "steward")`. Empty string fails the equality, returns `ErrAuthorizationDenied`. The "no authenticated caller" branch (line 916-921) ALSO returns `ErrAuthorizationDenied`. Fail-closed in both gaps. Confirmed by `TestAssertOwnerStateGateMoveActionItemStateAgentRejected` (caller has `AuthRequestPrincipalType="agent"`) and the no-caller path in production code (no test, but inspected directly).

### 2.2 REFUTED — autent boundary-map persistence loss

`internal/adapters/auth/autentauth/service.go:217-227` — `IssueSession` writes `metadata["auth_request_principal_type"] = tillsynPrincipalType` BEFORE forwarding to `s.service.IssueSession`. autent persists session.Metadata (verified via autent module at `~/go/pkg/mod/github.com/evanmschultz/autent@v0.1.1/domain/session.go:16` and `app/service.go:170`). `Authorize` (line 758-761) reads `validated.Session.Metadata["auth_request_principal_type"]` back, falls back to `validated.Principal.Type` only when metadata is absent. `TestIssueSessionStewardBoundaryMap` exercises the full round-trip and asserts `result.Caller.AuthRequestPrincipalType == "steward"`.

The fallback path on missing metadata returns "agent" for a STEWARD principal whose session was issued before 3.19 — which is fail-closed (rejects future STEWARD operations). Acceptable migration cost given the dev's "delete `~/.tillsyn/tillsyn.db` between schema-touching units" pre-MVP doctrine.

### 2.3 REFUTED — Pointer-sentinel false positive on Update

`UpdateActionItem` at `app_service_adapter_mcp.go:703-711` only enters the gate-fetch path when `in.Owner != nil || in.DropNumber != nil`. Description-only updates skip the gate entirely (no fetch). Verified via `TestAssertOwnerStateGateUpdateActionItemDescriptionOnlyAgentSucceeds`. The wire layer (MCP `update_task`) constructs the request without populating these pointers (`extended_tools.go:1098-1108`), so absent-vs-empty is not even reachable through the wire today; the in-Go test path uses Go's literal `nil` vs `&""`/`&"value"` — no JSON marshal step that could auto-fill `""`.

### 2.4 REFUTED — L1 idempotent mutation rejection

`assertOwnerStateGateUpdateFields` uses `strings.TrimSpace(*wantOwner) != strings.TrimSpace(existing.Owner)` — equality bypasses the rejection. An agent passing `Owner: &"STEWARD"` on a STEWARD-owned item is permitted (idempotent). Verified by inspection; no specific test, but logic is straightforward.

### 2.5 REFUTED — L8 Reparent state-neutrality

`ReparentActionItem` (`app_service_adapter_mcp.go:841-862`) fetches existing, runs `assertOwnerStateGate`, runs the reparent. Gate keys on `existing.Owner == "STEWARD"` — the destination parent is irrelevant. Reparenting STEWARD-owned within STEWARD's subtree is rejected for non-steward callers, matching spec ("regardless of destination"). Verified by `TestAssertOwnerStateGateReparentActionItemAgentRejected`.

### 2.6 REFUTED — Test 7 deferral honesty (Supersede)

`rg -l Supersede internal/` returns only `internal/domain/handoff.go` (HandoffStatus=superseded) and `internal/tui/model.go` (TUI rendering of that status). No action-item Supersede method exists anywhere in `internal/app`, `internal/adapters/server`, or the CLI. PLAN.md:449 anticipates Drop 1's "always-on failed" introducing `till action_item supersede`, but that wire surface does not exist in HEAD. Builder's deferral is honest.

### 2.7 REFUTED — autent enum non-modification

`git status` clean; no edits to `~/go/pkg/mod/github.com/evanmschultz/autent@v0.1.1/...` (mod cache is read-only). `autentdomain.PrincipalType` enum unchanged; the boundary-map collapse to `PrincipalTypeAgent` happens in tillsyn's `service.go:208-210`.

### 2.8 REFUTED — Cross-test interference

Each test calls `newCommonLifecycleFixture(t)` (autentauth) or `newStewardGatedActionItem(t, fixture, "")` (gate suite), which create fresh `t.TempDir()` SQLite databases. `t.Parallel()` is used on every gate test; they cannot collide because each owns its own DB. No package-level `var` mutation in any of the new code. Verified by inspection + `mage test-pkg ./internal/adapters/server/common` passes deterministically.

### 2.9 REFUTED — Concurrency / race in the gate helper

`assertOwnerStateGate` and `assertOwnerStateGateUpdateFields` are pure functions — no shared state, no goroutines, no mutable globals. They take `ctx` and `domain.ActionItem` (by value) and read from `app.AuthenticatedCallerFromContext` (also pure). Two parallel callers each fetch + check independently. No race. The pre-existing TOCTOU between fetch and apply is not new — `app.Service.MoveActionItem` already had this property; 3.19 inherits it without amplification.

### 2.10 REFUTED — Raw `go` toolchain bypass

`mage ci` returns 1808/1808 green with all coverage thresholds met. No raw `go test` / `go build` / `go vet` / `go run` invocations introduced. No `mage install` calls in any new code or test fixture.
