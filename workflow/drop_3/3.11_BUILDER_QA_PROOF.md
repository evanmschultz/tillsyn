## 3. Findings

### F1 (NIT) — Comparator primary-axis branch unexercised by tests

The sort comparator at `child_rules.go:117-119` has two return paths:

```go
if out[i].StructuralType != out[j].StructuralType {
    return out[i].StructuralType < out[j].StructuralType  // line 118 — primary-axis branch
}
return out[i].Kind < out[j].Kind                          // line 120 — secondary-axis branch
```

The fixture sets `StructuralType: domain.StructuralTypeDroplet` on every kind in `t.Kinds`, so every resolution emitted by `ChildRulesFor` carries `StructuralType: Droplet`. The `out[i].StructuralType != out[j].StructuralType` guard always evaluates false in current tests — the line-118 inner return is never executed. A test mistakenly swapping `<` to `>` on the primary-axis comparator would not fail.

**Severity:** NIT. The behavior is correct (lexicographic on the underlying `string`-typed `StructuralType`); the secondary-axis tiebreaker is correctly asserted. Risk is low because (a) the production default.toml is unlikely to mix structural types under the same parent in a way that would hit this path frequently; (b) 3.14's `embed_test.go` round-trip will exercise the loaded default template and surface a regression there.

**Suggested follow-up:** in 3.14 (default.toml round-trip) or a future micro-droplet, add one unit test that constructs a fixture with mixed structural types under the same parent kind and asserts the primary-axis sort key. NOT a blocker for 3.11.

### F2 (NIT) — Per-file coverage cannot be asserted under the no-raw-`go` rule

`mage ci` reports per-package coverage only. Per-file coverage on `child_rules.go` (≥85% per the acceptance bullet) cannot be measured directly without raw `go test -coverprofile`, which CLAUDE.md forbids. Static trace analysis (§1.10) estimates file-level coverage at ~92.3% (12 of 13 statements covered, the missing one being F1 above). Package-aggregate of 96.0% is consistent with that file-level estimate.

**Severity:** NIT. The 85% file-level bar is met under the most defensible reading. **Not a blocker.** A future mage tooling improvement (e.g. a `mage coverFile <package> <file>` target wrapping `go tool cover -func`) would close this measurement gap.

### N1 (Note, not a finding) — `Persistent` + `DevGated` hardcoded false

Acceptance bullet 2.3.6 says `Persistent` + `DevGated` per L9 + L10. The implementation hardcodes both to `false` and documents that the `[child_rules]` TOML schema does not yet expose those bits. PLAN.md droplet 3.13 owns the schema fill-in for those axes; PLAN.md droplet 3.20 (auto-generator consumer) reads them from `KindRule` once available. The hardcoded-false carry is a documented forward-compat decision, not a regression. **Not a finding.**

### N2 (Note, not a finding) — Doc comment paragraph for stable-sort tiebreaker is precise

Lines 78-85 explain that stable sort preserves declaration order for equal-key pairs and call out a corner case ("templates that declare two rules with the same `(StructuralType, Kind)` under different parents — though the closed schema and the load-time cycle validator make that combination unusual"). This is exactly the kind of comment that makes the consumer's behavior unambiguous when 3.20 (auto-gen consumer) reads it. **Not a finding.**

### N3 (Note, not a finding) — `load.go` carries 3.9's `validateMapKeys` work in addition to 3.11's `errors.AsType` cleanup

The PROOF spawn prompt scope is droplet 3.11. The diff on `load.go` includes both 3.11's surgical `errors.AsType` swap AND 3.9's `validateMapKeys` validator addition (a parallel fix-builder's work). Both changes coexist in the working tree because parallel fix-builders converge on the same file. The `errors.AsType` swap is the only line-level change attributable to 3.11; the rest of `load.go`'s diff is 3.9's responsibility and is being reviewed there. No conflict between the two changes — they edit disjoint regions of the file. **Not a finding for 3.11.**
