### 1.1 NIT — `raiseRefinementsGateForgottenAttention` doc-comment claims idempotent error-swallowing the implementation does not perform

**Severity:** NIT.

**Reproduction analysis.**

`internal/app/auto_generate_steward.go:339-342` (the helper's doc comment) reads:

> "The helper is idempotent on attention_item id: re-running gate-close against an already-warned drop yields the same id and the storage layer rejects the duplicate insert. Callers ignore that case."

The implementation at `internal/app/auto_generate_steward.go:426-428` does NOT ignore the duplicate insert; it returns the error verbatim:

```go
if err := s.repo.CreateAttentionItem(ctx, item); err != nil {
    return fmt.Errorf("safety-net create attention: %w", err)
}
```

`Service.MoveActionItem` at `internal/app/service.go:873-877` then surfaces that error as a hard failure:

```go
if toState == domain.StateComplete && isRefinementsGate(actionItem) {
    if err := s.raiseRefinementsGateForgottenAttention(ctx, actionItem); err != nil {
        return domain.ActionItem{}, err
    }
}
```

Reproduction sketch (would not currently fire because of the terminal-state guard at `service.go:832` blocking re-transitions FROM `complete`, but the doc-comment is still misleading):

```go
// 1. STEWARD MoveActionItemState(gate, complete) with stragglers → attention raised, move succeeds.
// 2. Some override path (Drop 1 D3) eventually puts gate back to in_progress.
// 3. STEWARD MoveActionItemState(gate, complete) again with stragglers still present:
//      - safety-net helper rebuilds attention with id "refinements-gate-forgotten::<gateID>"
//      - repo.CreateAttentionItem returns sqlite UNIQUE constraint violation
//      - helper wraps + returns error
//      - MoveActionItem returns error after the gate's lifecycle_state was already persisted
//        at service.go:859 — caller sees an error but the gate is already at complete.
```

**Why this is a NIT, not a CONFIRMED counterexample.**

The terminal-state guard at `service.go:832` (`if domain.IsTerminalState(fromState) && fromState != toState`) blocks re-transitions FROM `complete` pre-Drop-1 D3 override auth, so the failure mode is unreachable on the live code paths today. The doc-comment is wrong-but-currently-harmless rather than runtime-broken. PLAN.md's 3.22 acceptance criteria (lines 513-525) do not address re-entrancy / idempotency.

**Routing recommendation.**

Two options: (a) tighten the doc comment to `"the helper is non-idempotent: a duplicate gate-close attempt fails on UNIQUE constraint and the move's lifecycle persistence is already committed."`; or (b) implement the claim — wrap the create in an `errors.Is(err, sqlite.ErrUniqueConstraint)` check that returns nil, OR pre-check existence via `GetAttentionItem`. Option (a) is cheaper and matches today's behavior; option (b) is the safer landing if Drop 1's D3 override re-opens this path. Surface as a Drop 3 refinement (`DROP_3_REFINEMENTS_RAISED`) — the doc-comment drift was introduced by 3.22's builder so this is a real builder-side observation, but it is below the droplet's acceptance bar.

### 1.3 NIT — `isRefinementsGate` would false-positive on any future STEWARD-owned numbered confluence type

**Severity:** NIT (future-risk, no live false-positive today).

**Reproduction analysis.**

`isRefinementsGate` at `internal/app/auto_generate_steward.go:329-344` checks three predicates:

```go
if strings.TrimSpace(item.Owner) != stewardOwner { return false }
if item.StructuralType != domain.StructuralTypeConfluence { return false }
if item.DropNumber <= 0 { return false }
return true
```

Today the only auto-generated `(Owner=STEWARD, StructuralType=Confluence, DropNumber>0)` action item is the refinements-gate (`auto_generate_steward.go:271-291` — the only `StructuralType: domain.StructuralTypeConfluence` site under `stewardOwner`). The 5 persistent STEWARD anchors carry `StructuralType=Branch` and `DropNumber=0`, the level_2 drop findings carry `StructuralType=Droplet`. So the detector is tight today.

Risk surface: if a future drop adds a SECOND STEWARD-owned numbered confluence (e.g., per finding 5.C.13's "second-pass review" gate, or any cascade-extension gate), `isRefinementsGate` would false-positive on it and the safety-net hook would raise spurious `refinements-gate-forgotten` attention items on that other gate's close. The detector relies on title-shape being the discriminator (per the doc comment "the title shape is asserted by the auto-generator's create path so it does not need to be re-checked here"), but the detector itself doesn't actually CHECK the title.

**Why this is a NIT, not a CONFIRMED counterexample.**

Today there is no second STEWARD-owned numbered confluence, so the false-positive is uninstantiated. PLAN.md's 3.22 acceptance criteria do not require future-proofing the detector. The doc-comment correctly describes the assumption; the assumption is currently held.

**Routing recommendation.**

Surface as a Drop 3 refinement (`DROP_3_REFINEMENTS_RAISED`): if any future cascade work adds a STEWARD-owned numbered confluence, either tighten `isRefinementsGate` to also assert `strings.HasPrefix(item.Title, "DROP_") && strings.Contains(item.Title, "_REFINEMENTS_GATE_BEFORE_DROP_")`, or split the detector predicate into a typed flag stored on the action item. Non-blocking for 3.22.

## 3. Most Damaging Counterexample

None at CONFIRMED severity. The most material NIT is §1.1 — the doc-comment drift on idempotency. The runtime is currently safe because the terminal-state guard at `service.go:832` prevents the re-transition path that would expose the duplicate-insert failure, but the doc-comment misrepresents the implementation's error-handling contract. A future Drop 1 D3 override that opens the from-terminal transition path would expose this — a STEWARD running gate-close twice in succession (once before override, once after) would hit a UNIQUE constraint violation at sqlite, the move's lifecycle state would already be persisted via `repo.UpdateActionItem(ctx, actionItem)` at `service.go:859`, and the caller would see an error AFTER the persistence had committed.

The fix is one of:

```go
// Option A — match doc comment to reality (cheap):
//   "The helper is non-idempotent. A duplicate gate-close attempt against an
//   already-warned drop fails on UNIQUE constraint, and the move's lifecycle
//   state has already been persisted before this hook runs. Pre-Drop-1 D3
//   override auth, the terminal-state guard at MoveActionItem prevents this
//   path from reaching live code."

// Option B — make the implementation match the doc comment (safer):
if err := s.repo.CreateAttentionItem(ctx, item); err != nil {
    if errors.Is(err, sqlite.ErrUniqueConstraint) {
        return nil
    }
    return fmt.Errorf("safety-net create attention: %w", err)
}
```

Either lands as a Drop 3 refinement; not in 3.22's scope.
