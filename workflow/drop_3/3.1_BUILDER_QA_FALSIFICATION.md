## 2. Refuted Attacks

### 2.1 — Regex narrowing attack (`[a-z]+` vs `[a-z-]+`) — REFUTED

- **Hit:** `(?m)^StructuralType:\s*([a-z]+)\s*$` at `structural_type.go:48`.
- **Claim under attack:** the tightened class might admit unintended values or reject valid ones at the line-anchor boundaries.
- **Probes attempted:**
  - **Leading whitespace before `StructuralType`** — `"  StructuralType: drop  "`: `(?m)^` matches start-of-line, not start-of-content; the literal `S` must follow `^`. Two leading spaces force no match. Returns `("", nil)`. Not a defect — matches Role's identical anchoring semantics.
  - **Tab-prefixed line** — `"\tStructuralType: drop"`: same anchor logic; tab is not whitespace-into-`^`. No match. Returns `("", nil)`. Consistent with Role.
  - **Multi-occurrence in description** — `"StructuralType: drop\nStructuralType: segment"`: `FindStringSubmatch` returns the first match. Test case `"two StructuralType lines — first wins"` (lines 98-102) covers this exactly. First-match semantics preserved.
  - **Mid-paragraph occurrence** — `"intro paragraph mentioning StructuralType: drop inline\nbut not anchored"`: `(?m)^` requires line start. Test case at lines 92-96 covers. Returns `("", nil)`.
  - **Capitalized value** — `"StructuralType: Drop"`: `[a-z]+` rejects `D`. Test case at lines 116-120 covers. Returns `("", nil)`. Consistent with the contract that only canonical lowercase form is recognized.
  - **Hyphenated value** — `"StructuralType: drop-let"`: `[a-z]+` rejects `-`. Test case at lines 122-126 covers. Returns `("", nil)`.
- **Mitigation:** No defect — the narrowing is acceptance-allowed (PLAN.md line 125) and the doc-comment block at `structural_type.go:33-47` documents the choice with explicit reference to finding 5.A.10.
- **Severity:** N/A.

### 2.11 — Regex anchor attack (junk after value) — REFUTED

- **Hit:** `(?m)^StructuralType:\s*([a-z]+)\s*$`.
- **Probes:**
  - `"StructuralType: drop xtra"` — after `[a-z]+` consumes `drop`, `\s*` matches ` `, then needs `$` but `xtra` follows. Backtrack fails; no match. Safe — junk-after rejected.
  - `"StructuralType: dr op"` — `[a-z]+` consumes `dr`, `\s*` consumes ` `, but `op` is not end-of-line. No match. Safe.
  - `"StructuralType: drop\r\n"` (CRLF — adjacent probe, not in the 11-vector spec): `\r` is whitespace per Go's `regexp` package; `\s*` consumes `\r`, then `$` matches before `\n`. Captures `drop`. This is permissive but not contract-violating — the contract says the line-anchored match wins; CRLF is just a line ending. Same behavior as Role's regex would exhibit.
  - `"StructuralType:drop"` (no whitespace separator) — `\s*` matches zero chars; `[a-z]+` captures `drop`; `\s*$` matches end. Returns `drop`. Permissive. Identical to Role's behavior with `Role:builder`. Parity preserved.
- **Mitigation:** No defect — the anchor design is intentionally tolerant of internal-whitespace variation while strictly rejecting trailing junk that isn't pure whitespace.
- **Severity:** N/A.
