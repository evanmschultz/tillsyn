## 1. Counterexamples

### 1.1 NIT — Methodology citation drift on `DropNumber` doc comments (PLAN-level, not builder-failure)

The `DropNumber` doc comments on both `ActionItem` and `ActionItemInput` cite `ta-docs/cascade-methodology.md` §11.2:

- `internal/domain/action_item.go:67` — "Domain primitive, not STEWARD-specific. Semantics per `ta-docs/cascade-methodology.md` §11.2."
- `internal/domain/action_item.go:135-136` — "Semantics per `ta-docs/cascade-methodology.md` §11.2."

But per the methodology spec, `drop_number` is NOT in the §11.2 ActionItem field list. §11.2 enumerates: `role`, `structural_type`, `persistent`, `dev_gated`, `owner`, `due_at`, `paths`, `packages`, `files`, `start_commit`, `end_commit`, `objective`, `acceptance_criteria`, `definition_of_complete`, `validation_plan`, `implementation_notes_*`, `transition_notes`, `blocked_reason`, `risk_notes`, `command_snippets`, `expected_outputs`, `decision_log`, `context_blocks`, `resource_refs`, `attempt_count`, `blocked_retry_count`, `last_failure_context`, `completion_contract`. No `drop_number`.

§11.3 (Concrete Types) is where `drop_number` appears: "**drop** — L1 cascade root, `structural_type = "drop"`, has a `drop_number`."

So `drop_number` is a §11.3 concrete-type attribute, not a §11.2 ActionItem extension. The doc comment SHOULD cite either §11.3 or `§11.2 + §11.3` (since Tillsyn's choice is to lift it onto every ActionItem rather than only the `drop` concrete type).

**Why this is NOT a builder failure:** PLAN.md droplet 3.17 acceptance line 409 literally says: *"Doc comments name the methodology section: `ta-docs/cascade-methodology.md` §11.2."* Builder followed the contract verbatim. The drift originates in the PLAN's wording, not in the build. Recording as a NIT against the PLAN to surface in 3.17 closeout for either (a) updating the doc comment to `§11.2 + §11.3` or (b) accepting the citation as "§11.2 lift" with a clarifying clause.

The other three fields (`Owner`, `Persistent`, `DevGated`) ARE genuinely §11.2 fields per the methodology, so their citations are correct.

### 1.2 NIT — Rollback-cost note enumerates "consumers except auth gate" but PLAN bullet lists 3.17 itself

The `DropNumber` doc comment (`action_item.go:62-65`) lists the affected droplets on rollback as "3.18 schema column, 3.20 read-side filters, 3.21 MCP plumbing, plus their tests." PLAN.md droplet 3.17 finding 5.C.16 enumerates "3.17 + 3.18 + 3.20 + 3.21 + tests."

The PLAN includes 3.17 itself in the rollback-cost set; the doc comment scopes "every consumer except the auth gate" which by interpretation excludes 3.17 (the field's home) since 3.17 is not a consumer but the definition site. Both readings are defensible — 3.17 obviously gets touched on rollback because the field would disappear. NIT only because the doc comment text is slightly narrower than the PLAN bullet's literal numbering.

## 2. Refuted Attacks

### 2.1 `Owner` validation gap attack — REFUTED (intended behavior)

Constructed: `Owner = "💩"`, `Owner = "RANDOM_UNICODE"`, `Owner = "1234"` — all accepted as valid Owner values after trim. The doc comment explicitly states "no closed-enum membership check ... any non-empty trimmed value is permitted so future template-defined owned kinds can pick their own owner names without a domain-package change." The auth gate (3.19) keys on the literal string `"STEWARD"`; arbitrary non-STEWARD owner strings are valid domain values that simply don't trigger the STEWARD gate. This is the documented L13 reframe: Owner is a domain primitive, not a STEWARD-specific tag. **Intended.**

### 2.2 `DropNumber` overflow attack — REFUTED

`DropNumber int` is a signed int. The validation `if in.DropNumber < 0` (`action_item.go:258`) is a signed comparison — `math.MinInt` (whether -2147483648 on 32-bit or -9223372036854775808 on 64-bit) compares as `< 0` and rejects. There is no arithmetic that could overflow into the positive range at this comparison site. Refuted.

`go env GOARCH` reports `arm64` on this build host — `int` is 64-bit. The validation is correct on every Go-supported word size; no platform-specific corner case.

### 2.3 Existing-test regression attack — REFUTED

Baseline (before 3.17 build): `mage test-pkg ./internal/domain` reports 143 tests, all green. After 3.17 build: 159 tests, all green. The +16 delta corresponds to 4 new top-level tests with 12 subtests (5+3+2+2) plus the 4 top-level functions themselves counted in the runner's tally. `TestNewActionItemRoleValidation` and `TestNewActionItemStructuralTypeValidation` are present in the post-build run and pass.

`mage testFunc ./internal/domain "TestNewActionItem.*"` (race-enabled, count=1) reports 41 sub-tests passed across the family — race-clean.

### 2.4 Order-of-validation attack — REFUTED

The validation chain in `NewActionItem` (`action_item.go:175-260`) preserves early-return discipline. Confirmed sequence:
1. ID + ProjectID + ParentID + ColumnID + Title + Position (lines 184-201)
2. Priority (lines 203-208)
3. Kind (lines 209-215)
4. Scope (lines 216-228)
5. Role (lines 234-237)
6. **StructuralType** — empty rejects, invalid rejects (lines 243-249)
7. **Owner** — trim only (line 255)
8. **DropNumber** — negative rejects (lines 258-260)
9. LifecycleState (lines 261-266)
10. Then UpdatedByType, actor fields, labels, metadata, return literal.

A malformed input with empty `StructuralType` AND `DropNumber=-1` rejects on `ErrInvalidStructuralType` first, NOT `ErrInvalidDropNumber`. This is the correct early-return ordering — caller fixes one error at a time.

The `Persistent` and `DevGated` bools have no validation step; their `false` zero-value is the dominant case and they round-trip without error path.

### 2.5 `Persistent` / `DevGated` shadowing attack — REFUTED

Both fields default to `false` (zero value) and the existing test fixture helper `NewActionItemForTest` (`internal/domain/action_item_test_helpers.go`) does NOT set them, so all pre-3.17 fixtures construct with `Persistent=false, DevGated=false`. `mage test-pkg ./...` reports 1570 tests across 20 packages, all green — confirms the fixture chain across `internal/app`, `internal/adapters/storage/sqlite`, `internal/tui`, `cmd/till`, `internal/adapters/server/common`, `internal/adapters/server/mcpapi` is unaffected.
