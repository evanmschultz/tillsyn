## Findings

### F1 — Driver citation misattributes the embedded SQLite binary (editorial)

**Severity:** editorial.

**Where:** `PLAN.md:280` (unit 1.14 phase-4 prose).

**Counterexample:** The prose says *"The Go binary's driver is `github.com/ncruces/go-sqlite3 v0.23.3` (embeds SQLite 3.49.1, per `~/go/pkg/mod/github.com/ncruces/go-sqlite3@v0.23.3/embed/README.md`)"*. But `internal/adapters/storage/sqlite/repo.go:16` blank-imports `github.com/asg017/sqlite-vec-go-bindings/ncruces`, whose `init.go` sets `sqlite3.Binary = wasmBinary` and ships its own embedded SQLite WASM. The `sqlite-vec-go-bindings/ncruces` README (`~/go/pkg/mod/github.com/asg017/sqlite-vec-go-bindings@v0.1.6/README.md:85`) states verbatim: *"This replaces the `github.com/ncruces/go-sqlite3/embed` module in that project, so do NOT include them both."* The Go binary is therefore running the sqlite-vec WASM binary, not the vanilla ncruces embed. The cited file (`@v0.23.3/embed/README.md`) does NOT describe what SQLite version actually runs inside the Go binary.

**Why this does not escalate:** The technical conclusion in the prose (SQLite ≥ 3.35.0, supports native `DROP COLUMN`) holds for both environments:
- Dev runs the script via local `sqlite3 3.51.0` CLI (plan's claim; not the Go binary).
- The sqlite-vec-embedded WASM ships a SQLite well past 3.35.0.

So no behavioral risk — only a mis-citation. A correct phrasing would be *"embeds a SQLite ≥ 3.46 class binary via `sqlite-vec-go-bindings/ncruces`, which replaces the ncruces/go-sqlite3 embed; and dev runs the script via `sqlite3 3.51.0` CLI anyway — both well past the 3.35.0 `DROP COLUMN` floor"*. Editorial: does not block the build. Flagged for cleanup in a future rounds' triage if dev wants it factually accurate.
