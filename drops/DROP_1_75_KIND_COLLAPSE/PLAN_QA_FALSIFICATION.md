# Plan QA Falsification — Round 7

**Verdict:** fail

## Summary

Round 6's F4 mitigation wraps unit 1.14 Phase 4 in `PRAGMA foreign_keys = OFF; … PRAGMA foreign_keys = ON;`, but the unit's Acceptance explicitly wraps the whole script in `BEGIN TRANSACTION` for rollback. SQLite docs (https://devdocs.io/sqlite/pragma) state verbatim: *"This pragma is a no-op within a transaction; foreign key constraint enforcement may only be enabled or disabled when there is no pending BEGIN or SAVEPOINT."* The F4 fix as specified is a no-op — the very CASCADE hazard it was meant to mitigate (17 tables reference `projects.id ON DELETE CASCADE`, including `action_items`, `columns_v1`, `project_allowed_kinds`, `attention_items`, `capability_leases`, `auth_requests`, `handoffs`, `comments`, `change_events`, and more) reopens the moment the 12-step rebuild hits `DROP TABLE projects_old`. If FK enforcement happens to be ON at the time (app code sets `PRAGMA foreign_keys = ON` per `repo.go:133, :146`; dev's interactive `sqlite3` CLI default is OFF but that's not guaranteed across environments / wrapper scripts / integrations), the rebuild catastrophically wipes every child table linked to the pre-rebuild projects.

## Findings

### F1 — PRAGMA foreign_keys is a no-op inside a BEGIN TRANSACTION (SQLite invariant)
**Severity:** blocker
**Unit:** 1.14
**Counterexample:** A dev environment where FK enforcement is already ON at script start (e.g., sqlite3 CLI invoked via a wrapper that runs `PRAGMA foreign_keys = ON` in an .sqliterc, or the script is applied through Go code that already opened the connection with FK ON per `repo.go:133`). The plan's 1.14 Phase 4 prose: *"wrapped in `PRAGMA foreign_keys = OFF;` … `PRAGMA foreign_keys = ON;` per Round-6 F4 so the `DROP TABLE projects_old` step inside the 12-step canonical rebuild does not CASCADE-destroy child rows"*. The plan's 1.14 Acceptance: *"Rollback on assertion failure (via `BEGIN TRANSACTION` + `SELECT RAISE(ROLLBACK, ...)` guards)."* Under the straightforward reading (entire script inside one `BEGIN TRANSACTION ... COMMIT`), both PRAGMA calls execute inside that transaction and SQLite silently ignores them per the docs above. FK stays ON through Phase 4. `DROP TABLE projects_old` then CASCADEs through every `FOREIGN KEY(project_id) REFERENCES projects(id) ON DELETE CASCADE` on `repo.go:167, :195, :224, :235, :250, :284, :299, :314, :333, :358, :423, :440, :470, :493, :527, :555, :860` — 17 child tables, including `action_items` (all 115 rows), `columns_v1`, `project_allowed_kinds`, `attention_items`, `capability_leases`, `auth_requests`, `handoffs`, `comments`, `change_events`. The dev's live DB gets gutted under cover of "defensive" PRAGMA wrapping.
**Evidence:**
- Context7 `/websites/devdocs_io_sqlite` `PRAGMA foreign_keys`: *"Query, set, or clear the enforcement of foreign key constraints. This pragma is a no-op within a transaction; foreign key constraint enforcement may only be enabled or disabled when there is no pending BEGIN or SAVEPOINT."*
- `internal/adapters/storage/sqlite/repo.go:133, :146` — app opens connections with `PRAGMA foreign_keys = ON`.
- `internal/adapters/storage/sqlite/repo_test.go:37-41, :71-75` — tests assert `foreign_keys = 1` after app open.
- `repo.go` FK grep: 17 child tables cascade on `projects.id`; additional FK on `project_allowed_kinds.kind_id REFERENCES kind_catalog(id) ON DELETE CASCADE` at `:334`; FKs from `template_node_templates.node_kind_id` and `template_child_rules.child_kind_id` to `kind_catalog(id) ON DELETE RESTRICT` at `:374, :392` (these justify Round 5's ordering of Phase 2 before Phase 3, which remains correct).
- Plan 1.14 Acceptance bullet: "Rollback on assertion failure (via `BEGIN TRANSACTION` + `SELECT RAISE(ROLLBACK, ...)` guards)." — single transaction wrapping intended.
**Suggested fix:**
- **Option A (preferred):** Move both `PRAGMA foreign_keys = OFF;` and the paired `PRAGMA foreign_keys = ON;` to run *outside* the `BEGIN TRANSACTION` block. Structure: `PRAGMA foreign_keys = OFF;` → `BEGIN TRANSACTION;` → Phases 1-7 (assertions inside, `SELECT RAISE(ROLLBACK, ...)` guards fire on fail) → `COMMIT;` → `PRAGMA foreign_keys = ON;`. Cons: on `RAISE(ROLLBACK)`, the transaction rolls back but FK stays OFF until the final PRAGMA fires. Rollback path must still re-enable FK — wrap the whole script in a client-side shell guard that always runs `PRAGMA foreign_keys = ON;` even on failure, or add an `ON ROLLBACK` trigger (SQLite doesn't support that directly; use a shell-level `trap`).
- **Option B:** Split the script into 3 phases across separate transactions:
  1. `BEGIN; DROP TABLE template_*; DELETE FROM kind_catalog; COMMIT;` (Phases 1-3, FK can be ON)
  2. `PRAGMA foreign_keys = OFF; BEGIN; CREATE TABLE projects_new; INSERT-SELECT; DROP projects_old; RENAME; COMMIT; PRAGMA foreign_keys = ON;` (Phase 4)
  3. `BEGIN; DROP TABLE tasks; UPDATE action_items; <asserts with RAISE(ROLLBACK)>; COMMIT;` (Phases 5-7)
- **Option C:** Use `ALTER TABLE projects DROP COLUMN kind` directly (SQLite 3.35.0+ supports it; the column is not PK/UNIQUE/FK-referenced/indexed/triggered/viewed per `repo.go:147-157` + FK grep). This eliminates the 12-step rebuild entirely, and therefore eliminates the CASCADE hazard F4 was trying to mitigate. See finding F3 for details on this simplification.

### F2 — 12-step table-rebuild is unnecessary for `projects.kind` (straight `ALTER TABLE DROP COLUMN` works)
**Severity:** major
**Unit:** 1.14
**Counterexample:** Plan Scope §6 and unit 1.14 both specify *"`ALTER TABLE projects DROP COLUMN kind` (or SQLite equivalent: `CREATE TABLE projects_new` + copy + drop + rename)"* and Phase 4 settles on the 12-step rebuild. Per SQLite docs (`/websites/devdocs_io_sqlite` `5. ALTER TABLE DROP COLUMN > 5.1. How It Works`): *"The DROP COLUMN command succeeds only if the resulting schema remains valid after the column definition is removed. If any references to the dropped column persist in other schema objects, the command will fail."* Inspection of `projects` table at `repo.go:147-157` shows `kind` is a plain `TEXT NOT NULL DEFAULT 'project'` column with no PK, no UNIQUE, no FK, no CHECK, no index, no trigger, no view referencing it. Direct `ALTER TABLE projects DROP COLUMN kind` is permitted since SQLite 3.35.0 (March 2021) and is an order of magnitude simpler than the 12-step rebuild. It also eliminates the F4 transaction-boundary hazard entirely — there is no `DROP TABLE projects_old` to CASCADE.
**Evidence:**
- `repo.go:147-157` — `projects` table definition (no PK on kind, no UNIQUE on kind, no FK).
- FK inbound grep: 17 tables reference `projects(id)`, zero reference `projects(kind)`.
- Index grep: `rg '(?i)CREATE INDEX.* ON projects' repo.go` → `idx_projects_*` indexes only on `id`/`archived_at`, none on `kind` (verified by schema inspection).
- Context7 SQLite `lang_altertable` — DROP COLUMN available since 3.35.0; modernc.org/sqlite targets 3.44+.
- Go app uses `modernc.org/sqlite` per project `CLAUDE.md` Tech Stack, which ships SQLite 3.44+.
**Suggested fix:** Replace Phase 4 of `scripts/drops-rewrite.sql` with a single `ALTER TABLE projects DROP COLUMN kind;` statement. Drop the `PRAGMA foreign_keys = OFF/ON` wrap (no longer needed — no `DROP TABLE projects_old` means no CASCADE). The `BEGIN TRANSACTION` + assertion-rollback structure stays intact and simpler. This is both a simplification AND the cleanest mitigation for F1. Add a dev-DB pre-flight check to unit 1.14 Acceptance: `sqlite3 --version` ≥ 3.35.0 (should always be true for modernc.org/sqlite consumers but cheap to assert).
